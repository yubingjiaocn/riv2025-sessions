# AWS re:Invent 2025 云勒索软件攻击剖析：实时模拟

## 会议概述

本次会议是AWS re:Invent 2025的一场独特体验，名为"云勒索软件攻击剖析：实时模拟"。会议采用沉浸式桌面演练的形式，由Rubrik公司创建的"零时刻"(Zero Hour)体验，帮助客户理解复杂网络攻击过程。演练基于真实的云攻击战术、技术和程序(TTPs)，以虚构的零售公司Horizon Retail为案例，展示了一次完整的勒索软件攻击过程。

会议通过五个场景展现了从攻击发现到最终决策的完整过程，包括一个视频场景和四个角色扮演场景。参与者被要求观察Horizon公司策略中的漏洞、反思并将学习成果带回自己的组织，以及参与讨论分享经验。整个演练强调了现代云攻击的复杂性、业务影响的严重性，以及传统恢复策略在面对高级威胁时的局限性。

## 详细时间线与关键要点

### 00:00-10:00 开场介绍与背景设定
- 介绍"零时刻"沉浸式桌面体验概念
- 设定虚构公司Horizon Retail：云优先的零售组织，在AWS上有关键业务部署
- 说明演练目标：观察策略漏洞、反思学习、参与讨论
- 介绍参与角色：运营分析师、云运营主管、CIO等关键人员

### 10:00-20:00 场景一：攻击发现（视频）
- **周日早晨6:00**：IT运营中心发现异常
  - EBS卷读取延迟激增，但读取I/O完全平稳
  - S3存储桶出现大量API调用：put object、delete object versions
  - 对象名称被重写，添加".locked"后缀
- **关键发现**：
  - CloudTrail显示数千次未授权SSEC加密调用
  - GuardDuty警报显示QA服务器异常活动
  - 攻击者在VPC内部进行就地加密，而非数据外泄

### 20:00-30:00 场景二：危机升级与攻击链分析
- **上午8:00**：东海岸门店准备开业，危机团队集结
- **攻击链分析**：
  - 初始入口：社会工程学获取标准员工账户凭证
  - 权限提升：利用未打补丁的QA服务器和配置错误的IAM角色
  - 横向移动：控制身份提供商，获得"王国钥匙"
- **业务影响**：
  - 手持扫描仪无法使用，仓库拣货系统瘫痪
  - 卡车路线优化系统离线
  - POS系统无法验证礼品卡余额或处理退货
- **上午11:00**：确认为Scattered Spider组织攻击，勒索金额400万美元

### 30:00-40:00 场景三：恢复尝试失败
- **周日晚9:00**：团队工作15小时，疲惫不堪
- **恢复挑战**：
  - 攻击者在环境中潜伏超过30天，恶意软件已复制到所有近期恢复点
  - 每个TB数据的深度取证扫描需要近一小时
  - 传统备份工具无法应对恶意管理员凭证的攻击
- **凌晨1:00**：发现看似干净的快照，恢复到隔离VPC
- **凌晨1:05**：恢复的系统再次被加密，恶意软件具有延时引爆功能
- **凌晨2:00**：CFO和法务总监加入，股价预计下跌8%，勒索金额提升至600万美元

### 40:00-50:00 场景四：数据泄露确认
- **周二下午**：攻击开始48小时后，股价暴跌超过10%
- **数据清点发现**：
  - S3清单显示920个存储桶，实际扫描发现965个
  - 45个存储桶完全未被跟踪、监控和备份
  - 未跟踪存储桶包含客户忠诚度项目的开发数据
- **数据泄露确认**：
  - 发现包含100万条记录的CSV文件
  - 包含完整姓名、电子邮件地址、邮寄地址和出生日期
  - 正式确认为数据泄露事件
- **法律后果**：
  - 需在72小时内通知加州总检察长（CCPA要求）
  - 面临集体诉讼风险
  - 事件成本呈数量级增长

### 50:00-54:00 场景五：最终决策（视频）
- **第三天**：三天的恢复失败、损失增加和公众羞辱
- **财务状况**：
  - 直接收入损失340万美元
  - 每天持续损失近100万美元
  - 完全重建需要3周时间
- **最终决策考量**：
  - CISO反对支付：会奖励犯罪行为，不能保证数据安全
  - CFO支持支付：重建时间过长，公司可能无法生存
  - 法务总监：支付是防止PII数据公开泄露的唯一杠杆
- **决策结果**：授权支付600万美元赎金
- **后续承诺**：下年度预算首要项目为数据安全和网络韧性策略全面改革

### 关键学习要点
1. 现代攻击特征：以"不和谐数据点合唱"开始，涉及多个云服务
2. 业务影响：云攻击即业务攻击，可瘫痪整个运营体系
3. 攻击速度：勒索软件以机器速度运行，可在几分钟内加密数十万对象
4. 恢复挑战：网络恢复与运营恢复根本不同，需要处理不可信的数据和身份
5. 真正的网络韧性：需要可生存的备份（不可变且气隙隔离）、主动数据治理和完整的敏感数据可见性管理
