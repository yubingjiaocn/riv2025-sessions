# AWS re:Invent 2025 - STG412 会话总结

## 会话概述

本次会话主题为"使用 AWS Backup 构建针对勒索软件的弹性防护"（STG412），由 AWS Backup 团队的三位专家主讲：全球解决方案架构师 Sabit Wenger Rajaladi、高级工程师 Ian Vilichkovich（自2018年参与服务开发）以及高级产品经理 Vive Mishra（专注于勒索软件防护和恢复）。

作为一场400级别的深度技术会话，演讲者们系统地探讨了网络弹性（Cyber Resiliency）的核心概念，即组织在遭受网络攻击时准备、抵御和恢复的能力，同时保持业务持续运营。会话重点关注勒索软件威胁环境的演变——从最初的小麻烦发展成为一个成熟的产业，以及如何通过 AWS Backup 构建有效的恢复策略。演讲团队强调了备份在网络攻击恢复中的关键作用，特别指出攻击者通常会首先尝试删除或破坏备份系统，因此保护备份成为恢复策略的核心。

整个会话围绕三大支柱展开：不可变性与隔离（Immutability and Isolation）、完整性（Integrity）和可用性（Availability）。演讲者通过具体的参考架构和最佳实践，展示了如何利用 AWS Backup 的各项功能——包括逻辑隔离保险库（Logically Air-Gapped Vault）、恢复测试（Restore Testing）和多方批准（Multi-Party Approval）等——来构建全面的数据保护和恢复解决方案。会话还涵盖了威胁建模方法、监管合规要求（如 DORA、NYDFS、GDPR 等）以及组织层面的实施策略。

### 备份与风险缓解
[18:45 - 23:30] 恢复弹性威胁建模（Ian Vilichkovich）
- 介绍使用"四个问题模型"进行威胁建模：
  1. 我们在处理什么？（了解系统组件、相互关系、依赖性和一致性要求）
  2. 可能出什么问题？（识别攻击者及其权限级别）
  3. 我们能做什么来缓解？（设计符合 RTO 的恢复策略）
  4. 我们做得好吗？（定期评估和演练恢复程序）

[23:30 - 26:15] 常见攻击向量
- 备份策略和备份计划
- 通知机制
- 恢复基础设施
- 直接删除备份
- 用于促进恢复的附加基础设施

[26:15 - 29:40] 备份安全策略
- 推荐 3-2-1 方法：三份数据副本，其中两份是备份，一份跨信任边界
- 根据灾难恢复和网络弹性目标决定备份位置（同区域或跨区域）
- 强调定期测试备份的重要性
- 提醒云环境中资源请求可能影响恢复时间

[29:40 - 33:20] 备份数据选择策略
- 基于两个因素：恢复所需数据和保护策略预算
- 识别系统关键性
- 区分原始数据和派生数据
- 强调备份本身可能成为威胁（数据泄露风险）
- 建议遵循最小权限原则保护备份
- 推荐开发和维护恢复运行手册

### AWS Backup 核心能力
[33:20 - 37:45] 三大支柱介绍（Vive Mishra）
1. 不可变性与隔离：备份无法被篡改且与生产系统隔离
2. 完整性：确保备份干净、未损坏且可恢复
3. 可用性：确保备份在需要时可访问和恢复

[37:45 - 40:30] AWS Backup 服务概述
- 2019年推出，提供集中式数据保护能力
- 核心设计原则：易用性
- 功能包括：备份调度、生命周期策略控制、恢复操作、审计
- 支持跨组织扩展、跨账户监控
- 目前支持22个 AWS 服务（从最初几个服务扩展而来）

### 支柱一：不可变性与隔离
[40:30 - 46:15] 逻辑隔离保险库（Logically Air-Gapped Vault）
- 2024年推出的解决方案
- 作为逻辑容器管理备份
- 关键特性：
  - **默认合规锁定**：备份始终处于锁定状态，攻击者无法删除或修改
  - **隔离存储**：备份存储在独立服务账户中，使用服务自有加密
  - **跨账户共享**：支持将备份共享到不同账户甚至不同组织
  - **快速恢复体验**：即使主账户被攻陷也能快速访问备份

### 支柱二：完整性
[46:15 - 50:30] 恢复测试（Restore Testing）
- 托管式备份测试解决方案
- 类似备份计划的定期测试机制
- 工作流程：
  - 选择要测试的备份和测试频率
  - 服务自动恢复备份
  - 客户执行自定义测试用例
  - 向服务报告测试结果
  - 服务记录结果用于审计
  - 自动清理测试资源
- 提供测试报告用于合规审计

### 支柱三：可用性
[50:30 - 55:45] 多方批准（Multi-Party Approval）
- 2024年推出的功能
- 由高度可信个人组成的团队形成法定人数
- 用于批准关键操作（如访问逻辑隔离保险库）
- 设置方式：
  - 在独立组织中定义多方批准团队
  - 使用 IDC 人类身份（5-10名成员）
  - 定义最小批准阈值（如10人中5人）
  - 在保险库创建时关联团队
  - 关联关系只能由批准团队解除
- 极端情况下的恢复：
  - 即使主账户被攻陷，批准团队可授权新恢复账户访问保险库
  - 支持跨组织恢复

### 参考架构
[55:45 - 65:00] 完整参考架构演示（Ian Vilichkovich）
- **AWS Organizations 结构**：
  - 使用委托管理员账户而非管理账户进行日常操作
  - 通过备份策略实施备份策略
  - 使用 Backup Audit Manager 跨组织监控
 
- **隔离 KMS 密钥账户**：
  - 独立账户存储 KMS 密钥
  - 防止攻击者获取密钥导致恢复不可能

- **工作负载账户**：
  - 每个账户/区域创建备份保险库
  - 配置必要的访问策略
  - 主资源和第一份备份副本（3-2-1策略中的"1"）
  - 集成 GuardDuty 扫描备份（标记为"0"，确保备份安全可恢复）

- **次要副本**：
  - 使用逻辑隔离保险库存储次要副本
  - 可直接作为主备份目标（最新功能）
  - 相当于将备份复制到另一个账户

- **灾难恢复账户**（可选）：
  - 如需考虑灾难恢复，在不同区域创建专用账户
  - 将备份副本存储到该账户

- **取证账户**：
  - 专用于取证活动
  - 定期运行恢复测试
  - 检查备份是否受恶意软件影响
  - 灾难情况下作为安全环境使用
