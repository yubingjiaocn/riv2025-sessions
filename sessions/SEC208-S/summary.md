# AWS re:Invent 2025 技术会议总结：安全设计原则与实践

## 会议概述

本次技术会议由Native公司创始人Gal和AWS身份团队的Sjana共同主讲，重点探讨了"安全设计"（Security by Design）的重要性及其在AWS云环境中的实施策略。Gal作为前AWS Security Hub产品负责人，分享了从检测式安全控制向预防式安全控制转变的必要性和实践方法。

会议强调了传统"修复跑步机"模式的弊端——即发现安全问题后进行被动修复的循环过程，并提出了通过AWS原生安全能力实现主动预防的解决方案。演讲者详细介绍了AWS提供的多种策略类型，包括服务控制策略(SCPs)、资源控制策略(RCPs)和声明式策略等，以及如何通过这些工具构建全面的安全架构。会议最后介绍了Native平台如何简化跨云环境的安全策略管理和实施。

## 详细时间线与关键要点

### 开场介绍 (0:00-2:30)
- **演讲者介绍**：Gal（Native公司创始人，前AWS Security Hub产品负责人）和Sjana（AWS身份团队）
- **背景说明**：Gal在AWS工作期间发现云提供商虽然提供强大的安全能力，但客户很难充分利用这些能力

### 问题场景描述 (2:30-5:45)
- **典型安全挑战**：安全团队发现1000个未加密的EBS卷需要处理
- **"修复跑步机"现象**：联系资源所有者→查找创建原因→逐个加密→创建例外列表→数月后发现新增500个未加密卷
- **检测式方法的弊端**：始终处于被动状态，消耗大量资源和时间，组织在修复期间面临风险

### 安全设计理念 (5:45-8:30)
- **从检测到预防的转变**：实施策略禁止创建新的未加密EBS卷
- **安全设计的核心**：从被动响应转向主动预防
- **一次性清理vs重复修复**：安全团队花费几小时实施预防控制，而非数月重复清理工作

### 安全设计应用领域 (8:30-11:15)
- **配置管理**：扩展到EFS、S3、KMS等所有AWS资源的默认加密
- **云足迹管理**：控制AWS资源部署区域，防止影子IT，满足数据驻留要求
- **外部访问管理**：管理合作伙伴和第三方集成的安全控制
- **数据边界**：AWS数据边界框架的应用
- **特权操作**：控制数据库删除等敏感操作，防止安全服务被篡改

### AWS原生安全控制 (11:15-15:30)
- **服务控制策略(SCPs)**：为所有主体（AWS服务、用户、联合用户、角色）设置权限边界
- **资源控制策略(RCPs)**：从资源角度控制访问，无论访问者是内部用户还是外部合作伙伴
- **声明式策略**：定义和强制执行AWS资源和服务的配置
- **其他策略类型**：VPC端点策略、基于资源的策略、身份策略

### 策略实施示例 (15:30-18:45)
- **EBS加密策略**：拒绝创建encrypted条件为false的EC2卷，防止篡改账户级加密设置
- **EFS加密策略**：拒绝创建或更新encrypted条件为false的文件系统
- **复杂性挑战**：不同资源需要不同的条件、操作和控制方式，需要仔细设计以避免阻断业务流程

### 数据边界框架 (18:45-21:30)
- **身份边界**：只有受信任的身份可以访问资源，只允许从受信任网络访问
- **资源边界**：身份只能访问自己的资源或受信任的资源，只能从受信任网络访问
- **网络边界**：防止他人使用私人凭证通过网络访问
- **实施复杂性**：需要在多种策略类型之间实现完美平衡和协调

### Native平台解决方案 (21:30-26:00)
- **公司理念**：专注于云原生安全控制，利用内置机制实现零延迟、可扩展的安全执行
- **Amy Herzog引言**：AWS CISO表示，实现安全设计的最快路径是使用环境内置的保护机制
- **跨云挑战**：AWS、Azure、GCP等不同云平台的安全机制各不相同

### 端到端解决方案流程 (26:00-29:30)
- **发现阶段**：分析当前有效策略和安全状态，讨论安全目标并进行转换
- **规划阶段**：确定执行内容、执行方式和推出计划
- **模拟阶段**：评估对现有资源的影响
- **实施阶段**：将安全需求转换为30多个策略声明，提供实施指导和推出计划
- **运营化阶段**：管理策略效果、例外请求和策略漂移

### 平台价值与总结 (29:30-31:00)
- **统一治理**：从单一位置管理跨云提供商的治理策略和流程
- **跨云专业知识**：AWS专家无需成为其他云平台专家，可直接应用现有策略
- **自适应覆盖**：随着云平台更新和业务需求变化快速适应
