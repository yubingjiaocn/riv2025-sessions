# AWS re:Invent 2025 会议总结：Whirlpool 虚拟产品开发民主化

## 会议概述

本次会议由 AWS 和 Whirlpool 联合呈现，重点介绍了 Whirlpool 如何利用 AWS 云服务实现虚拟产品开发的民主化。会议首先由 AWS 代表介绍了产品工程领域的行业趋势和 AWS 的解决方案，随后 Whirlpool 的 Axel 详细分享了公司在冰箱等家电产品开发中的创新实践。

Whirlpool 面临着产品日益智能化、工程复杂度增加以及成本压力加大的挑战。为应对这些挑战，公司开发了一套基于 AWS 的虚拟产品开发框架，通过仿真模拟、机器学习和多目标优化，大幅缩短了产品开发周期，降低了成本，并提高了产品性能。该方案的核心是将传统孤立的 CAD、CAE、PLM、EDA 系统整合到 AWS 平台上，利用高性能计算、机器学习和生成式 AI 能力，实现了从 5.5 万亿种可能组合中高效筛选最优产品架构的目标。

关键技术包括：使用正交阵列设计实验（Design of Experiments）将仿真运行次数从万亿级降至千次以内；通过 AWS SageMaker 进行模型训练和超参数调优；使用遗传算法进行多目标优化；以及通过 Elastic Beanstalk 实现知识民主化。整个解决方案采用无服务器架构，通过 API Gateway 和 Lambda 实现灵活调用，大大提高了开发效率和成本效益。

## 详细时间线

00:00 - 开场介绍
- 演讲者欢迎参会者，介绍会议主题：Whirlpool 如何通过 AWS 实现虚拟产品开发民主化

01:30 - 行业趋势分析
- 产品智能化：制造企业不再只销售产品，而是销售产品+服务的组合
- 工程复杂度增加：机械、电气、嵌软件、连接性等多领域集成
- 成本压力：市场竞争激烈，需要持续优化开发成本和质量成本

03:45 - AWS 解决方案概述
- AWS 提供 200+ 服务支持产品工程
- 打破 CAD、CAE、PLM、EDA 系统孤岛
- 通过迁移和现代化解锁数据价值
- 利用 AWS Bedrock 等生成式 AI 能力构建数字线程和数字孪生应用

05:20 - AWS 独特优势
- 弹性伸缩：根据测试需求灵活扩展和收缩基础设施
- 丰富能力：提供 HPC、AI/ML 等独特功能
- 成本优化：多种配置选项优化仿真和工程场景成本
- 数据解锁：打破数据孤岛，支持现代化转型

07:10 - Whirlpool 公司介绍
- Axel 登场，介绍 Whirlpool 作为领先家电公司的背景
- 旗下品牌：Whirlpool、KitchenAid、Jenn-Air、Maytag、Amana 等
- 年销售额 170 亿美元，全球约 44,000 名员工
- 拥有约 40 个制造和产品开发中心

09:00 - 项目目标与挑战
- 目标：优化冰箱系统级需求，实现最佳性能和最低成本
- 挑战：传统方法分散且次优，难以实现多目标协调优化
- 解决方案：多目标/多准则优化框架，涵盖能耗、容量、冷却、噪音振动等多个维度

11:30 - 技术方法论
- 系统工程和基于模型的系统工程（MBSE）
- 六西格玛设计（Design for Six Sigma, DFSS）
- 经过验证的仿真模型
- 代理模型（Surrogate Modeling）
- AWS SageMaker 加速模型训练和调优
- Elastic Beanstalk 实现知识民主化

14:00 - 仿真模型成熟度层级
- 开发级（Developmental）：探索是否可仿真
- 方向级（Directional，铜牌）：遵循物理规律的敏感性分析
- 可量化级（Quantifiable，银牌）：与物理测试数据交叉验证，适用于有限推理空间
- 验证级（Validated，金牌）：适用于所有架构，精度达标且获得所有利益相关者认可，仿真可替代物理测试

18:20 - 多需求仿真能力
- 内部容量：测量冷藏室和冷冻室的内部体积
- 性能系数（COP）与能耗：蒸发器设计与热增益的平衡
- 热增益：量化进入系统的热量，受墙体厚度、保温材料、真空绝热板（VIP）等影响
- 发泡工艺：单孔或多孔注入，环戊烷或 HFO 发泡剂选择

23:40 - 能耗系统级分析
- 控制板和软件逻辑
- 多种压缩机类型（系统心脏）
- 风扇电机实现温度分布
- 热交换器（冷凝器）
- 所有组件协同实现系统级性能

26:15 - 鲁棒性要求
- 外部冷凝控制：确保表面温度高于露点，避免外部出汗
- 其他鲁棒性：霜冻形成、内部冷凝等
- 成本平衡：保温成本与冷却组件成本的复合优化

28:50 - 其他仿真能力
- 内部气流分布管理
- 结构刚度：满载情况下的变形控制
- 噪音振动（NVH）：空气传播和结构传播噪音
- 发泡仿真：单孔注入的填充过程模拟
- 包装：运输和搬运过程中的损坏防护

31:20 - 设计空间复杂度
- 34 个因子（2 水平）+ 3 个因子（4 水平）+ 1 个因子（5 水平）
- 总组合数：5.5 万亿种（超过银河系恒星数量）
- 即使使用高性能计算（HPC）也无法穷举所有组合

33:00 - Whirlpool Tester Recommender 工具
- 基于六西格玛的实验设计（DOE）方法
- 使用正交阵列（Orthogonal Arrays）
- 部署在 AWS Elastic Beanstalk 上，后端使用 R 和 Python
- 通过成对平衡和独立的方式高效收集数据

35:40 - 正交阵列优势
- 最高效的数据收集方式
- 完全平衡且正交，无共线性问题
- 成对覆盖：每个因子的每个水平至少与其他所有因子的所有水平组合一次
- 覆盖整个设计空间

37:50 - 机器学习模型构建
- 从最简单的线性回归开始（因正交阵列无多重共线性问题）
- 如精度不足，进行数据增强（Augmentation）
- 能耗模型运行快（秒级或分钟级），可快速生成数千次运行
- 热增益模型运行慢（小时级），依赖正交阵列基础
- 采样计划通常不超过 1,000 次运行（相比 5.5 万亿）
- 模型精度超过 95%

41:10 - 多目标遗传算法优化（MOGA）
- 关键步骤 1：约束种群生成，排除不可行方案和线性约束（如墙体厚度对称性、门齐平度）
- 关键步骤 2：性能预测，使用机器学习模型进行推理
- 关键步骤 3：进化选择，通过交叉（Crossover）和变异（Mutation）实现局部开发和全局探索
- 避免梯度下降的局部最优问题，寻找帕累托前沿的最佳平衡解

45:30 - AWS 云架构详解
- API Gateway：模块化 API 调用，可集成到生产应用
- Lambda：无服务器方法，按需使用，零开销
- ECR（Elastic Container Registry）：自定义容器镜像，包含 Dockerfile、requirements.txt 和动态加载器
- 为何不使用 SageMaker 预打包容器：需要支持 scikit-learn、深度学习和优化的全部功能

48:20 - 动态加载机制
- S3 存储桶指向 ECR 镜像的动态加载器
- training_main 文件路由 SageMaker 训练作业
- Lambda 通过环境变量 operation_type 区分模型训练或调优

50:40 - 模型训练流程
- 用户点击模型训练按钮
- Lambda 解析参数，触发 SageMaker 作业
- 生成模型 pickle 文件（前置加载，避免用户端实时训练）
- 生成 SHAP 值文件用于可解释 AI（特征重要性和偏依赖图）
- 输出存储到 S3 训练输出桶

53:00 - 模型调优流程
- 使用 Optuna 库进行超参数调优
- 与架构优化使用相同的遗传算法
- S3 存储桶包含调优配置 JSON 文件（神经元数量、dropout、批归一化、学习率等）
- 生成调优结果 pickle 文件
- 用户查看测试集精度，满意后触发模型训练

55:30 - 架构总结
- 完全无服务器架构
- API 驱动，灵活按需调用
- SageMaker 实例可根据需求选择
- 模块化设计，易于集成和扩展
