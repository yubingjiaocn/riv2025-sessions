# AWS re:Invent 2025 - Amazon Redshift 数据共享会议总结

## 会议概述

本次会议由 AWS 分析团队的高级专家解决方案架构师 Saman Irfan 主讲，重点介绍了 Amazon Redshift 数据共享功能如何提升组织敏捷性。会议时长约 20 分钟，深入探讨了通过实时数据共享来支持分析和 AI 工作负载的能力。

Amazon Redshift 数据共享是一项成熟的功能，已被数千家客户在生产环境中使用。该功能解决了传统数据共享的痛点，包括手动数据复制、复杂的 ETL 管道维护、数据延迟以及安全治理挑战。通过利用 Redshift 的存储与计算分离架构，数据共享能够在同一 AWS 账户内、跨账户以及跨区域实现即时、细粒度和高性能的数据访问，无需手动复制和移动数据。

会议还介绍了三种数据共享管理选项（Redshift 托管、Lake Formation 托管和 AWS Data Exchange），四个主要使用场景（工作负载隔离、跨团队协作、数据即服务、开发敏捷性），以及 Peloton 和 Fannie Mae 等客户的成功案例。最后，讲师分享了最佳实践和实施建议。

02:30 - 04:00 | 传统数据共享的问题
- 手动数据复制和移动过程繁琐耗时
- 需要构建和维护复杂的 ETL 管道
- ETL 延迟导致数据过时
- 每次数据复制都带来新的安全治理挑战
- 团队使用不一致和不完整的数据，导致决策失误

04:00 - 05:30 | Amazon Redshift 数据共享解决方案
- 跨 Redshift 数据仓库安全轻松地共享实时数据
- 支持同一 AWS 账户内、跨账户和跨区域共享
- 支持读写工作负载
- 提供即时、细粒度和高性能的数据访问
- 无需手动复制和移动数据
- 已是成熟功能，数千客户在生产环境中使用

05:30 - 07:30 | 数据共享架构原理
- 基于 Redshift 存储与计算分离架构
- 通过 Redshift 托管存储层实现数据共享
- 多个生产者数据仓库可共享数据用于读写
- 多个消费者数据仓库可访问共享数据
- 消费者数据仓库也可以成为生产者共享其拥有的数据
- 无需复制数据，用户和应用程序可看到最新一致的信息

07:30 - 10:30 | 三种数据共享管理选项
- **Redshift 托管数据共享**：在生产者数据仓库中创建数据共享，直接与消费者共享，适合简单拓扑
- **Lake Formation 托管数据共享**：通过 AWS Lake Formation 集中管理权限，适合复杂拓扑
- **AWS Data Exchange for Amazon Redshift**：将数据作为数据产品发布到 AWS Marketplace，支持数据货币化

10:30 - 11:00 | 四大使用场景概述
- 工作负载隔离
- 跨团队协作
- 数据即服务
- 开发敏捷性

11:00 - 14:30 | 使用场景一：工作负载隔离
- 从单体 Redshift 数据仓库演进到多数据仓库架构
- 将 ETL、BI、数据科学、即席查询等工作负载分离到独立数据仓库
- 每个数据仓库可根据工作负载需求独立调整规模
- 支持暂停和恢复生产者/消费者集群
- 即使生产者数据仓库暂停，消费者仍可查询数据
- **客户案例 - Peloton**：健身设备公司，将单体数据仓库拆分为多数据仓库架构，每年节省 30 万美元

14:30 - 16:30 | 使用场景二：跨团队协作
- 不同团队（财务、销售、市场、研发）拥有独立的 Redshift 数据仓库
- 根据各自需求调整数据仓库规模
- 跨业务团队无缝协作进行分析和数据科学
- 每个数据仓库既可以是生产者也可以是消费者
- **客户案例 - Fannie Mae**：构建中央数据市场，将数据发布为数据产品，支持数据科学、BI 仪表板、自助分析等场景，实现企业级数据目录和数据网格生态系统

16:30 - 17:30 | 使用场景三：数据即服务
- 在组织内部和外部提供数据和分析即服务
- 通过 AWS Data Exchange 与外部方共享数据
- 数据提供者保留数据集控制权
- 可监控和跟踪数据使用情况

17:30 - 18:30 | 使用场景四：环境间数据共享
- 在开发、测试和生产环境之间共享数据
- 避免环境间的数据复制和重复
- 提升组织敏捷性
- 支持任意粒度的数据共享

18:30 - 21:00 | AWS Lake Formation 简化复杂拓扑治理
- 在多对多的复杂数据共享拓扑中，权限管理可能成为噩梦
- Lake Formation 作为中心层集中管理权限
- 通过 Lake Formation API 或控制台管理权限授予、查看访问控制和审计
- 集中管理权限，一次实施策略，跨多个数据仓库执行
- 无需手动脚本和复杂查询

21:00 - 24:00 | 最佳实践和注意事项
- **计费模型**：生产者仅为数据存储付费，消费者为计算资源付费；同区域数据共享免费，跨区域需支付数据传输费用
- **查询性能**：完全取决于消费者集群的计算能力，对生产者集群零影响
- **跨区域延迟**：跨区域数据共享可能有更高延迟
- **并发扩展**：生产者和消费者数据仓库都支持并发扩展功能，用于处理突发工作负载
- **审计**：提供系统表用于审计数据共享使用情况和变更
- **安全性**：传输中数据完全加密，静态数据使用 KMS 密钥加密，生产者和消费者可使用不同的 KMS 密钥
