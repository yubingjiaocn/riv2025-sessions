# AWS re:Invent 2025 会议总结：伦敦证券交易所集团的市场数据转型之旅

## 一、会议概述

本次会议由AWS金融服务首席解决方案架构师Rohit Singh主持，伦敦证券交易所集团(LSEG)实时业务集团总监Jason West和实时架构师Ed Johnson共同分享了LSEG如何利用AWS云服务实现大规模市场数据智能化转型。

LSEG作为全球领先的金融市场基础设施公司之一，每天需要处理来自575个交易所和交易场所的海量市场数据，服务全球170个国家的44,000个客户。会议重点介绍了他们的Matrix Signals平台——一个基于AWS无服务器架构构建的创新解决方案，该平台每天处理和分发2740亿条消息，管理着75 PB的历史tick数据和超过9000万个市场交易工具。

这次转型不仅仅是技术现代化，更是为了应对市场数据量的指数级增长、降低基础设施成本（降低至原来的五分之一）、提高系统灵活性，并利用Amazon Bedrock等生成式AI服务提供更深入的市场洞察。通过采用AWS的S3、Glue、Athena、Lambda、OpenSearch等服务，LSEG成功构建了一个可以弹性扩展、零存储成本、响应迅速的现代化数据平台。

## 二、详细时间线与关键要点

### **00:00 - 开场介绍**
- Rohit Singh介绍会议议程和演讲嘉宾
- 会议将涵盖资本市场生态系统、LSEG与AWS的合作历程、Matrix Signals平台技术深度剖析

### **02:30 - 资本市场生态系统挑战**
- 全球有数百个交易所全天候产生海量数据
- 数据聚合商需要收集、处理并全球分发这些数据
- 消费者（银行、对冲基金）依赖这些数据做交易决策
- 系统需要处理575个不同数据源，每个都有独特的格式、协议和时间要求

### **04:45 - 市场数据规模与复杂性**
- 每一层都面临独特挑战：生产者需要超低延迟和高吞吐量
- 聚合商必须标准化和分发数据
- 消费者需要安全访问和PB级数据处理能力
- 市场数据量逐年增长，传统基础设施难以应对

### **07:20 - Jason West介绍LSEG业务背景**
- LSEG拥有300年创新历史（1850年首次使用信鸽传递消息）
- 目前服务170个国家的44,000个客户
- 公司包含多个业务部门：数据与分析、外汇交易、交易后清算、指数业务、证券交易所

### **09:15 - LSEG实时业务规模数据**
- 每天传输2740亿条消息
- 从575个交易场所摄取数据，延迟仅4毫秒
- 自1996年以来存储所有交易tick数据，目前达75 PB
- 每天新增5-10 TB数据
- 管理超过9000万个交易工具
- 全球34个收集和分发节点

### **11:40 - 市场波动性挑战**
- COVID-19期间市场数据量显著波动
- 俄乌冲突导致数据量增加40%且未回落
- 美国关税政策推动消息量达到每秒近2000万条
- 系统保持20-25%的容量余量以应对突发情况

### **14:30 - 业务挑战分析**
- 传统本地基础设施需要每3-5年更换，成本高昂
- 数据量呈指数级增长，预计到2029年将达到每秒5000万条消息
- 客户需要实时洞察和快速市场接入
- 成本压力和竞争市场要求
- 需要创新速度提升（开发团队每天发布5-10次）

### **17:00 - 转型历程关键举措**
- 历史数据迁移：将CSV格式转换为Parquet格式，减少存储空间并提高查询效率
- 实时平台开发：在34个全球位置实现弹性扩展
- Matrix Signals平台和大规模ETL处理
- 云原生数据室feeds覆盖从超低延迟到量化数据解决方案的全谱系

### **19:45 - 业务成果**
- 地理扩展：满足客户在全球任何位置的需求，特别是受监管国家
- 更快的客户接入：某些产品从数周缩短到2分钟
- 按需容量扩展：市场波动时可动态调整数据更新频率
- 加速创新周期：快速响应Gen AI和新技术趋势

### **22:30 - Ed Johnson技术深度剖析开始**
- LSEG实时平台本质上是一个全球分布式的有状态FIFO消息总线
- 覆盖9000万个市场工具流，每个都保持严格的消息顺序
- 数据标准化处理，无论来源如何都提供统一的API和数据结构

### **25:10 - 河流与湖泊类比**
- LSEG实时feed如同数据之河，575个交易所如支流汇入
- 客户在河岸观察数据流
- Tick History产品如同河流尽头的湖泊，存储自1996年以来的所有数据

### **27:40 - 全球基础设施概览**
- 收集、核心和分发基础设施遍布全球
- 包括LSEG私有云数据中心和AWS分发位置
- 通过世界领先的软件定义网络连接
- 旧的信号检测基础设施成本不断攀升

### **30:20 - 问题陈述**
- 需要预测和优化容量管理，检测热点
- 降低本地基于数据包捕获的信号检测系统成本
- 加快运营团队和领导层的决策速度
- 在极端市场波动时更快响应

### **32:50 - 数据流示例（Nvidia股票）**
- 展示标准化数据格式的7条消息示例
- 每条tick包含股价更新、交易价格和交易量
- 这只是1秒钟的数据，某些工具每秒可能有超过10,000次更新
- 数据必须保持顺序才有意义（FIFO特性）

### **35:30 - 架构设计核心理念**
- 将实时feed（河流）中的数据转化为Tick History（湖泊）中的洞察
- 在云环境中对Tick History优化数据集进行分析
- 将分析结果反馈到本地环境和全球分发基础设施
- 快速响应热点问题

### **37:45 - 解决方案架构详解**
- 从Tick History获取当日数据捕获
- 将数据聚合为高度优化的时间序列格式
- 提取检测热点所需的参数和洞察

### **39:20 - 技术栈选择**
- **Amazon Athena**：首选工具，低成本、轻量级，可直接查询S3中的Parquet数据，零拷贝
- 几个月内完成概念验证，无需部署基础设施
- **AWS Glue**：添加ETL层进行超优化的预聚合
- 生成所有可派生的时间序列及其他参数
- **Amazon OpenSearch**：索引和嵌入数据，提供API访问
- **Lambda和EventBridge**：协调整个流程

### **42:00 - 三个关键维度**

#### **维度1：转换数万亿tick数据（42:00-44:30）**
- 使用S3访问点和Glue Serverless实现零拷贝
- 将tick-by-tick存储转换为数十亿个优化数据点
- Glue提供更多性能调优选项
- 灵活扩展：根据周末/工作日、市场开盘等情况自动调整
- 与参考信息和其他参数关联
- 实现零存储成本

#### **维度2：创建灵活建模层（44:30-47:15）**
- 从数十亿优化数据点创建特定信号模型
- 保留灵活的骨干架构，因为无法预先知道所有问题
- Athena层自动检测新数据并运行所有模型
- 查询示例：
  - 精确定位美国地区市场开盘时的所有100毫秒热点
  - 找出今天与昨天相比最活跃的交易工具
  - 分析特定工具监视列表的计算和性能需求
- 使用Trino/Athena SQL查询，支持多租户
- 维持超低成本，几乎无需为环境付费

#### **维度3：自动索引到OpenSearch（47:15-50:00）**
- 将Athena输出的Parquet格式数据自动索引到OpenSearch
- 两大优势：
  1. 提供基于API的数据访问
  2. 以特定方式聚合并原生提供API
- 通过API快速提供洞察
- 集成Amazon Bedrock用于知识库用例
- 获取数据的嵌入结果，以GenAI原生方式检测和关联热点
- 为整个组织提供API或GenAI集成的容量数据

### **50:00 - 新闻关联层**
- 市场波动通常由突发新闻驱动
- 添加新闻关联层，将波动信息与路透社提供的市场领先新闻关联
- 不仅识别热点，还能解释热点产生的原因
- 这只能通过GenAI摘要和RAG（检索增强生成）实现

### **51:30 - 多租户实现**
- 通过Fargate实现Athena的多租户访问
- 为组织内不同用户提供经过身份验证的安全结构化API

### **52:45 - 端到端信号检测流程**
- Tick-by-tick数据 → AWS Glue Serverless聚合作业预聚合
- 每个市场工具的数据可用于检测和查询
- 示例：某工具1秒内有5000条消息，其中4000条出现在100毫秒突发中
- Glue Serverless作业完成后，自动运行Amazon Athena SQL热点检测模型
- 检测数据流中的异常突发（例如突发消息比例显著高于总量）
- 自动索引所有热点输出，触发基于突发的监控API
- 快速模型检测 → 自动索引 → API视图 → 触发基础设施操作、监控或突发修复

### **55:00 - 会议总结**
- 成功构建无服务器解决方案，在极端规模下运行
- 处理2740亿条消息、75 PB历史数据、9000万+交易工具
- 无需管理传统基础设施和扩展
- 基础设施成本降低至原来的五分之一
- 解决方案随市场条件变化灵活扩展
- 利用Amazon Bedrock提供GenAI驱动的市场洞察
