# AWS re:Invent 2025 会议总结：爱立信如何利用 AI 优化移动网络

## 会议概述

本次会议由 AWS 电信业务部门的高级解决方案架构师 Christian Finalei 主持，重点介绍了爱立信如何与 AWS 合作，利用人工智能技术转型其业务和产品线。会议分为两个主要部分：第一部分由 Christian 和 Ibrahim 讲解爱立信如何通过 AI 优化客户的移动网络体验；第二部分由 Dag 和 Bastian 介绍爱立信如何利用 AI 赋能内部软件开发团队，提升研发效率。

爱立信作为全球领先的电信设备供应商，目前为全球超过 50% 的 5G 网络提供技术支持，拥有超过 60,000 项专利。面对日益复杂的移动网络系统，爱立信选择与 AWS 深度合作，构建了基于 Amazon Bedrock、SageMaker 等服务的完整 AI 技术栈。通过认知网络解决方案（CNS）和系统理解实验室（System Comprehension Lab），爱立信不仅提升了网络运营效率，还显著改善了内部研发流程，实现了从传统 ML 到 Agentic AI 的全面转型。

会议强调了构建成功 Agentic AI 解决方案的关键要素：信任与可解释性、上下文管理、多智能体价值归因、端到端可观测性，以及高效的 AI SDLC 能力。这些要素对于电信这样的国家关键基础设施运营至关重要。

- **03:30** - 在 AWS 上虚拟化网络功能，提高弹性、可扩展性和业务连续性

### 成功 Agentic AI 的关键要素（07:30 - 11:00）
- **07:30** - 强调构建 Agentic AI 不仅是选择 LLM 和 SDK，准确性和经济可行性是成功的关键
- **08:00** - 信任：需要在最细粒度级别授权访问数据和知识
- **08:30** - 数据溯源：追踪数据来源以满足数据主权要求
- **09:00** - 可解释性：能够剖析智能体行为、推理过程和工具调用
- **09:30** - 上下文为王：在正确时间向正确智能体提供简洁的知识和数据（因为规模意味着成本）
- **10:00** - 价值归因：在多智能体解决方案中，需要确定每个智能体的价值贡献和 ROI
- **10:30** - 可观测性：端到端监控智能体性能，包括 LLM 运营和 ML 运营

### Ibrahim 关于可解释性的重要性（11:00 - 12:30）
- **11:00** - Ibrahim 强调 CSP 将国家基础设施的网络运营外包给爱立信时，需要极高的信任度
- **11:30** - 可解释性对于理解智能体决策、工作流设计和工具选择至关重要
- **12:00** - AWS 技术栈为 CSP 提供了实现可解释性和可观测性的工具和能力

### AWS AI 技术栈介绍（12:30 - 15:00）

- **13:00** - 底层：数据科学家使用 AI 计算（GPU 到 Amazon 自研芯片）进行训练和推理
- **13:30** - 中层：Amazon SageMaker AI 用于完整 MLOps 流程（数据准备、训练、评估、托管、监控）
- **14:00** - 上层：工程师使用 Amazon Bedrock 大规模安全运行智能体，提供多种基础模型选择
- **14:30** - 开发者利用 Agentic SDLC 应用提升编码和软件维护效率

### 爱立信的影响力与创新传统（15:00 - 18:00）
- **15:00** - Ibrahim 介绍爱立信的使命和社会影响
- **15:30** - 全球 340 个 5G 网络中有 189 个使用爱立信技术（超过 50%）
- **16:00** - 中国以外超过 50% 的全球 5G 流量运行在爱立信设备上
- **16:30** - 在拉斯维加斯，AT&T、Verizon、T-Mobile 的用户 100% 使用爱立信技术
- **17:00** - 拥有 150 年创新历史，超过 60,000 项移动技术专利
- **17:30** - 74% 使用爱立信设备的全球 CSP 在市场基准测试中排名第一

### 自主网络框架（18:00 - 21:00）
- **18:00** - 爱立信通过 AI 转型其产品组合，涵盖移动技术栈和网络运营
- **18:30** - 介绍 TMF 定义的自主网络框架，分为五个级别
- **19:00** - 目前不到 10% 的 CSP 达到或超过三级（AI 开始介入的级别）
- **19:30** - 无法在不嵌入 AI 的情况下达到四级网络自主性
- **20:00** - 自主网络是向 CSP 提供 AI 能力的重要框架
- **20:30** - 强调与 AWS 合作对实现这一愿景的重要性

### 认知循环（Cognitive Loop）（21:00 - 24:30）
- **21:00** - 介绍爱立信的"认知循环"——网络运营的五阶段工作流
- **21:30** - 测量阶段：测量智能体理解网络数据并捕获所需信息
- **22:00** - 保障阶段：分类器和根因分析智能体识别网络异常和根本原因
- **22:30** - 建议阶段：AI 推荐系统和智能体提出下一步最佳行动
- **23:00** - 评估阶段：确保最佳决策在国家基础设施中实施前经过评估
- **23:30** - 执行阶段：执行和驱动，循环继续
- **24:00** - Agentic AI 架构贯穿整个认知循环，通过专门的智能体实现

### 开放架构与数据暴露（24:30 - 27:00）
- **24:30** - 爱立信的架构基于开放性和水平化两大目标
- **25:00** - 通过 AWS 服务将网络数据暴露给上层智能体
- **25:30** - R-apps（无线应用）作为智能体的工具，提供理解和执行网络变更的能力
- **26:00** - 可观测性极其重要，不仅针对智能体层，还包括整个 AI 技术栈
- **26:30** - AWS 技术栈中的不同服务支持这一架构

### 技术架构详解（27:00 - 30:30）
- **27:00** - Christian 详细介绍技术架构
- **27:30** - ETL 层：使用 AWS 无服务器服务、容器和托管服务（Athena、Glue）处理来自无线接入网的大量数据
- **28:00** - SageMaker AI：构建 ML 管道，进行数据准备、模型训练、评估和托管推理端点
- **28:30** - R-apps：ML 驱动的应用调用推理端点，生成网络性能预测
- **29:00** - 智能体还依赖其他数据源和知识库（通过 RAG 技术），包括产品指南、网络配置、数据结构和参数关联
- **29:30** - Amazon Bedrock Agent Core：为爱立信提供内存和可观测性能力，在 AWS 托管服务上运行
- **30:00** - 完整技术栈：SageMaker 生成准确预测，Agent Core 管理智能体全生命周期

### Cell Shapers - 强化学习智能体（30:30 - 34:00）
- **30:30** - Ibrahim 介绍 R-apps 工具示例：Cell Shapers
- **31:00** - Cell Shapers 是强化学习智能体，具有网络的数字孪生表示
- **31:30** - 解决从 2G 到 6G 的多代问题：如何优化覆盖范围、容量和质量
- **32:00** - 传统方法是在部署前进行模拟，而强化学习智能体基于实时流量重建无线环境
- **32:30** - 通过多目标强化学习算法持续学习网络变化
- **33:00** - 在覆盖、容量和质量之间提供最佳权衡
- **33:30** - 适应流量变化、新频谱引入、站点密集化、新技术（如 Cloud RAN、XG）等因素

### 多种智能体与实际成果（34:00 - 36:00）
- **34:00** - Cell Shapers 属于认知循环的"建议阶段"
- **34:30** - 爱立信开发了多种智能体，正与 AWS 合作部署到其技术栈
- **35:00** - 涵盖改善用户体验、网络运营和运营效率等不同领域
- **35:30** - 展示的数据来自真实现场部署，而非实验室结果
- **36:00** - 能够测量影响，理解爱立信技术对社会和 CSP 的深远影响

### 系统理解实验室介绍（36:00 - 39:00）
- **36:00** - Dag 接手，介绍系统理解实验室（System Comprehension Lab）
- **36:30** - 强调智能不仅来自观察网络，还来自爱立信 40 年构建移动网络基础设施的经验
- **37:00** - 智能网络需要代表公司的机构智能、人才和数据资产
- **37:30** - 系统理解实验室旨在实现多个目标
- **38:00** - 当今移动网络极其复杂，尤其在美国等大国
- **38:30** - 网络是复杂的分布式系统，部署和管理都很复杂

### 系统复杂性挑战（39:00 - 42:00）
- **39:00** - 设计也很复杂：多种无线产品、2G/3G/4G/5G/6G 共存、数百万配置参数
- **39:30** - 爱立信与 AWS 的 AI 之旅始于认识到这种大规模系统复杂性
- **40:00** - 这不是爱立信或电信特有的问题，飞机、核电站等领域都面临类似挑战
- **40:30** - 核心问题：我们是否受限于理解能力（comprehension bound）？
- **41:00** - 爱立信研发团队有数万人，在分布式系统领域工作
- **41:30** - 构建 6G 或下一代 5G 时，是否受限于传统的成本、范围和时间三要素？

### 理解能力限制（42:00 - 44:30）
- **42:00** - 实际上更受限于参与人员能够应用的理解能力
- **42:30** - Dag 承认虽然全球一半 5G 流量通过爱立信产品，但仍可以做得更好
- **43:00** - 作为开发移动网络基础设施的人类，可能受限于理解能力
- **43:30** - AI 使命：如何让 AI 在复杂系统工程的极限下工作
- **44:00** - 这适用于电信、核电、飞机、汽车等任何复杂系统

### 系统理解实验室的使命（44:30 - 47:00）
- **44:30** - 实验室从拥有数百万份内部文档开始
- **45:00** - 三四十年来一直在没有 AI 的情况下工作
- **45:30** - 第一个使命：利用所有内部数据资产，提升设计移动网络的能力
- **46:00** - 通过 AWS Bedrock 等服务建立机构知识和智能基础
- **46:30** - 在此基础上叠加实时网络数据流，解释其含义并采取行动

### 智能集成与前沿 AI 差距（47:00 - 50:00）
- **47:00** - 智能集成到数据流中，为产品理解人类意图奠定基础
- **47:30** - 整合机构知识，改进产品设计，叠加真实网络遥测数据
- **48:00** - 指出前沿 AI 能力（竞赛级编程、物理、数学）与工业/产品开发中 AI 实际影响之间存在巨大差距
- **48:30** - 这个差距比可视化显示的要大得多
- **49:00** - 爱立信自身能够产生的 AI 价值与最先进技术之间的差距
- **49:30** - 与 AWS 紧密合作，探讨如何为爱立信规模的公司缩小这一差距

### 缩小差距的挑战（50:00 - 52:30）
- **50:00** - 爱立信是相当大的研发组织，需要缩小差距以实现 Ibrahim 所说的价值
- **50:30** - 挑战一：每个组织都有大量数据，但没有人有干净的数据
- **51:00** - 竞赛编程等演示背后是大量的数据工作
- **51:30** - 需要在复杂设计空间中结合不同类型的数据资产
- **52:00** - 挑战二：多步推理使问题更难——每一步都可能偏离轨道
- **52:30** - 除非能够约束推理，否则会很快发散到无意义的结果

### 信息融合与架构（52:30 - 55:30）
- **52:30** - 这些挑战促使爱立信与 Bastian 和 AWS 合作开发 Agent Core
- **53:00** - 核心是**信息融合**：在嵌入式系统公司的背景下
- **53:30** - 爱立信的流程：收集需求、系统化、开发、测试、集成、发布、部署
- **54:00** - 产品在现场运行十年或更长时间
- **54:30** - 需要以前所未有的方式融合整个流程中极其复杂的数据资产集
- **55:00** - 关键问题：需要什么架构才能在数万开发者规模上实现这一目标

### 解耦架构与 MCP（55:30 - 58:00）
- **55:30** - 第一个目标：实现某种解耦
- **56:00** - 使用 Amazon Q Developer 或 Kiro 作为面向用户的前端工具
- **56:30** - 但这些工具本身不了解爱立信的内部研发资产
- **57:00** - 通过 MCP（模型上下文协议）标准化前端工具与内部设计之间的解耦层
- **57:30** - 投入大量精力构建专门的 AI 智能体和技术，而非端到端 AI 产品
- **58:00** - 专注于爱立信差异化的专业模型、开发工具和智能体

### Bastian 介绍系统理解实验室关键支柱（58:00 - 61:00）
- **58:00** - Bastian 接手，讨论系统理解实验室的关键因素
- **58:30** - "理解能力受限"的概念对所有有历史的企业都很重要，尤其是研发密集型企业
- **59:00** - 系统理解实验室的关键支柱
- **59:30** - 智能体与人类交互，但需要正确的数据
- **60:00** - 不同的数据孤岛需要融合
- **60:30** - 虽然多步推理有其自身问题，但对数据融合至关重要

### 超越传统 RAG（61:00 - 63:30）
- **61:00** - 必须超越传统或"朴素"RAG（仅在文本空间中查找相似文本）
- **61:30** - 不可能仅通过这种方式获得有意义的结果
- **62:00** - 需要评估用户查询、查询不同数据源、组合数据的系统
- **62:30** - 可能需要退后一步重新思考，或再次与用户交互以获取智能体所需的正确数据
- **63:00** - 这是在正确时间为智能体获取正确、简洁数据的过程

### 自定义模型与多模态处理（63:30 - 66:00）
- **63:30** - 智能体需要采取行动，有时需要更进一步
- **64:00** - 上下文依赖于传统 AI/ML 模型，但也可能需要自定义大语言模型
- **64:30** - 多模态处理也很重要
- **65:00** - 摄入的数据不是结构良好的文本文档，有许多不同的模态
- **65:30** - Bastian 承诺在后续幻灯片中展示更多细节

### 下游工具集成（66:00 - 69:00）
- **66:00** - 智能体获得正确数据后，还需要集成到下游工具
- **66:30** - 智能体需要与物理或数字世界交互
- **67:00** - 工程师使用多种工具：项目规划工具、知识库、源代码控制工具等
- **67:30** - 需要访问这些工具并消除所有摩擦点
- **68:00** - 将任务交给智能体处理
- **68:30** - MCP 是连接层：通过 MCP 提供下游系统工具
- **69:00** - 智能体可以使用这些工具，工具之间也可以相互使用

### 用户界面与可观测性（69:00 - 72:00）
- **69:00** - 需要向用户展示信息
- **69:30** - 爱立信遵循在用户想要工作的地方提供信息的原则
- **70:00** - 有时在浏览器中进行深度研究任务是正确的选择（专用前端）
- **70:30** - 有时希望直接在 IDE 中进行编码（通过 Amazon Q Developer 或 Q）
- **71:00** - 智能体有正确的工具和数据，但还有更多非功能性要素
- **71:30** - 可观测性：理解调用了哪些工具、是否选择了正确的工具、是否与任务相关

### 可观测性的深度（72:00 - 74:30）
- **72:00** - 需要了解工具是否获得正确输出、上下文是否正确
- **72:30** - 通过多步推理创建的上下文是否与问题或任务相关
- **73:00** - 可观测性超越智能体层：这些系统是复杂的，有许多活动部件
- **73:30** - 包括常规应用、模型、数据源、处理数据的管道等
- **74:00** - 需要跟踪很多事情，因此可观测性至关重要
- **74:30** - 可扩展架构：不过多关注技术细节

### Agent Core 与无服务器体验（74:30 - 结束）
- **74:30** - 爱立信使用 Bedrock Agent Core 作为核心组件之一
- **75:00** - 提供无服务器体验，轻松运行和扩展智能体
- **75:30** - 会议在此处结束，强调了爱立信与 AWS 合作构建可扩展、可观测、可信赖的 AI 解决方案

总结：本次会议全面展示了爱立信如何利用 AWS 的 AI 技术栈实现双重转型——既优化了为 CSP 客户提供的网络运营服务，又提升了内部研发团队的效率。通过认知循环、强化学习智能体、系统理解实验室等创新方法，爱立信正在引领电信行业的 AI 革命。
