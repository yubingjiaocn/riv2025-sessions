# AWS 技术会议总结：使用 MCP 和 AWS 构建可扩展的微代理平台

## 会议概述

本次会议由 AWS GenAI 创新中心的首席机器学习工程师 Moon Kim 主持，与 Prudential Financial 的数据科学副总裁 Rohit Kapa 和机器学习工程总监 Subir Das 共同分享了如何为金融服务和保险行业构建可扩展的 AI 代理平台。

会议重点介绍了 Prudential Financial 面临的挑战：如何在企业范围内管理和扩展数百甚至数千个 AI 代理，同时保持治理、安全性和合规性。传统的单体架构方法在面对大规模代理部署时会遇到瓶颈，包括部署延迟、所有权边界模糊、以及难以管理敏感数据。为解决这些问题，团队采用了基于微服务的模块化架构，利用模型上下文协议（MCP）和 AWS 服务构建了一个企业级代理平台。

该平台已成功部署，支持超过 10 万名财务顾问使用人寿保险顾问 AI 助手。这个多代理系统通过编排代理协调多个专业子代理（快速报价、表单、产品信息、插图和业务账簿），将顾问的响应时间从 6-8 周缩短到 3-4 周，显著提升了运营效率。平台化方法还实现了组件复用、标准化框架、以及跨业务单元的快速扩展能力。

## 详细时间线

### 开场介绍
[00:00 - 02:30]
- Moon Kim 介绍自己是 AWS GenAI 创新中心的首席机器学习工程师
- 概述会议议程：客户挑战、Prudential 的 AI 计划、具体用例、平台架构、影响和未来方向

### 金融服务行业的 AI 挑战
[02:30 - 06:45]
- FSI 企业面临的核心挑战：需要快速交付大量 AI 解决方案（可能达到数千个代理）
- 开发团队需要灵活性和自主权，但必须符合企业治理和品牌保护要求
- 传统方法的问题：多个解决方案堆积在单一代码库中，导致部署时间从分钟延长到数天
- 所有权边界模糊，难以管理 PII/PHI 等敏感数据
- 需要架构转型：模块化设计、关注点分离、集中化治理

### Prudential Financial 业务背景
[06:45 - 11:20]
- Rohit Kapa 介绍 Prudential：美国金融服务公司，提供保险、退休和投资管理服务
- 在全球 40 多个国家运营，服务个人和机构客户
- 三大业务单元：退休、团体保险、人寿保险
- 每个业务单元都有类似的职能：分销与销售、承保与风险、客户与理赔服务、产品开发与营销
- 共同挑战：工作流程分散、个性化不足、手动流程缓慢、创新周期长

### 人寿保险顾问 AI 助手用例
[11:20 - 18:45]
- 财务顾问的典型工作流程：客户参与、需求评估、解决方案设计、产品演示、申请支持、长期服务
- 核心痛点：顾问需要与 50-60 个不同的 IT 系统交互（如果服务 5-6 家保险公司）
- 快速报价流程问题：顾问提交医疗信息后，承保人需要 1-2 天才能回复
- 表单和产品信息检索耗时 10-15 分钟
- 顾问难以维护 10-15 年的客户关系上下文

### 多代理系统架构
[18:45 - 24:30]
- 新系统提供统一的聊天界面，作为顾问的单一入口
- 编排代理理解意图并路由到专业子代理：
  - 快速报价代理：基于数百份承保文档训练，自动询问后续问题（如糖尿病的 A1C 值、癌症的缓解日期）
  - 表单代理：访问数百种表单，根据交易类型和州提供正确表单
  - 产品代理：智能搜索产品功能和适用性
  - 插图代理和业务账簿代理：提供政策演示和下一步最佳行动建议
- 系统已上线，支持超过 10 万名顾问使用

### 平台部署架构
[24:30 - 32:15]
- Subir Das 介绍部署挑战：扩展到 10 万用户、支持意图驱动编排、组件可复用、安全合规
- "随时间扩展"的概念：GenAI 领域快速变化，需要采用最新的提示管理、上下文工程、SDK 和管道
- 微服务架构流程：
  1. 用户通过 SSO 认证访问 UI
  2. 生成安全令牌传递给代理层
  3. 代理层验证令牌并生成上下文特定的窗口 ID
  4. 编排代理使用令牌和窗口 ID 维护会话和上下文
  5. 子代理通过 LLM 网关访问模型，通过知识管理系统检索文档

### 代理内部架构
[32:15 - 36:40]
- 单个代理的内部结构：
  - 访问数据平面
  - 使用 MCP 和 ReAct 框架
  - 短期和长期内存管理
  - A2A（Agent-to-Agent）协议支持跨代理通信
  - 可与其他多代理系统集成（如退休规划、智能文档处理）
- 正在开发基于 ACE 框架的上下文工程框架，以增强大规模上下文管理

### 企业代理平台
[36:40 - 40:20]
- 平台化方法的必要性：管理复杂性、随时间扩展、实现 MCP 和 A2A
- 平台组件：
  - 传统模型空间：SageMaker 推理（通过 MCP 访问）
  - GenAI 代理：向量存储、Bedrock 代理、LLM 网关、Web 管理、评估框架
  - 企业服务：AgentOps、GitHub Actions、DevSecOps
  - 企业数据和 AWS 基础设施
- 平台用户：数据科学家和机器学习工程师（开发部署），业务用户和应用（消费）

### 业务成果和收益
[40:20 - 45:30]
- 周转时间缩短：从 6-8 周减少到 3-4 周
- 新代理可在 4-5 周内开发并部署到生产环境
- 标准化解决方案：从单一用例扩展到多个用例，跨业务单元复用
- 减少技术债务：集中升级上下文框架、A2A 协议、SDK、可观测性系统
- 更快的业务反馈循环：开发人员专注于代理逻辑而非基础设施
- 负责任的 AI 在平台层面强制执行
- 改进的监控、代理跟踪和可观测性

### 未来方向和总结
[45:30 - 结束]
- 持续添加更多代理和 MCP 工具访问支持
- 增强监控、可观测性和评估框架
- 扩展到其他业务职能（承保、理赔、产品开发、营销）
- 目标：在整个企业范围内实现 AI 驱动的超个性化服务
