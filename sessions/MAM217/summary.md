# AWS re:Invent 2025 MAM217 会议总结

## 会议概述

本次技术深度研讨会（MAM 217）主题为"使用 AWS Transform 构建可组合的 AI 代理以支持合作伙伴解决方案"。会议由 AWS Transform 团队的产品管理负责人 Mortaza Chowri 主持，并由同事 Ravi 和 Alexi 共同演讲。会议重点介绍了 AWS Transform 服务如何从分散的迁移和现代化工具演进为统一的代理平台，以及如何通过可组合性（composability）使合作伙伴能够将其专有工具和代理集成到 AWS Transform 生态系统中。

AWS Transform 于 2025 年 5 月正式发布，目前支持多种专业化代理，包括 VMware 迁移、大型机现代化、Microsoft .NET 和 SQL Server 现代化，以及自定义转换功能。该服务提供了统一的工作空间体验，支持团队协作、角色管理、作业编排和持续学习循环。会议的核心信息是：AWS Transform 不仅整合了 AWS 自身的工具，还为合作伙伴提供了集成框架，使客户能够在单一平台上获得无缝的迁移和现代化体验，同时保持安全性、可观测性和企业级治理。

会议通过多个现场演示展示了代理集成、部署和实际应用场景，特别强调了基于 Base Agent 模块的开发体验、与 Amazon Bedrock Agent Core Runtime 的集成，以及如何通过知识库增强和代理间通信实现复杂的业务转型目标。

- **00:29:50 - 00:32:20** - 强调集成优势：清晰的执行阶段可见性、易于故障排查、无需额外工具即可获得结构化日志和进度追踪

### 核心能力介绍
- **00:32:20 - 00:35:40** - 安全性：与企业身份提供商无缝集成、所有操作需要授权、自动处理凭证生命周期管理（获取、刷新、续订、撤销）、内置审计跟踪

- **00:35:40 - 00:38:15** - 协作与治理：通过工作空间和角色委派职责、审批机制提供关键监督、安全的工件存储支持静态数据加密、可选的客户管理密钥（KMS 集成）、内置滥用预防和检测

- **00:38:15 - 00:41:30** - 工作流编排：作业计划管理覆盖代理整个生命周期、支持单代理和多代理编排流程、支持人机交互（Human-in-the-Loop, HITL）模式和代理间交互

- **00:41:30 - 00:44:20** - 部署与运行：与 Amazon Bedrock Agent Core Runtime 内置集成、按需计算能力自动扩展、可选的自主模式（无需人工干预）、检查点机制提供容错和故障恢复能力

- **00:44:20 - 00:46:50** - 监控与可观测性：内置日志、追踪和指标集成、支持构建仪表板和设置告警、便于事件响应

### 可组合工作流
- **00:46:50 - 00:49:30** - 定义可组合工作流：工具和代理与 AWS 及其他合作伙伴的代理无缝协作，实现单独无法完成的复杂转型目标

- **00:49:30 - 00:52:45** - 介绍可组合性组件：Base Agent 模块（与 AWS Transform 能力集成）、知识库集成（注入领域特定知识）、Bedrock Agent Core Runtime 集成（安全 VPC 环境部署）、代理注册表（实现代理发现和消费）

- **00:52:45 - 00:56:20** - 展示完整的转型流程：开发环境使用 AWS Transform Agent SDK 和 Base Agent 模块 → 本地测试 → 构建 Amazon ECR 镜像 → 上传到 AWS 账户并在 Agent Core Runtime 上运行 → 通过代理注册表使其可被发现 → 通过聊天或代理间交互使用

- **00:56:20 - 00:58:40** - 总结代理构建体验：构建阶段（生命周期管理、内存、知识库集成）→ 本地测试（正确性验证）→ 部署（通过 AWS Marketplace 等渠道分发）→ 运维监控（大规模运行）

### Alexi 演示：高级用例
- **00:58:40 - 01:01:30** - Alexi 提出可组合性的核心问题：如何让代理更强大？如何通过知识增强代理推理能力？如何同时驱动业务和技术转型？

- **01:01:30 - 01:04:15** - 类比团队智慧：智能团队不仅需要工具、知识库和内存，更需要成员间的有效沟通和知识共享；引入"灵活推理"概念，代理通过直接通信和信息共享产生新数据和新兴能力

- **01:04:15 - 01:07:40** - 介绍实际用例场景：金融行业应用管理者面对欧洲 DORA 法规（数字运营弹性法案）合规要求，需要自动检测大型机应用是否符合新法规并进行修复

- **01:07:40 - 01:10:20** - 解决方案架构：创建 DORA 代理 + 复用现有大型机业务规则提取代理（BR Agent）+ 使用 Cursor IDE 构建代理系统和自动修复；BR Agent 提取业务规则 → DORA 知识库描述法规要求 → Cursor 指导文件定义正确架构模式

### Cursor IDE 集成演示
- **01:10:20 - 01:13:45** - 演示 Cursor 中的 MCP 服务器：自动化构建 Base Agent、通过对话了解 AWS Transform 和 Base Agent 概念、Cursor 提供分步指导

- **01:13:45 - 01:16:30** - Base Agent 类型说明：编排器 Base Agent（负责大规模推理）和子代理（任务特定）；使用工具装饰器声明子代理、注册工具即可继承所有 Base Agent 能力

- **01:16:30 - 01:19:20** - 演示编排器实现：Cursor 生成示例代码和通信流程图，展示两个核心工具：注册表查询工具（检索可通信的子代理和编排器列表，控制代理可见性）和代理间消息传递工具

- **01:19:20 - 01:22:00** - 强调关键特性：代理间通信基于结构化消息传递、支持异步通信、内置重试和错误处理机制、所有通信自动记录到工作日志

### 实际业务场景演示
- **01:22:00 - 01:25:30** - 展示 DORA 合规性检查实际执行：用户通过聊天界面提问"我的应用是否符合 DORA 法规？"、DORA 编排器代理调用 BR Agent 提取业务规则、同时查询 DORA 知识库获取法规要求

- **01:25:30 - 01:28:15** - 代理协作过程：BR Agent 分析大型机 COBOL 代码、提取事务管理和客户通知相关业务规则、DORA 代理对比法规要求识别合规差距、生成详细的合规性报告

- **01:31:00 - 01:33:40** - 展示新兴能力：代理不仅执行预定义任务，还能通过知识库组合产生新见解、自动生成之前不存在的合规性文档和修复代码、展示真正的"推理"而非简单的模式匹配
