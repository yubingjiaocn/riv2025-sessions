# AWS re:Invent 2025 会议总结：使用 Amazon Bedrock 和 Kiro 加速开发与 DevSecOps

## 会议概述

本次会议主要介绍了如何利用 Amazon Bedrock 和 Kiro（原 Amazon Q）通过 AI 代理（Agentic AI）加速软件开发生命周期（SDLC）。演讲团队由 AWS EMEA 合作伙伴解决方案架构团队负责人 Dana Vien、来自慕尼黑的 Fabio 以及 Ovia Cloud 的 CTO George Kine 组成。

会议强调了传统 SDLC 面临的严重生产力损失问题。根据行业数据，78% 的软件工程师将至少 30% 的时间花费在重复性手动任务上，另有 20% 的时间因上下文切换和技术债务而浪费。以 250 人的开发团队为例，这相当于每年约 1340 万美元的生产力损失。演讲者提出了一种平台驱动的方法，通过将 AI 代理集成到 SDLC 的各个阶段来解决这些挑战。

该解决方案的核心是利用 Amazon Bedrock Agent、模型上下文协议（MCP）服务器以及项目知识库，为业务分析师、解决方案架构师、开发人员和运维工程师等不同角色提供支持。演讲团队展示了超过 20 种架构模式，并通过实际演示说明了如何将软件发布周期从 18 个月缩短至 6-8 个月。Gartner 预测，到 2028 年，90% 的企业软件工程团队将使用 AI 代码助手，这标志着从 2024 年初的 14% 实现了 6 倍的增长。

### 实际挑战与解决方案（15:00-20:00）
- **15:00** - 三大挑战类别：
  - 工具集和开发环境碎片化
  - 集成和互操作性问题（数据孤岛）
  - 企业级安全性、可观测性和可扩展性
- **17:00** - 现场调查：约 30-40% 的参会者已在构建代理，部分已部署到生产环境
- **18:00** - AI 代理定义：利用 LLM 进行推理、规划并代表人类或系统完成任务的自主软件系统
- **19:00** - 对比：生成式 AI 是一次性交互，而 AI 代理可以读取需求、与工具交互、理解标准并执行多步骤任务

### AI 代理工作原理（20:00-25:00）
- **20:00** - AI 代理的四个关键组件：
  1. **内存**：保留会话间的上下文（短期和长期记忆）
  2. **工具集成**：执行任务（如在 Jira 中创建任务）
  3. **规划能力**：反思、自我批评、思维链
  4. **行动**：自动执行或人工监督
- **22:00** - 代理设计最佳实践的四个关键领域：
  - 框架选择（考虑复杂性、模块化、集成能力）
  - 工具和知识库（网络搜索、代码解释器）
  - LLM 选择（推理能力、执行速度、成本）
  - 执行技术（ReAct、计划执行、反思）

### AWS Agentic AI 服务（25:00-30:00）
- **25:00** - AWS 提供三种方法：
  1. **代理驱动应用**：Amazon Q Suite、Kiro IDE、Kiro CLI
  2. **完全托管服务**：Amazon Bedrock Agents（内置编排）
  3. **DIY 方法**：Multi-Agent Orchestrator SDK、Nova Act SDK
- **27:00** - Amazon Bedrock Agent Core：提供身份访问管理、网关、可观测性、代码解释器、网络浏览等功能
- **28:00** - 介绍 Kiro：规格驱动开发的 IDE，用户只需描述需求即可生成实现
- **28:30** - Kiro CLI：命令行界面，可扫描代码错误、安全漏洞，自动修复并实现工作流自动化

### 平台驱动架构（30:00-38:00）
- **30:00** - 解决方案架构概述：基于 React.js 的自定义 UI，使用 Amazon Cognito 进行基于角色的访问控制
- **31:00** - 后端：使用 Amazon Bedrock Agent 构建 SDLC 代理，集成 MCP 客户端和 Fast API
- **32:00** - 架构模式 1：业务分析师模式 - 通过 MCP 服务器集成 Confluence 和 Jira
- **33:00** - 架构模式 2：项目知识库 - 使用 Amazon Bedrock Knowledge Base 和 S3 存储所有角色的历史数据
- **34:00** - 架构模式 3：架构设计模式 - 使用 draw.io 创建架构图和 OpenAPI 规范
- **35:00** - 架构模式 4：开发者模式 - Kiro 集成知识库以理解需求和设计
- **36:00** - 架构模式 5：测试模式 - 通过 MCP 服务器集成 JMeter
- **37:00** - 架构模式 6：运维模式 - 集成 Splunk、PagerDuty 和 Slack，实现异常检测和 RCA 分析
- **37:30** - 展示 QR 码：提供 20+ 架构模式和示例代码的访问链接

### 架构优势总结（38:00-40:00）
- **38:00** - 平台驱动方法的优势：
  - 在整个 SDLC 中实现 AI 集成
  - 灵活添加/删除工具，打破数据孤岛
  - 为不同角色定制 UI
  - 项目知识库保存整个 SDLC 的信息

  - 代理管理复杂编排
  - 使用 Guardrails 实现负责任的 AI

### 关键技术要点
- **MCP（模型上下文协议）**：标准化应用程序向 LLM 提供上下文的开放协议
- **Amazon Bedrock Agent Core**：完全托管的代理平台，提供编排、安全和可观测性
- **项目知识库**：使用 Amazon Bedrock Knowledge Base 和 S3 向量存储，实现跨角色的上下文共享
- **基于角色的访问控制**：使用 Amazon Cognito 确保不同角色只能访问相关功能
- **规格驱动开发**：Kiro 的核心方法，从高级需求自动生成实现

### 预期成果
- 将软件发布周期从 18 个月缩短至 6-8 个月
- 减少 30-50% 的重复性手动任务时间
- 改善跨团队协作和上下文共享
- 通过自动化文档和代码生成提高一致性
- 降低技术债务和上下文切换成本
