# AWS技术会议总结：奥迪通过AI革新焊接检测系统

## 会议概述

本次AWS re:Invent技术会议展示了奥迪汽车如何利用人工智能技术彻底改造其焊接检测系统。会议由AWS客户解决方案经理Vina Kovvagillis、奥迪制造规划专家Matias Maya以及AWS专业云架构师Fabrizio Manfred共同主持。

演讲聚焦于制造业面临的核心挑战：全球前500强制造商每年因生产瓶颈导致的低效率损失高达1.4万亿美元，而计划外停机的成本在2019年至2024年间翻了一番。为应对日益增长的生产复杂性和成本压力，奥迪与AWS合作开发了数字化生产平台(DPP)，该平台已成功运行5年，连接了大众集团旗下120多家工厂，实现了从车间到云端的数据集成。

会议详细介绍了两个创新用例：电阻点焊分析系统和基于计算机视觉的焊接飞溅检测系统。这两个系统展示了如何通过AI技术将质量检测从每天1万个抽样点提升到500万个全量检测点，同时将检测周期从静态的每周5辆车优化为动态的每天1辆车，极大提升了生产效率和质量保障能力。

### 用例一：电阻点焊分析系统 (11:30 - 25:00)

11:30 - 车身车间概况：
- A6和A5生产线每天约1000辆车
- 约1200台机器人，800个焊接机器人
- 每辆车5500个焊点
- 每天总计500万个焊点

13:00 - 传统质量检测方法的局限性：
- 质检人员每天使用超声波设备检测约1万个点
- 仅检测随机样本(约1辆车)
- 存在漏检风险：如果抽样恰好选中合格零件，则无法发现批次中的不良品

14:30 - AI解决方案优势：
- 从检测1万个点提升到500万个点
- 从静态检测转变为动态检测
- 实时显示缺陷位置和问题区域

15:45 - 检测频率优化：
- 过去：每周5辆车(每天1辆)
- 验证阶段：每周3辆车
- 当前：仅1辆车进行静态检测(满足质量保证要求)，其余全部动态检测

17:00 - AI模型架构说明：
- 数据来源：800台焊接机器的实时数据
- 每个焊点包含约400个数据点的电阻曲线
- 结合工艺参数和领域专家知识
- 训练两个模型：回归模型(基于超声波检测结果)和异常检测神经网络

### 技术架构详解 - 电阻点焊系统 (25:00 - 35:00)

25:00 - 数据规模挑战：
- 500万个焊点
- 每个焊点最多5万个数据点
- 每条曲线采集3个核心参数(电阻、电压、压力)，每毫秒采集一次
- 焊接时长超过5秒，产生5000个数据点/曲线
- 总数据量达到数十亿级别

26:30 - 数据采集架构：
- 焊接控制器通过MQTT协议发送数据到边缘网关
- 制造环境按域划分，生产线有严格的网络隔离限制
- 引入H网关作为中间层，连接生产线与云端

28:00 - 数据传输优化策略：
- 实时控制数据：使用MQTT
- 遥测数据：使用流式传输
- 批量数据：文件上传模式
- 通过打包和批处理减少传输调用次数

30:00 - 云端架构设计原则：
- 数据传输一次，多用例复用
- 解耦生产者和消费者
- 数据存储在数据中心(Data Hub)，正在迁移到数据网格(Data Mesh)

31:30 - MLOps实施：
- 数据科学家访问数据进行实验和模型调整
- 模型训练和部署自动化
- 推理结果可视化供车间操作员使用
- 推理不需要亚毫秒级响应(每个工位约1分钟)，因此在云端执行

33:00 - AWS服务应用：
- 边缘：AWS IoT Greengrass(包含MQTT代理、流管理器、文件上传组件)
- 数据摄取：Kinesis Data Stream(输入)和Kinesis Firehose(输出)
- 数据延迟：20-30秒
- 控制平面：AWS IoT Core(网络延迟约数百毫秒)

34:00 - 多账户架构：
- 摄取账户：数据接入
- 数据中心账户：集中存储和处理，多用例共享
- 数据科学家账户：使用SageMaker进行模型开发
- 应用账户：运行推理，结果存储在Time Stream时序数据库
- 每个工厂独立部署实例，避免多租户协调问题

34:45 - 自动化部署：
- 使用基础设施即代码(IaC)
- 遵循DevOps最佳实践
- 数分钟内可在新工厂部署用例

### 用例二：焊接飞溅检测系统 (35:00 - 48:00)

35:00 - 焊接飞溅问题介绍：
- 焊接过程中产生的金属飞溅物
- 金属板材配合不完美时会产生"爆炸"效应
- 在车身表面形成小缺陷

36:30 - 焊接飞溅的危害：
- 可能损坏车身内部线束
- 员工操作时存在受伤风险
- 影响外观整洁度
- 可能导致腐蚀问题
- 增加额外成本

37:45 - 传统处理方式的挑战：
- 员工在返工工位手动打磨
- 工作量大且不可预测
- 不同员工处理结果不一致
- 时间限制严格：60秒周期时间内，仅20秒用于检测
- 需要检查约500个焊点
- 工作姿势不佳(车顶和车底)，人体工程学问题严重

39:00 - AI解决方案需求：
- 必须在20秒内完成推理
- 结果直接传输到PLC(可编程逻辑控制器)
- 实现"打磨引导灯"系统：AI识别飞溅位置，灯光指示员工打磨位置
- 检查点从500个减少到100-200个

40:30 - 系统工作流程：
- 车身进入工位，拍摄照片
- 机器人开始工作的同时进行AI分析
- 20秒内将结果传输到下一工位
- 引导灯指示需要打磨的位置

41:45 - 检测难度展示：
- 单张照片中的飞溅物非常小，肉眼难以快速识别
- A6车型尺寸大，员工需要在车身两侧快速移动检查

43:00 - 技术架构特点：
- 模型开发和训练仍在云端
- 推理在车间边缘执行
- 使用工业PC配备GPU
- 8个摄像头，每个2000万像素
- 单张图片约50MB
- 推理时间控制在20秒内
- 直接连接PLC，便于集成

44:30 - 未来发展方向：
- 从引导员工转向机器人自动打磨
- 内卡苏尔姆工厂正在进行仿真测试
- 预计圣诞节前完成组件集成
- 明年初实现车身表面自动打磨

### 技术架构详解 - 焊接飞溅检测系统 (48:00 - 55:00)

48:00 - 实时性要求：
- 目标响应时间：低于1秒
- 为未来机器人自动控制做准备
- 不仅识别位置，还要控制机械臂进行清理

49:00 - 边缘计算架构：
- 摄像头位于南向网络(Southbound)
- 边缘设备运行推理
- MQTT作为通信骨干
- PLC通过MQTT向边缘运行时发送命令
- 边缘设备触发拍照并分析图像

50:30 - 边缘网关演进：
- 从工业计算机迁移到Kubernetes集群
- 实现云端类似的弹性扩展能力
- 每个组件运行在独立容器中
- 适应制造环境的动态需求

51:45 - 数据传输策略：
- MQTT：控制平面，连接AWS IoT Core
- S3：上传所有分析过的图像
- 图像存储在数据中心，供其他用例复用

52:30 - 数据科学工作流：
- 访问图像数据进行模型训练
- 使用相同的SageMaker工具
- 构建计算机视觉异常检测模型

53:15 - 模型部署流程：
- 数据科学家部署流程保持一致
- 应用账户架构复用
- 新增：通过边缘网关将模型打包部署到本地
- 使用MQTT触发部署操作并验证成功

54:00 - 架构复用性：
- 基础设施即代码实现
- 易于在多个工厂复制
- 从近实时演进到实时架构
- 保持相同的设计原则

### 经验总结与最佳实践 (55:00 - 60:00)

55:00 - 核心经验教训：数据质量是AI用例的基础

55:30 - 电阻点焊项目的数据挑战：
- 需要向数据科学家解释每个焊点400个数据点的含义
- 理解工艺流程、故障原因需要大量领域知识

56:15 - 焊接飞溅项目的数据挑战：
- 不同专家对同一图像有不同判断
- 用6种不同标准训练模型导致准确率仅30%
- 必须统一标注标准才能获得高准确率和高可用性

57:00 - 常见数据问题：
- 同一对象在不同工厂有不同名称
- 测量单位不统一
- 人工标注不一致
- 历史数据依赖纸质记录
- 数据协调和对账困难

57:45 - 实施策略：从小做起，快速迭代
- 从连接一个焊枪开始
- 连接1个和连接1000个所需时间相近
- 先把一个做好，扩展到1000个就不是问题

58:30 - 可扩展性设计原则：
- 数据仅复制一次，避免设备过载
- 工厂设备通常已满负荷运行
- 多次提取数据会被车间阻止
- 优化资源消耗

59:00 - 可组合架构原则：
- 组件可插拔
- 可独立重构而不影响其他组件
- 支持持续改进和重构

59:30 - 文化转变挑战：
- 制造业传统思维：设计一次，运行10年不变
- DevOps思维：持续重构和改进
- 重构不是设计失败，而是持续优化的必要过程
- 需要改变"不要触碰运行中的生产系统"的观念
