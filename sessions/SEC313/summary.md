# AWS re:Invent 2025 会议总结：使用 Bedrock Agent Core Identity 保护代理访问

## 会议概述

本次会议由 AWS 首席产品经理 Rahul 和 Ben 共同主持,深入探讨了如何使用 Amazon Bedrock Agent Core Identity 来解决 AI 代理(Agentic AI)在生产环境中面临的身份验证和访问管理挑战。

随着企业对 AI 代理的采用快速增长,Gartner 预测到 2028 年,33% 的企业应用将包含 AI 代理元素(相比去年仅 1%),50% 的决策将由代理自主完成。这种快速增长带来了显著的生产力提升、情境感知体验和更快的上市时间。然而,将代理从原型阶段推向生产环境时,组织面临着三大核心挑战:如何安全地委托权限给代理、如何处理用户授权疲劳、以及如何避免编写大量自定义集成代码。

为解决这些挑战,AWS 推出了 Bedrock Agent Core Identity 服务。该服务为每个代理部署创建唯一的工作负载 ID,提供身份提供商无关的授权器,管理 OAuth 流程编排,并通过令牌保管库持久化存储用户授权。该服务支持代理访问 AWS 资源(通过 SIG v4 签名)和非 AWS 资源(通过三方 OAuth、两方 OAuth 和 API 密钥),同时保持框架无关性,允许开发者在任何运行时环境中使用。会议还宣布了多项新功能,包括增强的 MCP 服务器集成和改进的授权管理能力。

### Bedrock Agent Core Identity 介绍 (22:00 - 32:00)
- **22:00** - Ben 介绍解决方案:Amazon Bedrock Agent Core Identity
- **22:30** - 现场调查:有多少人听说过或使用过 Agent Core Identity
- **23:00** - 三大核心价值:
  1. 为代理提供安全的委托访问
  2. 简化 AI 体验,减少授权疲劳
  3. 加速代理构建,消除重复性工作
- **24:30** - Agent Core 生态系统组件:Runtime、Gateway、Memory、Observability、Browser、Code Interpreter
- **25:30** - Agent Core Identity 作为核心粘合剂,连接所有组件
- **26:00** - 入站和出站身份验证流程:
  - 入站:用户如何访问代理
  - 出站:代理如何访问外部工具
- **27:30** - 四大核心组件:
  1. 身份目录:为每个部署创建唯一工作负载 ID
  2. 授权器:身份提供商无关,支持 Okta、Entra、Ping 等
  3. 资源凭证提供者:管理 API 密钥和 OAuth 凭证
  4. 令牌保管库:存储令牌并管理 OAuth 授权流程
- **30:00** - 工作流程详解:用户获取访问令牌 → 调用代理 → 授权器验证 → 创建工作负载访问令牌绑定用户和代理 → 访问外部工具
- **31:00** - 重要特性:支持自带代理运行时(Bring Your Own Agent Runtime),可在 EKS 等环境中使用

### 访问模式详解 (32:00 - 40:00)
- **32:00** - Rahul 强调设计原则:框架无关、身份提供商无关、运行时无关
- **33:00** - 支持的访问模式:
  - AWS 资源访问
  - 非 AWS 资源访问(OAuth 和 API 密钥保护)
  - 代表用户行动和预授权场景
- **34:30** - AWS 资源访问:使用 SIG v4 签名协议
- **35:30** - 代理签名 API 调用 → 假定 IAM 角色 → AWS 验证 → 颁发 STS 凭证
- **36:30** - 非 AWS 资源访问:三方 OAuth 流程(Three-legged OAuth)
- **37:30** - 三方 OAuth 示例:Sarah 登录 → 身份提供商验证 → 颁发访问令牌 → 代理使用令牌访问 CRM、财务系统和文档工具
