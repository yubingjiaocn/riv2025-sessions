# AWS re:Invent 2025 - 使用 Kiro 进行规格驱动开发

## 会议概述

本次技术分享会由 AWS 高级开发者倡导者 Eric Hchett 和 Kiro 产品团队负责人 Nick Hill 共同主持,重点介绍了 AWS 最新推出的智能 IDE 工具 Kiro 以及其核心开发方法论——规格驱动开发(Spec-Driven Development)。

Kiro 是 AWS 在几个月前发布的新型智能 IDE,并在几周前正式进入通用可用(GA)阶段。该工具一经发布便引起了广泛关注,下载量达到数十万次,甚至需要设置等待列表来管理用户需求。与传统的"氛围编码"(Vibe Coding)方式不同,Kiro 强调在代码生成之前进行系统化的规划和设计,这种方法更符合传统软件开发生命周期(SDLC)的最佳实践,同时充分利用了 AI 的能力。

在演示环节中,两位讲师现场构建了一个求职面试准备应用(Job Seekers App),完整展示了从需求定义、设计文档生成、任务列表创建到代码实现的全流程。这个应用允许用户进行模拟面试,通过语音回答问题,并获得 AI 生成的反馈和评分。整个开发过程充分体现了规格驱动开发的优势:结构化、可追溯、易于协作,并且能够生成高质量的文档和代码。

### Kiro 产品介绍与发展历程 (05:30 - 12:00)

05:30 - Nick Hill 开始介绍 IDE 的演进历史

06:15 - 传统 IDE 的价值:集成编译器、代码检查、调试等功能于一体

07:00 - AI 编辑器的兴起:从代码补全到智能代理(Agentic)体验

08:00 - 开发范式的转变:从"开发者驱动"到"开发者引导 AI"

09:00 - "氛围编码"(Vibe Coding)的局限性:类似于冗长的 Slack 对话线程,缺乏结构化的决策记录

10:30 - 规格驱动开发的核心理念:在生成代码前进行规划,包括需求文档、设计文档和权衡分析

### Kiro 核心功能详解 (12:00 - 20:00)

12:00 - Hooks(钩子)功能介绍:在文件保存、创建等事件触发时自动执行任务

13:00 - Hooks 应用场景:自动更新文档、本地化处理、拼写和语气检查、React 组件单一职责原则验证

14:30 - Agent Steering(代理引导)功能:通过规则文件定义编码标准和设计规范

15:00 - 演示自动生成 Steering 文件:产品文档和技术栈文档

16:30 - MCP(Model Context Protocol)集成:展示已安装的 MCP 服务器,包括 AWS 文档、Bedrock Agent Core、Brave Search 等

17:45 - AWS 提供的 MCP 服务器:文档服务器作为通用选择,以及针对特定服务的专用服务器

18:30 - 一键安装 MCP 服务器功能介绍

### 实战演示:构建求职面试应用 (20:00 - 45:00)

20:00 - 项目背景介绍:构建一个帮助求职者准备面试的应用

21:00 - 应用核心功能:用户登录、设置个人资料、开始面试准备、问答流程、录音回答、生成定性和定量反馈

22:00 - 展示 Figma 设计原型

23:00 - 开始使用 Kiro:介绍 Kiro 吉祥物(幽灵图标)和 Kiro House 体验区(可获得免费积分)

24:30 - 选择 Spec 模式(而非 Vibe 模式)开始开发

25:00 - 上传 Figma 截图并输入提示词:"请使用这张图片创建一个求职者应用,将图片作为设计参考"

26:00 - Kiro 开始生成需求文档(Requirements Document)

### 需求文档审查与优化 (26:00 - 32:00)

26:30 - Nick Hill 讲解需求文档结构:用户故事(User Story)+ 验收标准(Acceptance Criteria)

27:30 - 审查生成的用户故事:登录、个人资料管理、面试准备会话、查看和回答问题、多问题导航、分析结果、趋势分析、界面设计

29:00 - 决定简化 MVP 范围:删除需求 1、2、7、8,保留核心面试功能

30:00 - 强调迭代式开发的重要性:不是一次性生成完美结果,而是通过对话不断优化

31:00 - 限制面试问题数量为 3 个,以便快速演示

### 设计文档生成与完善 (32:00 - 40:00)

32:00 - 进入设计阶段,Kiro 基于需求生成设计文档

33:30 - 审查高层架构:开始屏幕 → 问题屏幕 → 结果屏幕

34:00 - 技术栈确认:Vue 3、TypeScript、Vite、Tailwind CSS、本地存储、Web Audio API

35:00 - 组件接口定义:QuestionResponse、SessionState、各种视图组件

36:30 - 发现缺失功能:语音转文字和 AI 分析结果生成

37:30 - 添加新需求:"请转录用户说话的文本,并使用 Claude AI SDK 对结果进行额外分析"

38:30 - Kiro 自动更新设计文档和需求文档,确保一致性

39:00 - 介绍检查点(Checkpoint)功能:可以随时恢复到之前的状态

### 任务列表创建与调整 (40:00 - 48:00)

40:00 - 进入任务列表阶段

41:00 - 可选任务与必需任务:属性测试(Property-Based Testing)默认设为可选

42:00 - 审查任务列表:创建数据模型、实现音频录制服务、响应验证存储、AI 集成等

43:30 - 优化任务顺序:要求在前 5 个任务中实现可视化的 MVP

44:30 - 产品开发类比:先造滑板,再造自行车,最后造汽车(而非分别造轮子、引擎再组装)

45:30 - 添加 UI 到初始阶段,以便进行可视化测试

### 规格驱动开发的优势总结 (48:00 - 结束)

48:00 - 强调文档化的重要性:需求、设计、任务列表都有清晰记录

49:00 - 与传统 SDLC 的契合:Kiro 的工作流程与 AWS 内部的文档驱动文化一致

50:00 - Refine(优化)功能:可以手动编辑文档后让 Kiro 重新整合

51:00 - Update Tasks(更新任务)功能:当代码库发生变化时,自动更新任务列表

52:00 - 灵活性:支持测试驱动开发(TDD)等不同开发方法论

53:00 - 演示即将继续进行代码实现阶段(字幕在此处截断)
