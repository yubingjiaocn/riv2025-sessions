# AWS re:Invent 2025 会议总结：利用生成式AI加速电信行业主机迁移与现代化

## 会议概述

本次AWS re:Invent 2025分组会议聚焦于电信行业面临的最大挑战之一：如何对运行了数十年的传统基础设施进行现代化改造。会议由AWS首席解决方案架构师Afatnir Burgerer主持，AT&T企业架构和混合云平台工程负责人Rama Ragavan以及AWS高级解决方案架构师Sanjay Agaval共同参与。

会议深入探讨了电信行业在进行大型遗留系统转型时面临的独特挑战。这些系统通常包含数百万行代码，每天处理数十亿笔交易，服务数百万客户，同时必须保持业务连续性。传统的迁移方法由于缺乏所需的速度和规模能力，一直无法有效解决这些问题。然而，生成式AI的出现为解决这些长期存在的难题带来了新的可能性。

会议展示了AWS如何通过其生成式AI技术栈，特别是Amazon Q Developer、AWS Application Transformation和Hero等服务，实现了迁移和现代化流程的自动化。AT&T分享了他们正在进行的大型主机现代化项目的实际经验，展示了这些技术如何在真实世界场景中应用。会议的核心信息是：生成式AI能够加速和自动化70%的转型工作，同时不增加额外风险。

### 电信行业现状与挑战 (02:30 - 06:00)
- **02:30** - 行业现状：70%的电信IT系统仍在本地运行（2025年数据）
- **03:00** - 根据麦肯锡研究，70%的系统已运行超过20年
- **03:30** - 这些不是后台系统，而是计费系统和客户管理系统等关键业务系统
- **04:00** - 传统迁移平均需要1.5年，电信工作负载可能需要更长时间
- **04:30** - 电信转型的独特挑战维度：
  - 关键任务系统（5个9的可用性要求，每年仅5分钟停机时间）
  - 数据量和业务逻辑复杂性（20-30年积累的逻辑）
- **05:00** - 监管合规要求：CPNI（客户数据）、PCI DSS（支付）、GDPR（欧洲数据隐私）
- **05:30** - 集成复杂性：围绕订单管理系统的数十个应用程序形成的"意大利面条式"生态系统
- **05:45** - 遗留技术债务：根据Forrester研究，20%的IT支出用于维护遗留系统

### 传统迁移流程的局限性 (06:00 - 08:00)
- **06:00** - 评估阶段的挑战：缺乏足够的知识，需要查看代码本身来获取业务逻辑
- **06:30** - 需要对代码进行逆向工程才能理解系统
- **07:00** - 文档问题：业务逻辑的文档实际上就在代码本身中
- **07:15** - 核心转换工作：COBOL到Java的转换需要大量手工工作
- **07:30** - 必须维护20多年来添加的所有业务逻辑复杂性
- **07:45** - 迁移测试：需要测试多年来添加的大量边缘用例（闰年、时区等）

### 生成式AI的愿景 (08:00 - 09:00)
- **08:00** - AWS的目标：加速和自动化70%的转型工作

- **08:40** - 引入生成式AI作为解决方案

- **10:20** - 第三阶段：代理式AI（多代理系统，可以完全自主执行复杂流程）
- **10:50** - 从交付转变为完全自主的流程执行
- **11:20** - AI代理的工作循环：观察、推理、行动、反思

### AI代理工作原理详解 (12:00 - 15:30)

- **12:40** - 推理：确定需要采取的步骤和方法
- **13:00** - 行动：使用工具、调用API、编排变更
- **13:20** - 反思：评估结果并持续学习改进
- **13:40** - 电信客户服务示例：被动代理vs主动代理
- **14:00** - 被动代理示例：简单回答月费查询（$85/月）

- **15:00** - 主动代理可以代表客户执行套餐变更

### 多代理系统 (15:30 - 17:00)
- **15:30** - 单个代理无法执行复杂的综合流程

- **16:25** - 代码转换代理：执行代码转换
- **16:40** - 单元测试代理：构建测试
- **16:55** - 内部代理通信：协调所有代理工作

### AWS生成式AI技术栈 (17:00 - 20:00)
- **17:00** - 介绍AWS代理式AI技术栈
- **17:20** - 基础层：专为AI工作负载优化的基础设施（AWS Trainium、Inferentia、GPU）
- **17:50** - 模型和工具层：Amazon Bedrock提供基础模型访问和代理运行时编排
- **18:20** - 应用层：Amazon Q Developer、AWS Application Transformation、Hero
- **18:40** - Amazon Q Developer：不仅是开发者聊天机器人，能够理解、编写和操作AWS上的应用程序
- **19:10** - AWS Application Transformation：首个行业专用服务，支持VMware、.NET和主机转换
- **19:30** - 新功能发布：Windows全栈转换（SQL Server到Aurora PostgreSQL）
- **19:50** - AWS Transform Custom：可重复的转换模式，支持版本升级、运行时迁移等

### Amazon Q Developer深度解析 (20:00 - 22:30)
- **20:00** - Q Developer是专业化代理列表，覆盖整个软件开发生命周期
- **20:20** - 软件开发代理：在几分钟内实现应用程序变更（生产级功能）
- **20:40** - 单元测试代理：为新代码构建综合测试
- **21:00** - 文档代理：解决遗留应用程序文档缺失问题，提供代码可解释性
- **21:20** - 代码审查代理：像高级工程师一样查找错误、监管约束和配置错误
- **21:40** - 转换代理：处理Java版本升级等挑战
- **22:00** - Java版本升级问题：从Java 6/8升级到17/21，解决生命周期结束和安全漏洞

- **23:10** - 演示过程通常需要10-30分钟，取决于项目规模
- **23:30** - 展示后台推理过程：确定需要执行的活动
- **23:50** - 第一步：JDK版本升级，完整分析依赖关系

- **24:30** - 第二步：识别已弃用的代码并相应更改

### AT&T的主机挑战 (30:00 - 34:00)
- **30:00** - 需要匹配和超越现有规模
- **30:20** - 主机系统是高性能系统，云平台提供新能力
- **30:40** - 安全性：Linux平台的漏洞多于主机，但需要高度安全的演进
- **31:00** - 业务方面：降低业务风险，随着演进提供更多能力
- **31:20** - 当前资产挑战：代码、文档，许多人员已离职
- **31:40** - 发现现有系统的挑战
- **32:00** - 评估了退役平台的可能性
- **32:20** - 可以退役10-20%，但之后发现应用程序服务于多条产品线的关键业务功能
- **32:40** - 没有积极追求重新托管（不想在替代托管环境中运行另一个主机）
- **33:00** - 转而追求代码转换路径
- **33:20** - 市场上有多种产品，但存在"代码异味"问题
- **33:40** - 源代码转换后看起来像用Java编写的COBOL

### AT&T的转型方法 (34:00 - 37:00)
- **34:00** - 雇佣顶级Java开发人员时，他们会质疑转换后的代码质量
- **34:20** - 过去18-24个月产品不断改进，支持面向对象的Java代码
- **34:40** - 达到某种程度的转换，但仍有局限
- **35:00** - AI的演进和可能性引发探索
- **35:20** - 与AWS合作伙伴探索关键能力
- **35:40** - 不再受现有环境约束的限制
- **36:00** - 使用现有工具集可以迁移十分之一的百分比
- **36:20** - 相信通过新的生成式AI能力可以做得更多
- **36:40** - 目标：不仅仅是重新平台化代码，而是进一步重构和重新构想整个系统

- **37:40** - 生成式AI增加了第8个R：重新构想（Reimagine）
- **38:00** - 这一新增改变了一切
- **38:20** - 不仅是重新构想，重构方式也发生了巨大变化

### 重构的定义 (39:00 - 41:00)
- **39:00** - 明确重构的含义（不同人可能有不同理解）
- **39:20** - 示例：将主机上运行的COBOL应用程序迁移到AWS云
- **39:40** - 重构定义：应用程序在本地的工作方式不会改变
- **40:00** - 业务逻辑和功能需求保持不变
- **40:20** - 改变的是技术栈：COBOL转换为Java/Python，数据库转换为RDS上的SQL引擎
- **40:40** - 可能从单体架构重新设计为微服务架构
- **40:50** - 生成式AI允许从头到尾自动化完成所有这些工作

### AWS Application Transformation详解 (41:00 - 44:00)
- **41:00** - 介绍AWS Application Transformation
- **41:20** - 它是预训练代理的集合，并排工作
- **41:40** - 从查看代码一直到部署
- **42:00** - 代理协同工作，用户完全控制
- **42:20** - 深入了解：将COBOL应用程序提供给AWS Transform时会发生什么
- **42:40** - 代码分析代理接管，逐行检查代码并学习
- **43:00** - 创建应用程序蓝图
- **43:20** - 找出所有依赖关系，进行依赖关系映射，找出所有接口
- **43:40** - 创建独特的应用程序蓝图（以前从未有过）

### 文档生成代理 (44:00 - 46:30)

- **44:20** - 这是魔法开始的地方

- **45:00** - 业务文档：准确说明应用程序在做什么
- **45:20** - 技术文档：逐个函数说明系统如何设计
- **45:40** - 这是游戏规则改变者：迁移中最耗时的部分是测试
- **46:00** - 如果出现问题需要修复，没有技术文档是最困难的部分
- **46:20** - 这个代理完美地解决了这个问题，将迁移时间缩短一半
