# AWS re:Invent 2025 技术会议总结

## 会议概述

本次会议由来自新西兰奥克兰的 Ciro 平台高级首席工程师 Dashit Pandya 主讲，他同时也是 AWS 无服务器社区建设者。会议聚焦于云环境中的事件响应挑战，特别是在复杂分布式系统中如何应对深夜突发故障。演讲者提出了一个核心观点：在事件响应中，最大的挑战不是缺少数据，而是缺少上下文（Context）。为此，他创造了一个新术语 MTTC（Mean Time To Context，平均上下文获取时间），认为这比传统的 MTTR（Mean Time To Resolution，平均解决时间）更为重要。

演讲者通过一个真实场景展示了"上下文瘫痪"（Context Paralysis）问题：当认证服务在午夜宕机时，工程师会收到海量告警，但这些告警彼此孤立，无法形成完整的故障链路图。为解决这一问题，Pandya 提出了基于模型上下文协议（MCP）和 AWS 服务的智能化解决方案。他开发了三个专门的 MCP 服务器，结合 Amazon Bedrock 和 Claude 模型，将混乱的告警数据转化为可操作的智能洞察。这套方案不是要取代人类决策，而是通过 AI 组织信息、人类做出决策的方式，实现"智能放大"而非"指标倍增"。

会议的核心理念是"不要自动化混乱，而要工程化清晰"（Don't automate your chaos, engineer clarity）。演讲者强调，在分布式系统中故障不可避免，关键在于系统如何帮助工程师冷静地理解和响应故障。通过将上下文智能内置到系统中，保持人类在决策环节的主导地位，可以实现更加从容和高效的生产环境事件响应。

场景演示：上下文瘫痪
- 描述典型故障场景：午夜认证服务宕机，Slack 告警泛滥，工程师打开数十个 CloudWatch 标签页
- 展示问题本质：有大量数据但没有清晰度，告警提供细节但不讲述完整的关联故事
- 定义"上下文瘫痪"（Context Paralysis）：数据过载但缺乏共享的故障地图

会议议程
- 四个核心主题：
  1. 上下文瘫痪 - 传统方法为何在复杂事件中制造混乱
  2. 从混乱到平台智能 - 使用 MCP 和 AWS 服务的智能方法
  3. 从混乱到清晰 - MCP 蓝图技术实现
  4. 超越 MTTR - 在云中构建冷静的哲学

问题深度剖析：架构层面的混乱
- 展示电商服务无服务器架构案例
- 故障级联链：Cognito 告警 → API Gateway 4xx/5xx 错误 → Lambda 错误率飙升 → 冷启动增加 → Lambda 重试循环冲击 RDS
- 核心问题：告警是分散的，在战争室压力下，人脑难以重建完整的故障序列
- 人类在压力下会错过明显的模式

解决方案：从混乱到平台智能
- 方案不是另一个仪表板或告警系统
- 引入 MCP（模型上下文协议）服务器作为中间层
- MCP 服务器功能：关联信号、应用 AI 分析（通过 Amazon Bedrock）、呈现可操作的智能洞察

三个专门的 MCP 服务器

1. O-Flow Context（事件流上下文）
   - 功能：分析事件的智能引擎
   - 三个核心工具：关联 O 事件、获取认证健康状态、检测瓶颈
   - 数据源：Amazon CloudWatch、DynamoDB、EventBridge
   - AI 模型：Claude Sonnet 3.5（用于复杂模式识别）
   - 价值：主动响应而非被动响应

2. Service Dependency Mapping（服务依赖映射）
   - 功能：级联故障预防系统
   - 四个关键能力：映射服务依赖、预测级联、建议隔离、创建依赖可视化
   - AI 模型：Claude Haiku 3 v1（用于快速响应生成）
   - 价值：在问题扩散前几分钟内遏制

3. Recovery Coordination（恢复协调）
   - 功能：保持人类在决策环节
   - 能力：建议修复措施、执行低风险操作、风险分类（低/中/高）
   - 中高风险操作需要人类批准
   - AI 模型：Claude Sonnet 3.5（用于修复措施生成、风险评估、恢复流程预测）

技术架构蓝图
- 完全无服务器架构
- 工作流程：
  1. Claude Desktop 向 MCP 服务器发送请求
  2. MCP 服务器调用 AWS 客户端（CloudWatch、DynamoDB、Lambda、EventBridge）
  3. 收集实时数据并发送回 Amazon Bedrock
  4. Bedrock 选择相应的 Claude 模型
  5. 返回基于追踪、指标和日志的分析结果
- 强调：这不是理论方案，而是可立即在 AWS 环境中使用的生产就绪方案

实际演示片段

O-Flow Context 实战
- 场景：Slack 告警泛滥时，请求分析当前事件并提供上下文
- 输出：
  - 当前状态概览
  - 受影响的服务识别（授权器、订单服务、支付服务）
  - 检测到 Lambda 冷启动导致 800 毫秒延迟
  - 预测级联影响到用户和订单服务
- 价值：提供可操作的智能洞察，而非仅仅是指标

Service Dependency 实战
- 立即映射爆炸半径
- 提供清晰的级联路径可视化（使用不同颜色：琥珀色、绿色、红色）
- 显示故障路径：认证 → 用户 → 订单服务
- 估算影响并建议隔离措施（如用户服务的断路器）
- 价值：预测性事件响应，领先于问题而非追赶问题

Recovery Coordination 实战
- 提供具体操作计划和风险评估
- 低风险操作：用户可直接点击执行按钮
- 中高风险操作：需要人类批准（如考虑成本、实例增加等因素）
- 价值：生产就绪的事件智能响应

核心哲学：超越 MTTR
- 范式转变对比：
  - 传统方法：大量指标 → 阈值告警 → Slack 告警泛滥 → 人类压力
  - MCP + Bedrock 方法：实时关联 → Bedrock 分析 → MCP 提供清晰上下文 → 人类在环决策
- 核心理念：不是取代人类，而是赋能人类
- 智能放大通过上下文关联，而非指标倍增

关键教训
- 引用 Werner Vogels 名言："一切都会时不时失败"（Everything fails all the time）
- 问题不在于如何预防故障，而在于系统如何帮助你冷静地理解和响应故障
- 三个关键经验：
  1. 上下文胜过指标
  2. AI 组织，人类决策
  3. 为冷静而设计（AWS Lambda 和 MCP 服务器帮助构建）
- 智能放大通过上下文关联，而非指标倍增

最终思考
- 核心信息："不要自动化你的混乱，要工程化清晰"（Don't automate your chaos, engineer clarity）
- 问题分析：大多数工程师试图自动化现有混乱（更快的告警、更多仪表板、更多指标、更大的值班轮换），这只是在自动化问题本身
- 正确做法：
  - 将清晰度工程化到系统本身
  - 构建理解架构的上下文代理
  - 使用 AI 放大人类判断而非取代它
  - 创建在压力下始终如一工作的冷静界面
- 结论：下一次事件不可避免，但响应不必混乱
