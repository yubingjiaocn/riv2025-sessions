# AWS re:Invent 2025 会议总结：通过 Agentic AI 降低大型机退出风险

## 会议概述

本次 AWS re:Invent 2025 分组会议由 AWS 大型机转型产品管理负责人 Mina Vimbu Sububman 主持,重点探讨了如何利用 Agentic AI 技术通过重构方式降低大型机现代化转型的风险。会议邀请了两位重量级嘉宾分享实际案例：BMW 转型项目副总裁 Frank Uslab 和 Fiserv 技术高级副总裁 Vimmel Braum。

会议强调了当前大型机现代化面临的核心挑战：全球仍有超过 2200 亿行 COBOL 代码在生产环境中运行,支撑着 70% 的关键企业应用。尽管企业清楚现代化带来的成本优化、技术创新和人才招聘等优势,但多年的转型周期、高昂的验证成本以及业务关键应用的风险往往让团队望而却步。

AWS Transform 通过引入 Agentic AI 能力,将传统的多年转型项目压缩至数月时间。该方案结合了确定性流程(如依赖提取、代码转换)和基于 LLM 的智能代理(如业务逻辑理解、文档生成),在一个工作流中流畅协作,同时在关键决策点引入人工判断。特别是 Reforge 功能,能够将功能等效的重构代码进一步优化为更易读、更易维护的现代 Java 代码,显著提升了代码质量和开发者体验。

## 详细时间线与关键要点

### 00:00 - 会议开场与背景介绍
- **主讲人介绍**：Mina Vimbu Sububman(AWS 大型机转型产品管理负责人)介绍会议主题
- **嘉宾阵容**：BMW 的 Frank Uslab 和 Fiserv 的 Vimmel Braum 将分享实际转型经验
- **议程概览**：涵盖大型机现代化挑战、AWS Transform 的 Agentic AI 能力、客户案例和成功模式

### 03:00 - 大型机现代化的核心挑战
- **现状数据**：2025 年底仍有超过 2200 亿行 COBOL 代码在生产环境运行
- **应用关键性**：这些代码支撑着 70% 的关键企业应用
- **敏捷性限制**：运维团队难以快速适应业务需求变化
- **转型障碍**：多年的转型周期和高风险的验证过程成为主要阻碍

### 05:30 - 现代化的收益与风险平衡
- **预期收益**：成本优化(特别是应用扩展时)、访问现代开源技术、人才招聘更容易、快速创新能力
- **风险因素**：需要同时投资转型和日常运维、业务关键应用的验证风险、需要同时理解遗留系统和现代架构的复合型人才

### 08:00 - Agentic AI 改变游戏规则
- **类比说明**：Mina 分享了在机器人和自动化领域的背景,强调人机协作的重要性
- **技术突破**：Agentic AI 能够在一个工作流中流畅结合确定性流程和 LLM 智能代理
- **应用场景**：依赖提取、代码转换(COBOL 到 Java)、业务逻辑理解、自然语言文档生成
- **人工介入**：在高判断力决策点(如业务域划分)引入人工专家

### 11:00 - AI 驱动的转型效率提升
- **理解阶段加速**：大幅减少理解遗留系统和业务逻辑所需的时间
- **专家效能提升**：专家能够在相同时间内处理更多代码和应用
- **迭代优化循环**：通过"理解-转换-测试"循环持续学习和改进
- **风险降低**：向最终验证团队交付更高质量的输出

### 13:30 - AWS Transform 产品愿景
- **混合方法**：为每个步骤选择最合适的技术
- **全流程自动化**：从初始分析规划到重构、测试验证的端到端自动化
- **时间目标**：将多年转型项目压缩至数月
- **功能等效性**：确保现代化应用与遗留系统功能完全等效

### 15:00 - AWS Transform 转型三阶段
- **第一阶段 - 理解**：代码分析(静态视图)、活动分析(动态视图,基于 SMF 记录)、数据分析(依赖关系和数据字典)
- **第二阶段 - 转换**：支持多种模式(重新平台化、重构、重新构想)
- **第三阶段 - 测试与验证**：自动测试计划生成、测试数据捕获脚本、自动化测试执行

### 18:00 - 活动分析代理的价值
- **动态洞察**：分析 SMF 记录了解应用运行频率、CPU 利用率等
- **优先级决策**：帮助团队确定工作负载优先级
- **架构建议**：例如,中频短生命周期应用适合事件驱动架构

### 20:00 - Reforge 功能详解
- **核心理念**：如同重新锻造金属使其更强,Reforge 优化重构后的代码
- **多代理流程**：识别并消除 COBOL 语义和结构残留
- **代码优化**：重构为更原生的 Java 类、改进变量命名以反映业务上下文
- **开发者体验**：Java 开发者能够直观理解代码和业务逻辑的关联

### 22:30 - 功能测试集成
- **质量保证**：确保 Reforge 过程不引入功能变更
- **等效性验证**：保障达到功能等效性证明点
- **生产就绪**：加速应用投入生产的路径

### 24:00 - 测试加速能力
- **测试计划生成**：早期阶段自动生成测试计划,帮助评估成本和工作量
- **测试数据捕获**：生成 JCL 脚本在大型机上捕获特定测试数据
- **自动化测试脚本**：在大型机和现代化环境中执行自动化测试

### 27:00 - Fiserv 案例分享开始
- **公司介绍**：Fiserv 是美国领先的金融支付处理机构,处理美国三分之一的经济交易
- **规模数据**：支持 600 万商户、16 亿张发行卡、1 万家银行机构
- **峰值处理**：每秒处理 25,000 笔交易
- **转型复杂度**：在如此规模下进行大型机到云的转型极具挑战性

### 30:00 - 支付处理的日常影响
- **生活场景**：从早晨购物、通勤支付、订阅服务、午餐外卖到晚餐消费
- **交易量**：每天发生 3.5-4 亿次类似交易
- **峰值季节**：黑色星期五、感恩节前、网络星期一等是支付行业的高峰期

### 33:00 - Fiserv 的云转型历程
- **战略转向**：几年前决定与 AWS 合作,采用云优先策略
- **多种模式**：新应用全部构建在云上,部分应用迁移,部分采用 lift-and-shift
- **当前状态**：大量应用已在 AWS 运行,为大规模转型奠定基础

### 35:30 - HDC 应用详解
- **应用名称**：Host Data Capture(主机数据捕获)
- **核心功能**：支付注入和预结算处理,处理高容量数据
- **运行模式**：7x24 小时不间断运行
- **业务场景**：在线购物订单发货前,商户需要先收到款项,系统需匹配原始购买和商户请求进行对账

### 38:00 - HDC 应用的惊人规模
- **日处理金额**：每天处理超过 100 亿美元(B = Billion)
- **日交易量**：每天 1.5-2 亿笔交易
- **可用性要求**：7x24 小时运行,不能有任何延迟

### 40:00 - 现有系统的优势与挑战
- **平台成熟度**：在大型机上运行 20-30 年,数百万行 COBOL 代码
- **非功能性需求**：响应时间、容量、可用性表现优异
- **转型标准**：最低要求是保持现有功能不变,不能破坏任何功能
- **创新需求**：不转型将错过 AI 和 Agentic AI 带来的创新机会

### 42:30 - Fiserv 的转型策略
- **初始方案**：使用 AWS 的 Blue Age(现称 Refactor)将 COBOL 转换为类 COBOL 风格的 Java
- **保留专家**：保持 COBOL 专家的参与和兴趣,利用他们的知识
- **挑战发现**：转换后仍需额外 12 个月将代码现代化为基于 API 的事件驱动架构

### 44:00 - 引入 Reforge 的转折点
- **AWS 提议**：Mina 团队在 Reforge GA 前邀请 Fiserv 试用
- **决策**：决定结合 Refactor 和 Reforge,直接从 COBOL 风格 Java 转换为现代 Java
- **试点验证**：选取一个应用模块进行试点,验证自动化测试和功能等效性

### 46:00 - 转型成果与预期
- **时间节省**：原计划 30 个月,现预计 18 个月完成,已进行 3-4 个月
- **进度状态**：已完成约四分之一,对结果非常乐观
- **可维护性提升**：Reforge 将可维护性提升 40%,这是选择 Reforge 而非仅使用 Refactor 的关键原因

### 48:00 - 技术栈转换细节
- **遗留技术**：COBOL、VSAM、DB2、平面文件
- **目标技术**：Spring Boot、Apache Tomcat、Groovy(用于 JCL)
- **核心原则**：业务逻辑必须保留、功能等效性保证、处理能力不降低(保持 100 亿美元日处理能力,目标 200 亿)

### 50:00 - 用户体验保持
- **操作用户**：有操作人员使用应用输出进行退款和数据对账
- **界面保持**：不改变现有屏幕和标准操作流程
- **培训成本**：避免因应用现代化而重新培训整个组织

### 51:30 - 代码对比展示
- **左侧代码**：Refactor 生成的代码,类似用 Java 编写的 COBOL,保留 COBOL 结构
- **右侧代码**：Reforge 优化后的代码,现代 Java 开发者可以直接理解
- **关键改进**：变量名有意义、使用 Java 惯用法、数据类结构清晰

### 53:00 - 人才挑战的解决
- **招聘困境**：没有大学毕业生愿意从事大型机应用开发
- **Reforge 优势**：右侧的现代 Java 代码是大学教授的内容,新员工可以直接上手
- **长期价值**：解决了人才可持续性问题

### 54:30 - Fiserv 的转型预期
- **处理能力**：继续保持 100 亿美元日处理,目标提升至 200 亿
- **多区域部署**：利用云的多区域能力和 Lambda 按需扩展
- **零中断**：确保转型过程零业务中断
- **现代技术栈**：利用所有 AWS 服务和工具
- **AI 创新**：在欺诈和风险领域应用 AI(非交易处理本身)
- **开发者体验**：使用现代 IDE 而非大型机上的 VX Code

### 57:00 - 时间和成本节省总结
- **时间节省**：Reforge 节省约 12 个月
- **可维护性**：提升 40%,这是巨大的改进
- **现实预期**：转型不会完美无缺,会有挑战和经验教训,将持续与合作伙伴分享

### 58:30 - 形象比喻
- **换引擎比喻**：在汽车行驶过程中更换引擎,同时确保乘客在结账时不会洒咖啡
- **过渡到 BMW 案例**：引出下一位嘉宾 Frank 分享汽车制造业的转型故事

### 59:00 - BMW 案例分享开始
- **Frank 介绍**：BMW 集团 IT 云和现代化转型负责人
- **互动环节**：询问现场有多少人完成了 10,000 步(re:Invent 会场很大),用幽默方式类比转型的挑战

### 01:00:00 - BMW 产品创新背景

- **智能体验**：全景 iDrive 和全新电子架构,创造智能直观的数字体验

### 01:02:00 - BMW i3 与产品线扩展

- **创新展示**：展示将塑造整个 BMW 产品线的创新
- **扩展计划**：到 2027 年扩展至 40 款车型
- **新时代开启**：标志着 BMW 新时代的开始

### 01:03:00 - 产品变革驱动 IT 变革
