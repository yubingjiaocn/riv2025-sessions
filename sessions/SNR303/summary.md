# AWS re:Invent 2025 会议总结

## 会议概述

本次会议由 Steven Brozvich 主讲,他是 AWS 高管驻场团队的成员,拥有超过 26 年的亚马逊工作经验。会议的核心主题是**企业文化在技术变革中的重要性**,特别是在 AI 等新技术快速发展的背景下,如何保持和传承组织文化。

Steven 从自己在亚马逊的职业历程出发,分享了他从技术岗位转向人力资源领域的经历。他深入探讨了文化的三个核心组成部分:心智模型(Mental Models)、行为模式(Behaviors)和组织产物(Artifacts)。通过古希腊安提基特拉机械装置的考古学案例,他生动地阐释了当组织产物与其背后的心智模型脱节时会发生什么——人们会根据自己的理解重新诠释这些产物,导致文化逐渐侵蚀。

会议重点介绍了亚马逊的领导力原则,特别是"客户至上"(Customer Obsession)和"主人翁精神"(Ownership)这两个核心原则。Steven 通过具体案例展示了这些心智模型如何驱动员工行为,包括印度海得拉巴的工程师团队为确保圣诞节前交付包裹而亲自驾车送货的故事。他还详细介绍了亚马逊的"逆向工作法"(Working Backwards)流程,强调在开发任何产品前先撰写新闻稿的重要性。会议最后指出,如果不持续强化心智模型,组织文化会随时间流逝而退化,人们会根据自己的经验误解组织流程的初衷。

## 详细时间线

### 开场与个人背景介绍
[00:00 - 05:30]
- Steven Brozvich 自我介绍,说明自己是 AWS 高管驻场团队成员
- 分享 1999 年 8 月加入亚马逊的经历,当时亚马逊只有 4 个产品类别(书籍、音乐、VHS 录像带等)
- 回顾亚马逊技术演进:从单体架构到微服务、从瀑布开发到敏捷、从集中式资源到分布式自治团队
- 13 年前转入人力资源领域,专注于文化和人的因素

### 文化的定义与重要性
[05:30 - 10:00]
- 讨论亚马逊快速增长期(从 30,000 到 90,000+ 员工)对文化保持的担忧
- 提出核心问题:如何在技术变革(如 Agentic AI)中保持组织文化
- 定义文化的三个组成部分:
  - 心智模型(Mental Models):理解世界的方式
  - 行为模式(Behaviors):基于心智模型采取的行动
  - 组织产物(Artifacts):人类活动留下的可见痕迹

### 心智模型案例:交通信号灯
[10:00 - 12:00]
- 用西雅图和圣保罗的行人过马路习惯对比说明不同心智模型
- 西雅图:规则遵守型(红灯停,即使没车也等)
- 圣保罗:实用主义型(没车就过)
- 说明两种心智模型都合理,但会导致不同行为

### 考古学类比:安提基特拉机械装置
[12:00 - 18:00]
- 介绍 1901-1902 年在地中海发现的古代机械装置
- 该装置包含 33 个精密齿轮,可预测日食和月食,年代约为公元前 200 年
- 比同等复杂度的技术早了 1500 年
- 考古学的三个核心问题:
  - 这是什么?(What is it?)
  - 它如何到这里?(How did it get here?)
  - 为什么制造它?(Why was it made?)
- 这三个问题对应文化的三个层面:产物、行为、心智模型

### 心智模型驱动组织文化
[18:00 - 22:00]
- 强调心智模型对组织文化的影响力
- 提出问题:你的组织有哪些驱动性的心智模型?
- 介绍亚马逊的独特文化现象:会议开始时的静默阅读时间(5-20 分钟)
- 新员工的文化冲突体验

### 亚马逊的核心心智模型
[22:00 - 25:00]
- 亚马逊的使命:成为地球上最以客户为中心的公司
- 介绍亚马逊的 16 条领导力原则
- 强调通过明确表达心智模型获得的力量

### 领导力原则详解:主人翁精神
[25:00 - 30:00]
- "Ownership"(主人翁精神)原则详解:
  - 领导者是主人翁,长远思考
  - 不为短期利益牺牲长期价值
  - 为整个公司而非仅自己团队行动
  - 永远不说"这不是我的工作"
- 强调亚马逊 26 年来从未基于季度结算做决策
- 挑战人类的领地意识,要求跨团队思考
- 客户不在乎是谁的错,只在乎问题是否解决

### 领导力原则详解:客户至上
[30:00 - 35:00]
- "Customer Obsession"(客户至上)原则详解:
  - 从客户出发,逆向工作
  - 努力赢得并保持客户信任
  - 关注竞争对手,但痴迷于客户
- 强调不要从解决方案开始,要从客户问题开始
- 信任难以建立,容易破坏
- 竞争对手本身也落后于客户真实需求

### 客户至上的实践案例
[35:00 - 42:00]
- 海得拉巴软件团队在肯塔基州希伯伦履行中心的故事
- 发现一个包裹错过了圣诞节前最后一班卡车
- 团队彻夜寻找解决方案,考虑包机但因节俭原则放弃
- 最终在返回印度途中在纽约 JFK 转机时,租车驾驶 100 英里到康涅狄格州亲手交付包裹
- 强调这种客户至上行为对长期客户信任的影响

### 心智模型的侵蚀问题
[42:00 - 48:00]
- 核心观点:未经强化的心智模型会随时间侵蚀
- 解释侵蚀机制:
  - 时间、地理位置、组织层级的距离增加
  - 产物持续存在,但背后的心智模型和行为变得模糊
  - 人们变成"业余考古学家",根据自己的心智模型重新诠释产物
- 三种常见反应模式:
  - 拒绝(Reject):认为这很愚蠢,用自己的方式
  - 盲从(Comply):不知道为什么但照做,导致质量下降
  - 其他误解

### 组织产物的普遍性
[48:00 - 50:00]
- 所有组织都会产生产物
- 有些永久(如公司园区建筑)
- 有些短暂(如会议照片)
- 每个新员工都会遇到这些产物并进行诠释

### 逆向工作法介绍
[50:00 - 58:00]
- 介绍亚马逊的"Working Backwards"(逆向工作法)流程
- 核心:在写任何代码前先写新闻稿
- 新闻稿结构:
  - 第一段:客户是谁?问题或机会是什么?
  - 接下来几段:产品如何改善客户生活(不是技术细节)
  - 后续段落:产品如何融入亚马逊整体业务
- 限制:单页,有字体和边距要求
- 如果无法在一页内清晰表达,说明思考不够深入
- FAQ 部分:收集客户和内部利益相关者的问题
- 用户体验部分:视觉呈现或 API 示例

### 逆向工作法的应用
[58:00 - 62:00]
- AWS 的每个产品都始于新闻稿
- AWS 作为业务本身的新闻稿经过 47 次迭代
- 这是低成本测试想法的方式
- 可以在第 7 次迭代时就放弃不好的想法
- 这个流程已使用超过 10 年
- 与客户和合作伙伴共同使用

### 逆向工作法的误用案例
[62:00 - 结束]
- 在高管培训中发现的问题案例
- 一位总监级别高管(由 Jeff Bezos 亲自选拔)自豪地说自己三年来从未写过新闻稿
- 他的观点:这是官僚主义,阻碍工作
- 他与副总裁关系良好,直接要资源就能获得
- 这是真实的个人经历,但基于对流程的误解
- 说明当组织不正确使用机制时,即使高层也会误解其目的
