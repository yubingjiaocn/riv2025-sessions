# AWS re:Invent 2025 会议总结：使用 AI Agent 实现业务流程现代化

## 会议概述

本次会议由 AWS 高级合作伙伴解决方案架构师 Pnesha 和 Cedric Bodon，以及 AWS 大使 Gita 共同主持，重点探讨了如何利用 AI Agent 和生成式 AI 技术对传统业务流程进行现代化改造。

会议通过一个生动的寓言故事开场——讲述了森林邮局如何通过引入不同动物的专长来优化邮件配送流程，从而将配送时间从数周缩短到数小时。这个故事完美地比喻了业务流程服务（BPS）行业正在经历的根本性转变。演讲者指出，BPS 市场预计将从 2024 年的 2140 亿美元增长到 2028 年的 2890 亿美元，而生成式 AI 和 Agentic AI 正在从根本上改变每个组织的运作方式。

会议强调，AWS 合作伙伴在这一转型中扮演着关键角色——不仅是技术实施者，更是值得信赖的顾问和战略思考者。合作伙伴需要帮助客户识别类似"老邮局"的传统流程，并将其现代化为自主智能的解决方案。演讲者通过一个高端食品杂货配送商的实际案例，详细展示了如何将一个涉及 12 个步骤、多次人工交互的繁琐流程，简化为 8 个步骤的智能化工作流，大幅提升了客户体验和运营效率。

### 用例分析：高端食品杂货配送商 (08:00 - 12:30)
- **08:00** - Cedric 介绍用例背景：配送商需要改进流程以满足客户即时需求
- **08:30** - 展示传统工作流程的 12 个步骤：
  - 客户通过 WhatsApp 发送请求
  - 客户支持记录并转发给销售团队
  - 销售团队检查产品可用性和折扣条款
  - 向客户发送提案
  - 客户接受后下单
  - 订单转回支持团队
  - 路由至物流安排配送
  - 客户需主动请求状态更新
- **10:00** - 指出传统流程的三大挑战：客户体验差、效率低、客户留存风险高
- **10:45** - 展示引入 Agent 后的优化流程（8 个步骤）：
  - Agent 直接接收客户咨询
  - 提供成本和产品推荐
  - 自动生成提案草稿
  - 自动下单
  - 与物流协调安排配送
  - 自动向客户提供更新
- **12:00** - 强调人工干预仅在异常情况下介入（如客户拒绝提案或配送时间不合适）

### Agentic 转型旅程框架 (12:30 - 15:00)
- **12:30** - 介绍典型的 Agentic 旅程四个阶段：
  1. **评估（Assess）**：从业务和技术角度评估工作流程
  2. **启用（Enable）**：准备技术栈以集成 Agent
  3. **构建（Build）**：设计和运行 Agent
  4. **扩展（Scale）**：实现生产级规模化
- **14:00** - 详细说明技术评估的重要性，以订单应用为例（SOAP 应用需要现代化）

### 技术实现：使用 Amazon Q Developer (15:00 - 25:00)
- **15:00** - Pnesha 开始演示如何使用 Amazon Q Developer 进行代码学习
- **15:30** - 展示传统 SOAP 应用的架构和流程
- **16:00** - 演示 Q Developer 如何分析代码库：
  - 理解业务逻辑
  - 分析表示层
  - 识别系统流程
  - 理解数据库架构
  - 进行代码审查和安全分析
- **18:00** - 生成高层设计方案：从 SOAP 迁移到 REST API，从 XML 到 JSON
- **19:30** - 创建需求和设计规范文档
- **20:45** - 生成详细设计文档：基础设施、数据库、解决方案架构、错误处理
- **22:00** - 生成实施计划：将设计文档分解为独立的小任务
- **23:30** - 开始基础设施和数据库迁移实施
- **24:30** - 强调 AI 生成内容需要人工审核（人机协同）

### Agent 框架：Multi-Agent Orchestrator (25:00 - 30:00)
- **25:00** - 介绍 Multi-Agent Orchestrator（AWS 开源 SDK）

- **29:00** - 列举 20+ 个开箱即用的生产就绪工具：

  - 图像读取器
  - 各种客户端

### 部署平台：Amazon Bedrock Agents (30:00 - 33:00)
- **30:00** - 介绍 Amazon Bedrock Agents 作为扩展平台
- **30:30** - 支持多种框架：Multi-Agent Orchestrator、LangGraph、CrewAI、自定义框架
- **31:00** - 展示架构图：如何在 Bedrock Agents 上部署 Multi-Agent Orchestrator
- **31:45** - 关键能力介绍：
  - 长时间运行支持（最长 8 小时）
  - 每个用户交互独立的虚拟机（CPU 和内存隔离）
  - 可观测性
  - 状态和上下文管理（通过 Bedrock Memory）
  - Agent Gateway 用于工具交互
  - IAM 用于身份验证和授权

### 设计考虑因素 1：编排模式 (33:00 - 38:00)
- **33:00** - 介绍三种编排模式
- **33:30** - Agent as Tool 模式：
  - 单个编排器 Agent 处理用户交互
  - 使用多个工具做出决策
  - 示例：临床 Agent 作为领域专家
  - 适用场景：直接的用例
- **35:00** - Swarm 模式：
  - 类似项目经理利用专业资源
  - 多个 Agent 独立工作解决一个问题
  - 适用场景：复杂工作流、创意问题解决、研发工作流、市场研究分析
- **36:30** - Graph 模式：
  - 确定性工作流
  - 订单处理示例：检查订单 → 产品库存 → 推荐 → 计算价格 → 返回信息
  - 适用场景：有多阶段依赖关系和分支决策的场景

### 设计考虑因素 2：推理和智能能力 (38:00 - 43:00)
- **38:00** - 强调 LLM 是 Agent 的"大脑"
- **38:30** - 提出核心问题：应该为 Agent 选择哪个 LLM？
- **39:00** - 介绍选择框架：任务复杂度 vs 延迟/成本
- **39:45** - 订单管理系统的模型选择示例：
  - **客户 Agent**（简单任务）：Amazon Micro Model（高吞吐量）
  - **销售 Agent**（多模态数据处理）：Amazon Nova Lite 或 Pro
  - **仓库 Agent**（复杂推理任务）：Anthropic Claude 或 Amazon Nova Pro
- **41:30** - 行业特定场景的领域知识模型：
  - 医疗行业：John Snow Labs Medical LLM
  - 研发场景：BioMistral
- **42:15** - 多模态嵌入模型用于 RAG 应用
- **42:45** - 强调模型评估的重要性：成本、质量、性能

### 设计考虑因素 3：上下文和状态管理 (43:00 - 47:00)
- **43:00** - 指出 AI Agent 和模型本质上是无状态的
- **43:30** - 举例说明记忆的重要性：避免用户重复输入信息
- **44:00** - 互动：询问观众对上下文管理的了解
- **44:30** - 解释上下文管理的信息来源：
  - 系统提示
  - 用户输入
  - RAG 应用提供的文档
  - 工具提取的外部信息
- **45:15** - 上下文管理的目标：
  - 最小化信息过载
  - 确保上下文相关性
  - 在正确的时间向正确的 Agent 提供正确的信息
- **46:00** - 介绍 Amazon Bedrock Agent Memory：
  - **短期记忆**：持续 7-365 天的用户交互
  - **长期记忆**：多年的持久化用户交互管理
  - 类比：长期记忆如硬盘，短期记忆如 RAM

### 设计考虑因素 4：可观测性 (47:00 - 50:00)
- **47:00** - 强调可观测性的重要性：理解工作流程和可追溯性
- **47:30** - 可观测性的用途：
  - 测试和调试
  - 提高效率和性能
  - 完整的追踪能力
- **48:00** - 介绍 Amazon Core Observability 托管服务：
  - 提供追踪、指标和日志
  - 兼容 OpenTelemetry
  - 集成 CloudWatch
  - 提供 360 度的 Agent 工作流程和性能视图
  - 监控 Token 使用和成本
- **49:00** - 提及合作伙伴解决方案：Arize AI 和 Phoenix
  - 提供类似能力
  - 可视化和模型评估功能
  - 加速实施速度

### 工具构建：Model Context Protocol (MCP) (50:00 - 54:00)
- **50:00** - 强调统一数据访问的重要性
- **50:30** - 互动：询问观众是否使用 MCP
- **51:00** - 介绍 MCP 协议：
  - 基于服务器和客户端架构
  - Agent/AI 解决方案通过客户端交互
  - 统一集成多个数据源
  - 避免为每个数据源构建单独的集成
- **52:00** - 两种 MCP 实现模式：
  - **一对一模式**：安全/合规要求，Agent 只能访问特定信息
  - **一对多模式**：集中管理系统
- **53:00** - 指出 MCP 服务器的构建、部署和维护挑战
