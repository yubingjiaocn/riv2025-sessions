# AWS re:Invent 2025 COP202 会议总结：AWS 优化工具新功能

## 会议概述

本次会议由 AWS 优化工具产品管理团队负责人 Rick Oaks 主持，重点介绍了 AWS 在成本优化领域的最新功能和工具更新。会议涵盖了 Compute Optimizer、Cost Optimization Hub、Savings Plan 等多个优化工具的重要更新。特别值得关注的是，AWS 推出了 EBS 自动化优化功能，允许用户自动清理未附加的 EBS 卷并升级卷类型。此外，还发布了成本效率指标（Cost Efficiency Metric）这一全新的 FinOps KPI，帮助组织更好地衡量和平衡资源优化与承诺购买之间的关系。

会议还邀请了 Delta Airlines 的中央 FinOps 团队负责人 Bonnie Fernall 分享实战经验。她详细介绍了 Delta 如何通过建立机会仪表板、创建自动化流程以及实施成本优化文化转型，最终实现了 27% 的月度成本节省。Delta 的案例展示了从手动优化到自动化执行的完整演进路径，以及如何通过有效的沟通和团队协作克服组织层面的挑战。会议最后部分还预告了 Savings Plan 购买体验的改进，特别是针对多数据库环境的优化建议功能。

## 详细时间线

### 开场与背景介绍
[00:00 - 02:30]
- Rick Oaks 介绍自己作为 AWS 优化工具产品管理团队负责人的角色
- 团队成员介绍：Bonnie Fernall（Delta Airlines 中央 FinOps 团队负责人）和 Mary Nachimov（Savings Plan 购买体验负责人）
- 现场调查：大部分参会者来自 FinOps 组织或工程团队，许多人已使用过 Compute Optimizer 和 Savings Plan 工具

### EBS 自动化优化功能发布
[02:30 - 10:00]
- 宣布 EBS 优化自动化功能，于感恩节前一周（约 11 月下旬）正式发布
- 功能包括：自动快照并删除未附加的 EBS 卷；自动升级卷类型（GP2 到 GP3 可节省 20% 成本，IO1 到 IO2 提升弹性）
- 用户可以立即执行优化或创建自动化策略，支持按账户、标签、区域等条件设置规则
- 支持设置维护窗口，每次最多实施 100 个建议
- 提供安全保障：自动回滚功能（最长一年）、实施前验证检查、失败自动停止

### EBS 自动化功能演示
[10:00 - 15:30]
- 现场演示 Compute Optimizer 控制台中的自动化功能
- 展示如何多选建议并快速应用（仅需 4-5 次点击即可节省成本）
- 演示创建自动化规则的流程：选择组织或账户级别、选择建议类型、添加标签条件（支持 AND/OR 逻辑）
- 展示如何设置执行计划、添加新标签以及查看历史执行事件和节省金额

### NAT Gateway 空闲检测功能
[15:30 - 17:00]
- 推出 NAT Gateway 空闲检测建议
- 功能特点：分析 30 天历史使用数据，检查路由表关联状态，识别真正未使用的 NAT Gateway（非故障转移备份）
- 帮助用户与网络团队协作，清理遗留的未使用 NAT Gateway

### 成本效率指标（Cost Efficiency Metric）
[17:00 - 22:00]
- 发布全新的成本效率 KPI，约一周半前上线
- 解决 FinOps 领域缺乏统一优化成功指标的问题
- 计算方法：成本效率 = 1 - (总潜在节省金额 / 可优化总支出)
- 自动生成 90 天历史数据，支持按账户、区域查看
- 平衡资源优化与承诺购买，避免单一指标的局限性
- 每个百分点对应实际发票金额，便于向领导层解释价值

### Delta Airlines 案例分享 - 背景与挑战
[22:00 - 28:00]
- Bonnie Fernall 介绍 Delta 在云迁移三年后建立 FinOps 实践的背景
- 面临的核心挑战（5W 分析）：
  - Where：开发者和管理者需要不同的访问入口
  - Who：1200 个账户生态系统中需要明确责任归属
  - What：许多开发者对云优化缺乏了解
  - When：优化工作难以获得优先级
  - Why：尽管有意愿但工作未能执行
- 这些挑战与 2025 FinOps Foundation 报告中的发现一致

### Delta 的解决方案 - 机会仪表板
[28:00 - 33:00]
- 创建集中式机会仪表板，整合所有建议并添加组织上下文（领导者、应用名称等）
- 两级报告结构：
  - 摘要报告：跨团队和机会类型查看最大节省潜力
  - 详细报告：资源级别信息，包括账户、节省估算、使用指标
- 为每个优化机会创建 5 分钟快速阅读的 Runbook，降低开发者学习成本

### Delta 的标签策略
[33:00 - 35:00]
- 采用轻量级标签策略：使用 10 字符代码作为主键，代表业务能力
- 优势：业务能力通常不变，减少维护负担；集中式架构仓库管理上下文信息
- 高质量数据且维护成本低

### 决策流程与验证机制
[35:00 - 40:00]
- 早期教训：并非所有建议都能转化为节省（因使用模式变化等原因）
- 在仪表板中加入团队决策点：实施、拒绝或需要测试
- 要求提供拒绝理由，便于后续对话
- 添加实施时间线和延迟原因（超过 30 天）
- 建立三类报告：未执行机会、已拒绝机会验证、已实施机会节省跟踪

### Delta 的自动化实践 - EBS 卷清理
[40:00 - 44:00]
- 发现团队与领导层在实施时间线上存在根本性错位
- 转向自动化解决方案
- 首先自动化未附加 EBS 卷清理：
  - 零自主权顾虑
  - 大量卷可产生可衡量的节省
  - 复制本地策略，行为非新增
- 如果当时有 AWS 自动化功能会直接使用

### Delta 的自动化实践 - 卷类型转换
[44:00 - 47:00]
- 第二步：卷类型转换（主要是 IO1 到 GP3）
- 需要更多教育工作：使用指标证明 GP3 满足系统需求；说明 GP3 是新技术而非设计决策
- 采用基于规则的方法：仅在团队选择加入的账户和资源上运行
- 允许通过管道更新的团队保持环境整洁
- 自动化后更新漂移的 CloudFormation 堆栈，避免创建技术债务
- 发送邮件通知团队完成的工作

### Delta 的自动化实践 - EC2 右调整
[47:00 - 52:00]
- 最大挑战：EC2 右调整（之前其他团队使用不同工具造成过破坏）
- 选择 Compute Optimizer 的原因：建立的信任基础；可自定义建议
- 初始策略：设置最高 CPU 和内存利用率阈值；排除不兼容实例类型
- 第一轮仅获得约 3 个批准
- 改进措施：适应组织的设计和弹性标准；识别负载均衡需要相同大小的实例；为不同环境/弹性层创建不同模式；允许有特殊供应商要求的团队选择退出
- 最终从选择加入转变为选择退出模式，实现更大节省

### 沟通模式与成本优化文化
[52:00 - 56:00]
- 强调沟通的关键性，建立固定模式：告知（通常多次）→ 开放反馈 → 透明处理反馈
- 实施成本优化冠军计划：4 人小团队通过每个成本中心指定的冠军扩大影响力
- 双周会议：教育新机会、流程变更；冠军开始向社区分享他们发现的优化
- 建立信任和领域知识的结合

### Delta 的成果与未来计划
[56:00 - 58:30]
- 实现 27% 的月度成本节省
- 文化转型：团队开始在设计阶段左移优化，FinOps 成为工作方式而非额外工作
- 合作伙伴关系建立信任，团队更愿意接受新优化和自动化
- 为领导层提供数据驱动决策的基础
- 未来计划：一直保持适度的承诺购买基线，现在优化状态更好，2026 年将重点关注 Savings Plan 作为成本控制的重要杠杆
