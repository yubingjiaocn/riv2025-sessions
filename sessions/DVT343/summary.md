# AWS re:Invent 2025 - Kiro Powers 会议总结

## 会议概览

本次会议由 Arun（Kiro 高级解决方案架构师经理）和 Dan Kuna（Kiro 专家解决方案架构师）主讲，重点介绍了 AWS Kiro 的最新功能 —— Kiro Powers。Kiro 是一个面向开发到生产全流程的智能代理 IDE，通过规范化的方式帮助开发者利用 AI 代理完成复杂的开发任务。会议首先回顾了 AI 如何改变软件开发的历程，从自动补全到代理辅助，再到 2025 年能够端到端完成开发任务的智能代理时代。

Kiro Powers 是昨天刚刚发布的新功能，旨在解决 AI 代理在专业化开发任务中遇到的挑战。传统的 AI 代理虽然能够处理通用开发任务，但在面对 UI 开发、API 开发、数据库开发等专业领域时，往往缺乏针对性的最佳实践指导，导致上下文过载和效率低下。Kiro Powers 通过可共享的最佳实践包（包含 MCP 服务器、引导文件和自动化钩子）来扩展 Kiro 代理的能力，并且这些 Powers 是由 ISV 合作伙伴精心策划的。发布时已与 Figma、Supabase、Stripe、Neon、Datadog、Dynatrace、Netlify、Postman 等合作伙伴推出了相应的 Powers。

会议通过实时演示展示了如何使用 Neon Power 快速为前端应用添加后端数据库，以及如何使用 Netlify Power 部署应用。整个过程无需离开 IDE，代理能够按需激活相应的 Power，动态加载所需的上下文和工具，避免了上下文窗口的浪费，大大提升了开发效率。

## 详细时间线

00:00 - 开场介绍
- Arun 和 Dan 介绍自己，Arun 是 Kiro 高级解决方案架构师经理，Dan 是 Kiro 专家解决方案架构师
- 会议议程：什么是 Kiro、现有挑战、Kiro Powers 介绍、实时演示

02:30 - AI 如何改变软件开发
- 回顾 AI 发展历程：自动补全时代 → 2024 年代理辅助时代（处理模块、函数、SDK）→ 2025 年端到端任务完成时代（全栈应用、迁移、现代化）

04:15 - Kiro 产品介绍
- Kiro 是面向开发到生产的智能代理 IDE
- 核心特性：通过 Spec 模式创建规范（使用 markdown 格式和 EASY 语法），将需求转化为设计产物，再分解为具体任务
- 解决了代理陷入循环的问题，通过结构化的软件开发生命周期方法指导代理

06:45 - Kiro 现有功能回顾
- Agent Chat：主要交互模式，支持广泛提示和编码
- Spec 模式：通过自然语言创建规范
- MCP 配置：支持本地和远程 MCP 集成，支持 OAuth
- 引导规则（Steering Rules）：通过 markdown 文件指导代理特定行为（如指定测试框架）
- Agent Hooks：基于 IDE 事件的自动化（如文件保存时自动更新 README）

09:20 - 现有工具生态系统
- MCP 服务器：提供跨客户端的工具通信标准
- Skills：类似 Claude 的指令和资源集
- 动态工具处理：按需发现和加载工具
- IDE 规则：类似 Cursor 的自定义指令

10:30 - 专业化开发任务的挑战
- 通用任务（代码结构、审查、测试、文档）vs 专业任务（UI 开发、API 开发、数据库开发、AI 代理开发）
- 当前挑战：AI 代理难以专业化、文档过载、多个 MCP 服务器和工具导致上下文混乱、需要大量手写引导文件、缺乏开发工具的规范性最佳实践、领域专家知识未编码化

12:45 - Kiro Powers 解决方案
- 问题陈述：AI 代理需要动态访问来自用例专家的定制化上下文，但会导致上下文膨胀
- Kiro Powers：通过可共享的最佳实践扩展 Kiro 代理能力
- Power 包含：MCP 服务器、引导文件、操作钩子（可包含一个或全部）

14:20 - Kiro Powers 的优势
- 覆盖开发到部署的用例（全栈开发、后端开发、API 开发、数据库开发、可观测性等）
- 为代理提供针对性上下文以快速决策
- 由 ISV 策划的最佳实践（如 Stripe 支付集成由 Stripe 自己创建）

15:30 - Power 文件结构
- 展示 Supabase Power 的 markdown 文件示例
- 包含：名称、显示名称、描述、关键词
- 按需激活：当用户提示包含相关关键词（如"数据库"、"PostgreSQL"）时自动激活

17:00 - 合作伙伴生态
- 发布合作伙伴：Figma、Supabase、Stripe、Neon、Datadog、Dynatrace、Netlify、Strands、Postman
- 用户也可以创建自定义 Power 并在组织内共享

18:15 - 实时演示开始
- Dan 展示 Kiro 界面的新 Powers 面板
- 两个主要部分：已安装的 Powers 和合作伙伴策划的 Powers

19:30 - Neon Power 演示场景
- 场景：后端开发者需要为前端 Next.js 应用添加数据库
- 安装 Neon Power（几秒钟完成）
- Neon Power 包含：MCP 服务器 + 最佳实践引导文件

21:00 - 添加后端数据库
- 简单提示："为我的前端应用添加后端数据库"
- 代理识别并激活 Neon Power
- 按需加载上下文，避免上下文膨胀
- 通过 OAuth 自动授权 Neon 账户（首次使用需要授权弹窗）

23:45 - 数据库初始化
- 代理检查现有 Neon 项目
- 创建后端文件、编辑前端文件、设置集成
- 提示添加数据库 URL 到环境变量文件

26:30 - 动态 MCP 调用
- 代理进行多次 MCP 调用获取项目信息和连接字符串
- 修改 .env 文件中的密钥变量
- 检查是否需要额外设置

28:15 - 创建数据库表
- 代理使用 MCP 工具运行 SQL 命令创建表
- 整个过程无需离开 IDE，无需手动登录 Neon 控制台

29:45 - 上下文使用指示器
- Arun 指出底部的新功能：当前会话的上下文使用指示器
- 使用 Powers 可以更有效地管理上下文窗口

31:00 - 运行本地开发服务器
- 提示运行本地开发服务器
- 遇到依赖缺失错误，代理自动安装依赖
- 成功启动本地服务器

33:20 - 添加示例数据
- 前端显示无数据
- 提示："向后端添加示例数据"
- 代理通过 MCP 工具运行 SQL 插入命令
- 自动推断需要产品数据（基于前端应用的元数据）
- 插入 6 个产品，刷新页面后显示数据

36:45 - Neon 分支功能演示
- Neon 特性：可以为后端创建分支
- 提示："我想修改数据库，请创建一个分支"
- 代理检索项目元数据并创建分支
- 可以在分支上修改架构而不影响主表，完成后合并

39:00 - 部署应用（Netlify Power）
- Arun 提问：如何部署应用？
- 安装 Netlify Power
- 简单提示："请部署我的应用"
- 代理激活 Netlify Power 并与 Netlify 后端交互
- 需要设置 Netlify CLI 授权
- 几分钟内完成部署，输出部署 URL

42:30 - Powers 的价值总结
- 无需花费数小时设置引导文件和寻找正确的 MCP 服务器
- 从前端应用外壳到完全部署的应用，全程在 IDE 内完成
- 简化工作流程，快速达成目标
