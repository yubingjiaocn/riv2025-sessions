# AWS re:Invent 2025 会议总结：IBM Power 系统与 AWS 混合云架构

## 一、会议概述

本次 AWS re:Invent 2025 分会场聚焦于 IBM 中端系统(IBM i 和 AIX)在混合云环境中的现代化策略。会议由 Light Edge 首席云官 James Rory 和 IBM 混合企业云解决方案副总裁 Richard Dleski 共同主讲。

会议的核心主题是如何帮助仍在运行 IBM Power 系统的企业实现云转型。与其他 re:Invent 会议不同,本场会议较少提及 AI,而是专注于解决一个特定但关键的问题:如何在不完全重构关键业务应用的前提下,将运行在 IBM i/AIX 平台上的工作负载与 AWS 云服务整合。演讲者强调,虽然许多企业制定了"云优先"战略,但 IBM Power 系统往往承载着企业最关键的 ERP 系统(如 JD Edwards、SAP HANA、Manhattan 等),这些系统无法简单地"拖放"到 AWS 中。

Light Edge 提出的解决方案是在 AWS 区域附近建立专用数据中心,通过高速 Direct Connect 连接实现混合架构。这种方案允许企业将 IBM Power 工作负载托管在 Light Edge 数据中心,同时将相关的 Windows/Linux 虚拟机迁移到 AWS,从而实现 70-90% 的 IT 基础设施云化,同时保持核心系统的高性能和稳定性。会议通过多个真实案例展示了这种渐进式现代化路径的价值。

## 二、详细时间线与关键要点

[00:00-02:30] 开场与会议背景介绍
- 演讲者自我介绍:James Rory(Light Edge 首席云官)和 Richard Dleski(IBM 混合云解决方案副总裁,37 年 IBM 平台经验)
- 会议定位:专注于 IBM 中端系统(非大型机),针对运行 IBM i 和 AIX 的企业
- 强调这是 re:Invent 中"最少提及 AI"的会议,聚焦于特定的基础设施挑战

[02:30-05:45] IBM Power 系统面临的主要挑战
- 引用 Gartner 调查数据,针对 1000 家 IBM Power 用户的调研结果
- 主要挑战包括:网络安全、高可用性/灾难恢复、合规性、**技能短缺**和**现代化需求**
- 技能短缺和现代化是 IBM i/中端系统特有的挑战
- 尽管面临挑战,企业仍在继续投资和购买 IBM i 硬件
- 用户对平台满意度高,特别是制造业企业视其为业务命脉

[05:45-08:20] 当前部署现状与云迁移障碍
- 数据显示:大多数 IBM Power 系统仍运行在本地环境(on-prem)
- 仅 15% 采用混合云模式
- 主要原因:IBM Power 系统无法像虚拟机那样简单迁移到 AWS
- 没有"云原生"的 IBM i 解决方案
- 本地部署虽然稳定,但在安全性、弹性和现代化方面存在长期隐患

[08:20-11:30] 本地部署的风险与现代化需求
- 安全态势:IBM Power 系统不能成为企业安全防护的薄弱环节
- 虽然 IBM Power 系统历史上很少被黑客攻击,但数据泄露风险依然存在
- 弹性需求:业务要求 7×24 小时运行,需要并发补丁和维护能力
- 传统做法(要求业务停机数小时进行维护)已不可接受
- 合规性、灾难恢复、高可用性、集群等能力需要专业技能支持

[11:30-14:45] IBM Power 系统的战略重要性
- IBM Power 系统是"记录系统"(System of Record),是企业的核心引擎
- 处理数百万笔交易,直接驱动企业收入
- 需要将记录系统安全地暴露给"参与系统"(Systems of Engagement)
- 现代 IT 消费模式与 10 年前、甚至 5 年前完全不同
- 传统的硬件刷新周期(3-4 年)已不足以满足业务演进需求
- 客户期望 7×24 全天候服务,系统维护不应影响业务运营

[14:45-18:30] Light Edge 混合云架构解决方案
- 现场调查:运行 AWS 生产工作负载的听众远少于运行 IBM i/Power 的听众
- 核心问题:IBM 工作负载需要极低延迟和高性能,难以在纯云环境实现
- Light Edge 解决方案:在 AWS 区域附近建立数据中心,通过高速 Direct Connect 连接
- 架构优势:一个 IBM LPAR 可与 100 个 Windows 虚拟机通信,或 50 个 LPAR 与 5000 个虚拟机通信
- 实现效果:将 70-90% 的传统 IT 资产迁移到 AWS,同时保持 IBM 工作负载在托管数据中心

[18:30-22:15] 混合架构的技术优势
- 支持旧版本和生命周期结束的 IBM 技术
- 提供升级路径,无需破坏现有应用
- 解决距离延迟问题:本地环境中延迟不明显,但云环境中距离会严重影响性能
- Light Edge 在 AWS 区域(如 Ashburn, Virginia)建立多个可用区
- 实现低于 2 毫秒的延迟连接
- 允许企业逐步现代化,而非一次性重构

[22:15-26:40] 高可用性架构设计
- Light Edge 是全球唯一在 Ashburn, Virginia 提供多可用区 IBM Power 系统的供应商
- 架构包含两个 IBM 云可用区(A1 和 A2),连接到 AWS 的六个可用区(a1-a6)
- 实现真正的高可用性,避免单点故障
- SAN 基础设施层面的高可用性:SAN 1 和 SAN 2 之间的复制
- 基于策略的复制,实现接近零数据丢失和接近零停机时间
- 使用 IBM DB2 Mirror 等产品实现数据库集群

[26:40-30:20] 数据保护与备份策略
- 并发维护能力:通过负载均衡,在一个集群维护时将用户切换到另一个集群
- 用户无感知的维护操作
- 备份不仅是归档,更重要的是恢复能力
- Flash Copy 功能:每天进行完整系统保存(IBM i 的 Option 21)
- 多层备份:两个完整系统备份(加密),第三层可选 S3 存储实现不可变性
- 逻辑复制:针对基于事务的系统,支持事务级别的回滚
- 防止勒索软件攻击,提供不可变备份

[30:20-34:50] 应用现代化路径
- 支持 SAP、JD Edwards、PeopleSoft、Lawson、Manhattan 等商业 ERP 软件
- 提供 90-120 天的云就绪迁移时间表
- 三种现代化路径:
  1. 保持 IBM 平台,仅现代化"低垂果实"(易迁移工作负载)
  2. 1-2 年项目进行代码重写,适应现代业务需求
  3. 混合方式:逐步减少 IBM 消耗,增加 AWS 消耗
- 无需硬件刷新,无需新的维护合同
- 灵活的性能配置:支持周期性负载(如零售业的黑色星期五高峰)
- 按使用付费模式,避免为峰值购买硬件

[34:50-38:30] 渐进式现代化策略
- "爬行、行走、奔跑"(Crawl, Walk, Run)方法
- 减轻"必须立即采取行动"的压力
- 从低垂果实开始:将虚拟机迁移到 AWS,实现快速现代化
- 中期:利用 AWS 原生服务(如 RDS、FSX、RedShift)
- 长期:可选择完全现代化,脱离 IBM Power 平台
- 强调企业可以根据自身节奏选择现代化速度
- 许多企业选择长期保留 IBM Power,同时充分利用 AWS 服务

[38:30-43:15] 案例研究一:大型物流 SaaS 公司
- 背景:运行自研 ERP 系统,追踪全国范围内的运输车队
- 初始状态:完全本地部署,多个数据中心,零 AWS 工作负载
- 迁移方案:IBM 系统迁移到 Light Edge 数据中心,数百个虚拟机迁移到 AWS
- 快速成果:迁移数据库到 RDS,使用 RedShift 进行分析
- 显著成效:停机时间减少 98%
- 后续发展:关闭所有三个本地数据中心,全面采用 AWS
- 新工作负载:全部构建在 AWS 原生服务上,使用 Bedrock、SageMaker 等 AI 服务
- 核心系统:IBM 仍在 Light Edge 数据中心运行,使用定制化 DB2 解决方案

[43:15-47:30] 案例研究一的技术细节
- 使用 IBM DB2 Mirror 产品实现 98% 的停机时间减少
- 业务场景:40 万辆卡车依赖系统获取运输文档和路线信息
- 原有问题:每季度 2-3 小时的计划维护窗口导致所有卡车停工
- 司机被迫打印文档,存在文档遗失和安全风险
- 解决方案:维护窗口从 2-3 小时缩短到不到 2 分钟
- 现在可以每周进行包构建和部署,对车队零影响
- 低延迟优势:AWS 服务对 IBM 系统的 I/O 和事务量增加 4-5 倍
- 业务创新:引入"钱包"功能,自动支付燃料费用,AI 推荐最优加油站
- 物流优化:使用 AWS 卫星跟踪能力优化路线和运营

[47:30-结束] 总结与关键要点
- 强调混合云架构的价值:无需完全重构即可实现云转型
- Light Edge 提供端到端托管服务,作为 IT 部门的延伸
- 企业无需再购买硬件,由 Light Edge 负责硬件升级(零成本)
- 客户可在最新或 N-1 架构上运行,获得更高可用性和数据保护能力
- 技能短缺问题得到解决:托管服务提供专业支持
- 实现真正的业务价值:从成本中心转变为业务驱动的技术平台
- 为企业提供灵活的现代化路径,支持长期战略规划
