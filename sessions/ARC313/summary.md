# AWS re:Invent 2025 架构会议 313 总结

## 会议概述

本次会议由 AWS 高级首席解决方案架构师 Andrew Baird 主讲,重点探讨了如何将传统的面向服务架构(SOA)原则应用于构建 AI 智能体系统。Andrew 在 AWS 工作了 13 年半,主要专注于开发者导向的技术领域,包括 DevOps、无服务器架构和 CI/CD。他强调,过去 18 个月中生成式 AI 和智能体技术的发展为软件开发者带来了前所未有的能力和机遇。

会议的核心观点是:智能体系统本质上仍然是面向服务的架构。尽管智能体技术看起来具有"观察"、"学习"和"自主行动"等拟人化特征,但从技术角度来看,它们仍然是由 Docker 容器、LLM 模型、标准化集成协议(如 MCP)和依赖关系管理组成的分布式系统。软件工程师多年来积累的架构设计经验和技能完全适用于构建智能体系统,开发者应该对此充满信心。会议后半部分由 WEX 公司的解决方案架构师 Dan Deluro 分享了他们在三个月内将智能体系统从试点推向生产环境的实战经验,目前已有超过 2000 名内部用户使用该系统。

## 详细时间线

### 开场与背景介绍 (00:00 - 05:30)
- **00:00** - 会议开始,Andrew Baird 自我介绍,欢迎与会者参加架构 313 分会场
- **01:15** - 介绍个人背景:在 AWS 工作 13.5 年,专注于开发者导向技术(DevOps、无服务器、API、CI/CD)
- **02:30** - 强调过去 18 个月是职业生涯中最激动人心的时期,生成式 AI 带来的能力提升令人震撼
- **03:45** - 说明会议目标:帮助软件开发者将现有技能应用到智能体系统构建中
- **04:30** - 介绍会议议程:前三分之一由 Andrew 讲解智能体与 SOA 的关系,后续由 WEX 的 Dan Deluro 分享实际案例

### 智能体的技术本质 (05:30 - 15:00)
- **05:30** - 提出核心问题:当智能体被描述为"观察、行动、学习"时,是否意味着面向服务架构已经过时?
- **07:00** - 深入技术层面解析智能体组成:LLM 模型 + 智能体应用容器(Docker)
- **08:30** - 介绍 SDK(如 Strands)如何简化与 LLM 的集成,处理提示结构、对话轮次、内存管理和工具集成
- **10:00** - 定义"智能体":Docker 容器化应用 + LLM 模型的组合
- **11:15** - 解释依赖关系管理:通过 MCP(模型上下文协议)标准化与数据库、文档库、其他服务的集成
- **12:30** - 强调 MCP 协议的成熟(约 7 个月前达到企业生产级别)
- **13:45** - 总结:智能体系统本质上就是面向服务架构,包含容器、集成机制、依赖定义等熟悉元素

### 传统 SOA 原则在智能体中的应用 (15:00 - 25:00)
- **15:00** - 开始讨论设计原则:传统 SOA 原则如何映射到智能体系统
- **15:45** - 松耦合原则:容器部署、异步处理、无共享关注点、独立扩展能力等原则完全适用
- **17:00** - 模块化原则:警告反模式 - 避免构建处理过多任务的单一智能体,应像微服务一样考虑智能体粒度
- **18:30** - 建议按业务逻辑、所有权或安全分段来划分智能体边界
- **19:15** - 可重用性原则:MCP 服务器应该像 REST 服务一样可重用,避免每个智能体都构建自己的集成代理
- **20:45** - 推荐由拥有相关业务领域的团队构建 MCP 服务器,而不是由智能体团队各自构建
- **22:00** - 可发现性原则:需要维护智能体目录,包含能力说明、集成文档、可观测性接入点等
- **23:30** - 强调自然语言描述使智能体比传统 API 更容易被人类理解

### 演进的设计原则 (25:00 - 35:00)
- **25:00** - 介绍需要调整的设计原则
- **25:30** - 无状态 vs 上下文记忆:传统架构追求无状态以提高弹性和可扩展性,但智能体需要持久化上下文记忆来提升多轮对话能力
- **27:15** - 警告:会话状态和记忆的存在会影响可扩展性和部署安全性,需要考虑对话中断时的用户体验
- **28:45** - 编排 vs 自主协调:传统工作流是确定性的(如 Step Functions),智能体允许运行时动态协调
- **30:00** - 说明智能体可以根据请求上下文动态选择下游专业智能体,但也可以结合确定性工作流
- **31:30** - 服务契约 vs 能力涌现:传统 API 强调向后兼容和严格契约,智能体的能力可以在运行时动态发现
- **33:00** - 强调工具描述变化时,智能体无需代码更新即可发现新能力

### 全新的设计原则 (35:00 - 42:00)
- **35:00** - 介绍智能体系统特有的新设计原则
- **35:30** - 从 CRUD 到目标导向:传统软件开发围绕创建、读取、更新、删除操作,智能体开发需要转向目标定义
- **37:00** - 建议将业务价值简洁地描述为目标,模型会理解并确定所需的数据和上下文
- **38:15** - 推理透明性:推理作为可部署技术是全新的,需要新的可观测性方法
- **39:30** - 传统日志只记录元数据和事件,现在需要捕获和分析智能体的"思考"过程
- **40:15** - 自我纠错能力:智能体倾向于通过多条路径自我纠正以达成目标,需要明确定义错误场景下的期望行为
- **41:30** - 非确定性:需要新的测试方法,提到当天上午发布的 Agent Core Evaluations 功能

### AWS 服务和工具推荐 (42:00 - 47:00)
- **42:00** - 推荐采用智能体框架(如 Strands SDK)而不是从零开始构建
- **43:15** - 介绍 Strands SDK 在模型推理能力成熟时发布,在抽象层次上达到了快速开发和生产就绪的平衡
- **44:30** - 推荐工具:Kiro CLI、Kiro IDE、Amazon Q Developer(可使用 Bedrock 模型)
- **45:45** - 介绍 Bedrock Agent Core Runtime:无服务器基础设施平台,提供运维卓越性和安全性保障
- **46:30** - 说明 Agent Core Runtime 也可以部署 MCP 服务器

### WEX 案例分享开始 (47:00 - 55:00)

- **48:30** - 介绍演讲主题:WEX 如何使用 Bedrock Agents 和 Agent Core 改善运维支持
- **49:00** - 强调从试点到生产仅用三个月,目前有超过 2000 名内部用户
- **49:45** - WEX 公司介绍:全球商务平台,为 200 多个国家提供移动、福利和支付解决方案
- **50:30** - 运营全球最大的专有车队网络之一,帮助消费者管理 HSA、FSA 等福利账户
- **51:15** - 去年处理超过 2300 亿美元交易,涉及 20 多种货币
