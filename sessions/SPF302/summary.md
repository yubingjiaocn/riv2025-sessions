# AWS re:Invent 2025 会议总结：GenAI 在足球运动中的应用

## 会议概述

本次会议由 AWS 企业架构师 Steve Brew 主讲，探讨了生成式 AI（GenAI）和 AI 技术在足球运动中的创新应用。作为一名拥有 40 多年球迷经历的足球爱好者和数据专家，Steve 分享了 AWS 服务如何影响现代足球的比赛方式。会议重点介绍了英超联赛和德国德甲联赛俱乐部的实际案例，展示了从数据平台建设、性能分析、球探系统到智能代理（Agentic AI）的技术演进。

现代足球俱乐部已经发展成为复杂的组织机构，不仅仅是 11 名球员之间的对抗。在最高水平的竞技中，微小的优势可能决定胜负，因此越来越多的俱乐部开始利用数据和 AI 技术来获取竞争优势。这些技术的应用涵盖了训练准备、性能分析、球员发展和球探等多个领域，帮助俱乐部在遵守规则和财务限制的前提下，以可持续的方式提升竞争力并建立社区。

会议强调，虽然技术在不断进步，但其核心目的是增强而非取代人类专业知识。数据驱动的方法与传统的足球智慧相结合，正在重塑这项美丽运动的未来。

## 详细时间线

### 开场介绍
时间戳：开始
- Steve Brew 自我介绍，AWS 企业架构师，专注于与战略合作伙伴和客户合作
- 分享个人背景：40 多年的足球球迷，曾参与伦敦奥运会数据 API 开发
- 引用 Brian Clough 名言："如果上帝想让我们在云端踢球，他会在那里铺上草坪"，以此引出 AWS 服务在足球中的应用

### 足球运动的复杂性
时间戳：早期
- 强调足球是全球最受欢迎的运动之一，从公园到可容纳 10 万人的体育场，世界杯决赛有数十亿观众
- 现代足球俱乐部是复杂的组织，拥有众多角色支持球员提升表现
- 在最高水平的竞争中，微小的优势至关重要
- 俱乐部必须在规则和财务限制内运营，同时建立可持续的社区

### 技术应用的核心组成
时间戳：中早期
- 介绍技术使用的三个主要组成部分：
  1. 用例：需要解决的问题，寻求优势的领域
  2. 数据 + 传统分析 + AI/ML/GenAI 技术
  3. 洞察和价值：获得更好的竞争优势
- 展示足球俱乐部使用技术的常见用例：
  - 训练准备和计划安排
  - 性能分析（球队和球员表现趋势）
  - 球探系统（培养自有球员和识别潜在引援目标）

### 数据基础
时间戳：中早期
- 强调数据是基础，使 AI 和 GenAI 服务能够产生洞察并获得优势
- 足球环境中的数据源：
  - 外部数据源和内部球队数据
  - 视频录像（比赛精彩片段和特定部分）
  - 文本数据（教练报告、球探报告）
  - 大量表格数据（数千个数据字段和数百万个数据点）
  - 球员表现 KPI、GPS 追踪数据等

### 数据管道和架构
时间戳：中期
- 在使用 GenAI 之前，仍需构建传统应用程序
- 需要数据库支持用户体验
- 标准数据管道包括：
  - 数据摄取（批处理或实时）
  - 结构化和非结构化数据处理
  - 数据转换和初步洞察
  - 数据治理考虑（隐私、安全、PII 数据保护）

### AI/ML 的持续作用
时间戳：中期
- AI/ML 仍然重要，主要应用于三个方面：
  1. 大量数据的统计分析
  2. 趋势和模式检测
  3. 特征识别（在海量数据中找出重要字段）
- AI/ML 作为数据处理的另一层，可被 GenAI 主题使用

### GenAI 的应用领域
时间戳：中期
- 数据提取：智能文档处理、球探手写报告、提取特定数据属性
- 洞察生成：聚合报告和信息
- 数据民主化：使数据池易于查询，让球探、教练等角色轻松访问

### 案例研究 1：德甲俱乐部的性能分析
时间戳：中期
- 一家德国德甲俱乐部构建关键绩效指标（KPI）组
- 从 6,800 个不同数据点中提取数据特征组（控球 KPI、铲球 KPI 等）
- 进行长期数据处理：单个球员、多个球员、球员对比、跨赛季分析
- 架构组件：
  - 第三方数据 API
  - 数据摄取和 ETL 流程
  - 数据存储（使用 Redshift 作为示例）
  - 可视化前端（自定义 UI 或 QuickSight）
- 展示简单仪表板，可查询和比较球员的 KPI 组

### 案例研究 2：Holstein Kiel（荷尔斯泰因基尔）
时间戳：中期
- 德国德甲俱乐部，强调数据帮助在短时间内生成重要的客观信息
- 俱乐部生成大量数据但难以转化为竞争优势
- 采用数据架构方法：
  - 为每个球员创建主 ID
  - 长期跟踪球员在场上、训练中的表现进展
  - 识别六个关键领域：球队管理、工作量管理、球探等
- 五个关键技术原则：
  1. 可扩展性
  2. 安全的数据基础
  3. 模块化架构
  4. 事件驱动
  5. 逻辑分离（减少未来的技术开发和维护）
- 重点：俱乐部专注于足球，而非技术提供商

### Holstein Kiel 的 GenAI 应用
时间戳：中后期
- 使用 Amazon Bedrock 上的 OpenAI 获得进一步竞争优势
- 三个应用领域：
  1. 使用推理模型（如 OpenAI）通过自然语言提问
  2. LLM 考虑所有可用数据点
  3. 从静态仪表板演进
- 自然语言搜索架构：
  - 用户通过自定义 UI（Amplify 托管）提问
  - 查询传递到 Amazon Bedrock Knowledge Base
  - 转换为 SQL 查询
  - 在 Redshift 中执行查询
  - 数据返回并由 LLM 处理
  - 支持交互式后续问题

### 推理能力的重要性
时间戳：中后期
- Holstein Kiel 选择 Bedrock 上的 OpenAI 是为了推理能力
- 示例：为什么我们的前锋进球贡献下降了 40%？
- 推理 LLM 的优势：
  - 将问题分解为多个任务
  - 多步骤规划
  - 调查更多数据源
  - 发现冲突信息时修订计划
  - 使用工具（SQL 查询、运行代码）提供更准确的答案
  - 包含验证步骤

### Holstein Kiel 的数据价值生成飞轮
时间戳：中后期
- 数字化转型的指导原则（"北极星"）：
  - 不同团队/部门如何协同工作
  - 如何丰富拥有数十年经验的员工的知识
  - 如何衡量新方法的成功
- 飞轮流程：
  1. 从可用数据开始
  2. 思考潜在用例
  3. 快速 POC 和迭代
  4. 开发完整功能解决方案
  5. 如果成功，投入更多资源
  6. 贡献回整体平台价值
- 关键引言："部署人类专业知识的运营改进是关键"——技术用于增强员工能力

### 案例研究 3：Darmstadt（达姆施塔特）点球分析
时间戳：中后期
- 德甲俱乐部，极其数据驱动
- 点球案例：不常见的事件，但转化率约 78-79%
- 小优势可能带来重大影响（进球事件）
- 数据源：
  - 外部数据提供商（球员点球位置坐标、助跑方式）
  - 教练手动观看视频
  - 第三方数据源
- 技术选择：
  - 使用 SageMaker Studio Lab 轻松设置环境
  - 使用 SageMaker 训练模型
  - 零扩展、相对低成本
- 实际应用：守门员教练和守门员使用分析结果，将对手前锋可能射门方向的列表贴在球门后的水瓶上
- 关键信息：不需要最复杂的架构来解决问题或获得洞察

### Darmstadt 的数据驱动文化
时间戳：中后期
- 引言："情感驱动足球，但数据驱动进步"
- 俱乐部在每个步骤都使用数据

### 智能文档处理
时间戳：后期
- 除了统计和数据分析，还有其他数据集需要洞察
- 足球俱乐部有大量教练、球探和其他人员
- 处理手写报告、平板电脑报告等
- 目标：将这些数据与表格数据一起查询
- 架构方法：
  - 手写报告照片上传到门户
  - 存储到 S3
  - Step Functions 启动处理
  - 使用 LLM 进行 OCR/智能文档处理
  - 提取关键洞察和实体
  - 验证球员姓名等固定变量
  - 写回数据存储
  - 与现有仪表板一起可视化

### 跨多种数据类型搜索
时间戳：后期
- 随着数据类型增加，需要跨多种数据类型和数据存储搜索
- 为不同数据类型选择最佳数据存储以获得更好性能
- 下一次迭代：
  - 自定义应用（Amplify 托管）
  - 使用 Bedrock Agents 处理用户查询
  - 连接到结构化和非结构化数据存储
  - 提取相关信息
  - 组合并呈现给用户

### Agentic AI 简介
时间戳：后期
- 仍在实验阶段，而非生产环境
- Agentic AI 的三个关键输入：
  1. 记忆（之前做过什么）
  2. 工具（可用的工具）
  3. 目标（用户试图实现什么）
- 流程：采取行动 → 查询数据 → 观察 → 迭代

### 单代理方法示例
时间戳：后期
- 任务：比赛分析——我们的压迫战术今天表现如何？
- 流程：
  1. 用户查询传递给代理
  2. 代理从规划角度分析
  3. 将任务分解（评估请求、分析数据等）
  4. 调用工具/操作（查询数据存储）
  5. 历史进攻表现分析
  6. 球员级别还是球队级别分析？
  7. 代理迭代计划，勾选要点，可能添加新要点
- 优势：更灵活、更自主的方法，随着发现新工具或信息而演进

### Agent Core 架构
时间戳：后期
- 使用 Agent Core Runtime 托管代理
- 组件：
  - 用户前端（左侧）
  - Agent Core Runtime 中的代理部署
  - 代理指令（足球数据分析的规则和指导）
  - 本地工具（历史报告、球员报告、球队分析、球员统计）
  - 查询不同数据存储的操作
  - 调用 Bedrock 和 LLM
  - 处理信息并实现用户目标

### 多代理架构
时间戳：后期
- 回到 Holstein Kiel 的六个领域
- 随着系统演进，采用多代理方法
- 专门针对特定任务的代理（例如比赛分析）
- 优势：
  - 模块化代理
  - 灵活性（快速添加/删除代理和功能）
  - 工作流管理（编排代理选择正确的工具/代理）
  - 可能只需调用单个代理，或查询 4-5 个领域并聚合结果
- 提供灵活性：不再是严格定义的规则，而是呈现大量数据和工具，LLM 根据用户请求或新发现的信息自动定制和规划操作

### 多代理部署架构
时间戳：后期
- 使用 Agent Core Runtime 部署各个代理
- 模式：通过 Agent Core Gateway 将常见工具和功能公开为 MCP 服务器
- 常见功能、报告托管在 Agent Core Gateway 中
- 可供所有运行的代理使用
- 可以接入现有 API 端点或 Lambda 函数
- 与 Agent Core Gateway 接口，供任何代理使用

### 何时使用不同的 AI 方法
时间戳：后期
- 并非所有事情都需要 Agentic AI
- 根据用例考虑组合使用：
  - **生成式 AI**：文档信息提取、报告摘要、报告关联和总结
  - **预测性 AI**：统计数据的趋势分析
  - **Agentic AI**：规划和根据发现采取行动
- 某些组件/工作负载作为传统代码运行更合理

### 入门指南
时间戳：后期
- 两套指导/团队可以帮助：
  1. **AWS GenAI 创新中心**：丰富的技术经验，帮助更快交付项目
  2. **APN AI 能力合作伙伴**：合作伙伴网络中的深度行业专业化

### 案例研究 4：West Ham United（西汉姆联）
时间戳：后期
- 英超联赛俱乐部，人才发展的悠久传统
- 被称为"足球学院"，在识别、培养和建设人才方面有深厚背景
- 技术招募团队构建端到端 GenAI 球探平台
- 英超首家端到端使用 GenAI 的俱乐部
- 与 AWS 和 APN 合作伙伴 Crayon 合作

### West Ham 的球探流程
时间戳：后期
- 球探流程挑战：
  - 从全球 100,000 名球员数据库开始（180 个国家）
  - 初步数据分析以缩小候选名单
  - 基于统计、报纸报道等创建观察名单
  - 球探观看视频，进行进一步分析
  - 创建文本和视觉报告（耗时）
  - 确定优先级列表
  - 球探实地观看球员（统计和视频无法捕捉一切）
  - 创建更长的报告（20-30 页）
  - 决定是否继续、投入更多时间/金钱
- 大量人工元素：报告创建、报告分析、数据收集
- 这些时间本可以更好地用于实际观看球员

### West Ham 的 GenAI 目标
时间戳：后期
- 使用技术帮助球探和招募球员
- 补充现有球探方法
- 加快流程
- 更好地覆盖更多球员
- 使用正确的技术获得更好的球员覆盖范围
- GenAI 提供生产力提升，帮助扩展规模，提供更多细节和洞察

### West Ham 人才识别平台总结
时间戳：接近结束
- 目标：
  - 减少球探和教练的时间投入
  - 提高分析水平
  - 扩大球员覆盖范围
- 数据输入考虑：
  - 人工制作的球探报告、比赛报告、评级
  - 其他数据模式：事件级数据、追踪数据、身体数据
  - 大量信息用于帮助做出进一步决策
- GenAI 组件：
  - 处理不同模式和数据类型
  - 提供支持决策的能力
