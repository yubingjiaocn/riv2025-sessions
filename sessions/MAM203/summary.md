# AWS VMware 迁移会议总结

## 会议概述

本次AWS技术会议聚焦于VMware工作负载迁移的成功案例、策略和路线图。会议由AWS主持，邀请了三位来自不同行业的客户代表分享他们的迁移经验：Envision Healthcare的云和基础设施交付副总裁Tyler Barheight、墨西哥Aeromexico航空公司的基础设施副总监Angel Gonzalez，以及加州大学系统的首席信息官Molly Greek。

AWS强调其作为最早帮助客户将VMware工作负载迁移到云端的供应商，提供多种灵活的迁移路径，包括使用Amazon EVS（Elastic VMware Service）进行快速迁移、通过AWS Application Migration Service重新托管到EC2、容器化改造、使用托管服务以及完全重构应用程序。三位客户分享了各自独特的迁移旅程，从应对紧急性能问题到战略性数据中心整合，展示了不同的迁移策略和实施方法。

会议重点讨论了迁移过程中的经验教训，包括建立FinOps实践的重要性、与AWS作为业务合作伙伴密切合作、建立治理框架以及利用AWS的各种支持计划。所有三家组织都强调了迁移带来的显著收益，包括性能提升、成本节约、安全性改善和运营效率提高。会议最后探讨了未来的现代化计划和持续优化策略。

- **10:00** - Tyler开始分享Envision Healthcare的迁移经历
- **10:20** - 初始状态：拥有本地基础设施和云足迹，存在大量碎片化

- **11:30** - 重点关注数据中心整合、运营效率和成本节约
- **12:00** - 使用的工具：Application Migration Service、Application Discovery Service、EC2、RDS Custom
- **12:20** - 早期专注于SQL工作负载迁移到RDS和RDS Custom
- **12:40** - 2025年成功退出一个数据中心，采用混合方法：数据中心间迁移、直接迁移到AWS、迁移到Azure AVS
- **13:00** - 计划在2026年底前退出最后两个主要数据中心
- **13:30** - 最初尝试现代化所有内容，但这减慢了进度
- **14:00** - 调整策略，保持灵活性，继续推进迁移
- **14:20** - 进行了两次EBA（Experience-Based Acceleration）研讨会，非常有帮助
- **14:40** - 与合作伙伴Presidio和AWS合作，建立结构化方法
- **15:00** - 利用AWS Skill Builder进行团队技能提升
- **15:20** - 整合了本地团队和云团队为一个组织

### Aeromexico迁移故事 (15:30-22:00)
- **15:30** - Angel开始分享Aeromexico的迁移旅程
- **15:50** - 迁移始于4年前，主要策略不是简单的lift-and-shift
- **16:10** - 首先进行架构设计，然后开始迁移和现代化应用
- **16:30** - 目标：从3个主要数据中心迁移出来，完全转向云端，仅使用AWS
- **16:50** - 计划迁移346-350个应用程序
- **17:10** - 策略分为两类：关键工作负载的重构/重新平台化，以及其他应用的不同选项
- **17:40** - 面临重大技术更新需求，决定评估AWS选项
- **18:00** - 与AWS合作进行评估，使用Countdown服务和专家支持，开始研究EVS
- **18:30** - 选择EVS的原因：业务团队需要专注于重构其他应用，基础设施团队负责剩余应用
- **19:00** - 挑战：需要在云中保留IP地址，避免对应用进行重大更改
- **19:20** - 需要改善网络安全漏洞
- **19:40** - 与合作伙伴DXC合作，他们对本地应用非常了解
- **20:00** - 决定使用EVS是一个冒险的决定，但最终效果很好
- **20:20** - 在仅一个月内迁移了96台服务器和50多个应用程序
- **20:50** - 意外收获：发现运营应用有更多容量，应用运行速度更快
- **21:20** - 客户反馈非常积极：性能、可靠性和安全性都得到提升
- **21:40** - 使用HCX技术扩展网络，在夜间迁移工作负载，零运营影响
- **21:50** - 作为24/7运营的航空公司，对客户没有任何影响

### 加州大学迁移故事 (22:00-29:00)
- **22:00** - Molly开始分享加州大学的迁移经历
- **22:20** - 迁移不是战略性的，而是从紧急情况开始
- **22:40** - 第一个紧急情况：HR薪资系统服务28万员工，存在严重性能问题
- **23:00** - 系统由供应商在不同平台上管理，每次运行薪资时间越来越长
- **23:30** - 面临无法按时运行薪资的风险，决定迁移到AWS
- **23:50** - 迁移耗时约6个月，与AWS合作伙伴密切合作
- **24:10** - 迁移后所有性能问题消失，供应商承认确实是基础设施问题
- **24:40** - 第二个紧急情况：本科生申请系统，每年25万申请者
- **25:00** - 截止日期前15分钟约有1.5万人提交申请
- **25:20** - 连续两年系统在最后15分钟崩溃，成为新闻头条
- **25:40** - 大学校长要求第三年不能再发生，尝试了多种性能管理方法
- **26:00** - 这是典型的峰值负载场景，无法为短暂峰值配置足够服务器
- **26:20** - 与AWS合作迁移系统，AWS团队在午夜与他们一起调整参数
- **26:50** - 已经是第三年，AWS仍在午夜时分提供支持确保顺利运行
- **27:10** - 第三个项目：退出大型机，大型机成本非常高
- **27:30** - 需要保留数据50-99年，将所有数据迁移到AWS
- **27:50** - 关闭大型机节省了300万美元，与Deloitte合作完成
- **28:10** - 在多次成功后，决定将所有内容迁移到单一平台
- **28:30** - 恰逢VMware变更时期，加速了迁移计划以避免VMware成本
- **28:50** - 现在100%在AWS上运行，团队经验丰富且更加自信

### 经验教训 - Envision Healthcare (30:30-34:00)
- **30:30** - 开始经验教训环节，Tyler首先分享
- **30:50** - 最大挑战：由于碎片化存在大量"丢失的玩具"，比15台僵尸服务器多得多
- **31:10** - 许多系统不知道是什么、上面有什么、存在多久、谁在使用
- **31:30** - 强调理解生态系统的重要性，需要良好的CMDB或发现工具
- **31:50** - 识别出大量可以保留的系统
- **32:10** - 面临VMware成本问题，在9个月内从约5000核减少到22核
- **32:30** - 通过推动IT与业务应用所有者的对话实现
- **32:50** - 识别出约400个长期存在的不需要的系统
- **33:10** - 经验：不要将"保留"作为第一答案，不要将第一个答案视为真相，需要持续推动
- **33:30** - 有时需要给业务"选择的错觉"，引导他们做出正确决定
- **33:50** - 与业务合作伙伴建立共同愿景

### 经验教训 - 加州大学 (39:00-43:30)
- **39:00** - Molly分享加州大学的经验教训
- **39:20** - 早期错误：给团队很多人AWS访问权限，让他们自由探索
- **39:40** - 不建议这种策略：人们做了花费大量资金、不安全、价值不高的事情
- **40:00** - 例如：将应用放在AWS但数据库保留在数据中心
- **40:20** - 必须收回控制权，当时很不受欢迎
- **40:40** - 引入有经验的云领导者，建立标准、治理和FinOps
- **41:00** - 月度成本从5万美元降至1万美元，大幅提升安全性
- **41:20** - 教训：从一开始就需要治理、标准和有经验的IT领导
- **41:50** - 第二个教训：HR薪资系统迁移项目，预算已规划但COVID导致预算削减
- **42:10** - 团队不想告诉供应商，认为"他们只会向我们推销东西"
- **42:30** - 将供应商视为业务合作伙伴，坦诚沟通预算问题
- **42:50** - AWS帮助找到了他们不知道的项目和计划，使项目得以继续
- **43:10** - 教训：将供应商视为业务合作伙伴，坦诚沟通问题，他们可能有意想不到的解决方案
