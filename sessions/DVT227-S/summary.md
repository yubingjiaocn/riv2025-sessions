# AWS re:Invent 2025 会议总结：现代化可靠性监控

## 概述

本次演讲由 Checkly 的 CEO 兼联合创始人 Hannes Lank 主讲，主题聚焦于软件可靠性在现代开发环境中面临的挑战。演讲指出，尽管软件开发和部署速度已经从十年前的年度发布演进到如今几乎由 AI 代理实时构建和部署，但可靠性保障方法却仍停留在 2013 年的水平。这种不匹配导致了现代应用程序在复杂依赖关系下的可靠性问题日益突出。

Hannes 强调，传统的"更多人员、更多流程、更多工具"的解决方案已经不再适用。测试和监控团队虽然使用不同的工具和流程，但实际上都在回答同一个核心问题：应用程序从最终用户角度是否正常工作？演讲提出了高绩效团队普遍采用的三大原则：可预测性（在发布到生产环境前预测应用表现）、问责制（清楚了解变更内容、责任人和时间）以及弹性（快速回滚能力）。

Checkly 的解决方案是将测试和监控统一到一个工作流中，通过代码化的方式（Playwright UI 自动化测试、JavaScript/TypeScript API 测试、运行时监控）实现从本地开发到生产环境的一致性语言。演讲最后提出了新的理念："你构建它，你测试它，你监控它，你拥有它"，将可靠性责任整合到整个软件开发生命周期中，特别是在 AI 代理参与开发的未来场景下。

## 详细时间线

开场介绍
- 演讲者自我介绍：Hannes Lank，Checkly 的 CEO 兼联合创始人
- 公司使命：帮助工程师从 pull request 到生产环境全程掌控可靠性

软件开发演进历史
- 十年前：软件每年构建一次，按季度、月度、周度、日度逐步加快发布节奏
- 现在：AI 代理几乎可以即时构建和部署软件
- 问题：可靠性保障方法仍停留在 2013 年水平

应用复杂性的变化
- 过去：应用简单，单一团队，少量依赖（1-3 个），问题易于定位
- 现在：应用复杂，大量依赖关系，每个新依赖都引入潜在故障风险
- 说明：这些集成（支付提供商、数据库等）虽然带来风险，但也加速了开发

传统解决方案的局限
- 传统答案：更多人员、更多流程、更多工具
- 团队分工：QA 团队负责测试，开发人员构建应用，运维团队监控生产环境
- 核心矛盾：不同团队使用不同工具回答同一个问题——"应用从最终用户角度是否正常工作？"

测试与监控的共同目标
- 在预发布环境验证构建是否正常工作
- 在生产环境持续验证应用是否正常运行
- 在用户发现问题前捕获故障
- 验证集成和 API 端点的稳定性
- 监控和断言性能表现

高绩效团队的三大原则
- 基于 1,300 家付费客户的实践总结
- **可预测性**：能够在发布到生产环境前预测应用表现
- **问责制**：清楚了解变更内容、责任人、原因、时间和问题来源
- **弹性**：构建弹性应用，最坏情况下能快速回滚以避免影响用户

Checkly 的解决方案
- 将测试和监控统一到一个工作流中
- 技术栈：Playwright UI 自动化测试、JavaScript/TypeScript API 测试、运行时监控、告警、状态页面
- 特点：全部版本控制，为软件开发生命周期而设计，注重速度
- 工作方式：将检查和测试部署到平台，在全球范围内持续运行，在用户发现问题前主动通知

理念演进
- 过去十年的口号："你构建它，你运行它，你拥有它"（You build it, you run it, you own it）
- 新理念：这个口号已经过时
- "你"的定义扩展：不仅是开发人员，还包括所有接触客户用户体验的角色，包括 AI 代理、云代码等工具
- 未来愿景："你构建它，你测试它，你监控它，你拥有它"

核心价值主张
- 将测试提升为监控
- 定义告警和状态页面
- 快速检测、沟通和解决问题
- 关键整合：将监控堆栈集成到 CI/CD 堆栈中
- 统一语言：CI/CD 管道和可靠性管道使用相同的框架和测试
- 一致性：从本地开发到生产环境使用相同的语言和工具
