# AWS re:Invent 2025 会议总结：曼彻斯特机场集团应用 Agentic AI 确保航班准点

## 会议概述

本次会议介绍了曼彻斯特机场集团(Manchester Airport Group, MAG)如何应用 Agentic AI 技术解决机场运营中的复杂挑战，特别是员工临时缺勤管理问题。MAG 每天处理超过1000个航班，拥有约9000名员工，在英国运营三个机场。会议由 AWS 高级 AI 策略师 Tom Chester、团队成员 Margarita 以及 MAG 的 Steve 共同主讲。

演讲团队详细阐述了为什么选择 Agentic AI 而非传统自动化工具来应对机场的非确定性、高复杂度运营流程。机场环境涉及多方协作（MAG 员工、边境管理、航空公司、地勤服务等），且乘客安全和安保是首要任务。传统自动化工具需要预编程数百种工作流程排列组合，仍无法覆盖所有边缘情况，而 Agentic AI 能够动态思考和适应不同场景，具有模块化和可扩展性，能够处理不完整或模糊的用户输入。

技术实现方面，解决方案基于 Amazon Bedrock Agent Core 构建，采用"大胆构想、小步启动、快速扩展"的框架。首个用例聚焦于员工临时缺勤报告和排班重组，这是一个具有明确业务价值和投资回报的实际问题。通过减少加班成本、提高安检效率以及将第三方服务内部化，该方案展现了显著的成本节约潜力。

### 首个用例：临时缺勤报告 (16:30-22:00)

16:30 - Steve 介绍首个用例：临时缺勤报告，需要实时重新排班

17:00 - 业务需求：特别是在夏季高峰期和圣诞节前，这是关键且相关的业务需求

17:30 - 关注成果：减少整体运营成本，减少加班支付，降低第三方服务费用

18:00 - 临时缺勤类型：主要是疾病（尤其在暑假和圣诞节期间），也包括交通问题、家庭紧急情况等

18:45 - 当前流程说明：员工致电报告缺勤 → 第三方热线处理 → 资源团队重新排班 → 安排替班人员 → 通知直线经理 → 更新 HR 记录

19:30 - 流程标准化需求：三个不同机场和不同团队的流程略有差异，需要标准化并确保结果准确

20:00 - 价值案例：减少加班成本、提高安检效率增加航站楼消费、将第三方服务内部化

21:00 - 关键考虑因素：安全性（用户身份验证）、准确性（缺勤时长对排班和薪资的影响）、可靠性（7×24小时运营、低延迟）

21:45 - 合规要求：处理个人身份信息(PII)、影响员工薪资、关键国家基础设施、受法规约束

### 为何选择 Agentic AI (22:00-28:30)

22:00 - Tom 解释选择 Agentic AI 的原因

22:30 - AI 解决方案演进谱系：从生成式 AI 助手（聊天机器人、文档处理）到 AI 代理（更多自主性、关注成果）再到完全自主的 Agentic 系统

23:30 - MAG 的定位：临时缺勤报告用例属于 AI 代理类别，数字化员工工作场所的终极愿景是完全自主的 Agentic 系统

24:15 - 代理能力说明：被赋予目标和指令、访问工具集（数据源、API）、拥有记忆（短期和长期交互）、推理能力（分解任务、制定计划、采取行动、观察影响）

25:30 - 多代理系统：代理可以与其他 AI 代理交互，每个代理都有专门的目标和工具集

26:00 - 为何不用简单自动化工具：机场是动态环境，具有非确定性流程，需要跨多个相互依赖功能的协调

26:45 - 复杂性示例：三个不同机场、多种流程（基于工作类型）、多种缺勤类型（各有 HR 政策），快速产生数百种工作流程排列组合

27:30 - 跨系统影响：一个区域的人员决策可能受另一区域设备故障影响，再加上大量航班延误

28:00 - Agentic AI 优势：模块化和可扩展设计、处理异常和不完整输入的灵活性、自然语言理解能力

### 技术实现：工具适配 (28:30-35:00)

28:30 - Margarita 开始技术深度讲解

29:00 - 当前工作流程分析：员工报告缺勤 → 联系人收集信息 → 了解 HR 政策 → 分类缺勤类型 → 记录到 HR 系统 → 通知相关人员 → 如果是排班员工则更新班次记录

30:00 - 自动化思路：通过 API 连接 HR 排班系统、通过知识库查询 HR 政策、通过 LLM 分类缺勤、通过 Amazon SNS 通知经理

31:00 - 确定性工作流的局限：如果员工提供的信息不完整，工具无法正常工作

31:30 - 引入 Agentic 工作流：需要自然对话模式来收集所有相关信息，代理连接所有工具并通过提示词指导行为

32:15 - 代理实现：使用 ReAct 模式（推理-行动-观察-响应），通过 Strands 实现，每个推理步骤记录在 S3 存储桶中以便观察

33:00 - 工具示例：提取用户信息供代理采取后续步骤的工具

33:30 - 第一个重要教训：需要为代理适配工具

34:00 - 工具适配要点：
- 设计详细的错误捕获（代理需要知道错误原因才能找到替代路径）
- 使输出人类可读（虽然 LLM 能理解 JSON，但直接告知所需信息会更快）
- 合并顺序工具（对延迟很重要，如果工具总是按相同顺序调用，就合并它们）

### 技术实现：扩展性与架构 (35:00-42:00)

35:00 - 扩展性挑战：文本到文本解决方案在笔记本电脑上运行良好，但需要支持50名员工同时使用

35:45 - 传统方案复杂性：在 Lambda 上托管需要考虑 WebSocket 状态管理，需要数周工作

36:15 - Agent Core Runtime 解决方案：一个下午就解决了扩展性问题

36:45 - Agent Core 生态系统介绍：提供规模化和安全部署代理的能力，兼容所有开源基础设施（如 MCP 服务器和各种框架）

37:30 - 使用 Strands 代理和 Model Context Protocol (MCP)

38:00 - Agent Core Runtime 优势：无服务器解决方案、直接与前端连接、具有状态持久性，每个员工都能得到答案

38:45 - 未来扩展考虑：如果员工报告不同的事情（如自动扶梯损坏）怎么办？

39:15 - 单体架构的局限：需要拆解和调整整个解决方案，重新验证所有功能

39:45 - 关注点分离：将工具的关注点与代理的关注点分离

40:00 - MCP 服务器方案：提供标准化的工具暴露方式，可以更改工具，也可以在不同代理中重用这些工具

40:45 - Agent Core Gateway：确保代理能够自动发现所有工具

41:15 - 工作流程：员工提问 → Agent Core Runtime → 连接 Agent Core Gateway → 搜索正确工具 → 查询工具 → 获取结果 → 回答用户问题

41:45 - 动态扩展：可以即时添加更多工具，所有操作通过单个 API 调用到 Agent Core Gateway 完成

### 技术实现：安全与防护 (42:00-47:00)

42:00 - 安全审查挑战：安全审查员提出问题——如何防止有人通过社会工程学让代理执行恶意操作

42:30 - 初步方案：添加防护栏(Guardrails)

43:00 - 标准防护栏的局限：对极端情况有效，但如果用户更狡猾或试图通过多步骤对话进行工程化攻击呢？

43:30 - 高影响场景：机场环境中，如果缺勤记录错误，会影响检查点、航班系统，具有高影响

44:00 - 第二个重要教训：需要超越标准防护栏的思考，为合规和负责任的 AI 编写自定义防护栏

44:30 - 当前 Bedrock Guardrails 的局限：只查看用户写的最后一条消息

45:00 - 防护栏设计原则：
- 最小权限原则（确保代理使用的工具只能由经过身份验证的人使用）
- 定义不需要的对话模式
- 构建具有对话上下文的防护栏，能够跟踪对话模式并在模式偏离标准时做出反应

46:00 - 实现方案：通过 Lambda 添加上下文感知防护栏，基于恢复代理记忆并使用 LLM 作为防护栏

### 技术实现：语音交互 (47:00-54:00)

47:00 - 新场景：Macy 开车去上班途中遇到交通堵塞，无法按时到达

47:30 - 旧方案：打电话给经理说"对不起，我迟到了"

48:00 - 新方案的问题：需要拿出手机、打开应用、输入文字——这些在开车时都无法做到

48:30 - 解决方案：需要语音到语音(Speech-to-Speech)解决方案

49:00 - Amazon Nova Sonic 介绍：能够进行语音交互的语音到语音模型

49:30 - 实时流式复杂性：Nova Sonic 需要连续音频输入并输出连续音频，无法使用请求-响应模式

50:00 - WebSocket 连接：通过 ECS 管理，缓冲音频帧并协调双向流

50:45 - 提示词挑战：尝试使用与文本模型相同的提示词

51:00 - 第三个重要教训：大声朗读语音到语音模型的提示词

51:30 - 语音模型提示词要点：
- 语音到语音模型在语音上训练，无法像文本到文本模型那样进行复杂思考
- 不要依赖思维链(Chain of Thought)——没有人会在说话前说出自己在想什么
- 写出枚举模式——在文本提示中很好，但没有人说话时会枚举句子
- 考虑拼写——模型可能不知道所有不同国籍的名字拼写，帮助模型处理这些边缘情况

53:00 - 重新起草提示词：将提示词从基于文本改为基于语音

53:30 - 连接挑战：语音模型需要连接到为文本到文本模型构建的所有工具

54:00 - 工具输出问题：语音到语音模型需要详细的、可以大声朗读的答案，但这些工具主要是软件输出

### 技术实现：混合架构 (54:00-结束)

54:00 - 顿悟时刻：基础设施中已经有一个部分经过训练可以提供人类可读响应——那就是文本到文本模型
