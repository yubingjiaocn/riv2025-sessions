# AWS re:Invent 2025 会议总结：从打孔卡到AI - 开发者的新思维模式

## 会议概述

本次会议由 Andakat Najiro 主讲，探讨了软件开发从打孔卡时代到AI辅助开发时代的演变历程，以及开发者思维模式的转变。演讲者强调，虽然我们构建的内容和构建方式已经改变，但真正有趣的是我们对构建过程的思考方式发生了怎样的变化。作为一名在软件开发、设计和系统思维交叉领域工作的专家，演讲者分享了与社区成员关于AI对工作影响的讨论，包括如何与AI协作、哪些技能在当今最重要，以及如何保持相关性等核心问题。

演讲者通过历史回顾展示了开发者所需的核心软技能——批判性思维、协作、沟通和精确性——在每个时代都至关重要，并且在AI时代依然不可或缺。会议重点介绍了两种开发方法：Vibe Coding（快速原型开发）和 Spec-Driven Development（规范驱动开发），并通过 Amazon Kira IDE 的实际演示，展示了如何在AI辅助开发中应用规范驱动的方法来确保系统的可维护性和长期发展。

演讲最后鼓励开发者通过实验来适应新工具和策略，强调设计意图的重要性，以及将AI视为协作伙伴而非简单的指令执行者的新思维模式。

## 详细时间线

开场介绍 (0:00-2:30)
- 演讲者 Andakat Najiro 介绍自己在软件开发、设计和系统思维交叉领域的工作背景
- 提出开发者面临的核心问题：如何与AI协作、哪些技能重要、如何保持相关性
- 向观众分发特制的打孔卡笔记本作为礼物

打孔卡的历史意义 (2:30-3:30)
- 解释打孔卡的概念：一张带孔的打孔卡相当于现代GitHub仓库中80个字符的一行代码
- 介绍笔记本的设计选择既是历史回顾也是实用工具

软件开发历史回顾 - ENIAC时代 (3:30-5:00)
- 展示第一台通用电子计算机ENIAC的编程方式：手动设置开关和插拔电缆
- 强调当时需要大量规划、文档记录和系统化的调试方法
- 团队协作和项目目标对齐的重要性
- ENIAC没有内存，信息通过打孔卡输入和存储

打孔卡编程流程 (5:00-6:30)
- 打孔卡上的孔代表数字信息的存在
- 使用机器手动打孔：一张卡片=一行代码，一叠卡片=一个程序
- 将卡片输入机器后需要等待，有时需要排队
- 发现错误可能需要数小时，然后需要检查设计、文档，找到错误的卡片

汇编语言时代 (6:30-7:30)
- 程序员开始使用符号而非电缆，但仍然非常接近机器层面
- 开发者需要处理寄存器、内存操作和指令序列
- 仍然使用打孔卡输入信息，但程序现在加载到逻辑内存中执行

高级编程语言出现 (7:30-9:00)
- Fortran和COBOL等首批高级编程语言诞生
- 金融行业至今仍在使用这些语言编写的系统
- 开发者开始更多地处理数学公式和业务逻辑，而非单个指令
- 演讲者的父母在大学学习Fortran，使用代码表设计代码，然后由机器自动打孔
- 整个过程仍然耗时，需要排队、团队协作和调试

Pascal及后续语言 (9:00-10:30)
- 演讲者父母从Fortran转向Pascal编程
- Pascal是演讲者在家中看到的第一种编程语言
- 应用程序规模和生命周期增长，编程成为更加集体化的密集过程
- 团队采用版本控制、模块化设计和代码审查等新实践

敏捷方法论的兴起 (10:30-12:00)
- 互联网普及推动了远程在线协作
- Scrum在90年代出现（现场几乎所有人都使用过Scrum）
- 2001年多种类似方法在敏捷宣言下正式化
- 敏捷强调：持续适应、短迭代、持续反馈、用户参与、增量交付
- 敏捷将人际交往技能置于软件实践的中心，重视个人互动胜过流程和工具

跨时代的核心软技能 (12:00-13:30)
- 批判性思维、协作、沟通和精确性在每个时代都至关重要
- 以精确性为例：从手动编程到传递卡片、编写指令，现在与AI系统交互时同样需要精确
- 强调需要警惕AI生成的"看起来正确但实际错误"的代码

AI辅助开发时代 (13:30-15:00)
- 进入自然语言编程时代，使用提示词（prompts）
- 介绍社区库promptz.dev
- Vibe Coding流程：选择AI助手、使用提示词、点击提交、等待结果
- 现场几乎所有人都尝试过Vibe Coding
- Vibe Coding优势：快速获得第一版本和原型
- Vibe Coding劣势：实现新功能和修改时变得耗时

Spec-Driven Development介绍 (15:00-16:30)
- 规范驱动开发采用传统方法：设计系统、构建规范、明确意图
- 规范清晰后再进行编码
- 在软件开发生命周期中的应用：从需求和设计开始，意图明确后进入开发、测试和部署
- 在开发阶段可以与AI系统来回交互，应用Vibe Coding或提示工程技能

两种方法的对比 (16:30-17:30)
- Vibe Coding适合原型开发，可以快速做决策
- 如果决定推进产品，应先实现规范，设计意图，然后再前进
- 演讲者与客户讨论AI时的常见问题：如何深入理解用AI构建的系统以确保可维护性

提供更好的上下文 (17:30-19:00)
- 引用Anthropic网站的对比图：提示工程vs上下文工程
- 更好的上下文包括：团队最佳实践、公司最佳实践、类似项目样本、使用和不使用的语言、使用和不使用的工具
- 将这些上下文提供给AI，基于上下文推进系统开发

Amazon Kira IDE演示 (19:00-25:00)
- Kira是Visual Studio Code的分支
- 右侧有与Kira交互的新界面
- 两种模式：Vibe Coding模式（探索、测试和原型）和Spec-Driven Development模式（先构建规范）
- Hero工具：specs agent、hooks agent、steerings和MCP服务器
- Agent steering文件提供更好的上下文：GitHub提交消息指南、代码行数限制等
- 可以添加MCP服务器和AWS标准工具

Tic-Tac-Toe游戏演示 (25:00-28:00)
- 使用规范驱动开发构建井字棋游戏
- 第一步：Kira构建需求（演示中接受了所有建议，实际应该审查和调整）
- 第二步：设计阶段（审查设计后继续）
- 第三步：定义任务（白色为必需任务，灰色为可选任务，测试任务为可选，演讲者将一个设为必需）
- 开始执行任务循环

任务执行过程 (28:00-29:30)
- 第一个任务"设置项目结构和依赖项"耗时最长
- 遇到权限问题、缺少安装等问题需要来回解决
- 第一个任务验证后，其他任务执行较快
- 在任务4.1（测试任务）停止

测试和调试 (29:30-30:30)
- 任务4.1是任务4的测试任务
- 运行测试时出现大量错误，代码不符合要求
- 经过4-5次来回：修复代码、执行测试、失败、再修复
- 最终测试通过，游戏运行成功

Spec-Driven Development的优势 (30:30-32:00)
- 游戏成功运行，演示效果良好
- 规范驱动开发不仅限于简单游戏
- 修改规范很容易：只需修改规范中的相关部分，然后让AI修改相关代码
- 对比Vibe Coding：在聊天中请求修改，可能超出上下文窗口，变得耗时，难以追踪规范演变

互动环节 (32:00-33:30)
- 邀请观众拿出笔记本和笔
- 笔记本中有一个问题：写下未来6个月有意培养的一项软技能或硬技能
- 建议设置6个月后的日历提醒，检查进展

总结 (33:30-35:00)
- 工作在演变，有新的策略和工具，但一切始于实验
- 目前可能没有完美的工作流程，但会逐步形成（就像当年开发者不信任编译器一样）
- 核心要点：设计你的意图，明确想要实现的目标，引导机器
- 这更像是协作，而不仅仅是告诉机器执行什么指令

结束 (35:00)
- 演讲结束，观众鼓掌
