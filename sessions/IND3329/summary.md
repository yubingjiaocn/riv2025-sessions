# AWS re:Invent 2025 会议总结：Cox Automotive 的 AI Agent 生产化之路

## 一、会议概述

本次 AWS re:Invent 2025 分会场由 AWS 高级解决方案架构师 Ravi、Cox Automotive 企业架构副总裁 Brian Lloyd Newberry (BLN) 以及首席架构师 Tabari Goen 共同主讲。会议聚焦于一个核心挑战：如何将 AI Agent 从原型阶段推向生产环境并实现规模化运营。

会议开场时的现场调查揭示了行业现状：许多组织正在试验 AI Agent，但真正将其部署到生产环境并大规模运营的企业寥寥无几。Cox Automotive 作为汽车行业的技术领导者，分享了他们在短短五周内成功将五个 AI Agent 产品推向市场的实战经验。他们的案例展示了如何利用 Amazon Bedrock Agent Core 和 Strands 框架，克服从原型到生产的关键障碍。

Cox Automotive 拥有海量的汽车行业数据（5.1 万亿条车辆洞察、数亿次客户交互），已在生产环境中运行超过 150 个 AI 模型。他们从 2023 年全面投入生成式 AI，在一年内推出了三个主要产品，其中一个产品每年节省 75 万美元成本，另一个产品使经销商的客户响应率提高了 50%。本次会议详细剖析了他们如何在 2024 年 8 月接受挑战，在劳动节前（仅五周时间）推出五个全新的 Agent 产品，以及背后的技术架构、安全策略和实施方法论。

## 二、详细时间线与关键要点

### **00:00 - 03:30 | 会议开场与挑战定义**
- **00:00** - Ravi 介绍演讲嘉宾，进行现场调查：有多少人在试验 AI Agent？有多少人已将其部署到生产环境？
- **01:15** - 揭示核心问题：从原型到生产的巨大落差
- **01:45** - 会议议程介绍：Amazon Bedrock Agent Core 简介、Cox Auto 的 Agent 之旅、参考实现与经验教训
- **02:30** - 定义 AI Agent：能够推理、决策、使用工具并自主执行目标的系统

### **03:30 - 08:00 | Agent 生产化的五大挑战**
- **03:30** - 挑战一：运行时（Runtime） - 需要可扩展的基础设施来运行长时间工作流，避免性能开销和成本超支
- **04:15** - 挑战二：内存（Memory） - Agent 需要跨会话维护上下文，进行有意义的对话
- **04:45** - 挑战三：安全（Security） - 需要安全访问 Agent、工具和数据
- **05:15** - 挑战四：集成（Integration） - Agent 需要通过工具发现现有系统，并与其他 Agent 无缝通信
- **05:45** - 挑战五：可观测性（Observability） - 由于 Agent 的非确定性特性，需要完整的执行可见性以进行追踪、调试和评估

### **08:00 - 12:30 | 服务技师助手用例详解**
- **08:00** - 实际场景：客户带着引擎故障灯来到维修店
- **08:45** - Agent 工作流程：分析仪表盘照片、引擎噪音、客户描述
- **09:30** - 知识源整合：OEM 手册、技术服务公告、召回通知
- **10:15** - 动态适应：根据技师发现调整诊断建议
- **11:00** - 零件推荐：检查库存、定价、考虑客户偏好（OEM 优质零件 vs 预算意识）
- **11:45** - 跨系统操作：在传统供应商系统中下单
- **12:15** - 会话持久性：维修可能持续数小时，客户数周后返回时 Agent 需记住所有信息

### **12:30 - 16:00 | Amazon Bedrock Agent Core 解决方案**
- **12:30** - Agent Core 五大核心服务介绍
- **13:00** - Runtime：安全、可扩展、无服务器，支持多模态输入（最大 100MB），长达 8 小时的会话
- **13:45** - Memory：短期记忆存储对话状态，长期记忆学习客户交互；内置策略（用户偏好、语义事实、摘要）
- **14:30** - Identity：安全认证和凭证管理，支持 OAuth 和 IAM
- **15:00** - Gateway：将现有 API 和 Lambda 函数转换为 Agent 就绪的 MCP 工具，提供语义搜索
- **15:30** - Observability：完整的 Agent 路径可见性，CloudWatch 仪表板，支持 OpenTelemetry 集成

### **16:00 - 18:30 | 特殊用途工具**
- **16:00** - Browser 工具：使 Agent 能够执行复杂的 Web 自动化任务
- **16:45** - Code Interpreter：允许 Agent 在安全沙箱中运行任何语言的临时复杂计算
- **17:30** - BLN 接手演讲，介绍 Cox Automotive

- **20:00** - 经销商软件套件：ERP、CRM、库存管理、金融应用
- **20:45** - 数据规模：5.1 万亿条车辆洞察，数亿次客户交互
- **21:30** - AI 能力：生产环境中运行 150+ 个 AI 模型
- **22:15** - 技术历程：2018 年全面采用 AWS，从 50 个数据中心迁移到 3 个加 AWS 区域

### **23:00 - 28:00 | Cox Auto 的生成式 AI 之旅**
- **23:00** - 2022 年 11 月 30 日：ChatGPT 发布引发关注
- **23:45** - 2023 年 3 月：全面投入生成式 AI，一个月内访问多个模型提供商
- **24:30** - 2023 年成果：推出三个主要产品，一个产品年节省 75 万美元
- **25:15** - 客户响应率提高 50% 的经销商产品
- **26:00** - 2024 年从 re:Invent 返回后启动 Agent 计划

### **28:00 - 32:00 | "疯狂实验"：五周五产品挑战**
- **28:00** - 2024 年 7 月第二周末：CPO 提出劳动节前推出五个产品的目标
- **28:45** - 起点：五个团队，大多数从未构建过 Agent 应用
- **29:30** - 空白页问题：不知从何开始
- **30:15** - 加速三要素：设定目标、设定截止日期、激发兴奋感
- **31:00** - 召集五个团队，举办大型启动活动
- **31:45** - 资产盘点：AWS 深度合作、Bedrock 早期客户、小型内部专家团队

### **32:00 - 36:30 | 技术决策与架构**
- **32:00** - 参考架构白板：识别已解决的问题（灰色）和待解决的问题（白色）
- **33:00** - 需要解决：成本管理、模型监控、编排、Agent 工作流、工具、API 管理
- **33:45** - 关键决策：采用 Agent Core（当时尚未正式发布）
- **34:30** - 风险承担：首次在市场产品中使用第一代技术
- **35:15** - 框架选择：Strands（AWS 支持的开源框架）
- **36:00** - 专注策略：五个团队使用同一框架，而非分散学习

### **36:30 - 40:00 | Strands 框架简介**
- **36:30** - Agent 构建的核心要素：编写提示词（文本）、配置、连接模型、附加工具
- **37:30** - Strands 简化开发：只需编写 Python 配置
- **38:15** - 代码示例：导入 Agent、导入计算器工具、提出问题
- **39:00** - LLM 数学问题解决方式：依赖确定性工具
- **39:45** - 核心信息：专注于功能特性，而非框架和基础设施

### **40:00 - 44:00 | Tabari 接手：产品团队视角**
- **40:00** - Tabari Goen 自我介绍，Cox Automotive 首席架构师
- **40:45** - 2022-2023 年：LLM 进入主流，团队开始探索应用场景
- **41:30** - 2023 年底：启动首个生成式 AI POC
- **42:15** - 2024 年初：推出 Predictive Insights（人在回路的消息生成器）
- **43:00** - 产品成功：经销商喜爱一键生成个性化客户消息
- **43:45** - 新挑战：超过一半的潜在客户在下班后到达，无人点击按钮

### **44:00 - 48:00 | 全自动虚拟助手架构**
- **44:00** - 核心问题：如何构建用户信任的自动化 AI 解决方案？
- **44:45** - 需求：了解经销商声音和品牌，全天候服务，无需人工监督
- **45:30** - 架构概览：基于 Agent Core 和 Strands 的虚拟助手
- **46:15** - 编排器（Orchestrator）：理解客户意图，路由到子 Agent
- **47:00** - 子 Agent：销售、服务等专门领域，独立处理对话
- **47:45** - 工作流程：编排器汇总结果，生成回复，持续对话

### **48:00 - 52:00 | 基础设施：Agent Core 的价值**
- **48:00** - 对话助手的关键需求：数据隔离
- **48:45** - AI 框架快速变化的现实
- **49:30** - Agent Core 的优势：会话数据隔离、框架可切换性
- **50:15** - 实际案例：从 Agent Squad 迁移到 Strands
- **51:00** - 迁移时间：通常需要数周，实际只用了两周
- **51:45** - 建议：从可演进的基础开始，专注于领域问题

### **52:00 - 56:30 | 安全策略：红队测试**
- **52:00** - 红队测试定义：主动尝试让系统偏离轨道
- **52:45** - 测试方法：外语提问、不可读字符、诱导泄露系统提示词
- **53:30** - 红队 vs 测试：测试检查什么有效，红队尝试破坏系统
- **54:15** - 实施时机：Alpha 前、Beta 前、持续进行
- **55:00** - 流程：记录每个漏洞、修复、迭代
- **55:45** - 每次代码部署和提示词更改后都进行红队测试
- **56:15** - 价值：向利益相关者展示系统如何失败及应对措施

### **56:30 - 60:30 | 护栏策略：硬护栏与软护栏**
- **56:30** - Bedrock Guardrails 介绍
- **57:15** - 客户服务场景：仅阻止会导致糟糕的客户体验
- **57:45** - 硬护栏（Hard Guardrails）：完全阻止系统，不与 LLM 交互，返回"无法帮助"
- **58:30** - 配置方式：Bedrock Guardrails
- **59:00** - 软护栏（Soft Guardrails）：使用 LLM，通过工作流或提示词设计引导对话
- **59:45** - 实际案例：客户询问低信用贷款，助手回应"这是财务团队的好问题，让我安排预约"
- **60:15** - 效果：在保持安全的同时继续提供帮助

### **60:30 - 65:00 | 自动化评估**
- **60:30** - 场景回顾：Agent Core、红队测试、护栏配置完成
- **61:15** - 问题：传统测试无法应对 LLM 的概率性特性
- **61:45** - 手动审查的局限：适用于数十或数百次对话，无法扩展到每日数万次交易
- **62:30** - LLM as a Judge：使用另一个 LLM 评判响应是否正确
- **63:15** - 流程：生成测试对话 → 运行系统 → 评估结果 → 跟踪趋势
- **64:00** - Cox Auto 的评估指标：相关性、完整性、语气
- **64:45** - 建议：找到对业务最重要的指标

### **65:00 - 68:30 | 断路器机制**
- **65:00** - 最后一个问题：LLM 可能做出意外行为，导致巨额账单
- **65:45** - 断路器（Circuit Breakers）：在系统失控前停止运行的硬限制
- **66:30** - Cox Auto 的两个硬限制：成本限制和轮次限制
- **67:15** - 成本限制示例：对话达到 P95 成本阈值时停止
- **67:45** - 轮次限制示例：对话达到 20 轮时停止
- **68:00** - 失败处理：将对话移交给经销商人员
- **68:30** - 建议：从第一天就设置限制，了解边界，优雅失败

### **68:30 - 71:00 | 项目成果与总结**
- **68:30** - 当前状态：产品处于 Beta 阶段
- **69:00** - 经销商反馈：客户在白天和下班后都能获得所需答案
- **69:30** - 计划：2025 年 Q1 初正式发布
- **70:00** - 核心建议：从正确的基础（Agent Core）开始，专注于重要的模式
- **70:45** - 最终目标：创建用户信任的自动化 AI

### **71:00 - 74:00 | BLN 总结：五个项目的最终结果**
- **71:00** - 现场互动：有多少人认为只有一个项目成功？有多少人认为全部成功？
- **72:00** - 实际结果：五个项目中有三个应用进入生产 Beta 阶段
- **72:45** - 第一个全面上线：Tabari 介绍的虚拟助手
- **73:15** - 第二个产品：库存定价自动化 Agent（代替人工完成定价工作）
- **73:45** - 价值转变：从提醒人们做工作到"我已为你完成工作"

会议核心要点总结：
1. 使用 Agent Core 作为可演进的基础设施
2. 选择一个框架并专注（推荐 Strands）
3. 持续进行红队测试以发现漏洞
4. 实施硬护栏和软护栏的组合策略
5. 自动化评估以应对 LLM 的概率性
6. 设置断路器以防止成本失控
7. 快速行动、设定截止日期、激发团队兴奋感
