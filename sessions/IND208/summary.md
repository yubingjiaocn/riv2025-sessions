# AWS re:Invent 2025 会议总结：利用 AI 减轻临床医生文档负担

## 会议概述

本次 AWS re:Invent 2025 分组会议重点探讨了如何通过 AWS 与 Veradigm 的合作，利用人工智能技术改善临床医生的工作体验。会议由 AWS 医疗保健团队产品负责人 Himanshu Jooshi 主持，Veradigm 的软件工程师 Ken Shepard 和 AWS 医疗保健 AI 团队的 Mira 共同参与。

会议核心议题围绕临床医生面临的严重文档负担问题展开。研究表明，临床医生每花 1 小时与患者交流，就需要花费 2 小时处理临床文档和行政任务，这导致了严重的职业倦怠。AWS 提出的解决方案是通过三层架构来解决这一问题：安全的医疗 AI 数据基础（使用 AWS HealthLake 等服务）、专门构建的医疗保健智能工具和代理（如 AWS HealthScribe）、以及无缝集成到临床工作流程中的用户体验。

Veradigm 的 Practice Fusion 电子健康记录系统成功集成了 AWS HealthScribe，为小型独立门诊提供商提供环境 AI 临床文档服务。用户反馈显示，该解决方案每天可为医生节省约 2 小时时间，65% 的用户无需培训即可使用，并显著提升了医患互动质量。会议还展示了 2025 年 HealthScribe 的新功能，包括患者上下文集成、专用 AI 模型、新的笔记模板以及支持 18 种以上专科。

## 详细时间线与关键要点

### 开场与问题陈述 (00:00 - 05:30)

00:00 - 00:45 - 会议开场
- Himanshu Jooshi 介绍自己是 AWS 医疗保健团队的产品负责人
- 团队专注于为医疗保健和生命科学客户构建专用服务

00:45 - 02:15 - 核心问题阐述
- 临床医生希望有更多时间面对患者和家属，而不是处理文档工作
- 介绍 AWS 与 Veradigm 的合作，通过自主 AI 临床记录和安全数据基础改变护理点体验

02:15 - 04:00 - 临床医生负担详解
- 临床医生被大量文书工作淹没
- 每 1 小时患者护理时间对应 2 小时的临床文档和行政任务
- 导致职业倦怠和"睡衣时间"（下班后仍需工作）

04:00 - 05:30 - 行政任务类型
- 就诊前：审查大量 PDF 文档，核对药物清单
- 就诊中：在与患者交谈时做速记
- 就诊后：填写事先授权表格，与医疗和计费团队沟通

### AWS 解决方案架构 (05:30 - 10:00)

05:30 - 07:00 - 三层解决方案架构
- 第一层：AI 就绪的数据基础，提供完整的患者上下文
- 第二层：医疗保健专用代理，针对特定任务构建
- 第三层：在护理点提供无缝的用户体验

07:00 - 08:30 - 实施挑战
- 患者上下文分散在多个系统和 PDF 中
- 多说话者、多语言、代码切换（如英语-西班牙语混合）
- 需要 AI 护栏确保安全性和可信度
- 需要无缝集成到临床医生工作流程

08:30 - 10:00 - AWS 服务栈
- 数据基础：HealthLake、Health Imaging、Health Omics
- 智能工具：AWS HealthScribe、Amazon Bedrock、Agent Core
- 通过 API 和 SDK 集成到 EHR 和医疗应用

### AWS HealthScribe 深度解析 (10:00 - 16:30)

10:00 - 11:30 - HealthScribe 核心功能
- 从患者-临床医生对话生成临床笔记的生成式 AI 服务
- 临床医生点击麦克风开始记录，就诊结束时自动生成笔记

11:30 - 13:00 - 三大关键特性
- 实时临床文档：生成实时转录，就诊结束后几秒内生成笔记
- 内置 AI 安全护栏：每个笔记都基于源数据，可追溯到原始对话
- 护理点 SDK：可轻松集成到任何应用或 EHR

13:00 - 16:30 - 2025 年新功能
- **患者上下文集成**：可将 EHR 中的人口统计、过敏等信息无缝整合到笔记中
- **专用 AI 模型**：针对护理点临床对话进行微调，提供精确的笔记生成控制
- **五种新笔记模板**：包括 SOAP、GIRP 等，支持行为健康专科
- **任务集成**：自动生成患者友好的就诊摘要
- **18+ 专科支持**：为不同专科（如心脏病学、皮肤科）定制笔记格式

### Veradigm 实施案例 (16:30 - 30:00)

16:30 - 18:00 - Veradigm 公司介绍
- Ken Shepard 介绍 Veradigm 的三个业务团队
- 提供商团队：EHR、实践管理、计费、患者参与系统
- 付款方团队：与医疗保险公司合作
- 生命科学团队：提供真实世界数据和证据

18:00 - 20:00 - Practice Fusion EHR 特点
- 针对小型独立门诊提供商（1-3 名医生）
- 100% 运行在 AWS 上
- "5 分钟上线"理念：任何人可在 5 分钟内开始使用
- 每月 500 万次患者就诊，每年 6000 万次

20:00 - 22:30 - 用户需求验证
- 用户调查显示环境技术是第一大功能需求
- 60% 的用户报告正在使用某种形式的环境技术
- 这是 10 年来首次医生主动要求新技术

22:30 - 25:00 - 选择 AWS HealthScribe 的原因
- 环境临床记录需要成为 EHR 的有机组成部分
- 需要可信赖的合作伙伴提供基础，然后在此基础上创新
- 采用混合构建-购买-合作策略
- 生成式 AI 成本高，需要实际部署才能建立经济模型

25:00 - 27:00 - 快速上市优势
- 单个 API 调用即可实现
- 小团队即可构建安全可靠的解决方案
- 基于平台使用 API 构建，加快发布速度
- 每两周发布一次，AWS 每周提供更新

27:00 - 30:00 - 负责任的 AI
- 医疗保健需要谨慎处理患者安全问题
- AWS 提供的负责任 AI 框架加速了 80% 的合规工作
- HealthScribe 提供源归因，转录与笔记并排显示
- 建立临床医生对技术的信任

### 实施成果 (30:00 - 35:00)

30:00 - 32:00 - 量化成果
- 每位医生每天节省约 2 小时
- 65% 的用户无需任何培训即可使用
- 4.6/5 的评分：减少了与患者在一起时的记录时间

32:00 - 33:30 - 笔记偏好洞察
- 对笔记完整性的满意度高
- 用户偏好多样：项目符号 vs 叙述、简短 vs 详细
- 代词使用偏好：性别、EHR 存储的代词、"他们"或"患者"

33:30 - 35:00 - 用户反馈引用
- "Scribe 让我能在看完患者后立即完成 95% 的笔记"
- "我不再需要输入 HPI 部分和计划，Scribe 捕捉得很好"
- 目标是实现零编辑体验

38:00 - 40:00 - 分析能力开发

- 测量编辑次数、编辑时间、具体更改类型
- 近实时反馈，用于风险缓解和安全保障
- 目标：实现零编辑

40:00 - 42:00 - 深度集成与新功能
- 更深入地集成到 EHR 中
- 逐步向所有用户开放（目前限制访问以便迭代）

### AWS HealthLake 数据基础 (42:00 - 结束)

42:00 - 45:00 - 数据孤岛挑战
- Mira 介绍数据孤岛问题
- 数据分散在 500+ 个表中
- 多种格式：CCDA、CSV、关系型数据库
- 本体论复杂性：ICD-10 代码、CPT 代码、单位值等

45:00 - 48:00 - 非结构化数据挑战
- 大部分数据嵌入在笔记中
- 需要挖掘数百或数千页的历史笔记
- 本体论不够清晰，难以提取信息

48:00 - 结束 - FHIR 通用数据模型
- 采用 FHIR（Fast Healthcare Interoperability Resources）作为通用数据模型
- 开源社区驱动的标准
- RESTful API 架构，标准化数据访问
- 引用式而非关系型数据结构
- 标准化数据持久化和暴露方式
