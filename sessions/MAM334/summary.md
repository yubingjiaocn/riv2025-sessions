# AWS re:invent 2025 - MAM334 会议总结

## 会议概述

本次会议主题为"如何利用 Agentic AI 能力大规模推进应用现代化"（MAM334）。会议由 AWS 企业解决方案架构师 Pratib Bakshi、Thomson Reuters 的 AI 解决方案架构师 Lalit Kumar 以及 AWS 微软工作负载技术负责人 Yogi Barot 共同主讲。

会议重点探讨了企业在将传统 .NET Framework 应用迁移到 .NET Core 平台过程中面临的挑战，以及如何通过 AWS Transform for .NET 这一 AI 驱动的工具来加速现代化进程。Thomson Reuters 作为实际案例，分享了他们如何利用该工具成功转型超过 400 个 .NET 应用程序，涉及超过 5 亿行代码。会议强调了现代化带来的三大核心价值：释放创新潜力、降低技术风险、减少总拥有成本（TCO）最多可达 40%。

通过采用 Agentic AI 技术，企业可以将传统需要数月甚至数年的现代化项目缩短至数周完成，同时保证代码质量、安全性和性能。会议还宣布了 AWS Transform 的多项新功能，包括多用户协作、支持更多 AWS 区域、以及与 Okta 和 Microsoft Entra ID 的集成等。

## 详细时间线

### 开场与介绍
[00:00 - 02:30] - 会议开场，主持人进行暖场互动，询问参会者是否首次参加 re:invent，并建议穿舒适的鞋子。Pratib Bakshi 介绍会议主题和演讲嘉宾。

[02:30 - 04:15] - 通过三个问题进行现场互动：询问参会者是否参与过现代化项目、是否有 AI 采用和降低 TCO 的目标、是否需要用更少资源做更多事情。带领观众做伸展运动活跃气氛。

### 现代化挑战与动机
[04:15 - 06:45] - Pratib 定义应用现代化的范围：将 .NET Framework 应用迁移到 .NET Core 平台，重点关注三个成果：释放创新潜力、降低技术风险、减少总拥有成本。

[06:45 - 09:20] - 讨论传统现代化面临的三大挑战：
- 速度慢：传统方法需要数年时间
- 流程顺序化：团队需等待每个步骤完成才能继续
- 技术老旧：难以找到熟悉旧技术的开发人员

[09:20 - 12:30] - 阐述现代化的技术动机：
- 性能提升：.NET Core 比 .NET Framework 4.7 快 600 倍
- 更好的支持和安全性
- 跨平台支持：可部署到 Linux 容器和 Graviton 实例
- 更大的开发者人才库

[12:30 - 14:00] - 商业动机：
- 节省高达 40% 的运营成本
- 消除昂贵的商业许可证
- 利用轻量级容器降低存储和计算成本
- 通过云的规模和性能支持业务增长

### 传统升级流程
[14:00 - 16:30] - 介绍传统的 .NET 升级流程：发现 → 分析 → 移植 → 验证 → 部署。这个过程劳动密集、容易出错，对地理分布的团队存在协作瓶颈。

### Thomson Reuters 案例分享
[16:30 - 18:45] - Lalit Kumar 介绍 Thomson Reuters 的背景：作为全球领先的内容驱动 AI 技术公司，已完成 95% 的云迁移，但仍有大量遗留 .NET 代码需要现代化。

[18:45 - 21:30] - Thomson Reuters 的 .NET 应用规模：
- 超过 400 个运行在 Windows 服务器上的 .NET 应用
- 超过 5 亿行代码
- 根据 2024 年云消费报告，迁移到 Linux 和 Graviton 可节省数百万美元

[21:30 - 24:00] - 展示复杂的依赖关系图，说明手动分析这些深度嵌套的依赖关系非常困难且容易出错。许多遗留 API 已被弃用或在 .NET Core 中行为不同。

### AWS Transform for .NET 介绍
[24:00 - 26:15] - 选择 AWS Transform for .NET 的三个原因：可靠性、智能性和速度。从 Beta 阶段开始合作，从单个解决方案转型发展到大规模 Web 体验转型。

[26:15 - 28:45] - 展示 Web 体验的作业计划功能：
- AI 代理通过简单的聊天体验创建端到端作业计划
- 统一的 Web 体验提供集中管理和跨团队协作
- 可定义团队成员角色（审批者、管理员、协作者）
- 提供详细的工作日志和转型摘要仪表板

[28:45 - 32:00] - 代码转换示例演示：
- 示例 1：将 XML 格式的 web.config 文件转换为基于代码的 program.cs 文件，保持功能不变但增强了 cookie 安全性
- 示例 2：将配置拆分为 appsettings.json 和 program.cs 两个文件，实现更现代和默认安全的代码

[32:00 - 34:30] - 讨论转型后的挑战：
- AI 无法完成 100% 的工作
- 需要维护与外部或第三方依赖的兼容性
- 某些组件用 C++ 编写，或存在不支持的库（如 Windows Win32 DLL、VB.NET）
- 需要人工参与解决这些挑战

[34:30 - 36:00] - 如何弥合差距：AWS Transform 将转换后的代码提交到 GitHub 新分支并创建 PR，团队在本地使用 Amazon Q Developer 等 AI 辅助工具完成剩余的转换工作。

### AWS Transform 深度解析
[36:00 - 38:30] - Yogi Barot 介绍 AWS Transform 的三大优势：
- 加速转型速度：比传统方法快 4 倍
- 成本节省：Windows 许可证成本比 Linux 高 40%，且工具本身免费
- 质量保证：使用 Amazon Bedrock 大语言模型和深度 .NET 专业知识

[38:30 - 40:15] - 支持大规模并行现代化：Thomson Reuters 同时运行约 10 个 .NET 现代化项目。

[40:15 - 42:45] - 两种体验模式：
- Web 体验：基于 React 的 SPA 前端，支持 GitHub、Bitbucket、GitLab、Azure DevOps
- IDE 体验：Visual Studio 集成，提供并排代码对比

[42:45 - 44:00] - 重大新功能发布（11:00 AM 生效）：
- 多用户协作支持
- 支持不同角色（IT 管理员、开发人员、DBA、项目经理）
- 合作伙伴可在同一工作区协作

[44:00 - 46:30] - IDE 体验特点：
- 并排代码对比，智能高亮显示差异
- 自动生成文档并链接到 .NET 官方文档
- 适合开发人员进行单个应用的移植

[46:30 - 48:00] - 使用场景建议：
- 开发人员或单应用移植：使用 IDE 体验
- DevOps 或大规模扫描：使用 Web 体验
- 可先用 Web 体验扫描所有应用，创建商业案例，再用 IDE 体验进行具体移植

### 工作原理详解
[48:00 - 50:30] - 转换流程：应用进入三态循环（分析 → 转换 → 验证），直到找到最佳方案。包括：
- 将 web.config 转换为 program.cs
- 使用数千个预定义代码转换模式
- 执行单元测试
- 需要人工审批（仓库、项目选择、部署）

[50:30 - 54:00] - 五个深度转换阶段：
1. 发现阶段：确定代码行数、应用类型（Windows Forms、Web Forms、WPF、MVC）
2. 分析阶段：依赖关系映射（内部和外部），支持 250+ NuGet 包，生成详细评估报告
3. 规划阶段：创建转型策略、时间线、资源需求，需管理员审批
4. 执行阶段：自动代码转换，使用预训练的转换模式，执行单元测试
5. 验证阶段：性能矩阵评估（确保性能相同或更好），生成转型摘要报告和 Linux 就绪报告

[54:00 - 55:30] - Linux 就绪报告：确保移植后的代码可在 Linux 上运行，列出缺失项并提供修复步骤。

### 安全性与合规性
[55:30 - 58:30] - 区域扩展公告：AWS Transform 现已支持 6 个新区域，包括加拿大中部。

[58:30 - 61:30] - 安全架构：
- 仅支持 IAM 认证用户
- TLS 1.3 加密连接
- AWS WAF 处理速率限制和请求过滤
- 临时处理：IDE 体验在本地工作，Web 体验在 VPC 中创建临时 EC2 实例，转换后销毁
- 使用 Amazon S3 临时存储，通过 AWS KMS 加密（支持自定义密钥）
- **新功能**：现支持 Okta 和 Microsoft Entra ID 认证
- AWS CloudTrail 记录所有 API 和审计日志
- Amazon CloudWatch 提供详细监控

[61:30 - 63:00] - 新功能总结：
- 可从失败的作业重新启动并继续
- 多团队协作，支持多种角色
- CloudWatch 日志现在对用户开放，提高透明度

[63:00 - 64:30] - 支持的源版本：
- .NET Framework 3.5 及以上
