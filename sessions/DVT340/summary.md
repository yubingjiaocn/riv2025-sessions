# AWS re:Invent 2025 - Kuro Steering 功能详解

## 会议概述

本次 AWS re:Invent 2025 分会场深入探讨了 Kuro（AWS 的 AI 驱动 IDE）中最关键的功能——Steering（引导文档）。演讲者 Ally Fortman（AWS 下一代开发者体验专家）和 Matt Lever（伦敦高级专家）共同阐述了为什么 Steering 是充分发挥 Kuro 价值的最重要概念。

会议强调，虽然大型语言模型（LLM）在编写代码方面表现出色，但它们缺乏组织特定的领域知识、编码标准和业务上下文。Steering 文档通过为 LLM 提供项目特定的上下文和指导原则，解决了代码质量感知不佳、控制力有限以及 AI 助手在复杂任务中表现不稳定的问题。演讲者将 Steering 比作驾驶时双手握住方向盘——如果不主动引导，AI 代理就会按照自己的方式行事，可能偏离开发者的意图。

会议还介绍了 Steering 文档的两大类型：基础文档（由 Kuro 自动生成，包括产品、技术栈和结构文档）和自定义文档（包含组织特定的标准、日志规范、测试框架等）。通过正确使用 Steering 文档，团队可以实现一致的代码生成、减少重复说明、提升团队协作，并建立可扩展的项目知识库。

## 详细时间线与要点

### **开场与背景介绍 (00:00 - 05:30)**
- **00:00** - Ally Fortman 开场，强调这是 re:Invent 第四天，表达对讲解 Kuro Steering 的兴奋
- **01:15** - 介绍 Kuro：AWS 的 AI 驱动集成开发环境（IDE）
- **02:00** - AI 在 IDE 中的演进历程：
  - 2023 年：内联代码补全
  - 2024 年：单一任务代理（如 /test 命令）
  - 2025 年：配对编程式的代理协作开发
- **03:45** - 代理开发的三大承诺：自主性、真正的协作、高质量输出

### **AI 开发面临的挑战 (05:30 - 08:00)**
- **05:30** - 现场互动：询问观众是否遇到过 LLM 行为难以理解的情况
- **06:00** - 三大核心挑战：
  1. **扩展性问题**：LLM 在简单任务表现良好，但在复杂任务或大型代码库中容易失败
  2. **控制力有限**：AI 可能陷入失败-修复循环，开发者失去控制权
  3. **代码质量感知**：输出结果未达到预期标准，因为缺乏必要上下文

### **为什么 Steering 至关重要 (08:00 - 11:30)**
- **08:15** - Steering 的核心比喻：就像驾驶时双手握住方向盘，保持车辆在车道内
- **09:00** - LLM 的三大局限性：
  - **无状态性**：只有当前提示、部分代码和聊天历史，没有长期记忆
  - **歧义处理能力弱**：人类擅长消除歧义，但 LLM 会直接猜测并执行
  - **缺乏领域知识**：不了解组织的编码、测试、安全标准和日志规范

### **Steering 的实现方式 (11:30 - 15:00)**
- **11:45** - 两种 Steering 类型：
  1. **基础文档**：Kuro 一键生成，基于现有代码库
  2. **自定义文档**：包含组织特定的领域知识
- **13:00** - 基础文档包含三个文件：
  - **Product.md**：项目的业务目标、目标用户
  - **Tech.md**：技术栈、框架、库、工具
  - **Structure.md**：文件组织、命名约定、架构模式
- **14:30** - 使用 Steering 后的效果：LLM 获得"记忆库"，提供更相关的响应

### **Steering 的长期价值 (15:00 - 17:00)**
- **15:15** - 四大核心优势：
  1. **一致的代码生成**：相同问题得到一致答案
  2. **减少重复**：无需反复提醒 LLM 遵循规范
  3. **团队协作**：整个团队使用相同的 Steering 文件
  4. **可扩展的项目知识**：Steering 文档随代码库动态演进

### **Matt Lever 演示与最佳实践 (17:00 - 35:00)**
- **17:30** - Matt Lever 接手，演示如何生成基础 Steering 文档
- **18:00** - 演示：在 Kuro 菜单中点击"生成 Steering 文档"
- **19:00** - Steering 文档作为"记忆库"的概念：存储项目描述、技术栈、架构模式
- **20:00** - 文档位置：可以是项目本地或全局（用户目录中的 .kuro/steering）

### **工作区文档 vs 全局文档 (22:00 - 26:00)**
- **22:15** - 工作区文档适用场景：
  - 项目特定的领域知识
  - 团队特定的工作规范
  - 特定的部署方法（如容器 vs 无服务器）
- **24:00** - 全局文档适用场景：
  - 架构师定义的组织级标准
  - 安全标准
  - 通用最佳实践（TDD、BDD、事件驱动架构）
  - 新应用的蓝图模板

### **自定义 Steering 文档最佳实践 (26:00 - 32:00)**
- **26:30** - 按领域组织：避免单一庞大文档，按主题拆分（如日志标准、测试规范）
- **27:30** - 日志标准示例：统一日期时间格式、关联 ID、结构化日志
- **28:45** - 提供示例：用具体代码示例展示"好的做法"和"应避免的做法"
- **30:00** - 管理上下文窗口：
  - 优先考虑质量而非数量
  - 建议每次任务使用 3-5 个文档
  - 根据任务类型轮换文档（如前端工作时不加载后端 Steering）

### **内容编写指南 (32:00 - 36:00)**
- **32:15** - 不要教 LLM 已知的内容："不要教奶奶吸鸡蛋"原则
  - LLM 已经知道：基本语法、标准库、REST 概念
  - 应该提供：业务特定逻辑、设计决策理由、自定义 API 约定
- **34:00** - 使用具体示例：代码示例帮助 LLM 理解具体期望
- **35:00** - Steering 文档即代码：像对待代码一样维护，进行代码审查，防止文档腐化

### **高级功能：MCP 集成 (36:00 - 40:00)**
- **36:30** - 使用模型上下文协议（MCP）连接外部数据源
- **37:15** - 场景：从集中式 Git 仓库动态获取架构指南、日志库标准
- **38:00** - 两种方法：
  1. 在 Steering 文档中引用 MCP 工具
  2. 使用相对文件路径引用相邻目录中的共享库

### **文档创建技巧 (40:00 - 42:00)**
- **40:15** - 使用 Kuro 编写 Steering 文档：让 LLM 帮助生成初始文档
- **41:00** - 从小处开始，记录痛点和领域知识
- **41:30** - 不要试图记录所有内容，专注于关键的领域特定信息

### **包含模式 (42:00 - 45:00)**
- **42:30** - 三种包含模式：
  1. **Always（默认）**：所有聊天交互中都包含
  2. **文件特定**：使用通配符匹配特定文件类型（如 .rs、.ts）
  3. **Manual（手动）**：按需手动激活，适用于偶尔的任务（如数据转换）

### **Kuro Powers 新功能 (45:00 - 47:00)**
- **45:15** - 昨天发布的新功能：Kuro Powers
- **45:45** - 预打包的知识包，包含：
  - MCP 服务器
  - Steering 文件
  - Hooks（钩子）
- **46:30** - 合作伙伴已发布 Powers：Figma、Stripe、Datadog
- **47:00** - 组织可以创建自定义 Powers 在团队内分发

### **客户案例引入 (47:00 - 48:30)**
- **47:30** - Matt 介绍 Nielsen Sports 的 Chandra 分享实际应用经验
- **48:00** - Chandra 开始分享从大型机时代到现代 AI 辅助开发的演进历程
