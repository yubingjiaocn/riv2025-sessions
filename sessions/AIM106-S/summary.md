# AWS re:Invent 2025 Zscaler 零信任与 AI 安全会议总结

## 会议概述

本次会议由 Zscaler 主讲，重点介绍了零信任架构在 AI 时代的演进以及与 AWS 的深度集成。主讲人分享了 Zscaler 在零信任安全领域 14 年的发展历程，从最初专注于用户到互联网的安全网关，逐步扩展到零信任网络访问（ZTNA）、云工作负载保护、物理站点安全，以及最新的 AI 基础设施安全。

会议特别邀请了 Amazon Robotics 的高级技术项目经理 Matthew Foreman 分享实际部署案例。Amazon Robotics 在 2023 年底完成了企业级零信任转型的第一阶段，将 454 个站点和 158,000 个应用程序迁移到零信任架构。Matthew 详细介绍了他们如何使用 Zscaler Private Access (ZPA) 作为核心解决方案，实现了无 VPN、无 Active Directory 依赖的安全架构，同时提升了安全性并降低了用户使用摩擦。

会议强调了零信任不仅仅是 ZTNA 用例，而是一个全面的架构理念，涵盖用户访问、工作负载间通信、多云环境、以及新兴的 AI 代理安全。Zscaler 将自己定位为流量源和目标之间的"策略引擎"或"交换机"，通过单一上下文和策略框架实现全面的零信任安全。

### 架构转型：从传统到零信任 (15:30-18:00)
- **15:30** - 传统架构问题：回程流量、复杂网络、冗余安全控制、管理复杂性
- **16:00** - Zscaler 简化模型：作为策略引擎，提供多种形态因子（客户端、物理设备、虚拟形态）
- **16:30** - 带上正确的上下文（用户、工作负载、IoT 设备）构建策略
- **17:00** - 单一上下文、单一策略，与业务需求绑定
- **17:30** - 用户零信任：支持任何设备、任何位置、非托管设备、第三方承包商

### 用户零信任详解 (18:00-21:00)
- **18:00** - 保护到互联网和 SaaS 应用（Office 365、Google Apps）的流量
- **18:30** - 保护企业自有应用（数据中心或 AWS 中），应用不暴露在互联网上
- **19:00** - 核心原则：IP 命名空间在互联网上"隐身"，这是 Private Access 服务的作用
- **19:30** - 安全不应以用户体验为代价：全球 165 个接入点（PoP）
- **20:00** - 可扩展到私有或混合部署
- **20:30** - 数字体验监控：识别终端设备、网络 ISP、应用性能问题并提供根因分析

### 成本节约与技术整合 (21:00-24:00)
- **21:00** - 采用新技术需要淘汰旧技术以节省成本
- **21:30** - Zscaler 如同"云中的 DMZ"，取代大量入站/出站 DMZ
- **22:00** - 替代传统网络架构，采用零信任架构
- **22:30** - 分段需求：网络分段、用户分段、应用分段
- **23:00** - Zscaler 将分段作为所有产品的基础策略构造，无需单独的 VLAN 或 IP 分段工具
- **23:30** - 微分段部署困难：大型银行（30 万用户、2000 人安全团队）难以大规模部署微分段

### AI 驱动的智能分段 (24:00-27:00)
- **24:00** - 引入智能分段或 AI 策略概念（几年前推出）
- **24:30** - 学习行为：了解谁访问什么，特别是在客户命名空间内
- **25:00** - 案例：机架下运行 10 年的 Windows 服务器，不知道谁在访问，不敢拔掉或阻止
- **25:30** - Zscaler 理解访问模式，提出基线策略建议，可立即实施
- **26:00** - 客户可从 0% 分段立即达到 40-50% 分段
- **26:30** - 深入分段：识别部门内特定用户访问特定端口的应用，创建细粒度分段策略
- **27:00** - 通过 SCIM 推送用户组到目录，实现细粒度分段；客户可达到 80-90% 分段率

### Amazon Robotics 业务背景 (30:00-33:00)

- **30:30** - 业务目标：创建安全、高效、可持续的工作场所（安全包括信息安全）

### 遗留应用挑战 (33:00-36:00)
- **33:00** - 企业工程完成了主要工作，但存在 80/20 规则：总有 20% 的尾部问题
- **33:30** - 遗留应用定义：需要用户和应用之间私有网络访问的应用
- **34:00** - 可能是服务、服务器、文件存储、SQL 服务器等
- **34:30** - 许多是商业现成（COTS）应用，无法自行现代化，只能向供应商请求

### Amazon Robotics 零信任四大原则 (36:00-39:00)
- **36:00** - 原则 1：消除企业网络
- **36:30** - 原则 2：移除所有 Active Directory 依赖
- **37:00** - 原则 3：无 VPN，所有内容迁移到原生 AWS 云（本地裸机服务器、本地虚拟化服务器、或现有 EC2 资源）
- **37:30** - 原则 4：提高安全态势，最小权限访问，自动化用户配置和基线
- **38:00** - 保护使用本地不安全协议的应用
- **38:30** - 在每个级别进行分段：不仅分段应用，还分段基础设施
- **39:00** - 每个应用都是独立的 AWS 账户，无东西向流量，实施管理层面的最小权限

### Secure Application Link (SAL) 架构 (39:00-42:00)
- **39:00** - 内部称为"Secure Application Link"（SAL），简单命名约定
- **39:30** - 最初是"Securely Link Users to Applications"（SLOOA），但 SAL 更顺口
- **40:00** - 利用 Zscaler 零信任交换作为交换机，用户连接到应用而非网络
- **40:30** - 在零信任交换中对每个事务进行策略执行
- **41:00** - ZPA 提供分段、加密、时间限制、最小权限、态势验证、MFA 合规的连接
- **41:30** - 如果在玩"信息安全宾果游戏"，现在应该赢了（幽默）

### ZPA 部署详情 (42:00-45:00)
- **42:00** - Amazon Robotics 是"ZPA 优先"客户，这不是典型的 Zscaler 部署模式
- **42:30** - 大多数客户从 ZIA 开始或 ZIA+ZPA 组合，但 Robotics IT 提供差异化服务，不需要 ZIA
- **43:00** - Zscaler 团队提供了仅许可所需服务的途径
- **43:30** - 与 IDP 原生集成，打包 Zscaler Client Connector 为一键安装，预配置环境变量
- **44:00** - 这些都是 Zscaler 开箱即用的功能
- **44:30** - 在 AWS 云中部署应用连接器（App Connectors），以 AMI 形式在 AWS Marketplace 提供

### 高可用性与可扩展性 (45:00-48:00)
- **45:00** - Zscaler 承担大部分更新和维护负担
- **45:30** - 以高可用方式部署在跨多个可用区的自动扩展组中，根据应用需求可能跨多区域
- **46:00** - 用户无法直接访问应用，应用无需入站连接
- **46:30** - 所有连接都是从 VPC 内部向外的
- **47:00** - ZPA 使 Amazon Robotics 能够提供可扩展的 SAL 解决方案
- **47:30** - "第一天"文化鼓励为工程师构建高效解决方案，为 Amazon 创造长期价值

### 零信任分支应用场景 (51:00-54:00)
- **51:00** - 本地原型实验室和制造设施使用 3D 打印机、CNC 机器、增材/减材制造设备

- **52:00** - 需要担心数据管理和原型文件访问权限
- **52:30** - 通过部署零信任分支，可以在本地仅网络区域操作所有设备
- **53:00** - 同时获得在线可用的所有好处，通过零信任交换交换机访问
- **53:30** - 这是 Zscaler 帮助专注客户并减少工作流程摩擦的另一个例子
- **54:00** - 通常添加安全会增加摩擦，但这里做到了相反：增加安全性同时减少摩擦

### 云工作负载零信任 (59:30-63:00)
- **59:30** - Davo 感谢 Matthew 的精彩分享，讨厌分析师创造的缩写词
- **60:00** - 讨论云工作负载零信任，这是非常有趣的领域
- **60:30** - 5-6 年前开始涉足这一领域，目标是保护所有云中的所有工作负载（AWS、Azure、GCP）
- **61:00** - 虽然有相似构造，但不完全相同：区域、可用区、不同连接形式（VPC 对等、Direct Connect）
- **61:30** - 持久工作负载（EC2）和非基础设施工作负载（Lambda 函数）都需要保护
- **62:00** - 三大定位：
  1. 保护工作负载到互联网访问（数千客户使用）
  2. 工作负载到工作负载分段
  3. 多云安全
- **62:30** - 传统做法是部署南北向防火墙或虚拟防火墙

### 工作负载安全挑战 (63:00-66:00)
- **63:00** - 防火墙只查看会话，不解密流量，除了维护会话外没有真正的安全性
- **63:30** - 部署更多虚拟防火墙会变得更加复杂
- **64:00** - 案例：United Airlines 等大型客户在不部署任何防火墙的情况下进行 PB 级出站流量
- **64:30** - 工作负载到工作负载分段变得有趣，因为工作负载类似用户（访问互联网或相互通信）
- **65:00** - 但与用户不同，工作负载没有持久身份，身份成为挑战
- **65:30** - 许多客户尝试微分段，在工作流或网络流级别分段，但过于复杂
- **66:00** - 成功客户的做法：缩小范围，定义宏分段边界

### 宏分段与微分段 (66:00-69:00)
- **66:00** - 宏分段示例：预生产暂存 VPC 不应暴露在任何地方，只对生产 VPC 可见，在互联网上隐身
- **66:30** - 数据中心中与云工作负载通信的特定子网不应在其他地方广播
- **67:00** - 先实施宏分段用例，然后在其上分层微分段
- **67:30** - 目标是使用一致的策略框架实现这一点
- **68:00** - 多云安全视角：第一个用例是混合云（物理站点/数据中心连接到应用）
- **68:30** - 可以提供非常具体的策略：数据中心中的特定 IP 子网访问特定 VPC，构建策略和访问

### 多云与跨区域部署 (69:00-72:00)
- **69:00** - 第二个用例：多云（有不同变体）
- **69:30** - 长期与 AWS 标签和 Macy 标签集成，通过 API 连接到租户，消费标签作为策略标准
- **70:00** - 示例：标记为 SAP 的任何内容不能与某些应用通信
- **70:30** - 标签成为工作负载的身份，还读取 VPC 名称作为控制台中的位置，围绕它构建策略
- **71:00** - 大规模客户的复杂性：跨区域、多区域部署
- **71:30** - 多云是现实：主要云是 AWS，但收购在 Google 运行 BigQuery 的小公司，需要连接应用
- **72:00** - 区域内场景：受监管行业（医疗保健、金融）需要在 VPC 内运行的工作负载级别实施 PCI 特定策略

### 全面的安全能力 (72:00-75:00)
- **72:00** - 甚至基于网络流的入站与出站策略
- **72:30** - 从更广泛的角度：提供分段框架、完全解密
- **73:00** - 不需要担心仅进行基于域的过滤（大多数客户使用防火墙的做法）
- **73:30** - 可以提供完全 SSL 解密、数据安全、工作负载 DLP、完全沙箱、高级威胁防护
- **74:00** - 如果流量保持在 AWS 边界内，不会将流量带出 AWS
- **74:30** - 实际上在 AWS 内扩展 Zscaler 足迹，策略执行也在 AWS 内部进行
- **75:00** - 工作负载级别需要构建的特定规则集

### 微分段服务 (75:00-78:00)
- **75:00** - 引入微分段服务，但它是在 ZPA 或 ZTNA 服务之上分层的
- **75:30** - 客户已经在使用应用分段，微分段作为专门用例分层，无需从头重建
- **76:00** - 获得网络流的完全可见性：进出工作负载的流量
