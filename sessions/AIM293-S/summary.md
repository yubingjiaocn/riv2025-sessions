# AWS re:Invent 2025 会议总结：用 Agentic AI 重新构想软件开发和 DevOps

## 会议概述

本次分享由 GitHub 的 Cameron Stewart 主讲，主题聚焦于如何利用 Agentic AI（智能代理 AI）重新定义软件开发和 DevOps 实践。演讲首先回顾了 DevOps 的本质——它不仅仅是工具和技术栈，而是一种文化理念和哲学，旨在通过统一团队、流程和技术来持续为最终用户交付价值。DevOps 的核心是优化整个系统的流程，消除瓶颈，而非仅关注单个环节。

随着生成式 AI 工具的兴起，开发者能够以前所未有的速度生成代码。然而，Stewart 强调，即使在 AI 可能生成 90% 代码的未来，开发者仍然不可或缺——他们需要专注于那关键的 10%，以及验证、测试、安全审查等环节。演讲详细介绍了 GitHub Copilot 从代码补全助手演进到智能编码代理的过程，展示了如何通过 Copilot Coding Agent 将整个任务（如实现新功能、修复 bug、增加测试覆盖率）委托给 AI 异步完成。GitHub 内部已将 Copilot Coding Agent 作为核心代码库的头号贡献者，有效解决了技术债务和积压任务。

## 详细时间线与关键要点

### 开场与 DevOps 回顾 (0:00 - 5:30)
- **0:00** - 演讲开始，Cameron Stewart 介绍会议主题：用 Agentic AI 重新构想软件开发和 DevOps
- **0:45** - 议程概览：DevOps 回顾、当前工具技术、开发者赋能、支持整个 SDLC 的其他工具
- **1:15** - 强调 DevOps 的本质：DevOps 不是工具、不是 YAML 文件或管道，而是一种理念和哲学
- **2:00** - DevOps 起源：应对开发、运维、安全、QA 等团队各自为政、激励机制不一致导致的功能失调
- **3:00** - DevOps 借鉴制造业原则（如《凤凰项目》和精益思想），强调优化整个系统而非单个组件
- **4:00** - DevOps 的定义：人员、流程和技术的结合，持续为最终用户交付价值
- **4:45** - CI/CD、IaC、GitOps 等技术是为支持 DevOps 文化而产生的，而非 DevOps 本身

### AI 时代的开发者角色 (5:30 - 10:00)
- **5:30** - 引用前 GitHub CEO Thomas Dohmke 的观点：开发者永远不会消失，即使 AI 生成 90% 的代码
- **6:15** - 强调基础实践的重要性：小批量 PR、代码检查工具、代码扫描、代码审查在 AI 时代更加关键
- **7:00** - 即使在 AI 之前，开发者也只花费少量时间编写代码，大部分时间用于调试、测试、协调等
- **8:00** - AI 加速代码生成后，验证和确认（verification and validation）成为新的关注焦点
- **8:45** - 比喻：从拥有汽车进化到拥有 F1 引擎，但需要配套的刹车、转向、安全带等支持系统
- **9:30** - 如果只加速代码生成而不优化其他环节，会在流程中产生瓶颈，压力会传导到整个系统

### 生成式 AI 工具的演进 (10:00 - 17:00)
- **10:00** - 从变更范围、交互频率、内循环/外循环、开发者画布等维度分析 AI 工具演进
- **10:45** - 2022 年 GitHub Copilot 发布（比 ChatGPT 早一年），提供代码补全和对话式聊天
- **11:30** - Copilot 最初定位为"AI 结对编程伙伴"，实时完成代码片段，响应时间以毫秒计
- **12:30** - 演进到对话式聊天和多文件编辑，响应时间扩展到秒级，仍聚焦于 IDE 内的内循环
- **13:30** - 进一步演进到可以处理完整任务的代理：创建新功能、新分支、提交 PR，响应时间达到分钟级
- **14:30** - 强调这些能力不仅限于 GitHub Copilot，市场上有多种类似工具提供代码补全、对话和代理功能

### Copilot Coding Agent 介绍 (17:00 - 22:00)
- **17:00** - 介绍 Copilot Coding Agent：可以将整个 GitHub Issue 分配给它，自动解析并完成工作
- **17:45** - 支持的任务类型：增加测试覆盖率、编写单元测试、实现 UI 暗黑模式等
- **18:30** - 集成范围扩展：从 GitHub Issues 到 Azure DevOps Boards、Linear，即将支持 Jira 和 Slack
- **19:15** - 异步工作模式：可以同时分配 5 个任务给 Coding Agent，然后去开会、吃饭或休息，回来后查看 5 个草稿 PR
- **20:00** - 虽然不是 100% 完美，但即使需要修改，也能节省大量时间，有时甚至一次性完成
- **21:00** - 从"辅助"到"代理"的转变：从"AI 结对编程伙伴"演进为"同级程序员"，可以像分配给初级开发者一样分配任务

### 现场演示：Copilot Coding Agent (22:00 - 28:00)
- **22:00** - 演示开始：展示一个物流应用的 GitHub 仓库，包含多个待处理 Issue
- **22:45** - 选择 Issue："帮助中心页面导航无法重定向"，包含问题描述和截图
- **23:30** - 将 Issue 分配给 Copilot，可选择目标仓库和自定义代理
- **24:00** - Copilot 开始工作：创建草稿 PR，承诺保持 PR 描述更新，预计耗时约 10 分钟
- **24:30** - 切换到 VS Code 演示另一个场景：创建购物车页面
- **25:15** - 介绍 VS Code 中的不同模式：Ask、Edit、Agent、Plan，Agent 模式最常用
- **26:00** - 展示自定义代理列表：API Specialist、TDD Green、TDD Red 等，定义在 .github 目录中
- **27:00** - Copilot 在 Agent 模式下工作：检查仓库结构、创建购物车上下文、更新路由等，跨多个文件进行修改

### Copilot Coding Agent 会话详情 (28:00 - 32:00)
- **28:00** - 查看 Copilot Coding Agent 的工作会话：检出代码、设置 Node、启动 GitHub 和 Playwright MCP 服务器
- **28:45** - MCP（Model Context Protocol）服务器：默认加载 GitHub 和 Playwright，可自定义添加其他 MCP 服务器
- **29:30** - Copilot 的工作流程：探索仓库结构、定位相关文件、检查组件、理解构建和测试设置
- **30:30** - 问题诊断：发现帮助中心链接使用的是普通 href 而非 React Router Link
- **31:00** - 制定计划：更新 Footer 组件使用 React Router Link、运行项目、构建成功
- **31:45** - 对比两种代理体验：IDE 内的 Agent 模式（需要等待）vs Coding Agent（完全异步委托）

### GitHub 内部使用案例 (32:00 - 36:00)
- **32:00** - 回到演示文稿，介绍 GitHub 内部真实使用场景
- **32:30** - GitHub 内部广泛使用自己的平台和工具，包括 Copilot Coding Agent
- **33:00** - 2025 年 3 月数据：Copilot Coding Agent 是团队中最高产的成员之一
- **33:30** - 不仅创建新功能，还能处理技术债务——这是传统开发中常被忽视的领域
- **34:00** - 技术债务 vs 新功能的永恒战争：新功能总是优先，技术债务积压不断增长
- **34:45** - Copilot 能够处理开发者不愿意做的枯燥任务，帮助清理积压
- **35:00** - 展示 GitHub 核心仓库的贡献者统计：Copilot Coding Agent 是第一大贡献者
- **35:30** - Copilot Code Review Agent 排名第三，仅运行几个月就产生巨大影响
- **36:00** - 现场展示 GitHub 内部仓库的真实数据，证明这不是营销宣传而是实际使用情况

### 上下文和自定义指令的重要性 (36:00 - 42:00)
- **36:00** - 转向讨论支持 AI 工具的其他基础设施：上下文是一切的关键
- **36:45** - 上下文的演进：从提示词优化到 RAG、MCP 等工具引入外部系统的上下文
- **37:30** - 最佳实践：自定义指令（Custom Instructions）
- **38:00** - 自定义指令的层级：顶层 copilot-instructions.md 文件适用于整个仓库
- **38:45** - 更精细的控制：针对特定语言或框架的指令文件（如 CSS 文件 vs Playwright 测试）
- **39:30** - 自定义指令适用于多种场景：IDE 中的 Copilot、Agent 模式、Coding Agent、Code Review
- **40:00** - 用途：定义编码标准、合规指南、代码格式要求等
- **40:45** - 强烈建议每个仓库都使用自定义指令文件
- **41:15** - 演示自定义指令文件：API 指令文件，指定适用的目录和文件类型、审查指导、API 原则、检查清单
- **41:45** - 可以排除特定场景：某些指令只给 Code Review 看，某些只给 Coding Agent 看

### 自定义代理演示 (42:00 - 结束)
- **42:00** - 展示自定义代理示例：可访问性代理（Accessibility Agent）
- **42:30** - 自定义代理定义在 .github/agents 目录中，可以针对特定任务创建专门的代理
- **43:00** - 演示继续进行中，展示如何使用自定义代理处理特定类型的任务
