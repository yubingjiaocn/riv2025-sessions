# AWS re:Invent 2025 - Amazon Bedrock Agent Core 与 Dynatrace 可观测性集成

## 会议概述

本次会议由 Dynatrace 开发者体验团队负责人 Sean Odell 和 AWS 首席解决方案架构师 Simone 共同主持,重点介绍了 Amazon Bedrock Agent Core 服务以及如何通过 Dynatrace 实现智能体的端到端可观测性。会议强调了将 AI 智能体从概念验证阶段推进到生产环境所面临的挑战,以及 AWS 和 Dynatrace 如何通过合作解决这些问题。

Bedrock Agent Core 是 AWS 推出的一套服务,旨在帮助企业大规模运行智能体工作负载。该服务提供了无服务器运行时、身份管理、工具网关、内存管理、浏览器访问和代码解释器等核心能力。然而,仅有基础设施支持还不够,企业还需要强大的可观测性能力来监控智能体的状态、性能和行为。Dynatrace 通过集成 Bedrock Agent Core 的可观测性功能,利用 OpenTelemetry 标准提供跨平台、跨模型的统一监控视图,帮助开发者在 IDE 中直接获取洞察,减少复杂性,提升开发体验。

03:30 - 构建智能体应用的挑战
- 概念验证阶段构建智能体相对容易
- 将智能体扩展到生产环境非常困难
- 智能体需要访问多个工具和其他智能体
- 管理基础设施和应用层的复杂性是主要障碍

04:45 - Gartner 预测
- 到 2027 年,超过 40% 的智能体 AI 项目将被取消
- 原因是从概念验证到生产环境的转换极具挑战性

05:15 - 智能体生产化的六大挑战
1. 需要专用的基础设施和运行时(智能体运行时间不确定,可能持续数分钟到数小时)
2. LLM 是无状态的,需要内存能力来记录会话和历史交互
3. 智能体需要访问第三方工具和内部工具(数据库、Google Drive、存储系统等),需要身份管理
4. 智能体需要发现和管理工具,大规模连接和管理工具具有挑战性
5. 智能体可能需要高级功能,如浏览器访问或代码生成执行
6. 需要可观测性来了解智能体的行为、推理步骤、成功和失败情况

07:30 - Bedrock Agent Core 解决方案介绍
- 提供三类服务:部署、增强和监控智能体

08:00 - 部署智能体:运行时和身份
- **运行时**:提供代码,包括框架和模型(可以是任何框架,AWS 或非 AWS 的模型)
- Agent Core 在无服务器运行时上自动扩展运行
- **身份管理**:为智能体提供身份认证,支持与各种服务交互,包括 SigV4、OAuth 和 API 密钥

09:15 - 增强智能体功能
- **Agent Core Gateway**:中央 MCP 服务器,智能体可以看到所有可用工具(称为 targets)
- Targets 包括 API、数据库、外部工具(Google Drive、Salesforce、Slack)
- **Agent Core Memory**:短期和长期内存
  - 短期内存:记录电商购物车中的产品、用户信息、配送地址等
  - 长期内存:记录历史交互,实现个性化体验

11:00 - 高级功能
- **Agent Core Browser**:完全托管的浏览器,支持数十到数百个并行隔离会话
- **Agent Core Code Interpreter**:LLM 擅长生成代码但不擅长确定性计算,代码解释器执行生成的代码(如 Python 计算器)获得确定性结果

12:30 - 监控智能体:可观测性
- **Bedrock Agent Core Observability**:监控智能体状态和性能
- 提供基础可观测性仪表板
- 暴露指标、日志、跨度、追踪和会话数据
- Dynatrace 使用这些数据提供高级可观测性

13:45 - Sean 介绍 Dynatrace 的价值
- 在可观测性领域有 8-9 年经验
- 每次新技术出现,挑战都是如何获取信息
- AWS 构建 Agent Core 时考虑了数据导出的灵活性
- 支持 CloudWatch,也支持通过 OpenTelemetry 导出到其他平台

15:00 - Dynatrace 的差异化价值
- 企业使用的不仅仅是 Bedrock Agent Core
- Dynatrace 提供端到端可观测性,覆盖云原生服务、AI 智能体甚至大型机应用
- 目标是在所有场景下提供可见性

16:30 - 95% AI 试点项目失败的统计
- Sean 对这一统计数据持保留态度,认为可能是历史数据
- Agent Core 提供的智能能力有助于提高成功率
- 使用 OpenTelemetry 等开源解决方案可以更好地消费信息
- 强调这是第 0 天的对话,而非第 2 天(从开发初期就考虑可观测性)

18:00 - 开发者体验优先

- Dynatrace 希望在开发者所在的位置提供服务
- 提到与 AWS 合作发布的 Kiro
- 目标是在 IDE 中直接提供可观测性

19:15 - 多模型支持
- 组织通常使用多个 LLM 和小语言模型
- Dynatrace 不仅支持 Bedrock Agent Core,还支持第三方(如 Nvidia、Anthropic、OpenAI)
- 提供统一的可观测性视图

20:30 - 两种集成方式

- OpenLLMetry(专门用于 LLM 可观测性)

21:00 - 演示 Dynatrace 仪表板

- 提供详细的元数据和属性
- 支持在 IDE 中使用自然语言搜索

23:00 - 跨平台支持
- 不仅支持 Agent Core,还支持其他解决方案(如 ServiceNow)
- 以统一、可预测的方式提供可观测性信息

24:00 - 体验至上的理念
- 行业面临的挑战是复杂性
- 开发者希望减少复杂性,保持工作流程
- 快速进出调试和故障排除模式是关键
- Agent Core 和 Bedrock 团队确保了可用性和体验

25:30 - 基础设施可观测性
- 所有应用都运行在基础设施上
- 开发者可能不喜欢与基础设施交互,但从可观测性角度必须关注
- 端到端可观测性不仅包括 LLM 和 AI,还包括日志、追踪、指标和基础设施信息
- 根据场景和问题,提供不同层次的信息
- 没有单一的起点,取决于具体情况
