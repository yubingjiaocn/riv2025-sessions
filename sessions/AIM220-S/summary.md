# AWS re:Invent 2025 会议总结：AI驱动的自主可观测性代理

## 一、会议概述

本次AWS re:Invent 2025分会场由Coralogix公司的联合创始人兼CEO Ariel和AI负责人Luran共同主讲，主题聚焦于AI在可观测性领域的革命性应用。演讲者指出，当前软件系统的复杂性呈指数级增长，传统的仪表板和告警方式已经无法有效应对海量遥测数据带来的挑战。过去8年间，遥测数据增长了100倍，但可观测性并未相应提升，反而导致工程师将近一半的时间花费在问题分类上。

Coralogix推出了名为"Ollie"的全球首个自主可观测性AI代理，旨在从根本上改变可观测性范式。Ollie不是简单的AI助手或副驾驶工具，而是一个能够自主进行深度调查、关联数百万数据点、识别根本原因并提供修复建议的智能代理系统。演讲通过实时演示展示了Ollie如何在一分钟内完成通常需要数小时的故障排查工作，将问题解决时间缩短94%。会议还深入探讨了构建高质量AI代理系统的核心要素，包括代理架构、知识管理和评估体系，为与会者提供了构建自己AI代理的实用指导。

## 二、详细时间线与关键要点

### **开场与背景介绍 (0:00-3:30)**
- **0:00** - 会议开始，演讲者幽默地提到AI是本次大会"几乎没有讨论"的话题
- **0:30** - 介绍Coralogix公司背景：成立于2015年，每天处理超过300 PB数据，融资4亿美元
- **1:15** - 获得AWS EMEA年度合作伙伴奖，在8个区域运行包括联邦云
- **2:00** - 引出核心问题：AI在可观测性领域的最大价值不仅是让工程师工作更轻松，更是提升软件交付速度、稳定性和安全性

### **软件复杂性挑战 (3:30-8:00)**
- **3:30** - 展示现代微服务架构的复杂服务地图，指出大规模系统几乎不可能被单个工程师完全理解
- **4:45** - 对比15-17年前的单体架构时代：当时只需查看日志和少量指标即可理解系统
- **5:30** - 现代系统面临的挑战：微服务、多云部署、无服务器架构导致理解系统变得极其困难
- **6:15** - 关键数据：过去8年遥测数据增长了100倍，但并未带来更好的可观测性
- **7:00** - 批判"万能仪表板"思维：更多仪表板和告警并不等于更好的可观测性

### **当前可观测性困境 (8:00-12:30)**
- **8:00** - 展示典型客户现状：平均拥有数百个仪表板和告警，有些客户甚至有10,000-20,000个告警
- **9:00** - 92%的仪表板仅使用一周就被废弃，但每次事后分析的结论仍是"添加更多仪表板和告警"
- **10:15** - 工程师将近**50%的时间**花费在问题分类上，而非实际解决问题
- **11:00** - 大规模事故时所有服务同时触发告警，难以定位真正的根本原因
- **11:45** - 提出核心问题：数据增长速度超过了我们处理数据的能力

### **重新定义可观测性 (12:30-16:00)**
- **12:30** - 质疑"可观测性"这个术语：观察系统不是目标，保持系统可靠运行才是
- **13:30** - Coralogix在过去6年投入超过2.5亿美元研发，但仍难以跟上数据和系统演进速度
- **14:30** - 真正的目标应该是：可靠性、易管理性、快速解决问题、可衡量的性能、业务KPI
- **15:15** - 软件产生数百万甚至数万亿数据点，挑战在于如何将这些"面包屑"串联成完整故事

### **Ollie的诞生 (16:00-20:00)**
- **16:00** - Luran接手演讲，强调过去几年我们专注于"如何做更好的可观测性"而忘记了"为什么"
- **17:00** - AI新范式让我们能够重新思考可观测性，实现**10倍更好的可靠性**而非线性增长
- **18:00** - 介绍Ollie：**全球首个自主可观测性代理**
- **18:45** - 用Netflix观看《鱿鱼游戏》的例子说明：用户体验问题可能源于复杂软件栈中的任何环节

### **Ollie工作原理 (20:00-24:00)**
- **20:00** - Ollie的三大核心能力：
  1. 从数十亿数据点中聚焦到最相关的部分
  2. 消除噪音，专注于真正重要的信号
  3. 连接点成为完整故事，追溯到根本原因
- **22:00** - 演讲者表示"听起来好得不真实？是的。不。"引发笑声
- **22:30** - 宣布进行实时演示而非录制视频

### **实时演示：故障排查 (24:00-35:00)**
- **24:00** - 演示场景：扮演Sportify电商平台的平台负责人Adam
- **25:00** - 收到产品VP Sarah的消息：客户投诉激增，结账出现错误
- **26:00** - 在Slack中标记Ollie："嘿Ollie，能帮忙看看吗？"
- **26:30** - Ollie立即开始自主调查，实时展示调查过程
- **27:30** - 展示Ollie的工作方式：像团队中的另一位工程师，逐步查看日志、指标、追踪
- **29:00** - 一分钟内得出结果：
  - 发现过去2小时结账失败激增
  - 定位到购物车API服务的类型错误
  - 影响约2,000名用户
- **30:30** - Ollie提供证据支持：展示具体错误日志和Python堆栈跟踪
- **31:30** - 根本原因：代码第65-66行的bug，sum函数尝试将int和None相加
- **32:30** - Ollie提供建议修复方案和受影响用户分析图表
- **33:30** - 展示Ollie的完整调查过程：分析指标、运行查询、过滤日志等数十个步骤
- **34:00** - 关键数据：平均节省94%的调查时间，5小时的工作缩短到20分钟

### **Ollie的独特性 (35:00-42:00)**
- **35:00** - 强调这是真实环境的实时演示，不是模拟
- **36:00** - 澄清Ollie**不是什么**：
  1. 不是"可观测性的ChatGPT"
  2. 不是AI助手（不帮你构建仪表板）
  3. 不是MCP服务器
- **38:00** - Ollie的核心特征：
  - 具有上下文和知识
  - 完全自主决策，无预设剧本
  - 专注于实现可靠性而非平台效率
- **40:00** - AI助手vs AI代理的区别：助手让你在平台内效率提升20%，代理让组织可靠性提升10倍
- **41:00** - MCP只是暴露API给AI代理，没有知识、上下文或自主性

### **Ollie技术架构 (42:00-50:00)**
- **42:00** - Ollie的工作流程：数据 → 学习 → 知识库 → 专业代理团队
- **43:30** - Ollie不是单一代理，而是**专业代理团队**：
  - 日志探索器
  - 指标分析器
  - 安全分析师等
- **44:30** - 每个代理由**小型语言模型(SLM)**驱动：70亿-130亿参数，专注特定任务
- **45:00** - Ollie在首次连接时会学习客户的可观测性工作空间：应用、子系统、字段等

### **构建AI代理的五大核心组件 (50:00-58:00)**
- **50:00** - 介绍任何AI代理系统的五大核心组件
- **50:30** - 1. 模型(Model)：代理的引擎
- **51:00** - 2. 系统提示(System Prompt)：如"你是分析日志的专家"
- **52:00** - 3. 代理架构(Agent Architecture)：
  - 编排器模式：中央协调器分配任务
  - 交接模式：代理自主协作如传球
- **54:00** - 4. 知识(Knowledge)：如何为代理提供信息和工具
- **55:00** - 5. 评估(Evals)：最重要但最具挑战性的组件
- **56:00** - 强调：模型是商品化的，真正的差异在于后三个组件

### **上下文工程三角 (58:00-65:00)**
- **58:00** - 提出**上下文工程三角**概念：代理架构 + 知识 + 评估
- **59:00** - 深入探讨知识管理：
  - 挑战：无法将所有PB级数据推送给模型
  - 更多数据≠更好答案
  - 需要相关、结构化、针对LLM优化的数据
- **61:00** - 知识管理的三个关键：如何存储、如何结构化、如何查询
- **62:00** - 评估的重要性：传统软件有单元测试、集成测试，AI代理如何测试？
- **63:30** - 常见陷阱：修改系统提示修复一个用例，却破坏了之前工作的用例
- **64:30** - 强调评估在AI代理中**至关重要**，不可或缺

### **Ollie的技术实现 (65:00-结束)**
- **65:00** - 展示Ollie的内部架构图
- **65:30** - Ollie基于Coralogix强大的可观测性数据平台
- **66:30** - 再次强调：Ollie不是使用Coralogix的专家，而是实现可靠性的专家
- **67:00** - Ollie利用大规模数据平台的信号创建知识
- **67:45** - 演讲者表示：如果只能选择一个成功关键因素，那就是知识管理
- **68:00** - 会议结束（字幕在此处截断）

核心要点总结：
- 传统可观测性方法已无法应对现代系统复杂性
- Ollie通过AI代理实现自主故障排查，节省94%调查时间
- 构建高质量AI代理的关键是上下文工程三角：架构、知识、评估
- 目标不是更好的可观测性工具，而是10倍更好的系统可靠性
