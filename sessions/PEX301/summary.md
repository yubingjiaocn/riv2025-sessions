# AWS re:Invent 2025 保险承保中的代理式AI技术会议总结

## 会议概述

本次会议聚焦于金融服务和保险行业中的代理式AI（Agentic AI）应用，特别深入探讨了保险承保（Insurance Underwriting）这一具体用例。演讲者团队包括Bala Kapi（金融服务合作伙伴团队解决方案架构师）、Ayan Ray（生成式AI技术负责人）和Shinas Pendiala，他们共同展示了如何利用AWS的最新技术来转变传统的保险承保流程。

当前保险行业面临着严峻挑战：极端天气事件导致的经济损失逐年攀升，仅2025年初的洛杉矶火灾就造成了2500亿美元的损失。传统的承保流程高度依赖人工，承保人需要在15-20个不同系统中手动收集和分析数据，包括历史索赔记录、物业特征、天气风险等信息，这导致他们将大量时间花费在数据搜集而非核心的风险评估工作上。为应对这些挑战，保险公司正在采用四大策略：从邮政编码定价转向精细化定价、引入保险范围修改、重新评估高风险市场策略，以及提供风险缓解建议并给予保费折扣激励。

会议展示了一个基于多代理协作（Swarm设计模式）的AI解决方案，利用Amazon Bedrock Agent Core和Strands Agents框架构建。该方案通过多个自主AI代理（地理编码代理、天气风险代理、历史索赔代理、承保决策代理等）协同工作，自动从各个数据源收集信息并提供综合分析报告，大幅提升承保人的工作效率。演讲还包括了完整的架构演示、实时demo展示以及代码实现讲解，展现了如何用仅几行代码就能构建生产级的代理式AI应用。

### 开场与背景介绍
[00:00-02:30] - 会议开场，Bala Kapi介绍会议主题和演讲团队，说明将重点讨论保险承保用例以及如何转变承保人的工作流程

[02:30-04:45] - 展示极端天气事件造成的经济影响数据，强调这些事件已成为行业常态化的金融风险，特别提到2025年洛杉矶火灾造成2500亿美元损失

- 从邮政编码定价转向精细化定价（如同一街道不同建筑根据与水体距离差异定价）

- 重新评估并可能退出高风险市场

### 当前承保流程的挑战
[07:30-11:45] - 深入分析承保人的日常工作流程挑战：
- 需要验证物业是否属于批准的资产类别
- 查询历史损失和索赔记录（NAIC、CLUE报告等）
- 分析物业特征（建筑类型、屋顶年龄、消防栓距离等）
- 评估天气相关风险
- 需要在15-20个不同系统中手动收集和综合数据
- 大量时间浪费在数据搜集而非核心承保决策上

[11:45-13:30] - 介绍代理式AI解决方案如何改进流程：AI代理自动从批准的数据源收集信息，提供带引用来源的综合分析，让承保人有更多时间专注于核心专业工作

### AI代理技术深度讲解
[13:30-15:00] - Ayan Ray接棒，介绍AI代理的定义：能够推理、规划并采取行动以完成目标的自主软件系统

[15:00-17:30] - 详细讲解AI代理的核心组成要素（代理原语）：
- 大语言模型（推理、规划、工具调用）
- 数据（提供上下文）
- 工具（与外部系统交互）
- 记忆（跟踪对话历史）
- 系统提示（定义代理行为）
- 护栏（确保隐私、安全、合规）
- 可观测性（性能监控和改进）

[17:30-20:00] - 介绍Strands Agents开源Python SDK：
- 仅需4行代码即可构建首个AI代理
- 已有超过500万次下载
- 模型驱动而非复杂的确定性编排
- Amazon Q开发团队使用该框架，将开发周期从数月缩短至数周

[20:00-22:30] - 展示Strands Agents代码示例和特性：
- 支持Amazon Bedrock上的多种基础模型（默认使用Anthropic Claude Sonnet）
- 支持其他LLM提供商（Azure OpenAI、Google Gemini等）
- 内置工具集（知识库、记忆、网络搜索、API调用）
- 支持模型上下文协议（MCP）进行标准化工具访问
- 支持自定义工具创建
- 内置可观测性和OpenTelemetry支持

### Amazon Bedrock Agent Core服务介绍
[22:30-25:00] - 讨论将AI代理投入生产的常见挑战：
- 代理需求多样化（实时响应vs长时间运行）
- 计算资源配置困难
- 安全性和会话隔离要求
- 身份和访问管理复杂性
- MCP服务器构建和维护负担
- 透明度和可信度问题

[25:00-27:30] - 介绍Amazon Bedrock Agent Core服务套件：
- 消除构建和管理代理工作负载专用基础设施的繁重工作
- 支持任何框架、任何模型、任何协议

[27:30-30:00] - 详细介绍Agent Core Runtime：
- 安全的无服务器运行时，用于部署和运行AI代理
- 支持长时间运行工作负载（最长8小时）
- 真正的会话隔离（microVM级别，基于Firecracker技术）
- 支持大型负载（最高100MB）
- 快速启动时间

[30:00-32:00] - 介绍其他Agent Core服务：
- **Agent Core Identity**：为代理应用提供身份和访问管理，支持Okta、Microsoft Entra、AWS Cognito等
- **Agent Core Gateway**：轻松构建、部署、发现和连接AI工具，支持Lambda、OpenAPI、Smithy，一键集成Salesforce、Jira、Zendesk等SaaS应用
- **Agent Core Memory**：管理短期和长期对话记忆，提供智能化、个性化响应
- **Agent Core Observability**：监控、调试和可视化完整的代理追踪，包括中间输出、模型调用、推理行为等

[32:00-35:00] - 展示参考架构和Agent Core Runtime的差异化特性：
- microVM级别的真正会话隔离（非进程级）
- 使用Firecracker microVM技术实现快速启动
- 支持最长8小时的长时间运行并保留完整上下文
- 支持最高100MB的大型请求和响应负载
- 仅需几行代码即可将本地代理代码转换为云原生部署

[35:00-37:00] - 演示部署流程：
- 导入bedrock_agent_core包
- 初始化应用并定义入口点
- 代码、依赖和配置打包为ARM64兼容的Docker容器
- 推送至Amazon ECR
- Agent Core Runtime创建microVM和版本化端点

### 实时Demo演示
[37:00-39:00] - Shinas Pendiala接棒进行demo演示，介绍Swarm设计模式（多代理协作模式）：每个代理完全自主，共同完成特定任务

[39:00-41:30] - 解释代理的三个核心组件：
- 系统任务（通过系统提示表达）
- 工具集
- 基础模型（来自Amazon Bedrock）

[41:30-44:00] - 介绍demo架构中的多个代理：

- **天气代理**：从NOAA网站获取气候信息
- **历史索赔代理**：从数据源获取历史索赔数据
- **综合分析代理和承保代理**：汇总所有信息并提供建议

[44:00-46:30] - 强调"人在回路"（Human-in-the-loop）的重要性：AI代理是辅助工具，最终决策由人类承保人做出

[46:30-50:00] - 展示Streamlit应用界面，演示分步向导流程：
- 输入物业地址（休斯顿德克萨斯州的具体地址）
- 选择商业物业类型（零售店）
- 输入估计物业价值（80万美元）
- 选择评估选项（天气数据、地址验证、卫星图像分析、索赔历史等）

[50:00-55:00] - 展示代理执行过程的底层细节：
- **地理编码代理**：成功将地址转换为经纬度，进行位置分析（市区分类、应急响应基础设施、关键风险因素）

- 展示代理间的完美交接（handoff）机制

[55:00-58:00] - 天气风险评估代理执行：
- 调用NOAA网络服务REST API
- 分析历史天气事件（重大洪水、龙卷风等）

- 分析季节性风险模式
- 评估对保费结构的影响

[58:00-60:00] - 索赔分析代理执行：
- 从Amazon RDS获取历史索赔数据
- 分析该地区的索赔历史和风险影响
- 计算平均索赔金额
- 评估整体风险因素
- 再次展示代理间的协同工作（"像交响乐团一样"）

[60:00-结束] - 承保决策代理：汇总所有代理的分析结果，提供最终的虚拟承保建议（强调实际应用中需要人类承保人做最终决策）
