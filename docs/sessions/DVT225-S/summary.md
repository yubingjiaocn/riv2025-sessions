# AWS re:Invent 2025 分会场总结：容器化技术赋能 AI 代理

## 会议概览

本次分会场由 Docker 首席软件工程师 Jim Clark 主讲，探讨了 AI 代理（Agents）的发展现状以及容器技术在代理系统中的关键作用。演讲者回顾了从互联网、云计算到生成式 AI 的技术演进历程，指出当前 AI 代理时代与以往技术革命的本质区别在于其非确定性和超预期的能力表现。从 2022 年的内容生成，到 2023 年的问答交互，再到近年来编码代理的兴起，AI 代理正在展现出令人惊讶的自主工作能力。

演讲重点阐述了将 AI 代理部署到企业环境所面临的安全挑战，特别是"致命三要素"（敏感数据访问、外部系统通信、不可信数据暴露）带来的风险。Jim Clark 提出，容器技术作为云原生计算的基础设施，同样可以成为构建安全、可治理的代理系统的核心技术。通过将 MCP（Model Context Protocol）服务器容器化，可以实现工具的标准化分发、供应链安全保障和运行时隔离，从而为代理系统建立可信内容、安全运行时和治理策略三大支柱。

演讲最后邀请与会者前往 Docker 展位体验实际演示，包括在沙箱环境中运行 Claude、Codex 和 AWS Curo 等代理的案例，展示如何在不改变使用体验的前提下，通过容器化技术为代理系统增加安全边界和凭证隔离。

## 详细时间线

00:00 - 开场与自我介绍
- 演讲者 Jim Clark 介绍自己是 Docker 的首席软件工程师
- 回顾过去几十年软件行业经历的技术革命

00:30 - AI 代理时代的独特性
- 对比云计算时代：当时明确知道需求，惊讶于技术的可行性
- AI 代理时代：持续被其能力所惊讶，超出预期的表现
- 2022 年：生成式 AI 的输出超出预期
- 2023 年：与代理对话的体验前所未有
- 近 1.5 年：编码代理展现出有效的自主工作能力

02:00 - 编码代理的工作机制
- 编码代理是首次真正的代理循环体验
- 工具集相对简单：读文件、更新文件、运行 bash 命令
- 开发环境天然适合代理：单元测试、Git 工作流等已为协作做好准备
- 开发者已习惯邀请初级开发者加入工作流

03:30 - 代理的定义与特征
- 代理是一个循环系统：输入上下文 → 模型处理 → 执行操作 → 更新上下文
- 工具调用循环：读取文件 → 放入上下文 → 继续迭代
- 构建出复杂的工作流

04:00 - 非确定性的挑战与机遇
- 非确定性并非真正的障碍
- 类比：人类在喝咖啡前也是非确定的，但仍能构建复杂系统
- 企业实际上为非确定性付费（创造力）
- 需要以不同方式思考问题

05:00 - 从监督到自主
- 初期使用编码代理：逐步确认每个操作（读文件、修改代码、运行测试）
- 快速建立信任，转向"YOLO 模式"让代理自主工作
- 构建代理的难点：学会放手
- 类似管理团队：需要建立反馈循环、提供合适工具、允许自主性

06:30 - 代理面临的新问题
- 引用 Simon Willison 的"致命三要素"（Lethal Trifecta）：
  1. 访问敏感数据
  2. 与外部系统通信
  3. 暴露于不可信数据
- 代理缺乏边界意识的案例：LinkedIn 简历中嵌入指令，代理会执行（如发送食谱）
- 代理缺乏对抗性边界的原生概念

08:00 - 企业级代理的挑战
- 安全性：工具访问与凭证管理的分离
- 数据分区：区分指令与不可信数据
- 自主性需求：希望代理能做未预期的事情
- 工具范围：需要尽可能广泛的工具集
- CTO 们的担忧：缺乏资源应对这些新问题

09:30 - 代理与微服务架构的相似性
- 编码助手使用的工具链本质上是微服务架构
- 代理通过 CLI 调用各种服务
- 代理正在组合和构建新解决方案

10:30 - 容器技术赋能代理
- Docker 的简单性：pull 和 run 构建整个云
- 代理需要构建"工具微云"
- 1.5 年前开始将工具容器化
- MCP 标准出现后，确认了方向的正确性

12:00 - MCP 与容器的结合
- MCP：代理调用应用、工具和内容的协议
- 简化代理循环管理
- 关键是如何提供工具和上下文
- 无需重新发明分发标准

13:00 - 复用云原生经验
- 供应链安全：15 年的投资可直接应用
- 可信内容分发
- 安全运行时构建
- 所有云原生计算的经验都可复用

14:00 - 代理治理策略的基础
- 必须从可信内容开始
- 构建安全运行时
- 没有这些基础，治理层将难以构建

14:30 - 可信内容
- Docker 提供 MCP 服务器目录
- 所有工具已容器化，可立即使用
- 供应链安全：与过去 15 年相同的问题
- 镜像加固和漏洞跟踪

15:30 - 安全运行时
- 代理需要通用运行时（不确定下一步需要什么工具）
- 安全隔离环境
- 安全处理密钥和凭证
- 只有需要凭证的工具才能访问
- 容器化是解决方案

16:30 - 实际演示案例
- 在沙箱中运行 Claude、Codex 和 AWS Curo 代理
- 使用体验不变，但增加安全边界
- 防止删除硬盘、窃取 AWS 凭证等攻击
- 不影响使用，只是更安全

18:00 - 治理层构建
- 基于可信内容和安全运行时的基础
- 跟踪代理行为
- 部署管理
- 安全参数设置
- 工具访问控制
- 集中化策略管理

19:00 - 总结与展位邀请
- 代理不是微服务，但需要组合能力
- 代理可以容器化
- 容器是构建新系统的基础技术
- "类固醇版微服务"
- 可信内容、安全运行时、治理是三大关键支柱
- 邀请参观 Docker 展位观看实际演示
