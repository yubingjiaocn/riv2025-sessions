# AWS re:Invent 2025 多云开发者体验分享会总结

## 会议概览

本次分享会由AWS和Macquarie Group(麦格理集团)联合呈现,主题聚焦于如何简化多云环境下的开发者体验。演讲者Chris来自AWS,Brent Burgess来自Macquarie Group。会议探讨了在多云架构中开发者面临的挑战,以及Macquarie Group如何通过构建名为"Dex"(Developer Experience Daemon)的内部平台来解决这些问题。

Macquarie Group是一家全球性金融服务机构,在31个国家运营,拥有约20,000名员工,业务涵盖资产管理、资本管理、交易以及数字银行等多个领域。该公司已实施云优先战略长达11年,目前超过70%的基础设施和应用程序托管在公共云上,同时使用AWS和GCP两个云平台,为应用工程师启用了超过80项服务。

Macquarie的多云战略核心在于通过最佳云服务推动创新和敏捷性,使工程师能够快速为客户和客户交付价值。Dex平台基于Microsoft的开发容器技术构建,实现了快速入职、标准化开发环境、左移合规控制,并通过AI增强服务消费体验。该平台由仅一人维护,却支持着全球数千名工程师的日常开发工作,显著提升了开发速度和工程师体验。

- 介绍会议主题:简化多云开发者体验

00:01:00 - 会议议程概述
- 定义多云的含义
- 讨论如何简化开发者体验
- 探讨跨多云构建的挑战
- Macquarie Group的解决方案分享
- 经验教训和业务影响

00:02:00 - 多云定义
- 多云指使用一个或多个云平台来部署服务和实现目标
- 重点关注开发者体验层面

00:02:30 - 多云运行的挑战
- 可能需要使用不同云提供商的特定功能或服务
- 企业收购导致的多云环境
- 供应商工具可能分布在不同云平台上
- AWS从人员、流程和技术三个维度看待多云问题

00:03:30 - 开发者面临的具体挑战
- IDE配置:VS Code、IntelliJ、Cloud9等多种选择
- 需要配置个人偏好和组织标准(如代码格式化规范)
- Windows vs Mac等不同操作系统的兼容性问题
- 跨多云的身份管理复杂性
- 网络访问配置(如连接开发数据库)
- 密钥管理(SSH密钥等)
- 本地工具安装:Docker、Artifactory、CI/CD工具等
- AI工具的安全集成和护栏设置

00:05:30 - Brent Burgess介绍Macquarie Group
- 全球金融服务机构,在31个国家运营
- 约20,000名员工
- 业务多元化:资产管理、资本管理、交易、零售银行
- 在澳大利亚运营纯数字银行(无分支机构、无ATM)

00:06:30 - Macquarie的云战略
- 实施云优先战略已有11年
- 目标:推动创新和敏捷性,快速为客户交付成果
- 超过70%的基础设施和应用托管在公共云
- 多云战略:同时使用AWS和GCP
- 启用超过80项云服务供应用工程师使用
- 过去一年中,三分之二的工程师获得公共云认证或培训
- 超过1,000个应用原生托管在公共云

00:08:00 - Macquarie的多云意义
- 通过云服务提供商获取最佳服务
- 使应用工程师能够为客户交付价值和成果
- 带来的挑战:多种工具集、不同环境
- 业务多样性导致平台差异(资产管理、资本管理、交易、数字银行、核心银行)
- 需要支持多种框架和编程语言:Python、Java、Go、Ruby
- 框架持续升级变化
- 工程师需要具备多方面技能,能够使用不同工具和平台
- 在31个国家、20,000名员工、多个市场的背景下运营

00:10:00 - Macquarie的愿景
- 创造一个让工程师蓬勃发展的环境
- 赋能开发者:提供最佳工具、最新框架和语言
- 加速交付:让工程师专注于业务问题而非技术配置
- 减少在SSH密钥、Artifactory访问、AD组权限等方面的时间消耗
- 使工程师能够大规模交付业务成果

00:11:00 - Dex平台介绍
- Dex = Developer Experience Daemon(开发者体验守护进程)
- 基于Microsoft的开发容器(Dev Containers)技术
- 目标:快速入职、提供铺平的道路、左移合规控制

00:12:00 - Dex的核心功能
- 快速入职:消除AD组、Artifactory认证、Docker安装等复杂性
- 提供一致的开发环境,便于全球范围内管理和支持
- 作为受监管的金融服务公司,将合规控制和安全要求左移到本地开发环境
- 让工程师保持更长时间的心流状态,专注于业务成果

00:13:00 - Dex用户界面演示
- 每个工程师的笔记本电脑上自动安装Dex守护进程
- 通过浏览器访问localhost查看可视化仪表板
- 使用交通灯系统显示配置状态
- 配置缺失或错误时,点击交通灯可自动修复或跳转到文档
- 减少工程环境设置的复杂性

00:14:00 - Dex的开发容器模板
- 配置完成后,提供默认的开发容器基础镜像
- 支持多种操作系统和应用类型
- 提供数十种预构建模板,涵盖不同应用、工具和框架
- 特别注意:同时提供AWS和GCP框架模板,加速多云服务启用

00:15:00 - 多云启用的实际案例
- 两个角色:服务构建者(Service Builders)和服务消费者(Service Consumers)
- 服务构建者:云工程师,为应用团队启用多云服务

00:16:00 - 服务构建者的Dex模板
- 预构建模板安装所有框架组件
- 包含:精简版Kubernetes、Crossplane、Argo、LocalStack、Moto
- 将云工程左移到本地笔记本电脑
- 工程师可以利用笔记本电脑的计算能力,而非推送到共享开发环境
- 早期只有少数工程师时速度很快,但团队扩大后共享环境出现冲突
- 通过本地开发容器解决了团队协作冲突问题,加速了多云服务启用

00:17:30 - 服务消费者的AI增强
- 使用预训练模型,包含所有已启用的云服务信息
- 服务消费者可以在IDE中生成符合规范的Crossplane声明
- 基于之前启用的服务自动生成合规配置
- 加速服务启用和消费,减少工程师对控制和护栏的担忧
- 让工程师专注于为客户交付价值

00:18:30 - Dex的业务影响
- 快速入职:将入职时间从数天甚至数周缩短到极短时间
- 解决了延迟发现的问题(如一两周后才发现缺少SSH密钥或AD组权限)
- 不仅适用于新员工,也适用于内部换团队的员工
- 加速工程师在不同团队、应用甚至部门之间的流动
- 左移所有配置,利用笔记本电脑的计算能力
- 使多云工程师能够快速启用服务以满足业务期望
- 提高开发速度:启用最佳多云服务,大规模为业务交付价值
- 最终目标:赋能工程师蓬勃发展

00:20:00 - Chris的提问环节
- 问:构建这样的开发者体验需要多长时间?
- 答:花费了几年时间构建基础能力,优先解决最高价值的痛点,逐步构建基础镜像和模板
- 采用内源(Inner Sourcing)模式,工程师可以贡献到平台
- 多云服务启用模板就来自一位云工程师的贡献

00:21:00 - 平台支持团队规模
- 问:有多少人支持这个平台?
- 答:全球数千名工程师,仅有一个人支持该平台
- 平台在很大程度上自运行

00:21:30 - AI工具集成
- 问:如何使用Dex与内部工具交互?如何使用MCP增强开发者体验?
- 答:已启用AWS MCP服务器
- 也用于自己的云控制平面(运行了11年,非常成熟)
- 使用MCP分析构建失败
- 工程师可以在开发环境中使用聊天窗口
- MCP根据以前的示例自动提供声明的修正建议
