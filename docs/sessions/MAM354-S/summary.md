# AWS re:Invent 2025 技术会议总结

## 会议概述

本次会议由Zendesk的软件工程师Nick Hefty主讲，主题为"在大规模环境下发现噪声邻居客户性能模式"。Nick拥有7年半的Zendesk工作经验，从客户支持部门转入软件工程领域，为解决高基数环境下的可观测性挑战提供了独特视角。

会议重点讨论了如何通过改进可观测性策略来减少深夜紧急响应，特别是在拥有数十万客户的高基数环境中识别和处理"噪声邻居"问题。Nick分享了Zendesk在AWS基础设施上的实践经验，包括使用Aurora、ElastiCache、EC2等服务，以及如何平衡可观测性需求与成本控制。

## 详细时间线与关键要点

### 0:00-5:00 开场与问题背景
- 以感恩节后深夜被叫醒处理紧急事件的场景开场
- 介绍"疲惫工程师"概念和噪声邻居问题
- 说明Zendesk的AWS技术栈：Aurora、ElastiCache、EC2、容器化基础设施

### 5:00-10:00 高基数挑战分析
- 解释高基数概念：同一层级存在大量实体
- 展示复杂的基础设施图，说明数十万客户带来的挑战
- 讨论成本与可观测性之间的平衡问题
- 介绍数据保留、采样率、数据敏感性等考虑因素

### 10:00-15:00 噪声邻居定义与初始解决方案
- 用公寓楼比喻解释噪声邻居概念
- 展示初始客户流量模式图的局限性
- 介绍通过租户标记改进可观测性的方法
- 展示Datadog APM追踪的详细信息优势

### 15:00-25:00 REST方法论详解
- **识别(Recognize)**：确定需要观察的内容（热分区、积压、延迟峰值）
- **检查(Examine)**：分析如何获得可见性（顶级租户、标记、微服务）
- **塑造(Shape)**：制定长期策略和限制措施
- **测试(Test)**：验证解决方案有效性，定期清理无用告警

### 25:00-30:00 成本优化策略
- 讨论日志、APM、指标的不同成本层级
- 强调摄取与索引管道的管理重要性
- 介绍采样策略：成功请求10%采样，错误100%采样
- 将成本作为持续对话的一部分，而非年度突击检查

### 30:00-35:00 实施效果与常见陷阱
- 展示实施后的积极成果：更快响应时间、更好的资源配置、更满意的客户
- **避免的陷阱**：
  - 自由格式标记导致重复计费
  - 全量日志记录的高成本
  - 缺乏服务所有权映射
  - 告警疲劳和信噪比问题

### 35:00-41:00 关键要点总结与问答
- **核心建议**：
  - 优先建立标记标准而非花哨仪表板
  - 早期投资租户级KPI和热力图
  - 将成本视为一等维度
  - 内置上下文信息，修复元数据
- 强调客户体验的重要性：主动发现问题优于被动响应

- 开放问答环节
