# AWS re:Invent 2025 分会场总结：大规模部署AI代理的安全与治理

## 会议概述

本次分会场由AWS首席解决方案架构师Michael Espie和Rubrik AI总经理Dev Rishi共同主讲，重点探讨了企业如何安全、可控地大规模部署AI代理（Agents）。Dev Rishi此前是生成式AI基础设施初创公司Predase的联合创始人兼CEO，该公司于今年夏天被Rubrik收购。Rubrik是一家数据网络安全和弹性公司，从数据备份恢复起家，逐步演进为帮助企业应对云迁移和网络安全挑战的平台。

演讲者指出，当前企业面临的最大挑战不再是如何构建AI代理，而是如何将其从概念验证阶段推向生产环境。通过与180多家企业的深入交流，Rubrik发现超过半数的组织仍处于实验阶段，主要障碍在于缺乏有效的治理框架和风险管理机制。AI代理具有不可预测性、弱监督性和非人类身份等特点，使其在带来10倍效率提升的同时，也可能造成10倍的潜在损害。为此，Rubrik推出了Agent Cloud平台，通过监控可观测性、策略治理和AI代理回滚（Agent Rewind）三大核心能力，帮助企业在控制风险的前提下加速AI转型。

## 详细时间线与关键要点

00:00 - 开场介绍
- Michael Espie介绍自己是AWS首席解决方案架构师，在AWS工作超过9年
- Dev Rishi介绍自己是Rubrik AI总经理，此前是Predase公司联合创始人兼CEO

01:30 - 客户面临的核心挑战
- 客户希望利用AI代理技术，但在从POC到生产环境的过渡中遇到困难
- 主要关注点：如何获得更好的控制和可见性，以及如何实现成本效益
- AWS与Rubrik保持紧密合作伙伴关系

02:45 - Rubrik公司背景
- Rubrik是一家难以简单定义的公司，核心是帮助企业加速不同阶段的转型
- 从磁带到数字化时代的数据备份恢复起家
- 随着云迁移浪潮推出Rubric Security Cloud
- 现在进入AI转型时代

04:20 - AI代理的定义与趋势
- Rubrik将代理定义为"具有工具访问权限的LLM"，能够在生产环境中代表用户采取行动
- 过去4个月与180多家组织交流，涵盖IT、安全、工程和AI部门
- 最明确的趋势：AI代理不是即将到来，而是已经到来

05:30 - AI代理成熟度的四个阶段
1. 实验阶段（Experimentation）：构建第一个或第二个代理
2. 正式化阶段（Formalization）：从演示原型转向企业可消费的环境（最关键的转折点）
3. 扩散阶段（Proliferation）：从几个代理快速扩展到数百个
4. 自主阶段（Autonomous）：代理拥有完全写入权限，代表组织自主运作

07:00 - 客户现状调查结果
- 超过50%的组织仍处于实验阶段
- 约25%处于正式化阶段
- 13%尚未开始AI采用生命周期
- 现场互动：询问参会者所处阶段

08:15 - 风险管理成为核心关注点
- 从实验到正式化阶段的转变中，风险管理成为关键考量
- Rubrik的演进：从自然灾害防护→网络安全（勒索软件）→AI威胁
- AI代理可能造成10倍损害，且速度快10倍

09:30 - AI代理的治理挑战
- 代理具有不可预测性和弱监督性
- 使用非人类身份运作，难以大规模管理
- 企业困惑：如何将非确定性、概率性系统与生产系统相协调

10:45 - 实际案例：AI代理出错场景
- 推荐竞争对手产品
- 虚构退款规则
- 编码代理删除整个生产数据库以"优化"子程序

11:30 - 企业ROI困境
- 企业尚未看到AI的投资回报
- 原因：AI无法应用于最重要的问题，因为这些问题风险太高
- Dev Rishi作为Rubrik AI治理委员会成员，深刻体会到这些挑战

13:00 - 生产系统的架构挑战
- AI代理运行在Salesforce、Microsoft 365、AWS等生产系统上
- 这些系统最初并非为AI原生工作负载设计
- 企业需要回答的基本问题：
  - 当前有哪些代理和AI在运行？
  - 它们能访问哪些工具和数据？
  - 出错时如何恢复？

14:30 - Rubrik内部经验
- 构建初始概念验证只需2周
- 但每个项目都需要审批，导致时间线从几周延长到几个月
- 团队需求：提供一个框架，在框架内可以自由构建代理，无需每次都重新审批

16:00 - Rubric Agent Cloud产品发布
- 结合Rubrik的三大核心能力：
  1. 数据和元数据理解
  2. 身份系统管理
  3. LLM和AI平台（来自Predase收购）

17:15 - Agent Cloud三大核心支柱

支柱一：监控与可观测性
- 与AWS Bedrock Agent Core等环境集成
- 自动发现代理并填充代理清单
- 提供软件定义的非手动方式查看：
  - 生态系统中运行的代理
  - 关联的身份
  - 可访问的工具和数据

18:45 - 支柱二：治理
- 现场互动：询问有多少组织有AI治理委员会和政策
- 问题：许多治理政策难以在实践中执行
- 例如："AI代理不应提供财务建议"或"默认只读权限"
- Rubric提供定义和执行策略的能力，实时监控代理流程

20:30 - 支柱三：AI Agent Rewind（代理回滚）
- 基于Rubrik核心DNA的弹性和恢复能力
- 允许撤销代理的破坏性操作
- 如果代理错误删除生产数据库或错误编辑Salesforce列
- 可将代理操作与Rubrik备份中的健康快照关联
- 实现一键撤销功能

22:00 - 产品定位与客户价值
- 口号："释放代理，而非风险"（Unleash agents, not risk）
- 基于Rubric Security Cloud的相同基础架构
- 作为"客户零号"（Customer Zero），Rubrik自身面临：
  - 数百个代理用例
  - 复杂的跨部门需求（信息安全、法务、IT等）
  - 需要可审计的策略制定流程

23:30 - 产品可用性与总结
- Rubric Agent Cloud目前处于预览阶段
- 结合第一方经验和客户反馈构建平台
- 邀请参会者访问展位查看实时演示
- 展位在展览大厅显眼位置

24:30 - 会议结束
- 感谢参会者
