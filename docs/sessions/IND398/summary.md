# AWS re:Invent 2025 - Rovio 使用生成式 AI 加速游戏资产创作

## 会议概述

本次会议由 AWS 高级客户经理 Rupar 和 Rovio 首席机器学习工程师 Ignasio 共同主讲,深入探讨了《愤怒的小鸟》开发商 Rovio 如何利用生成式 AI 技术革新游戏开发流程。Rovio 是一家芬兰游戏工作室,在第 52 次尝试后创造了全球现象级游戏《愤怒的小鸟》,如今已发展成为一家现代娱乐公司,业务涵盖游戏、电影和动画制作。

会议重点介绍了 Rovio 如何应对艺术团队面临的巨大挑战——艺术家占公司员工三分之一,但他们在截止日期的压力下疲于应对,因为玩家消耗游戏内容的速度远超艺术资产的创作速度。每个新季节(万圣节、情人节、圣诞节)、新游戏功能和特殊活动都需要大量新艺术资产。ML 团队意识到生成式 AI 可以将某些任务从数天缩短到数小时,在特定用例(如赛季通行证背景)中节省 80% 的制作时间。这不仅仅是效率提升,更是游戏规则的改变者。

Rovio 的 AI 之旅始于 2022 年,当时扩散模型刚刚问世。团队设定了明确的原则:仅使用自有专有素材、确保数据安全(仅在本地或云端处理)、尊重所有人的知识产权。通过与 AWS 的合作,利用 Amazon SageMaker 和 Amazon Bedrock 等服务,Rovio 构建了名为 "Become Picasso" 的生成式 AI 艺术工具套件,成功地将 AI 技术融入游戏开发流程,同时保护了品牌的独特身份。

## 详细时间线

### 开场与背景介绍
[00:00 - 02:30] - Rupar 开场致辞,介绍这是 re:Invent 第四天的会议,主题是《愤怒的小鸟》如何使用生成式 AI。询问观众是否在尝试将 AI 整合到游戏开发流程中,以及是否在创意控制与生产效率之间难以平衡。

[02:30 - 04:45] - 介绍 Rovio 公司背景:芬兰游戏工作室,《愤怒的小鸟》是他们的第 52 款游戏。如今 Rovio 不仅仅是"用小鸟打猪",而是一家游戏优先的现代娱乐公司,在电影、动画和 AI 领域都有突破。

### Rovio 的 AI 发现之旅
[04:45 - 07:15] - 讲述 Rovio 如何"偶然发现" AI:一边是 ML 技术团队,另一边是占公司三分之一员工的艺术家团队,他们被截止日期压得喘不过气来。每个新季节(万圣节、情人节、圣诞节)、新游戏功能和特殊活动都需要大量新艺术资产。

[07:15 - 09:30] - 艺术家面临的真正挑战:他们需要专注于创作新艺术和资产,却被迫做重复性任务,如创建同一背景的多个变体或对现有资产进行增量更改。ML 团队的"顿悟时刻":如果能给艺术家创意超能力,将需要数天的任务变成几小时的工作会怎样?针对赛季通行证背景这一特定用例,可以节省 80% 的制作时间。

### Ignasio 的技术深入讲解
[09:30 - 12:00] - Ignasio 回顾 2022 年:ML 团队和艺术团队当时并未合作。扩散模型在 2022 年问世后,团队想用《愤怒的小鸟》品牌尝试这些模型。设定基本原则:始终使用自有专有素材、保持数据安全(仅在本地或云端)、尊重所有人的知识产权(包括提示词中的)。

[12:00 - 15:30] - 首次概念验证(2022 年):目标是训练《愤怒的小鸟》主角 Red 的模型,采用电影风格。展示第一个数据集和失败的结果——生成的图像在各个方面都失败了,Red 骑自行车的图像看起来很诡异。

[15:30 - 18:00] - 持续尝试和改进:到 2022 年 12 月,团队学会了如何生成电影风格的 Red 图像。询问观众对结果的看法,虽然 Ignasio 当时很自豪,但艺术家不满意——"相当好"还不够好。

[18:00 - 21:00] - 解释为什么生成 Red 如此困难:所有这些都是 Red,但看起来非常不同(经典版、电影版等)。展示不同游戏中的《愤怒的小鸟》角色:《Angry Birds Dream Blast》中的可爱婴儿版本、《Angry Birds Friends》中的经典版本、《Angry Birds 2》中的现代版本。

[21:00 - 24:00] - 2024 年的新尝试:使用《Angry Birds Dream Blast》风格训练模型。展示目标风格和 AI 生成结果,询问观众意见。虽然结果看起来很好,但艺术家发现了许多缺陷。

### 品牌质量标准
[24:00 - 27:00] - Rovio 有数百页幻灯片详细说明如何绘制《愤怒的小鸟》角色:眼睛、眉毛、脸颊如何表达情感,翅膀甚至脚的画法,以及如何绘制不同的面部表情(快乐、愤怒、惊讶)。艺术家有非常高的质量标准,因为这关乎品牌形象。AI 应该帮助提高标准,而不是降低标准。

### 关键突破和经验教训
[27:00 - 30:00] - 第一个突破:虽然为 Red 训练的模型效果不佳,但可以用来生成次要角色(如可爱的鳄鱼宝宝和鲨鱼),因为这些角色没有数百页的绘制指南。

[30:00 - 33:00] - 第二个发现:专注于非品牌核心资产,如背景的多个变体等重复性任务。这样艺术家就有更多时间专注于角色、情感和故事创作。

[33:00 - 36:00] - 三大挑战:
1. 控制问题:艺术家很难获得想要的结果,尤其是生成《愤怒的小鸟》角色时。但 AI 更适合不需要精确控制的任务,如创建新地点和背景。
2. 数量问题:需要生成数百张图像才能得到几张好的。学会接受一定程度的随机性,可以锁定种子并修改提示词来控制图像的某些部分。
3. 风格问题:不想要通用图像或风格,需要 Rovio 风格。

[36:00 - 39:00] - 关键学习:需要组建高质量数据集,20-30 张由艺术家精心制作的图像就足够了。艺术家标注图像比 ML 工程师更好,他们有更丰富的语言来描述图像内容,而且这能建立信任。

[39:00 - 41:00] - 最大的收获:需要艺术家来使用这项技术并审查输出。需要对视觉效果有深刻理解的艺术家,需要有丰富词汇来创建标注的艺术家,需要能够审查输出并在必要时进行编辑以达到高质量的艺术家。

### 技术实现细节
[41:00 - 44:00] - 三个主要步骤:
1. 选择基础模型(没有万能方案,不同用例适合不同模型)
2. 微调模型(与艺术家一起创建优质数据集)
3. 构建工作流(艺术家通常不直接与模型交互,而是使用工作流)

[44:00 - 48:00] - 基础模型训练原理:使用互联网上数十亿张带文本描述的图像训练。文本转换为嵌入(高维向量空间表示),图像添加噪声并转换为潜在表示。神经网络学习在文本引导下去除噪声。扩散模型的核心就是添加噪声和去除噪声。

[48:00 - 52:00] - 微调过程:从捕捉风格的优质数据集开始,需要触发词(trigger word,使用奇怪的词以避免与基础模型词典冲突)。图像存储在 S3 中,所有微调训练在 SageMaker 训练作业中进行,可以并行运行数千个训练,几小时内得到结果。

[52:00 - 55:00] - 展示训练过程视频:图像逐步变得更像 Rovio 风格,但在某个时刻会开始过拟合。每个步骤在 S3 中都有一个安全张量文件(模型工件)。通过向艺术家展示这些视频来决定选择哪个模型。

[55:00 - 58:00] - 推理过程:从 S3 获取安全张量工件并放入工具或工作流中。展示生成北极光的例子,从随机噪声开始,经过多个步骤后生成 Rovio 风格的北极光图像。

[58:00 - 61:00] - 工作流参数:步骤数(较少步骤生成更快但质量较低)、模型强度(生成图像与风格的相似度)、可以组合两个微调模型、引导比例(AI 遵循提示词的程度)。所有这些都由 EC2 G6E 实例提供支持,使生成在几秒内完成。

### 从 AI 图像到游戏资产
[61:00 - 65:00] - 艺术家工作流程:
1. 提出想法并找到提示方式
2. 使用 AI 工具生成大量图像
3. 选择最佳图像作为起点
4. 清理图像(可能有奇怪的伪影)
5. 扩展图像
6. 提升分辨率(AI 模型生成较低分辨率图像更快)
7. 创建不同图层并添加品牌角色
8. 嵌入游戏 UI

[65:00 - 69:00] - 赛季通行证内容:移动游戏中的限时特殊活动,通常持续一个月,玩家可以体验新的季节性内容并获得奖励。使用 AI 可以将生成所有资产的时间从几周缩短到几天,但艺术家仍然保持控制权。

[69:00 - 72:00] - 工作流程的改变:第一次尝试时,先头脑风暴决定要做什么,然后生成图像,花费的时间几乎与不使用 AI 相同。第二次迭代时,先生成大量图像,然后再进行头脑风暴,大大加快了流程。这不仅关乎 AI,还关乎改变内容创作流程。

[72:00 - 74:00] - 展示艺术家编辑图像并将其组合成最终结果的快速视频,这基本上是他们几天的编辑和故事创作工作。

### Become Picasso 工具套件
[74:00 - 78:00] - 艺术家对 AI 的不同态度:有些人一开始就喜欢,有些人觉得困惑,有些人甚至不想尝试。有些人失望因为不是一键生成,有些人失望因为以为是一键生成(他们习惯复杂工具)。ML 工程师喜欢用 Python 生成图像,但艺术家不是技术人员,他们更喜欢用详细的提示词(描述光照、相机角度、角色等)。

[78:00 - 81:00] - 目标明确:为合适的人提供合适的工具。因此构建了多个由 AWS 支持的工具,统称为 Become Picasso。Ignasio 在 Beacon 工作,这是一个为游戏提供服务和平台的中央单位,AI 工具是这一产品的一部分。所有工具都由 AWS 支持,可以与在 Amazon SageMaker 上训练的微调模型交互,数据永不离开 VPC。

[81:00 - 85:00] - Slack 机器人:第一个工具,因为公司每个人都有 Slack。首要目标是民主化并共同学习如何使用这些工具。面临许多采用障碍和担忧(数据隐私、有害内容、环境问题、内容泛滥、知识产权问题、偏见等)。

[85:00 - 88:00] - 偏见案例:一位产品经理提示"Rovio 的重要会议",只生成了白人男性,但 Rovio 是一家非常多元化的公司。对抗偏见的最佳方法是意识到它们的存在。Slackbot 的一大胜利是开启对话,公开讨论所有这些问题。

[88:00 - 91:00] - Slackbot 技术架构:选择 SageMaker(因为已经熟悉,已有许多机器学习模型在 SageMaker 上,并且支持推理所需的 GPU)。演示:ping Become Picasso 并给出提示词,机器人回复"okie dokie, working on it",几秒钟后返回生成的图像。

[91:00 - 93:00] - 后端架构:Lambda 回复第一条消息并触发 SageMaker 请求生成图像,图像存储在 S3 中,另一个 Lambda 将生成的图像发送回 Slack。所有消息通过 SNS 传递。第三个 Lambda 将元数据存储到 RDS 用于分析(生成时间、每天生成次数等)。

[93:00 - 97:00] - Picasso Pro Studio Cloud:Slackbot 太简单,对艺术家不够用。需要更高级的工具。有许多开源工具提供强大功能和灵活性,但难以使用且计算成本高。目标是让艺术家轻松尝试开源社区的前沿模型和工具,不想自己构建新 UI,因为领域变化太快。

[97:00 - 100:00] - 从本地笔记本开始:尝试不同的开源项目和模型。但艺术家越来越兴奋,需要为他们配置模型(这些是非技术人员),要处理 Python 虚拟环境、bash 脚本等。每次有新模型或更新都要重新配置。有些艺术家甚至订购了超贵的 GPU 台式机送到家里,声音像洗衣机但生成速度很快。这显然不可扩展。

[100:00 - 103:00] - 迁移到云端:将所有代码和项目移到 EC2 实例,允许艺术家通过 SSH 隧道连接。从 G6 系列开始,比艺术家家里的昂贵 GPU 台式机慢一点。但 Amazon 发布 G6E 实例后,比那些昂贵台式机快得多,艺术家停止使用家里的机器,转而使用云端模型。

[103:00 - 106:00] - 改进架构:仍需设置 SSH 隧道,没有友好的 URL 供艺术家在浏览器中收藏。展示开源工具 UI 示例,艺术家通过 VPN 或办公室访问,通过入站 Route 53 解析器提供友好的 DNS。

[106:00 - 109:00] - 自动扩展组:允许跨不同可用区配置实例。这很重要,因为 GPU 不多,如果一个可用区没有可用 GPU,可以在另一个可用区配置。也有利于容错和节省成本(GPU 很贵),在周末和工作时间外关闭这些机器。

[109:00 - 112:00] - CI/CD 流程:不只有一个 EC2 实例,有很多。AI 工程师需要创建新版本来创建新模型或微调不同的东西。之前的架构在 Terraform 中,AI 工程师不太熟悉。目标是让 AI 工程师也能轻松推送最新版本的开源工具或最新模型。

[112:00 - 115:00] - 自动化部署:AI 工程师将代码推送到 Git 仓库,触发 AWS CodePipeline,启动 EC2 实例(创建者实例),然后在另一个 EC2 实例上安装所需的一切(从 S3 下载基础模型、下载安全张量微调模型、使用不同脚本设置一切)。最终得到一个 AMI,可用于启动由强大 GPU 支持的新 EC2 实例供艺术家使用。

[115:00 - 118:00] - 三个工具总结:Slackbot(超级简单,面向全公司)、Picasso Pro Studio Cloud(面向高级用户,但非常复杂)。需要介于两者之间的东西,既能生成游戏内容,又对所有 Rovio 员工更易用。

### Amazon Bedrock 集成
[118:00 - 121:00] - 新挑战:此时已有许多微调模型,艺术家开始发现很难找到应该为其用例使用哪个微调模型。展示内部文档页面,艺术家需要在多个工作流中选择,仍然不够简单。

[121:00 - 124:00] - 目标:为最终用户隐藏所有这些复杂性,组织所有这些工作流并帮助他们提示。Amazon Bedrock 和 Claude 来救援。通常需要从其他团队(前端和后端开发人员)请求资源,需要等待一段时间。但团队最近开始使用 Bedrock。

[124:00 - 结束] - 创建新的推理配置文件并将其与 Claude 连接,开始编码。两名 AI 工程师开始构建解决方案...(字幕在此处结束)
