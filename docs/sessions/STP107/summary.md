# AWS re:Invent 2025 医疗AI扩展实战分享

## 会议概述

本次闪电演讲由Heidi公司的平台工程师Ocha主讲，分享了他们在将生成式AI应用于真实医疗场景中的扩展经验。Heidi是一家专注于AI医疗记录助手的公司，致力于通过AI技术减轻医生的行政负担，让医生能够专注于患者护理。演讲者从工程师角度深入分析了在医疗AI产品开发和全球扩展过程中遇到的技术挑战和解决方案。

Ocha作为Heidi的创始工程师之一，拥有四年的产品开发经验，从早期的独立开发到现在专注于平台工程。他在演讲中重点强调了三个核心观点：专注解决单一痛点问题、重视人在AI循环中的作用，以及从第一天就构建灵活的架构。整个分享为正在构建AI产品的开发者提供了宝贵的实战经验和技术洞察。

## 详细时间线与关键要点

### 0:00-2:30 开场与问题背景
- 演讲者自我介绍，分享re:Invent学习体验
- 现场互动：询问观众是否有过医生看诊时感觉被忽视的经历
- 问题定义：医生因需要处理行政任务（写病历）而无法专注患者
- Heidi使命：通过AI护理伙伴将全球医疗容量翻倍

### 2:30-4:00 Heidi产品演示
- 产品功能展示：实时转录生成临床笔记，无需修改
- 基于模板的笔记生成和患者解释功能
- AI临床研究支持和后续任务建议
- 核心价值：让医生专注患者，AI处理行政工作

### 4:00-6:00 公司发展历程
- 创始人Tom Kelly从医学生考试聊天机器人Oscer开始
- 发展为全面医疗护理平台
- 生成式AI出现后专注临床笔记生成单一工作流
- 成长数据：37万+临床医生用户，每月1000万次咨询，加拿大第一AI记录助手

### 6:00-8:30 第一个挑战：构建AI信心
- 个性化和定制化笔记的需求
- 医疗需要临床准确性，验证非确定性输出的挑战
- 早期解决方案：为医生提供Jupyter Notebook进行实验
- 扩展问题：不是所有临床医生都会编程
- 临床医生在循环中的评估流程：词错误率、模板遵循检查、幻觉率检查

### 8:30-11:00 第二个挑战：全球扩展的四层复杂性
- **数据主权**：严格的数据本地化要求（澳洲必须使用ap-southeast-2/4，美国使用us-east-1/west-2）
- **模型可用性**：不同地区模型可用性差异巨大
- **医疗现实**：不同地区的医疗实践、术语、咨询流程差异
- **监管环境**：生成式AI影响监管空间，不同地区合规要求不同

### 11:00-13:30 技术解决方案
- **标准化基础设施**：使用基础设施即代码确保跨地区部署一致性
- **混合模型策略**：Amazon Bedrock解决即时可用性，EKS支持自托管模型
- **EKS架构优势**：支持全球大部分地区，提供灵活的推理模型服务
- **人员本地化**：雇佣地区临床医生大使，确保本地医疗方言适配
- **合规网络**：内部法律合规团队配合外部医疗安全合作伙伴

### 13:30-14:30 核心经验总结
- **专注单一问题**：从广泛的护理平台转向专注单一工作流程
- **人的重要性**：在生成式AI时代，人比以往更重要，临床医生是产品核心
- **灵活架构**：从第一天就构建灵活架构，这关乎业务生存而非仅仅代码质量
- 结束语和联系方式分享
