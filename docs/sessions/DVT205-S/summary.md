# AWS re:Invent 2025 技术分享会总结

## 会议概述

本次分享会由Cortex联合创始人兼CTO Ganesh主讲,主题为"在AI时代保持卓越"(Maintaining Excellence in the Age of AI)。演讲者从第一性原理出发,探讨了在AI编码助手快速普及的背景下,工程团队如何在追求更快交付速度的同时,避免在软件质量和安全性方面做出妥协。Cortex是一个内部开发者门户平台,专注于服务目录管理、所有权定义、最佳实践标准制定等功能。

演讲核心观点是:AI是一个放大器,它会放大组织现有的优势和劣势。根据Cortex的调研,70%的工程领导者将安全性和质量回归列为采用AI编码工具时的首要关注点。演讲者强调,这些挑战并非全新问题,而是长期存在的工程基础问题,只是在AI时代被进一步放大。因此,解决方案不是发明新方法,而是加强已知的最佳实践:代码质量测试、监控与SLO设置、安全实践以及明确的所有权责任制。

演讲者提出了AI卓越的三阶段框架:1)建立稳定基础(生产就绪流程、测试框架、安全流程);2)衡量AI编码助手对客户影响的实际指标(而非仅关注AI使用率);3)充分利用AI工具的价值。Cortex通过其内部开发者门户(IDP)、AI数据引擎Mellin、评分卡系统和工程智能仪表板来帮助组织实现这些目标。

## 详细时间线与关键要点

### 开场与背景 (0:00-2:30)
- **0:00** - 演讲者自我介绍:Ganesh,Cortex联合创始人兼CTO
- **0:30** - 提出核心问题:在追求更快交付速度时,我们在牺牲什么?这是否是必然的权衡?
- **1:15** - 介绍Cortex产品:内部开发者门户,涵盖服务目录、所有权定义、评分卡标准、黄金路径等功能
- **2:00** - 演讲主题:在AI时代保持软件卓越
- **2:15** - 方法论:从第一性原理出发,定义软件卓越的含义,然后反向推导

### 工程领导者的关注点 (2:30-5:45)
- **2:30** - 核心问题:工程领导者今天在担心什么?
- **3:00** - 调研结果:Cortex今年对工程领导者进行了AI采用调研
- **3:30** - 关键发现:70%的工程领导者将安全性和质量回归列为首要关注点
- **4:00** - 解释原因:AI编写的代码可能不被完全理解,可能泄露密钥,引入新的风险因素
- **4:30** - 提出深入问题:质量和安全回归的二阶效应是什么?

### 质量回归的影响 (5:45-7:15)
- **5:45** - 质量回归的具体表现:
  - 更多事故发生
  - SLA/SLO违反
  - 维护成本上升(保持系统运行的时间增加)
- **6:30** - 最终影响:客户受到影响,失去客户信任
- **6:45** - 强调:这些都是可衡量的指标(事故数量、MTTR、SLA违反情况)

### 安全风险的影响 (7:15-8:45)
- **7:15** - 安全风险的两大关注点:密钥泄露和数据泄露
- **7:45** - 观察:消费者产品公司最担心数据泄露,因为客户信任是核心差异化因素
- **8:15** - 密钥泄露风险:当人类不再完全参与代码编写循环时,AI可能会将密钥复制到环境文件中
- **8:30** - 承认可以设置防护措施,但这些是真实存在的风险类型

### 为什么是现在? (8:45-12:00)
- **8:45** - 问题:为什么现在更加关注这些问题?这些不是一直存在的吗?
- **9:15** - 引用Dora报告:AI是一个放大器,擅长的会更好,不擅长的会更差
- **9:45** - AI对软件开发生命周期的影响分析:
  - AI编写更多代码
  - 开发者对代码的理解减少
  - 所有权变得不清晰
  - 在混乱中管理事故
- **10:30** - 代码理解问题:即使在人类编写所有代码时,跨团队理解就已经是问题,现在更加严重
- **11:00** - Pull Request堆积:审查速度加快但关注度降低,部落知识流失
- **11:30** - 所有权不清:服务级别的所有权已经是问题,现在代码级别的所有权也成为问题
- **11:45** - 事故管理困难:在不理解代码的情况下处理事故,风险增加

### 缓解风险的实践 (12:00-15:30)
- **12:00** - 提出解决方案:如何实际缓解这些风险?
- **12:30** - 实践1:代码质量和测试
  - 采用代理策略的团队发现编写测试很有价值
  - 人类应该定义测试边界,代理在边界内操作
- **13:15** - 实践2:设置监控和SLO
  - 即使不完全理解代码,也要能检测问题
  - SLO可以捕获客户影响
  - 至少衡量输出(客户影响),即使无法衡量输入
- **14:00** - 实践3:安全实践
  - 不要让仓库处于无人管理状态
  - 反模式:只关注已知的活跃或关键仓库
  - 所有仓库都包含潜在风险代码,需要在所有仓库执行安全实践
- **14:45** - 实践4:保持人类参与,创建明确责任制
  - 建立最新的所有权信息
  - 创建正确的文化:当我知道责任在我时,会投资于测试和更好的代码
  - 明确的可问责所有权比以往任何时候都更重要

### 行业共识 (15:30-17:30)
- **15:30** - 调研验证:所有被调研的工程领导者都认同安全、测试、纪律和所有权清晰是安全扩展AI编码助手的关键
- **16:00** - 重要观察:这些不是新概念
  - 大多数组织仍然没有强大的生产就绪或安全流程
  - 大多数组织只有"感觉"而非正式的生产就绪流程
- **16:45** - 生产就绪的组成部分:监控、SLO、良好的代码质量
- **17:00** - 核心论点:这些是长期讨论的话题(生产就绪、所有权、测试),但AI是放大器,应该关注AI正在放大的领域

### AI卓越框架 (17:30-20:00)
- **17:30** - 总结:如何自信地推出AI编码工具?
- **17:45** - 阶段1:建立稳定基础
  - 测试实践和框架
  - CI工具投资
  - 生产就绪流程
  - 安全流程
  - 这些无论如何都应该做,但在AI时代更加重要
- **18:30** - 阶段2:衡量编码助手对这些指标的影响
  - 不仅仅是速度提升
  - 关注客户影响:质量、MTTR、事故数量
  - 关注二阶影响,而非AI指标本身
- **19:00** - 挑战常见做法:不要纠结于"AI编写了多少百分比的代码"
  - 类比:我们不会问使用Vim vs IntelliJ的开发者谁更好
  - 关注最终结果和影响
- **19:30** - 阶段3:充分利用AI工具
  - 在AI编码助手上投入大量资金
  - 类比云迁移:新技术需要新能力,要确保获得投资回报

### Cortex解决方案 (20:00-24:00)
- **20:00** - IDP(内部开发者门户)如何驱动AI卓越的三个阶段
- **20:30** - 阶段1:驱动就绪性
  - Cortex的核心用例:定义最佳实践并在组织中扩展
  - 过去5年客户使用案例:电商和税务公司为黑色星期五/税季做生产就绪
  - 安全团队使用Cortex驱动安全标准和左移
- **21:15** - AI就绪评分卡:
  - 确保每个仓库有所有者
  - 启用密钥扫描和漏洞扫描
  - 报告代码覆盖率且不回归
  - 设置防护栏,让团队在其中自主操作
- **22:00** - 阶段2:衡量影响
  - 强调客户影响,而非仅仅是采用率
  - 关联AI使用与MTTR、事故、周期时间等瓶颈指标
- **22:30** - 阶段3:AI成熟度
  - 采用最佳实践
  - 在仓库中设置代理指令文件(适用于Q或其他工具)
  - Jira票据最佳实践
  - 规范优先开发:规范是否真的好?
- **23:00** - 挑战:不同团队和个人的成熟度差异很大
  - 这是新技术,需要带上所有人
  - AI成熟度评分卡类似生产就绪评分卡,定义"优秀"的标准

### Cortex功能特性 (24:00-25:30)
- **24:00** - Mellin AI数据引擎:
  - 自动映射整个工程生态系统并编目
  - 解决所有权难题:数千个仓库如何定义所有权?
  - 使用AI解决AI采用中的问题
  - 自动确定团队对仓库的责任
  - 自动映射PagerDuty轮值
- **24:45** - 开箱即用的功能:
  - AI就绪和AI成熟度评分卡
  - AI影响仪表板
  - MCP查询支持(工程领导者无需学习新工具)
  - 工程智能:直接在IDP中衡量影响

核心要点总结:
- AI是放大器,不是根本性改变,需要加强工程基础
- 关注客户影响指标,而非AI使用率指标
- 三阶段框架:稳定基础 → 衡量影响 → 充分利用
- 所有权和责任制在AI时代比以往更加关键
