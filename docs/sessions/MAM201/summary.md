# AWS re:Invent 2025 - Amazon Elastic VMware Service (EVS) 会议总结

## 一、会议概述

本次 AWS re:Invent 2025 分会场主要介绍了 Amazon Elastic VMware Service (EVS) 这一新服务。演讲者通过搬家的生动比喻来说明云迁移的复杂性——就像搬家时不同物品需要不同的搬运方式一样,不同的工作负载也需要不同的迁移策略。EVS 于 2025 年 8 月正式推出,旨在帮助客户将本地 VMware 环境无缝迁移到 AWS 云端,同时保持相同的人员技能、操作流程和技术工具。

该服务的核心价值在于提供了一种"直接搬运"的迁移路径,让客户能够在 AWS VPC 中运行完整的 VMware Cloud Foundation (VCF) 堆栈,而无需重新培训团队或改变现有的运维方式。与传统的托管服务不同,EVS 是一个完全由客户管理的解决方案,客户拥有完整的管理员权限,可以像在本地环境一样操作 VMware 环境。同时,客户还能充分利用 AWS 生态系统中的其他服务,如 RDS、S3、EFS 等,实现混合架构的灵活部署。

会议还邀请了纽约大学朗格尼医疗中心(NYU Langone Health)的云工程经理分享了他们的实际应用案例,展示了如何通过 EVS 将应用迁移时间从平均 320 天缩短到几周,并实现了业务连续性和灾难恢复的目标。

## 二、详细时间线与关键要点

### 开场与背景介绍 (00:00 - 05:30)
- **00:00** - 会议开场,演讲者通过互动问题引入话题:询问观众搬家次数
- **01:15** - 分享关键数据:美国 18-45 岁人群平均搬家 11 次
- **02:30** - 介绍会议筹备趣事:原计划在舞台上放置搬家卡车和箱子,但被产品营销团队否决
- **03:45** - 展示 AI 生成的客厅图片,引出核心比喻:搬家时不同物品需要不同的搬运方式

### 云迁移策略与 EVS 服务介绍 (05:30 - 15:00)
- **05:30** - 阐述核心理念:云迁移没有"一刀切"的方案,不同工作负载需要不同的迁移方法
- **07:00** - 介绍 AWS 的"七个 R"迁移策略:重新托管(Rehost)、重新平台化(Replatform)、重构(Refactor)、重新定位(Relocate)等
- **08:45** - 正式介绍 Amazon Elastic VMware Service (EVS):在 AWS VPC 中运行 VMware 工作负载的服务
- **10:20** - 强调 EVS 使用商用现成软件(COTS),而非定制版本的 VMware 堆栈
- **12:00** - 说明服务特点:运行在 EC2 裸机实例上,客户拥有完全管理员权限
- **13:30** - 介绍定价模型:支持按需付费、一年期和三年期实例节省计划,采用自带订阅(BYOL)模式

### EVS 的规模与合作伙伴生态 (15:00 - 20:00)
- **15:00** - 宣布服务扩展:从 8 月发布时的 6 个区域扩展到目前的 12 个区域
- **16:15** - 强调客户管理与合作伙伴管理的选择灵活性
- **17:30** - 展示合作伙伴生态系统:包括托管服务提供商(MSP)和 ISV 合作伙伴
- **18:45** - 说明第三方工具兼容性:NetApp、Pure Storage、Veeam 等工具可在 EVS 中正常运行

### "村庄"比喻与 AWS 服务集成 (20:00 - 25:00)
- **20:00** - 使用"从乡村搬到村庄"的比喻:EVS 让客户从孤立环境迁移到拥有丰富服务的 AWS 生态系统
- **21:30** - 说明 EVS 环境可与 RDS、S3、EFS 等 AWS 服务共存于同一 VPC
- **23:00** - 举例:客户可将 VMware 中的数据库迁移到 RDS,而应用程序继续在 VMware VM 上运行
- **24:15** - 强调 AWS 希望成为运行 VMware 工作负载的最佳平台

### EVS 技术架构 - 三大构建模块 (25:00 - 40:00)

#### 计算资源 (25:00 - 28:30)
- **25:00** - Arthy 接手演讲,介绍 EVS 的三大构建模块
- **26:00** - 说明部署过程:EVS 收集客户输入,使用 Cloud Builder 自动部署 VCF 堆栈
- **27:15** - 介绍共享责任模型:AWS 负责底层基础设施,客户负责 ESXi 层及以上的管理
- **28:00** - 计算实例:目前使用 i4i.metal 裸机实例,已通过 Broadcom 认证支持 VCF 5.2.1

#### 扩展性与集群管理 (28:30 - 32:00)
- **28:30** - 环境结构:最小 4 台主机起步,单个环境最多支持 16 台主机
- **29:45** - 扩展方案:可创建多个环境,通过 Transit Gateway 互联
- **30:30** - 支持多个工作负载域和集群配置

#### 存储选项 (32:00 - 36:00)
- **32:00** - 主存储:i4i.metal 实例自带 30TB 本地 NVMe 存储,使用 vSAN ESA(Express Storage Architecture)
- **33:15** - 存储策略管理:支持 RAID 级别和容错配置
- **34:00** - 扩展存储:集成 Amazon FSX for NetApp ONTAP 作为额外数据存储
- **35:00** - 第三方存储:支持 Pure Storage、Weka 等 ISV 存储解决方案
- **35:45** - 存储协议:支持 NFS 和 iSCSI 协议挂载额外存储

#### 网络架构 (36:00 - 40:00)
- **36:00** - 网络创新:在传统的 AWS 扁平三层网络中实现 VLAN 子网概念
- **37:00** - 自动化配置:EVS 自动处理 Overlay 和 Underlay 网络配置
- **38:00** - AWS Transit Gateway Route Server:作为 BGP 端点,自动传播路由信息
- **39:00** - 混合连接:支持通过 Direct Connect 或 VPN 连接本地网络

### 定价模型 (40:00 - 42:00)
- **40:00** - 三个定价维度:
  1. EC2 实例成本(支持节省计划和预留实例)
  2. VPC Route Server 端点费用(每个环境需要 2 个端点)
  3. 服务控制平面费用(按小时计费)
- **41:30** - 采用自带订阅(BYOS)模式,客户可使用现有 VCF 许可证

### NYU Langone Health 客户案例 (42:00 - 55:00)

#### 机构背景 (42:00 - 44:30)
- **42:00** - Jignesh Shah 介绍 NYU Langone Health:纽约市大型学术医疗系统
- **42:45** - 关键数据:320 天是企业应用迁移到云的平均时间
- **43:30** - 机构规模:5 个主要院区、400 个门诊中心、每天 200 万门诊量、53,000 名员工
- **44:00** - 医疗成就:多个专科排名第一,医学院免学费

#### 采用 EVS 的背景 (44:30 - 47:00)
- **44:30** - 触发事件:管理层要求将迁移时间从"数月"缩短到"数周"
- **45:15** - 评估过程:团队制定详细的成功标准,评估多个云服务提供商
- **46:00** - 业务目标:加速云迁移、实现业务连续性

#### 业务需求 (47:00 - 50:00)
- **47:00** - 保持控制权:需要对 vSphere 集群的完整管理员访问权限
- **47:45** - 运维延续性:保持现有的补丁、监控和安全流程
- **48:15** - 灵活许可:自带许可证(BYOL)模式
- **48:45** - 多层安全:结合 AWS 安全控制(SCP、RCP)和 NSX 安全功能
- **49:15** - 减少人员流失:利用现有技能,逐步提升能力
- **49:45** - 降低资本支出:从 CapEx 转向 OpEx 模式

#### 选择 EVS 的原因 (50:00 - 52:30)
- **50:00** - 在客户自己的 VPC 中运行(而非供应商管理的 VPC)
- **50:30** - 基础设施即代码:可使用现有 CI/CD 流水线
- **51:00** - 自带 VCF 许可证
- **51:15** - 小规模起步:从 4 节点试点集群开始,按需扩展
- **51:45** - 保持 ISV 合作伙伴生态系统
- **52:00** - 自动化安全配置
- **52:15** - 按节点统一定价:不按 VM 或 EC2 实例计费

#### 经验教训 (52:30 - 54:30)
- **52:30** - 运维化挑战:需要构建自定义模块集成 EVS 监控到现有服务管理系统
- **53:00** - 网络路由复杂性:Overlay 和 Underlay 网络之间的路由配置需要时间
- **53:45** - 数据入口成本:出现了意外的数据入口费用(通常 AWS 入口免费)
- **54:15** - CloudFormation 学习曲线:团队需要适应 CloudFormation 部署方式

#### 未来展望 (54:30 - 55:00)
- **54:30** - 期待多可用区(Multi-AZ)支持
- **54:45** - 希望更多 ISV 合作伙伴认证
- **54:55** - 期待 Terraform 支持

### 现场演示 (55:00 - 结束)
- **55:00** - Arthy 演示 AWS 控制台中的 EVS 部署界面
- **55:30** - 展示部署前的先决条件和规划信息
- **56:00** - 说明需要收集的信息:IP 地址空间、网络配置等
- **56:30** - 强调本地团队和云团队通常是分开的,需要提前规划
- **57:00** - 演示被截断(字幕未完整)
