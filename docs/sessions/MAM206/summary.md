# AWS re:Invent 2025 会议总结：企业云迁移与现代化转型实践

## 会议概述

本次 AWS re:Invent 2025 分会场聚焦于企业级云迁移、现代化和转型的实践经验。会议由 AWS 企业加速计划(EBA)全球市场负责人 Bhavna Sarati 主持,伦敦证券交易所集团(LSEG)云转型负责人 Brian Quinn 和 AWS 首席客户解决方案经理 Magnus Cumin 共同分享了真实的转型案例。

核心观点强调,成功的云转型不仅仅是技术问题,更重要的是人的因素。研究显示,企业在云迁移过程中遇到的 70% 的摩擦点实际上并非技术性问题,而是组织和人员相关的挑战。主要障碍包括:38% 的客户面临团队孤岛问题,45% 的客户存在优先级冲突,40% 的客户缺乏必要的 AWS 云技能。

AWS 的体验式加速(EBA)计划提供了一个系统化的解决方案,通过沉浸式、敏捷的实践方法,帮助企业在 2-3 天内完成原本需要数周或数月的工作。LSEG 的成功案例证明,通过正确的方法论和专业支持,即使是最复杂的迁移和现代化项目也能转化为成功故事。该公司已经运行了超过 11 次 EBA 活动,涉及 13 个团队、12 个应用程序,识别并消除了 50 多个关键障碍,实现了 26 项流程改进。

- **4:20** - 会议议程:为什么要进行云转型、LSEG 云旅程、实施建议

### 云转型的核心要素 (5:30-12:00)
- **5:45** - 核心观点:成功云转型的第一要素是"人",而非技术
- **6:30** - 强调组织准备度、利益相关者协调和专业能力建设的重要性
- **7:00** - 关键数据:70% 的摩擦点是非技术性的
- **7:45** - 客户面临的三大挑战:
  - 38% 的客户受团队孤岛阻碍
  - 45% 的客户面临优先级冲突
  - 40% 的客户缺乏 AWS 云技能
- **9:30** - 场景示例:开发人员完成容器化、微服务、数据库迁移等技术工作后,却因跨团队协调困难而陷入停滞
- **11:00** - 成功要素:技术能力、高管支持、团队协作、及时培训

### EBA 计划介绍 (12:00-18:30)
- **12:15** - 介绍体验式加速(EBA)计划:沉浸式、敏捷、实践性的能力建设方法
- **13:00** - EBA 的三大价值:
  - 打破组织孤岛,建立组织能力
  - 加速成果交付,实现快速决策
  - 让开发者专注于构建和创新
- **14:30** - EBA 实践:2-3 天内完成原本需要数周或数月的工作
- **15:20** - 分享 EBA 现场照片:应用团队、自动化平台、云卓越中心、安全团队共同参与,设有指挥中心快速解决障碍
- **16:45** - EBA 三大支柱:迁移、现代化、创新
- **17:30** - 强调 EBA 不是一次性活动,客户会规划多次 EBA 活动

### LSEG 公司背景 (18:30-23:00)
- **18:45** - Brian Quinn 介绍 LSEG:全球领先的金融市场基础设施和数据提供商
- **19:30** - 全球业务:在 65 个国家运营,员工超过 26,000 人
- **20:15** - 五大业务部门:数据与分析、富时罗素指数、风险情报、资本市场、交易后服务
- **21:30** - 重点关注交易后服务部门的 LCH 清算业务
- **22:00** - 云转型的四大业务驱动因素:
  - 业务韧性和可扩展性
  - 成本效率和技术投资优化
  - 创新和实验平台
  - 缩短上市时间

### LSEG 云旅程时间线 (23:00-25:30)
- **23:15** - 2020 年:LCH 云旅程启动,初步规划和监管沟通
- **23:45** - 2022 年:关键年份,启动云计划,团队动员,首个应用迁移到 AWS
- **24:15** - 2023 年:迁移门户和报告应用(所有 LCH Web 应用的中央访问点)
- **24:45** - 2025 年 Q2:重要里程碑,抵押品管理系统(CMS)上线
- **25:00** - 2025 年 Q3:现代化可观测性策略,寻求 AWS 支持
- **25:15** - 2026 年及以后:计划多个迁移和现代化项目,包括 EKS 迁移

### 面临的挑战 (25:30-28:30)
- **25:45** - 团队孤岛:传统运营模式的副产品,开发、运维、安全各自为政
- **26:30** - 解决方案:实施 DevSecOps 最佳实践,自动化平台,基础设施即代码
- **27:00** - 优先级挑战:平衡日常运维与云迁移活动,管理技术债务,处理遗留应用依赖
- **27:45** - 采用基于风险的优先级框架:先迁移小型、低关键性工作负载,积累经验后处理复杂工作负载
- **28:15** - 最大障碍是组织和人员问题,而非技术问题

### 第一次 EBA:Consent Controller 迁移 (28:30-33:00)
- **28:45** - 背景:团队寻求迁移 FX 交易清算服务的 Consent Controller 应用
- **29:15** - 问题:收到大量意见、PowerPoint 和视频会议,但应用仍在本地
- **29:45** - 转折:首席工程师发现 EBA 机制,决定尝试
- **30:30** - 初始目标:简单的提升和迁移(Lift and Shift)
- **31:00** - 成果:第一天完成迁移,随后实现容器化
- **31:30** - 额外成就:
  - 将应用部署到 Kubernetes 容器化 Pod
  - Oracle 数据库转换为 PostgreSQL
  - 集成回私有云
  - 在 EC2 上运行 Cucumber 测试
- **32:15** - 关键价值:不仅是迁移应用,更重要的是建立的模式、经验教训和流程改进
- **32:45** - 该团队随后又运行了三次 EBA

### 第二次 EBA:抵押品管理系统(CMS)迁移 (33:00-38:30)
- **33:15** - CMS 是 LCH 技术上最关键的应用之一
- **33:45** - 监管要求:被全球 40 多个监管机构(包括英格兰银行)正式分类为关键重要业务服务
- **34:30** - 严格的恢复目标:
  - RTO(恢复时间目标):2 小时
  - RPO(恢复点目标):零数据丢失
- **35:00** - 架构方案:
  - 多区域架构,每个区域三个可用区
  - AWS Direct Connect 本地连接
  - 容器化集群 Pod 运行在 Amazon EC2
  - 多可用区 RDS 提高韧性
- **36:00** - 挑战:如何向监管机构测试这种韧性水平?
- **36:30** - 创新方案:将 EBA 与混沌工程结合
- **37:00** - 三天内使用 AWS Fault Injection Simulator 运行超过 10 个混沌工程实验
- **37:30** - 实验类型:
  - EC2 实例故障转移和重启
  - RDS 故障转移
  - 网络延迟和严重网络降级
  - EBS 卷故障
- **38:00** - 主题:夏季主题,团队穿着主题服装,打破团队壁垒,建立友谊

### 第三次 EBA:可观测性现代化 (38:30-43:00)
- **38:45** - 背景:优化和增强可观测性策略,CloudWatch 近年来有重大改进
- **39:15** - 准备工作:深入研讨会、逆向工作会议、策略审查、沉浸日活动
- **39:45** - 挑战:原定参与的应用团队因冲突退出
- **40:15** - 敏捷应对:快速调整,找到两个新团队,推迟日期继续进行
- **40:45** - 关键时刻:启动时发现可能完全阻塞整个 3 天流程的问题,但因为合适的人在场,在赞助商启动会议时后台修复了问题
- **41:30** - 成果:
  - 在基础设施上部署全面的 CloudWatch 代理
  - 开发实时业务指标
  - 创建自定义指标和过滤器
  - 使用 CloudWatch 和 SNS 实现自动告警
- **42:00** - 延伸目标:真实用户监控(RUM)和 CloudWatch Synthetics
- **42:30** - 团队协作:两个之前不熟悉的团队从第一天坐在房间两侧,到第二天开始协作,分工攻克延伸目标,最终互相教授所学内容

### EBA 成果统计 (43:00-45:30)
- **43:15** - 数字成果:
  - 超过 270 名参与者
  - 215 次迁移庆祝(敲响牛铃)
  - 11 次以上 EBA 活动
  - 涉及 13 个团队
  - 12 个应用程序
  - 5 个迁移计划
- **44:00** - 最重要的两个指标:
  - 26 项流程改进:这些改进已自动化并在整个组织推广
  - 50 多个关键障碍被消除:避免了可能导致项目停滞数天或数周的问题
- **44:45** - 上下文切换的成本:每次上下文切换会损失 20-30% 的效率
- **45:00** - LSEG 建立了自己的 EBA 卓越中心(COE)
- **45:15** - EBA COE 团队在可观测性 EBA 中跟随 AWS 学习,现在独立运行 EBA,2026 年日程已排满

### 领导力的重要性 (45:30-47:00)
- **45:45** - Brian 强调领导力协调的重要性
- **46:15** - Rob(LSEG 市场和风险情报 CIO)是所有 EBA 的赞助商
- **46:30** - 领导不仅提供支持,还直接参与,出席最终演示并提供反馈
- **46:45** - 领导的参与激励团队继续推进

### 实施建议:奠定基础 (47:00-51:30)
- **47:15** - Magnus Cumin 开始分享实施建议
- **47:45** - 三个阶段:奠定基础、无缝交付、实现规模化
- **48:15** - 强大的高管赞助:
  - 基本层面:支持获取所需资源
  - 更重要:直接参与,接触利益相关者,消除障碍,跨孤岛工作
- **49:00** - 案例:生成式 AI 交付中,团队因需要新 AWS 服务白名单而受阻,高管赞助商直接与云安全团队沟通,提供安全专家时间,审查服务批准文档,最终获得批准
- **50:00** - 实验心态:INSEAD 商学院研究表明,这可能是数字化转型中团队成功的最重要决定因素
- **50:45** - 为什么重要:团队需要采用新工作方式、学习新技能和 AWS 服务,需要知道高层领导支持他们,即使出错也没关系
- **51:15** - 案例:某客户工程主管在每次启动会或全员会议上强调,尝试新事物是可以的,如果在过程中学到了新东西,失败也是可以的

### 实施建议:无缝交付 (51:30-55:00)
- **51:45** - 延伸目标:团队应有核心目标,但设置感觉略微遥不可及的延伸目标
- **52:15** - 经验法则:如果团队觉得有 50/50 的机会达成目标,那就差不多了
- **52:45** - 双重好处:
  - 团队始终有待办事项
  - 延伸目标是强大的激励因素
- **53:15** - 早期云旅程中,团队往往低估自己能完成的工作量,通过设置和实现延伸目标,给予他们自信
- **53:45** - 敏捷项目管理方法:
  - 短冲刺周期
  - 理想情况下团队集中办公
  - 专注于交付增量价值
  - 向组织展示构建成果
- **54:30** - 重要提醒:敏捷项目管理不等于需要最新的项目管理软件工具,每日站会、看板(已完成、进行中、受阻)才是核心
