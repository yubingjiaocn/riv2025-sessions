# AWS re:Invent 2025 STG213 会议总结

## 会议概览

本次会议主题为"Motability Operations 统一备份策略"(STG 213 Motability Operations Unified Backup Strategy),由AWS高级解决方案架构师Sabit Wing Rajabadi和Motability Operations平台经理Nick Ro共同主讲。会议详细介绍了Motability Operations如何从分散的备份方案转变为统一的、具有防御能力的不可变备份系统。

Motability Operations是一家为残障人士提供汽车租赁服务的英国公司,拥有超过90万客户和1500多名员工。面对日益严峻的勒索软件威胁和跨平台复杂性挑战,该公司决定与AWS合作,实施全面的不可变备份恢复系统。项目从2025年1月开始技术规划,历经设计研讨、概念验证、部署实施等多个阶段,成功建立了基于3-2-1-1策略的多层备份架构。

整个项目的核心围绕三大支柱展开:不可变性(immutability)、隔离性(isolation)和完整性(integrity)。通过使用AWS Backup、EventBridge、Lambda等AWS原生服务,结合Elastio平台进行备份完整性验证,Motability Operations成功构建了一个能够抵御勒索软件攻击、确保业务连续性的企业级备份解决方案。

## 详细时间线与关键要点

00:00 - 会议开场与介绍
- AWS高级解决方案架构师Sabit Wing Rajabadi介绍会议主题
- 欢迎Motability Operations平台经理Nick Ro作为客户代表分享实践经验

01:30 - Motability Operations公司介绍
- 公司业务:为残障人士提供轮椅无障碍车辆、踏板车和电动椅的全包租赁服务
- 规模数据:超过86万客户(实际已达90万),10万电动车驾驶员
- 客户满意度:9.4-10分的优秀评分
- 员工分布:1500多名员工分布在伦敦、布里斯托和爱丁堡
- 企业愿景:为残障人士提供可负担的可持续出行方案

03:00 - 核心价值观阐述
- 寻找解决方案、倾听客户需求
- 适应变化、共同解决问题
- 推动变革、引领出行创新和可持续发展
- 关怀客户和员工

04:00 - 为什么需要不可变备份
- 勒索软件和网络攻击防护:不可变备份被锁定和保护,不同于可能被破坏的传统备份
- 合规性和审计准备:满足监管合规要求,保证数据保留和审计追踪
- 数据完整性和可恢复性:确保每个备份都是原始状态,可信任的系统恢复
- 业务连续性和韧性:可靠的快速恢复,满足业务连续性目标

06:00 - 面临的挑战和技术驱动因素
- 需要已知良好的恢复状态:确保在遭受攻击时能够从干净的备份中恢复
- 跨平台复杂性:自2018年使用AWS以来,平台和服务规模不断增长,存在多种原生AWS备份解决方案
- 检测和响应能力差距:需要取证扫描能力来确保备份的可靠性

08:00 - 项目目标和可交付成果
- 创建Moscow需求框架(必须有、应该有、可以有、不会有)
- 评估多个SaaS服务提供商,最终选择与AWS合作
- 关键可交付成果:在所有AWS环境中部署不可变备份恢复系统
- 完整移交给运营团队,更新恢复运行手册并进行测试
- 增强安全态势,减少网络攻击的业务影响

10:00 - 转型方法的四个阶段
- 第一阶段:设计研讨会,架构师团队(包括AWS架构师)共同制定部署方案
- 第二阶段:概念验证(POC),验证方案可行性和成本效益
- 第三阶段:定义范围,确定哪些Motability服务和技术需要纳入
- 第四阶段:敏捷部署,过去一年中跨多个平台团队实施,设置不同里程碑

12:00 - AWS合作伙伴关系
- AWS团队与Motability工程团队紧密合作
- 进行全面的威胁建模,理解客户需求
- 采用AWS的"逆向工作法"(working backwards),从最终状态开始规划
- 共同开发全面的保护需求,为长期执行策略奠定基础

14:00 - 3-2-1-1策略适配
- 采用行业标准的3-2-1-1框架并针对Motability需求进行定制
- 三大核心支柱:不可变性(immutability)、隔离性(isolation)、完整性(integrity)
- 加上高可用性(high availability)作为成功恢复策略的定义支柱
- 持续监控项目成功度,评估设计和实施,根据需要调整行动方案

16:00 - 备份机制详解
- 事件驱动的复制工作流:使用EventBridge触发Lambda进行成功备份复制
- CloudWatch监控和重试日志记录
- CMK(客户管理密钥)策略:在所有平台中间保管库中实现一致的CMK部署
- IAM角色分离:为备份创建、管理和恢复定义不同角色,无用户分配重叠

18:00 - 解决方案架构
- 工作负载账户连接到集中式备份账户
- 取证账户(集成Elastio进行验证)
- 管理账户
- 恢复账户(独立的恢复环境)

20:00 - 备份工具构建
- 集中式不可变备份保管库(Tier 2):按平台分离保管库,锁定在合规模式,按保留策略分离
- 账户级备份(Tier 1):使用Amazon托管密钥的中间保管库,使用CMK加密
- 每个平台一个CMK密钥

22:00 - 平台特定实施
- Motability特定平台:Native、OSMO和NCW
- Tier 1和中间保管库锁定在治理模式
- 基于资源类型的备份工作流
- 按平台分离保管库,锁定在合规模式,按保留策略分离

24:00 - 勒索软件恢复的三大支柱
- 不可变性:备份数据存储在无法远程访问的气隙环境中
- 隔离性:备份数据无法被更改、删除或篡改
- 完整性:备份数据经过验证,干净、未受损且可恢复(使用Elastio)

26:00 - Elastio平台介绍
- 支持的AWS服务:EC2、S3存储桶、AWS Backup、GuardDuty、EFS等
- 检测隐藏的数据泄露:机器学习驱动引擎主动扫描备份中的加密和损坏
- 检测准确率:99.99%
- 确保未受损的恢复点
- 与AWS账户无缝集成,实施非常简单

28:00 - 项目时间线
- 2025年1月:技术规划启动
- 2025年2月:集中式AWS备份账户设置
- 2025年3月:取证账户和Elastio配置
- 3月后:持续推进各AWS账户部署(初期有挑战,后期流程化)
- 2025年11月:渗透测试和审计(发现问题很少,级别较低)
- 2025年12月-2026年2月:低级文档编写和项目收尾

30:00 - 结论与建议
- AWS与Motability Operations成功合作,实现大规模数据保护策略
- 建议从明确的愿景开始,定义最终状态
- 采用AWS的逆向工作法
- 做出高速双向门决策,保持灵活性和适应性
- 以最安全和最具成本效益的方式实现目标

32:00 - 最佳实践推荐
- 尽可能利用自动化
- 使用AWS提供的参考架构蓝图(取证账户、恢复账户、网络保管库账户)
- 实施最小权限许可和纵深防御策略
- 考虑使用AI代理自动化任务
- 使用AWS Well-Architected框架评估实施情况
- 利用AI代理大规模自动化恢复流程

34:00 - 会议结束
- 鼓励参会者通过re:Invent应用提供反馈
- 提供线下交流机会
