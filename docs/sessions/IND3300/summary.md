# AWS re:Invent 2025 技术会议总结

## 会议概述

本次会议由AWS首席解决方案架构师Hassan Tarik主持,Stripe公司的Chrissy和Christopher作为嘉宾分享了他们在合规领域应用AI代理(Agentic AI)的实践经验。

会议聚焦于金融服务等受监管行业面临的合规挑战。根据Forrester研究,全球企业每年在金融犯罪相关运营上的支出高达260亿美元,仅北美地区就达到610亿美元。更严峻的是,欧洲合规团队反馈监管要求每年增长高达35%。Experian的调查显示,合规团队约三分之一的任务可以实现自动化,这意味着每周可以为分析师节省8-12小时,让他们专注于更具战略性的分析和决策工作。

Stripe作为全球领先的支付基础设施平台,去年处理了1.4万亿美元的交易量(占全球GDP的1.38%),同比增长38%。面对业务的爆炸式增长,Stripe需要在不牺牲安全性和合规性的前提下实现规模化。他们开发了基于Amazon Bedrock的AI代理解决方案,用于自动化增强尽职调查(EDD)流程。该方案通过异步工作流编排实现并行处理,无缝集成现有合规工具,并能够7×24小时不间断工作,在审查人员打开工单之前就完成调查和洞察总结。

核心设计理念是"人机协作":AI代理负责繁重的数据收集和初步分析工作,而人类专家保留最终决策权。系统保持完整的审计追踪,记录每个决策及其背后的理由,确保符合监管要求。这种方式不仅提升了效率和一致性,还让合规分析师能够处理更多案例,深入进行更复杂的判断。

### 合规挑战详解 (13:00 - 20:00)
- **13:00** - 合规不仅是遵守法律,更是信任的基础
- **13:30** - 两个关键维度:生态系统完整性和用户保护
- **14:00** - 执行KYC(了解你的客户)、KYB(了解你的业务)和反洗钱措施
- **15:00** - 四个影响因素:增长、规模、时效性、技术创新
- **16:00** - 核心挑战:需要对高风险用户进行广泛审查
- **17:00** - 手动流程的两大障碍:专家审查员充当"导航员"而非分析师
- **18:00** - 可扩展性和司法管辖区切换的认知负荷
- **19:00** - 不同司法管辖区(如加州、阿联酋、新加坡)需要完全不同的思维方式
- **19:30** - 风险阈值是可变的,安全业务的定义不是静态的

### AI代理解决方案 (20:00 - 28:00)
- **20:00** - 解决方案:构建NLM驱动的AI研究代理进行自主调查
- **21:00** - 三种方式的重负载工作:异步工作流编排实现并行处理
- **22:00** - 直接插入现有合规工具,自然融入审查员熟悉的工作流
- **23:00** - 代理永不休息,可在审查员打开工单前完成调查和洞察总结
- **24:00** - 三个关键要点:监督(人类中心验证)
- **25:00** - 完整的审计追踪,记录每个决策及其理由
- **26:00** - 通过预取分析增强审查能力,使专家分析师能更深入、更快速地工作
- **27:00** - 构建可信、可审计、规模化的代理合规解决方案

### 技术实现 - 问题设置 (28:00 - 35:00)
- **28:00** - Christopher开始技术深入讲解
- **29:00** - 问题:非常长的手动审查流程
- **30:00** - 不能简单地将代理应用于整个工作流
- **31:00** - 代理可能在不需要的地方花费太多时间,在需要的地方花费不够
- **32:00** - 解决方案:将复杂工作流分解为DAG(有向无环图)
- **33:00** - DAG作为代理的"轨道",使用React代理
- **34:00** - 任务变得足够小,适合代理的工作记忆
- **34:30** - 使用代理的原因:能够调用工具

### React代理机制 (35:00 - 42:00)
- **35:00** - React代理的思考方式类似人类处理复杂问题
- **36:00** - 示例:计算10除以π,需要调用计算器工具
- **37:00** - 思考-行动-观察循环
- **38:00** - 在合规审查中,查询更加复杂
- **39:00** - 可能需要多次查询循环
- **40:00** - 代理需要深入挖掘,同时覆盖所有基础
- **41:00** - 提示词可能变得非常长
- **41:30** - 成本呈二次方增长(1+2+3+4...)

### Amazon Bedrock集成 (42:00 - 50:00)
- **42:00** - 输入令牌成本占主导地位
- **43:00** - 解决方案:提示词缓存(Prompt Caching)
- **44:00** - 缓存将二次成本降低为线性成本
- **45:00** - 使用Bedrock作为LLM提供商
- **46:00** - 内部LLM代理服务解决"嘈杂邻居"问题
- **47:00** - 集中式代理服务确保带宽不被其他团队占用
- **48:00** - 其他优势:可用性保证、授权、模型路由
- **49:00** - 支持模型回退配置

### 选择Bedrock的原因 (50:00 - 55:00)
- **50:00** - 隐私和安全的标准化
- **51:00** - 一次审核即可访问多个模型供应商
- **52:00** - 提示词缓存等功能可节省大量成本
- **53:00** - 微调功能允许按自己的时间表弃用模型
- **54:00** - 一个API访问多个模型,减少集成开销

### 代理服务架构 (55:00 - 65:00)
- **55:00** - 项目初期没有专门的代理服务
- **56:00** - 尝试将代理工作流集成到传统ML推理系统中,但被否决
- **57:00** - 传统ML与代理需要完全不同的硬件配置
- **58:00** - 传统ML:计算密集型,需要GPU或大量CPU,延迟一致,超时时间短
- **59:00** - 代理:等待LLM响应,需要大量线程,超时时间长(5-10分钟)
- **60:00** - 代理的控制流非常不确定
- **61:00** - 代理服务历史:Q1初不存在
- **62:00** - Q1末:一个月内建立单体服务,提供同步API
- **63:00** - Q2:添加评估、追踪功能,开发无代码代理UI
- **64:00** - Q3:大规模扩散,遇到容量限制,转向每个用例独立服务
- **64:30** - Q4:添加有状态流式API支持聊天机器人用例

### Stripe代理生态系统 (65:00 - 70:00)
- **65:00** - Stripe目前有超过100个代理
- **66:00** - 审查员通过Web UI与系统交互
- **67:00** - 审查工具作为整个DAG流程的编排器
- **68:00** - 代理可以在审查开始前预先运行研究
- **69:00** - React代理通过LLM客户端和工具与生态系统交互
- **69:30** - 支持MCP客户端和Python工具

### QA与质量保证 (70:00 - 72:00)
- **70:00** - 严格的QA流程(未在架构图中显示)
- **71:00** - 监管机构需要确保系统可靠运行
- **71:30** - 虽然LLM作为评判者很流行,但该项目坚持人工质量标准
- **72:00** - 所有内容都需要通过人工质量审核
