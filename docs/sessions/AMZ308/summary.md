# AWS re:Invent 2025 FinOps 会议总结

## 会议概述

本次会议由 Nathan Perry 主讲,他是 AWS Optics 团队的高级云 FinOps 架构师,在亚马逊工作了 11 年。过去 5 年中,他专门为亚马逊及其所有业务线提供 AWS 客户支持,帮助亚马逊理解如何在云端有效运营、降低成本、提升成本可见性和成本分配能力。

Perry 分享了亚马逊在 FinOps 现代化转型过程中的三个关键经验:首先,基于 AWS 账单和成本管理服务构建基础;其次,通过与业务目标一致的机制驱动效率;第三,通过智能自动化扩展 FinOps 实践。这些洞察旨在帮助处于不同阶段的组织加速其财务管理系统与 AWS 账单和成本管理的集成。

亚马逊的 FinOps 之旅始于自定义财务报告,提供月度云成本视图。如今,他们正在使用 AWS Data Exports、AWS 成本和使用报告(CUR)以及其他 AWS 账单和成本管理服务来转型这一基础。他们从月度粒度和账户粒度的成本可视化转向 ARN 粒度和小时粒度,将成本数据民主化,让构建者、领导者、财务和 FinOps 团队都能访问。通过部署全组织范围的标签策略、集成 Compute Optimizer 和 Cost Optimization Hub,亚马逊正在从集中式报告转向分布式成本智能模型。

## 详细时间线与关键要点

[00:00 - 02:30] 开场介绍与背景
- Nathan Perry 自我介绍,AWS Optics 团队高级云 FinOps 架构师
- 在亚马逊工作 11 年,过去 5 年专注于为亚马逊业务线提供 AWS 客户支持
- 独特的工作性质:为亚马逊工作,但客户也是亚马逊

[02:30 - 04:00] 三大核心经验概述
- 第一:基于 AWS 账单和成本管理服务构建基础
- 第二:通过与业务一致的机制驱动效率
- 第三:通过智能自动化扩展 FinOps 实践

[04:00 - 08:30] 数据基础建设
- 亚马逊所有业务线都在 AWS 上运营
- FinOps 之旅始于自定义财务报告,提供月度成本视图
- 现在使用 AWS Data Exports、成本和使用报告(CUR)进行转型
- 从月度/账户粒度转向 ARN 粒度和小时粒度
- 成本数据民主化,向所有团队开放
- 在整个组织中启用 Cost Explorer 实现自助成本分析
- 部署全组织标签策略以跟踪投资
- 集成 Compute Optimizer 和 Cost Optimization Hub
- 利用 AWS Organizations 建立一致的控制框架

[08:30 - 13:00] 业务背景整合与效率驱动
- 关键挑战:推动成本管理实践的广泛采用
- 核心洞察:将云成本与团队层面的业务成果简单连接
- 结合两个关键要素:CUR 的细粒度成本 + 团队实际关心的业务指标
- 将账户和基于标签的成本分配与业务背景、投资跟踪相结合
- 自动化 ROI 分析
- 利用 AWS Cloud Intelligence Dashboard(CID)简化成本可见性
- CID 是开源框架,为任何规模的客户提供可操作的洞察
- 将 AWS 服务特定的预算数据与实际使用情况对比
- 创建角色特定视图以实现更深入的服务分析

[13:00 - 18:00] 效率机制建设
- 认识到效率不是一刀切的
- 早期测量基本资源利用率指标:CPU 利用率、内存使用模式、网络吞吐量
- 不同工作负载需要不同方法(如 Amazon Retail vs Alexa vs Prime Video)
- 创建"信用评分"(Credit Score)指标来衡量资源效率
- 评分包含:Graviton 使用、容量利用率、存储类优化等
- 该指标衡量所有业务线的 FinOps 成熟度
- 团队每周收到效率评分和成本建议
- 建议可按技术类别分组:存储、计算、生成式 AI、数据库、网络
- 所有利益相关者都能从各自角度查看数据

[18:00 - 23:00] 自动化与智能循环
- 云财务管理不是孤立任务,而是持续的智能循环
- 通过集成 AWS 服务和自动化工作流,将手动反应式流程转变为自我改进系统
- 自动化之旅始于简单目标:为团队提供更好的成本可见性
- 在财务规划等关键流程中引入自动化
- 团队通过首选渠道接收优化机会通知
- 对于充分理解的场景,团队可定义触发自动响应的策略和阈值
- 自动化与监督之间的平衡对建立信任至关重要

[23:00 - 26:00] 信任与透明度
- 扩展 FinOps 的关键不仅是技术,还有通过一致的透明度建立信任
- 每个操作(自动或手动)都必须详细记录和跟踪
- 团队需要准确了解基础设施成本发生了什么以及原因
- 透明度优先的方法对推动自动化和规划能力的采用至关重要
- 特别是在亚马逊零售业务中看到了人类洞察与自动化分析结合的影响

[26:00 - 30:00] FinOps 自动化路线图
- 不需要一次性构建所有内容
- 从 AWS 服务作为基础开始,首先关注成本可见性
- 逐步自动化充分理解的流程(如财务规划和容量管理)
- 通过透明度建立信任后,扩展自动化的范围和复杂度
- 目标不是从流程中移除人类,而是将其角色从日常分析提升到战略决策

[30:00 - 36:00] 转型演进与具体实施
- AWS CUR 提供工程师和领导者从未意识到需要的细粒度数据
- Cost Explorer 将分析能力直接交到团队手中
- Organizations 实现大规模治理(管理数万个账户)
- 仅有数据不够,真正的转型发生在将业务成果与云成本连接时
- 示例:了解花费 10 万美元不如了解这 10 万美元带来 100 万美元收入有用
- Cloud Intelligence Dashboard 不仅显示成本,还显示团队层面的价值
- 从月度审查转向每日优化操作
- 智能系统检测异常、推荐优化,甚至自动实施改进

[36:00 - 40:00] 三年演进对比
- 三年前:使用多个工具、手动流程、成本可见性不清晰
- 今天:集成 AWS 服务、自动化洞察
- 展望 2026-2027:AI 增强优化、预测性操作、自动化容量规划
- 这些能力现在就可用

[40:00 - 43:00] 加速路径三层架构
1. 数据基础层:启用 CUR、实施标签策略、熟悉 Cost Explorer 和 Cost Optimization Hub
2. 集成层:部署 Cloud Intelligence Dashboard 连接业务指标、启用自动化建议
3. 优化层:实施效率建议、启用透明度控制、自动化响应操作

[43:00 - 46:00] 总结与关键启示
- 引用 Dunning-Krueger 效应:亚马逊最初对自定义解决方案过于自信
- 最快的路径往往不是最明显的,需要挑战假设
- 这些原则适用于任何规模的组织
- 最明智的选择往往是最简单的选择
