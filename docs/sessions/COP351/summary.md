# AWS re:Invent 2025 - Innovation Sandbox on AWS 会议总结

## 会议概述

本次闪电演讲介绍了AWS的创新沙箱(Innovation Sandbox)解决方案,这是一个专门为解决企业在AWS上进行创新和技能提升时面临的挑战而设计的工具。演讲者分享了自己在高度监管行业的工作经验,指出传统的服务白名单和审批流程严重阻碍了开发人员的技能提升和创新能力。许多企业虽然批准了某些服务的使用,但由于开发人员无法在真实环境中测试,这些服务最终从未被实际使用。

创新沙箱解决方案于2025年5月正式发布,是AWS解决方案团队经过三年半的开发和与数百位客户沟通后推出的产品。该解决方案通过CloudFormation模板部署,大约需要一小时即可完成安装。它提供了一个完整的用户界面,支持通过IAM Identity Center进行身份验证,并且完全由AWS支持和维护。该方案的核心目标是实现创新民主化,让更多用户能够在AWS上进行技能提升和构建应用,同时有效控制成本和时间。

演讲中通过现场演示展示了该解决方案的完整工作流程,包括账户池管理、租约模板配置、用户请求流程以及成本和时间控制机制。该解决方案特别适用于大学、初创公司、高度监管的企业、黑客马拉松活动以及生成式AI应用开发等场景。

## 详细时间线

0:00 - 0:45 - 开场介绍
- 欢迎参加re:Invent 2025第一场闪电演讲
- 主题:AWS上的创新沙箱(Innovation Sandbox on AWS)

0:45 - 2:30 - 为什么需要沙箱环境
- 演讲者分享在高度监管公司的工作经验
- 传统服务白名单限制了开发人员使用特定服务
- 这种限制造成了技能提升和创新的巨大瓶颈
- 客户案例:服务经过审批流程后从未被实际使用,因为开发人员无法在真实环境中测试

2:30 - 4:00 - 传统沙箱方法的挑战
- 管理员创建AWS账户并提供访问权限
- 通常由平台工程团队或身份团队负责
- 多个用户共享同一个沙箱导致资源管理困难
- 成本成为主要关注点
- 清理未使用资源的管理开销巨大

4:00 - 5:30 - 创新沙箱解决方案的开发历程
- AWS解决方案团队开发了Instance Scheduler、Quota Monitor、Landing Zone Accelerator等解决方案
- 三年半前开始构思创新沙箱
- 与数百位客户沟通以获取反馈和开发批准
- 来自大学、初创公司和高度监管客户的需求

5:30 - 6:30 - 解决方案发布和特点
- 2025年5月正式发布
- 获得了比其他解决方案更多的关注
- 适用于各个行业
- 通过CloudFormation模板部署
- 包含四个CloudFormation模板,部署时间约一小时
- 由AWS支持和维护,代码在GitHub上开源

6:30 - 8:00 - 三种用户角色
- 管理员(Admin):负责部署解决方案和底层AWS账户
- 管理者(Manager):无需技术背景,负责将沙箱账户分配给最终用户,管理成本和持续时间
- 用户(User):实际使用沙箱环境的开发人员或研究人员

8:00 - 10:00 - 工作流程概述
- 管理员预先创建一定数量的AWS账户(可以是2个到数千个)
- 账户存储在账户池中
- 管理者通过租约模板(Lease Templates)将账户租赁给沙箱用户
- 管理者可以设置成本和时间控制
- 所有操作通过用户界面完成
- 访问控制通过IAM Identity Center实现,支持第三方身份提供商(Okta、Entra等)

10:00 - 12:00 - 账户请求流程
- 用户通过IAM Identity Center认证
- 用户请求账户或租约
- 可选的审批流程(管理者可在UI中批准或拒绝)
- 沙箱账户被分配
- 用户获得账户访问权限
- 新功能:管理者可以直接将租约分配给用户(适用于黑客马拉松等场景)

12:00 - 14:00 - 基于时间和持续时间的操作
- 发送警报
- 冻结账户
- 清理/回收账户
- 管理者可以随时手动触发这些操作

14:00 - 17:00 - 完整流程示例(黑客马拉松场景)
- 账户池中有账户,管理者和用户准备就绪
- 创建5天持续时间的租约模板
- 5天后账户被冻结
- 用户失去访问权限,但管理者和管理员保持访问
- 管理者可以为特定用户延长租约(例如从5天延长到30天)
- 解冻后用户重新获得访问权限

17:00 - 18:30 - 账户弹出(Eject)选项
- 较少使用的模式
- 当构建了有价值的应用时,可以将账户移出解决方案
- 账户移至开发OU,不再作为沙箱账户管理

18:30 - 20:00 - 账户回收流程
- 最常见的模式
- 可以设置清理时间阈值(例如7-8天)
- 管理者也可以手动清理账户
- 账户移至清理OU
- 使用AWS Nuke清理所有资源
- 账户返回账户池供重复使用

20:00 - 22:00 - 现场演示开始
- 通过IAM Identity Center认证登录
- 展示用户界面
- 管理员可以看到管理标签页
- 管理者可以看到除管理标签页外的所有内容
- 用户只能看到主页

22:00 - 24:00 - 账户池管理演示
- 当前账户池:3个可用账户,4个活跃账户,1个冻结账户,共8个账户
- 显示账户详情和账户ID
- 管理员可以直接登录任何账户(管理员级别访问)

24:00 - 26:00 - 添加账户到账户池
- 点击"添加账户"
- 预先创建的账户移至入口OU
- 注册账户(会清除账户中的所有内容)
- 账户进入清理状态
- 清理过程需要5-10分钟,使用AWS Nuke

26:00 - 27:00 - 弹出账户功能
- 可以选择任何状态的账户并弹出
- 账户从解决方案中移除
- 适用于需要保留的账户或转为常规账户的场景

27:00 - 30:00 - 租约模板管理
- 租约模板是配置账户阈值的核心
- 通常由管理者角色创建(无需技术背景)
- 展示"Agentic Testing"模板
- 包含名称、描述和可见性设置
- 私有可见性:用户无法看到该模板,无法请求
- 可选审批功能

30:00 - 33:00 - 预算配置
- 可以设置最大预算(可选)
- 示例:$50发送警报,$75冻结账户,$100清理账户
- 建议始终保留清理操作作为防止成本失控的保障
- 可以选择性地配置不同的操作

33:00 - 35:00 - 持续时间配置
- 与预算配置类似,但基于时间
- 可以设置不同时间阈值和对应操作
- 示例:24小时后执行特定操作

35:00 - 37:00 - 成本报告功能
- 两周前发布的新功能
- 可以为租约模板分配元数据(如成本中心或团队)
- 每30天生成成本报告到S3存储桶
- 按成本报告组汇总成本
- 用户选择对应团队的租约模板,便于成本追踪

37:00 - 39:00 - 租约管理
- 当前有3个活跃租约
- 显示租约持有者、当前预算、到期时间和状态
- 有一个处于冻结状态的租约
- 管理者可以随时登录账户

39:00 - 41:00 - 修改特定租约
- 可以更改特定租约的阈值
- 每个租约从租约模板创建,继承时间和成本配置
- 可以针对单个租约进行调整
- 适用于黑客马拉松场景中延长特定用户的使用时间

41:00 - 42:30 - 分配租约功能
- 点击"分配租约"
- 显示四个租约模板
- "研究员租约模板"设置为私有可见性
- 用户无法请求此类租约,只能由管理者分配

42:30 - 43:30 - 审批管理
- 显示待审批请求
- 示例:AWS Todd的请求
- 可以在UI中批准
- 审批请求会发送电子邮件通知
- 管理者和管理员都可以审批

43:30 - 46:00 - 用户视图演示
- 切换到用户视角
- 通过IAM Identity Center认证
- 显示当前租约:共4个租约,2个活跃
- 可以登录活跃账户
- 显示当前预算和到期时间
- 1个待审批,1个已手动终止

46:00 - 48:00 - 用户请求租约
- 点击"请求租约"
- 显示公开的租约模板
- 选择模板并接受服务条款
- 服务条款可配置

48:00 - 50:00 - 全局配置
- 在Hub账户的App Config中管理
- 可以更改服务条款
- 可以设置单个用户最大沙箱数量(示例中设置为3)
- 可以设置租约模板的绝对最大值(例如不超过$500)
- 可以要求成本报告组
- 高度可配置

50:00 - 51:00 - API支持
- UI中的所有操作都有API支持
- 提供完整的OpenAPI规范
- 可以集成到现有流程中

51:00 - 52:00 - 完成请求演示
- 请求已完成
- 展示QR码获取更多信息

52:00 - 54:00 - 常见用例总结
- 自5月发布以来,已与50-100位客户交流
- 大学(非常常见)
- 高度监管的客户
- 黑客马拉松
- 生成式AI应用开发(最重要的用例)
- 构建Agentic应用
- 提供隔离、安全的环境,具有成本和时间控制

54:00 - 54:30 - 结束语
- 感谢参与
- 请求完成调查问卷
