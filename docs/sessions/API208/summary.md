# AWS re:Invent 2025 会议总结：使用 Amazon API Gateway 构建 API 卓越文化

## 会议概述

本次分会由 Amazon API Gateway 服务团队的高级产品经理 Shrian 主讲，主题聚焦于如何通过 Amazon API Gateway 及其工具构建 API 卓越文化。演讲者指出，每个成功的数字化转型都始于 API，但构建优秀的 API 只是成功的一半，另一半更困难的部分是让所有开发者能够发现、访问和使用这些 API。

演讲深入探讨了企业面临的 API 碎片化问题。当组织拥有多个业务单元、多个 AWS 账户，每个账户中又有数百个 API 时，API 总数可能达到数万甚至数十万个。这种规模的 API 生态系统带来了严重的发现困难、重复开发和安全盲点等问题。为解决这些挑战，AWS 推出了 Amazon API Gateway Portals——一个完全托管的开发者门户解决方案，帮助企业管理 API 混乱状态，实现跨账户的统一发现和治理。

演讲强调了"API 即产品"的思维方式，建议企业将 API 视为需要长期维护和增强的战略资产，从 API 消费者角度反向设计，并为每个 API 产品指定专门负责人。通过 API Gateway Portals，企业可以在四个简单步骤内从 API 提交到 API 采用，实现文档管理、访问控制、品牌定制和统一门户等功能。

## 详细时间线

00:00 - 00:45
- 演讲开场：强调数字化转型始于 API，但让 API 可被发现、可访问和可使用是更困难的挑战
- 演讲者自我介绍：Shrian，Amazon API Gateway 服务团队高级产品经理

00:45 - 01:30
- 介绍演讲框架：将涵盖三个关键领域
  1. 量化碎片化 API 生态系统导致的问题
  2. 定义 API 卓越文化的思维方式
  3. 介绍整合所有内容的平台解决方案

01:30 - 03:15
- 现场互动：引导听众思考三个问题来评估问题规模
  - 组织有多少个业务单元或主要产品领域？
  - 每个组织下有多少个 AWS 账户？
  - 每个账户中有多少个 API？（如果充分使用 API Gateway，可能约 600 个）
- 计算结果：企业级组织可能拥有数万个 API

03:15 - 05:30
- 列举 API 碎片化带来的五大挑战：
  1. 开发者如何发现这些 API？需要多少 Slack 消息、会议或工单才能找到正确的端点？
  2. 多个端点提供相同功能的重复问题有多频繁？
  3. 如何可靠地知道哪些 API 将被淘汰、哪些处于活跃状态？
  4. 当新账户加入（如收购）时，如何识别功能重叠？
  5. 谁具体负责保持 API 目录的清洁和更新？

### API 碎片化的影响 (05:30 - 07:00)

05:30 - 06:15
- 分析 API 碎片化的三大核心问题：
  1. **减慢发现速度**：导致团队入职机制不佳，开发者需要更长时间才能提高生产力
  2. **抑制重用**：开发者陷入"数字寻宝"，找不到合适端点就重新发明轮子，加剧 API 蔓延
  3. **隐藏风险**：创建安全盲点，大量未治理的 API 形成影子 IT，无法保护看不见的内容，导致合规风险

06:15 - 07:00
- 提出解决方案：API 蔓延的解决方案不是构建更多 API，而是组织和编目现有 API

### API 卓越思维方式 (07:00 - 09:00)

07:00 - 08:00
- 介绍 API 卓越思维框架的三大核心原则：
  1. **API 即产品**：将 API 视为长期战略资产，需要增强、维护和修改，服务内部和外部客户以实现业务目标
  2. **从消费者反向工作**：从 API 消费者而非后端出发进行设计，确保不走单向门（不可逆决策）
  3. **单一负责人**：每个 API 产品需要专门负责人，负责质量、成功、运营卓越性、可用性和采用率

08:00 - 09:00
- 强调这种思维方式将消除冗余，最大化 API 投资回报

### Amazon API Gateway Portals 解决方案 (09:00 - 15:30)

09:00 - 10:00
- 宣布推出 Amazon API Gateway Portals
- 产品定位：AWS 原生、完全托管的开发者门户，帮助管理 API 混乱并简化跨账户发现
- 核心功能：
  - 控制文档新鲜度
  - 启用范围访问控制
  - 统一编目所有 API
  - 无需管理额外基础设施，保持在 AWS 边界内

10:00 - 11:30
- **创建 API 产品**：
  - 可以选择端点、特定路径和阶段
  - 将这些元素添加到称为"API 产品"的抽象层中

11:30 - 12:15
- **跨账户共享产品**：
  - 在多个账户中创建产品
  - 将产品共享到构建门户的账户
  - 实现统一门户，为所有 API 提供统一前端

12:15 - 13:00
- **文档管理**：
  - 快速入门：自动从 API Gateway 运行时拉取文档
  - 灵活性：可以使用默认文档或覆盖并提供自定义文档

13:00 - 13:30
- **API 试用功能**：
  - 允许开发者直接从门户试用 API
  - 消除 Slack 消息往返的需求

### 门户配置与定制 (13:30 - 16:00)

13:30 - 14:15
- **访问控制**：
  - 使用 Cognito 用户池限制开发者门户的访问范围

14:15 - 14:45
- **域名配置**：
  - 可选择默认域名或自定义域名
  - 与组织品牌指南保持一致

14:45 - 15:15
- **门户指标**：
  - 提供基于门户的指标
  - 跟踪访问频率和使用情况

15:15 - 15:45
- **统一门户架构**：
  - 不仅可以添加当前账户的 API 产品
  - 还可以添加从其他账户共享的 API 产品
  - 可选择为每个账户创建独立门户或为整个组织创建统一门户

15:45 - 16:00
- **品牌定制**：
  - 自定义外观和感觉
  - 添加组织徽标
  - 遵循组织配色方案

### 发布流程 (16:00 - 17:30)

16:00 - 16:45
- **预览功能**：
  - 在发布前生成开发者门户预览
  - 检查外观、产品配置和 API 试用功能

16:45 - 17:30
- **发布门户**：
  - 确认一切正确后发布开发者门户
  - 向内部或外部 API 消费者公开门户 URL

### 四步实施流程总结 (17:30 - 18:30)

17:30 - 18:30
- API Gateway Portals 实现从 API 提交到 API 采用的四个简单步骤：
  1. 继续使用当前文档或编辑文档，创建 API 产品并添加到开发者门户
  2. 配置范围访问控制和自定义域名功能
  3. 控制开发者门户的品牌
  4. 公开门户 URL，让 API 消费者（内部或外部）试用托管的 API
