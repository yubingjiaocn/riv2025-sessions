# AWS 会议总结：SPS 318 - 利用生成式 AI 实现运营卓越

## 会议概述

本次会议由 AWS 高级企业支持经理 Rahul Singler 及其同事 Joe 和 Jason 共同主讲，重点分享了富达投资（Fidelity Investments）如何将企业级 AWS 健康事件和支持案例数据管理这一基础运营挑战转化为战略运营优势的创新故事。会议展示了从传统的被动监控模式向主动预防性运营的转变过程，以及如何应用生成式 AI 技术将解决方案推进到下一个阶段。

富达投资面临的核心挑战是在拥有数千个 AWS 账户、数百万资源的大规模企业环境中，如何有效管理和响应海量的运营事件数据。研究表明，拥有全面数据监控的组织能够比没有监控的组织快 3.5 倍地检测到事件，这在金融服务行业尤为关键，因为每分钟的停机时间都可能意味着数百万美元的收入损失和客户信任的损害。会议详细介绍了富达投资构建的云事件通知传输服务（CENT，也称为 Sense），这是一个能够在企业规模上摄取、丰富、路由和处理 AWS 健康和支持数据的数据管道和通知平台。

会议的亮点在于展示了如何在 Sense 等系统之上应用生成式 AI 模式，包括使用 Amazon Bedrock 进行事件分析、趋势识别、自动化摘要生成，以及通过 AI 代理实现增强运营。这种方法不仅降低了运营成本（减少约 57%），还显著提高了利益相关者的参与度和问题解决效率，最终实现了从数据收集到智能运营的完整转型。

[00:18:30 - 00:21:00] - 富达投资背景：1946 年成立，创始人是技术爱好者，始终利用技术差异化业务。2016 年部署第一个云应用，2019 年加速云迁移，目前约有 8500 个应用跨多个云平台。

[00:21:00 - 00:23:30] - 2019 年的起点：12% 的应用组合在云端，约 500 个 AWS 账户，20 万个资源。AWS Health 提供了良好的基础，尽管当时功能不如现在强大。

[00:23:30 - 00:26:00] - 第一版解决方案：保持简单，基于时间的拉取机制，从账户中获取事件和支持案例，聚合到中央处理。初期通过电子邮件发送通知，但很快发现这种方式效率低下。

[00:26:00 - 00:29:00] - 当前规模（会议时）：80% 的应用在云端，近 2000 个账户，近 500 万个资源，过去两年仅从 AWS 就聚合了近 15 万个事件。强调数据基础的重要性。

[00:29:00 - 00:32:00] - AWS Health 的演进：引入委托管理账户（在组织级别聚合所有事件）和 EventBridge 原生集成，这些都是重大改进。

[00:32:00 - 00:35:00] - 架构重新思考：从拉取机制转向事件驱动系统。单个 EventBridge 规则可以聚合 2000 个账户的事件，然后进行丰富化处理，添加元数据、依赖关系等上下文信息。

[00:35:00 - 00:38:00] - 通知系统：在用户所在的地方与他们会面，提供个性化偏好设置，支持多种通知渠道（Teams、Slack、电子邮件、工作项等）。

[00:38:00 - 00:41:00] - 架构详解：使用 Route 53 进行路由，通过 API 接入以支持多云实现和弹性架构。健康事件经过丰富化处理后存储在 DynamoDB 中，并注入 OpenSearch 集群供用户搜索。

[00:41:00 - 00:44:00] - 支持案例集成：利用平台注入框架，通过单一规则和跨账户角色包部署到 1900 个账户，实现支持案例数据的聚合。

[00:44:00 - 00:46:00] - 弹性设计：AWS Health 现在支持本地区域事件备份到备用区域，可以在主区域出现问题时切换。

[00:46:00 - 00:49:00] - 用户偏好系统：提供企业级系统让用户记录通知偏好，包括媒介类型、关心的事件类型和交付渠道。生产账户事件可能触发即时通知，而开发环境的弃用运行时可能只发送电子邮件。

[00:49:00 - 00:52:00] - 成果展示：成本降低约 57%，同时增加了新功能（如 OpenSearch 搜索平台）。利益相关者参与度显著提高，从发送大量无效电子邮件转变为针对性的可操作通知（事件、服务请求、工作日志项）。

[00:52:00 - 00:54:30] - 未来目标：从基础工作转向数据驱动运营，包括识别趋势、关联事件。例如，2000 个账户中的某个问题可能已经在其他地方解决，需要建立连接。

[00:54:30 - 00:56:30] - 实际案例：RDS 证书过期事件（8 月 22 日截止），通过一致的通知系统成功避免了客户影响。下一步是增强运营和自动化解决方案。

### Joe 介绍生成式 AI 应用
[00:56:30 - 00:58:30] - Joe 介绍 Maki 框架（Machine Augmented Key Insights），已在 AWS Samples GitHub 上发布。该框架将支持案例和健康事件数据输入 Amazon Bedrock。

[00:58:30 - 01:01:00] - 支持案例处理：使用参考数据（如案例类别）增强每个支持案例，然后发送到 Bedrock。这是标准的提示工程实践，使用 Converse API。

[01:01:00 - 01:03:00] - 结果展示：处理数千个支持案例后，能够识别不同的分类桶，实现相似事件趋势识别。

[01:03:00 - 01:05:30] - 分阶段处理：事件级分析使用轻量级快速模型（Amazon Nova Micro 或 Anthropic Claude Haiku），聚合级分析使用更复杂的模型（Anthropic Claude Sonnet 或 Opus）。强调为正确的工作使用正确的工具。

[01:05:30 - 01:08:30] - 弹性模式：根据事件数量选择推理方式。少量事件使用 Bedrock 按需推理，大量事件（如年度分析）使用批处理推理，避免令牌限制和节省成本。

[01:08:30 - 01:11:00] - 事件摘要：为每个支持案例生成摘要，提取对话中的关键信息，提供具体建议和文档链接。健康事件也进行类似处理。

[01:11:00 - 01:13:30] - 聚合级分析：将健康和支持数据结合在一起（它们通常是相关的），生成运营环境的整体视图，并基于弹性基础设施指导提供改进计划。

[01:13:30 - 01:16:00] - 代理架构：事件输入 OpenSearch 并使用 Bedrock Titan 嵌入模型创建嵌入。提供词法搜索、语义搜索和混合搜索工具，这些工具可供托管在 Kira 中的 MCB 堆栈使用（使用 Claude Sonnet）。

[01:16:00 - 01:18:30] - OpenSearch 索引结构：保持 AWS 健康事件元数据的结构化形式，同时也有嵌入。强调预生成式 AI 技术（如结构化数据）仍然很有价值。

[01:18:30 - 01:21:00] - 混合方法的重要性：结构化数据查询速度快且可预测，而嵌入提供语义搜索能力。两者结合使用效果最佳。

[01:21:00 - 01:23:30] - 代理工作流演示：展示如何使用自然语言查询（"显示上周影响 EC2 的所有健康事件"），代理选择合适的工具并返回结果。

[01:23:30 - 01:26:00] - 复杂查询示例：查询特定时间范围内的 RDS 相关支持案例，代理能够理解意图并执行正确的搜索。

[01:26:00 - 01:28:30] - 增强运营能力：代理不仅可以检索信息，还可以执行操作。展示了如何通过自然语言请求创建事件摘要报告。

[01:28:30 - 01:31:00] - 自动化修复的未来：讨论如何将代理与自动化工具集成，实现自动修复。例如，检测到弃用的 Lambda 运行时后自动更新。
