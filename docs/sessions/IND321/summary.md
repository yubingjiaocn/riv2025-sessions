# AWS re:Invent 2025 - Boeing PLM 现代化迁移至 AWS 会议总结

## 会议概述

本次会议由 AWS 专业服务团队的高级云基础设施架构师 Justin Arvani 以及 Boeing 公司的 3D Experience 首席架构师 Jim Gallagher 和首席系统集成工程师 Dan Meyering 共同主讲。会议重点探讨了产品生命周期管理(PLM)系统面临的挑战、现代化迁移的障碍，以及 Boeing 如何成功将其 PLM 运营迁移到 AWS 云平台的实践经验。

Boeing 采用 Dassault Systèmes 的 3D Experience 作为下一代 PLM 系统，目前已在数十个小型项目中投入生产，支持数千名用户。通过与 AWS 专业服务团队的紧密合作，Boeing 成功克服了传统本地基础设施的局限性，实现了基础设施即代码(IaC)、自动化部署和弹性计算。迁移后，环境部署时间缩短了 60% 以上，从原来的 9.5 小时降至 3.5 小时，同时显著提升了运维效率和成本控制能力。

会议详细介绍了 Boeing 如何利用 AWS 服务(如 EC2、EFS、RDS、ALB、Auto Scaling Groups)替代传统基础设施，通过 Terraform 实现基础设施即代码，使用 GitLab CI/CD 管道实现自动化部署，并建立了标准化的环境模板。这一现代化转型不仅解决了资源供应缓慢、配置不一致等问题，还为全球协作和快速创新奠定了基础。

## 详细时间线与关键要点

### 开场介绍 (00:00 - 02:30)
- **演讲者介绍**: Justin Arvani，AWS 专业服务高级云基础设施架构师
- **会议议程**: 讨论 PLM 当前挑战、现代化障碍、解决原则，以及 Boeing 在 AWS 上运营数百个环境的实践

### PLM 三大核心挑战 (02:30 - 06:00)
- **性能与可扩展性**: 3D 模型保真度提升、业务智能和元数据复杂性增加导致系统扩展困难
- **数据摩擦**: 传统本地架构和遗留系统集成造成功能孤岛和数据孤岛
- **全球协作**: 跨区域工程活动、并购和技术合作伙伴关系增加了协作复杂性

### 现代化迁移的障碍 (06:00 - 08:30)
- **服务中断风险**: 升级和迁移必须最小化业务中断
- **培训投资保护**: 需要保持现有团队知识和技能的连续性
- **复杂集成**: PLM 系统与 ERP、MES、SCM 等关键系统深度集成
- **迁移周期长**: PLM 迁移项目通常持续数月甚至数年，成本高昂

### PLM 三大支柱 (08:30 - 11:00)
- **数字孪生**: 代表产品定义的多个方面，包括 3D 产品表示和行为需求
- **数字线程**: 连接各种数据源，提供实时决策支持和上下文信息
- **数字织网**: 构建生态系统，实现全球连接和资源最大化利用

### AWS 现代化方法论 (11:00 - 13:30)
- **客户至上**: 理解客户需求，逆向工作分解复杂问题
- **可扩展架构**: 考虑架构和运营复杂性，通过自动化实现自助服务
- **速度与敏捷性**: 优先快速迭代交付客户价值，而非追求完美方案

### Boeing PLM 系统介绍 (13:30 - 16:00)
- **PLM 定义**: 产品生命周期管理系统，涵盖从设计到支持的全生命周期数据
- **应用范围**: CAD、CAM、CAE、仿真、工具规划、变更管理、配置管理
- **3D Experience**: Boeing 下一代 PLM 系统，已在数十个项目中投入生产

### 传统本地基础设施挑战 (16:00 - 20:00)
- **资源供应缓慢**: 基于年度预算和资本采购周期，审批流程漫长
- **成本模型复杂**: 不同服务有不同计费模型，需要开发和维护退款模型
- **资源利用率低**: 项目不愿归还资源，难以动态调整规模
- **手工流程繁重**: 多个团队分工管理不同组件，协调成本高

### 迁移至 AWS 的机遇 (20:00 - 23:00)
- **企业内部云**: Boeing 与 AWS 合作创建网络隔离的 GovCloud 区域
- **团队赋能**: 开通 AWS 账户，使架构师和运维团队能够自主设计和构建
- **学习曲线**: 需要分析、学习和迭代实验，与 AWS 专业服务团队合作

### 新的责任与技能要求 (23:00 - 25:30)
- **自主管理**: OS 管理、服务器跟踪、合规性、备份、存储、负载均衡等
- **AWS 服务学习**: 团队需要掌握 AWS 服务和最佳实践
- **架构模式建立**: 与 Dassault 和 AWS 专业服务紧密合作建立新架构模式

### 迁移目标与成果 (25:30 - 29:00)
- **加速基础设施运营**: 缩短供应和退役时间，加快应用升级
- **避免官僚主义**: 消除审批和请求周期
- **提升 AWS 熟练度**: 运维和架构团队理解并执行最佳实践
- **降低成本**: 实现弹性计算，按需供应而非按峰值供应
- **改进配置管理**: 通过基础设施即代码减少服务器差异

### 附加迁移成果 (29:00 - 31:00)
- **全球协作**: 利用 AWS 账户隔离，使印度团队能够管理非出口管制数据环境
- **技术更新**: 从 Red Hat 7 升级到 Red Hat 8，解决操作系统过时问题
- **避免影响发布**: 确保迁移不影响 3D Experience 主要版本发布计划

### 合作伙伴关系 (31:00 - 33:00)
- **AWS 专业服务**: 提供其他 3DX 客户经验和深厚 AWS 专业知识
- **Dassault 合作**: 持续合作确定如何将 3D Experience 架构映射到 AWS 基础设施
- **能力优先级**: 确定基础设施即代码、负载均衡、数据复制、灾难恢复等优先级

### 官僚主义挑战 (33:00 - 38:00)
- **传统流程**: 每个组件(VM、存储、数据库、DNS)需要向不同团队提交请求
- **协调成本高**: 大量邮件、电话、即时消息协调，延迟显著
- **请求表单复杂**: 需要特定格式，错误可能导致数周延迟
- **内部文档**: 团队维护详细的请求提交指南以最小化延迟

### 传统基础设施问题 (38:00 - 41:00)
- **配置不一致**: 服务器规格错误、存储区域错误、数据库配置不当
- **运营风险高**: 深夜故障频发，如数据库监听器未配置自动启动
- **过度供应**: 为应对不确定性而过度配置资源
- **迭代困难**: 难以进行实验和优化

### AWS 服务替代方案 (41:00 - 48:00)
- **Application Load Balancer**: 替代单点 Apache 服务器，提供高可用性和健康检查
- **EC2**: API 驱动的实例规格选择，自定义 AMI 减少手动设置
- **Auto Scaling Groups**: 支持自动或手动扩展，基于应用指标触发
- **EFS**: 替代网络附加存储，自动扩展，按使用量计费
- **RDS**: 托管数据库服务，提供高可用性、自动备份、时间点恢复

### 基础设施即代码实施 (48:00 - 53:00)
- **Terraform 选择**: 从 CloudFormation 转向 Terraform 以符合企业标准
- **GitLab CI/CD**: 代码存储在 GitLab，通过管道执行计划和应用
- **审批门控**: 确保预期结果后再触发应用阶段
- **审计历史**: 强制同行评审，创建可审计的变更历史

### 环境模板标准化 (53:00 - 56:00)
- **模板类型**: 小型、中型、大型和单体(所有服务在一个 EC2 上)
- **配置定义**: 每个模板定义基线 EC2 规格、RDS 类别、LDAP、DNS、ASG 配置等
- **减少开销**: 标准化规格确保部署符合性能和规模要求
- **消除手动调查**: 不再需要寻找类似环境并手动配置

### 模块化架构 (56:00 - 59:00)
- **环境定义**: 提供最小参数集定义有效环境规模
- **3DX 核心模块**: Terraform 模块为 AWS 中所有资源提供脚手架
- **智能默认值**: 使用智能默认值减少重复手动输入，同时允许覆盖
- **灵活性**: 支持固定旧版 AMI 或覆盖 Route 53 DNS 别名等特殊需求

### 并行部署能力 (59:00 - 62:00)
- **消除瓶颈**: 可并行部署多个环境，无需顺序等待
- **时间节省**: 从一天或一周缩短到几小时
- **被动时间**: 运维人员可在部署期间处理其他任务
- **规模测试**: 成功部署 80 个开发环境，仅受 IP 地址限制

### 代码库组织 (62:00 - 65:00)
- **多仓库分离**: AMI 仓库、3DX 核心基础设施仓库、PLM 环境仓库、先决条件仓库
- **AMI 管道**: Chef cookbooks 生成 AMI，测试后共享到其他账户
- **企业模块**: 使用企业 Terraform 模块维护公司标准
- **账户隔离**: 跨多个 AWS 账户分离 PLM 基础设施关注点

### 应用部署自动化 (65:00 - 69:00)
- **Ansible 配置管理**: 使用 Ansible 部署 3D Experience
- **动态清单**: 基于资源标签的动态清单，消除手动复制粘贴
- **角色重构**: 将线性 playbooks 重构为角色，简化脚本
- **CI/CD 集成**: 为更紧密的应用部署 CI/CD 集成奠定基础

### 部署模式对比 (69:00 - 72:00)
- **传统模式**: 多个服务共享单个主机，串行安装，单个服务问题影响其他服务
- **新模式**: 每个服务独立主机，并行安装
- **性能提升**: 部署时间从 9.5 小时缩短到 3.5 小时，减少 60% 以上
- **一致性改进**: 消除特殊情况处理，简化自动化

### 挑战与经验教训 (72:00 - 76:00)
- **学习曲线**: Terraform、GitOps、CI/CD、系统管理、数据库管理、合规性等新技能
- **爆炸半径风险**: 最初将多个环境放在单个 Terraform 文件中，后改为分离
- **精确执行**: Terraform 严格按代码执行，需要仔细验证关键资源变更
- **持续优化**: 通过实践不断改进流程和架构

### 总结
Boeing 的 PLM 现代化迁移展示了如何通过 AWS 云服务、基础设施即代码和自动化实践克服传统本地基础设施的局限性，实现更快、更可靠、更经济的运营模式。
