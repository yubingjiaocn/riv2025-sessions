# AWS re:Invent 2025 会议总结：Bloomberg 媒体 AI 平台架构

## 一、会议概述

本次 AWS re:Invent 2025 分会场由 AWS 专家解决方案架构师 Brandon Lindau 和 Robert Raver，以及 Bloomberg Media 媒体技术负责人 Luke Barau 共同主讲。会议重点探讨了如何构建端到端的智能媒体管理平台，用于分析、理解和处理大规模媒体库。Bloomberg Media 作为全球领先的商业新闻平台，每月拥有 6000 万独立观众，覆盖全球 4.37 亿家庭，通过 48 个主要流媒体平台进行分发。面对海量视频内容和快速变化的 AI 技术，Bloomberg 分享了其创新的"可抛弃式 AI 策略"（Disposable AI Strategy），强调平台架构比具体模型更重要。

会议深入讲解了如何利用 AWS 服务构建灵活、可扩展的媒体 AI 工作流，包括内容摄取、多模态分析、混合搜索、知识图谱以及 AI 代理编排等关键技术。Bloomberg 的核心理念是通过版本化管理、联合搜索和动态工作流设计，使平台能够快速适应新的 AI 模型和业务需求，而无需重构整个系统。这种架构设计使得 Bloomberg 能够在保持速度、准确性和可信度的同时，高效处理从历史档案到实时直播的各类媒体内容。

## 二、详细时间线与关键要点

### **开场介绍 (00:00 - 03:30)**
- **00:00** - Brandon Lindau 开场，介绍这是一个 300 级别的技术会议，将快速覆盖大量深度概念
- **01:15** - 强调会议重点：构建完整的端到端系统，而非单一服务或解决方案
- **02:00** - 介绍会议议程：Bloomberg 的愿景、元数据分析、PB 级内容摄取、混合搜索、知识图谱和 AI 代理工作流
- **03:00** - Luke Barau 登场，介绍 Bloomberg Media 背景

### **Bloomberg Media 平台介绍 (03:30 - 08:00)**
- **03:30** - Bloomberg Media 定位：为商业领袖提供数据驱动洞察的全球新闻平台
- **04:15** - 内容类型：文本新闻、音频（广播和播客）、视频（线性电视、移动端、网页和社交平台）
- **05:00** - 关键数据：每月 6000 万独立观众，覆盖 4.37 亿家庭，48 个流媒体平台
- **05:45** - 展示两个主要频道：Bloomberg TV（复杂的实时新闻整合）和 Bloomberg Originals（深度长篇内容）
- **06:30** - 现有媒体生态系统：混合全球平台，结合本地网络和云服务
- **07:15** - 平台设计目标：灵活应对业务变化、动态工作流、易于引入新服务

### **AI 应用挑战与策略 (08:00 - 14:30)**
- **08:00** - AI 在媒体领域的核心理念：**基于档案驱动，而非人工合成**
- **08:45** - 目标：利用现有内容提取洞察并重新组装，而非生成虚假内容
- **09:30** - 早期实验发现的问题：
  - 单个视频处理效果好，但长视频和复杂内容会失去上下文
  - RAG（检索增强生成）在大规模应用时变得复杂且昂贵
  - 工作流设计和测试周期长，AI 模型更新快导致难以跟上
  - 供应商锁定问题严重
- **11:00** - 解决方案：识别需要调整的"旋钮"（可调参数）
- **11:45** - 版本化策略：对模型、嵌入、服务、工作流和管道进行版本管理
- **12:30** - 多数据库架构：存储多种类型的嵌入（音频、视频帧、文本转录）
- **13:15** - 联合搜索：单一搜索服务智能选择正确的数据库，AI 层分析和重写查询
- **14:00** - 核心理念："平台重要，模型不重要" - 可抛弃式 AI 策略

### **内容摄取架构 (14:30 - 17:00)**
- **14:30** - Robert Raver 接手讲解技术实现
- **15:00** - 五大核心模块：摄取、分析、理解、搜索、创建
- **15:30** - 摄取源类型：
  - 基于文件的媒体
  - 实时内容流
  - 历史档案
  - 第三方供稿和贡献内容
- **16:15** - 设计目标：创建统一的"漏斗"，避免为每个来源创建独特的工作流
- **16:45** - 所有内容最终进入 S3，通过事件触发统一的分析工作流

### **多模态内容分析 (17:00 - 23:30)**
- **17:00** - 三种分析方法：
  1. **任务特定模型**（Task-specific models）
  2. **视觉语言模型**（Vision Language Models - VLMs）
  3. **嵌入模型**（Embedding models）
- **18:00** - 任务特定模型：生成标签、转录等预定义输出（如 AWS Rekognition、Transcribe）
- **18:45** - 视觉语言模型：
  - 自由形式查询，自然语言描述
  - 原生支持视频、图像、音频
  - 零样本推理，无需额外训练
- **19:30** - 演示：使用 AWS Nova 模型分析会议视频
- **20:15** - 模型输出示例："第二位演讲者是一位留胡子的男士，穿着浅蓝色衬衫和蓝色牛仔裤"
- **21:00** - 向量嵌入：将媒体转换为数字向量，实现语义理解和相似度比较
- **21:45** - 嵌入的优势：可以跨模态比较（音频、图像、视频片段）
- **22:30** - 关键结论：使用所有三种方法，发挥各自优势
- **23:00** - AWS 服务示例：Transcribe、Rekognition、Bedrock（托管 12 Labs Morango 3.0、Nova 多模态嵌入模型）

### **混合搜索架构 (23:30 - 31:00)**
- **23:30** - 搜索的重要性：用户和 AI 代理使用相同的搜索 API
- **24:00** - 传统搜索问题：依赖关键词和文件名，用户体验差
- **24:45** - 目标：让用户在几秒内找到所需内容
- **25:15** - 搜索的个性化：不同人对同一内容的描述方式不同
- **25:45** - 三种搜索数据类型：
  1. **关键词搜索**：精确匹配（如人名 "Robert Raver"）
  2. **向量搜索**：语义相似度（如"舞台上演讲的人"）
  3. **混合搜索**：结合两者优势
- **26:30** - 关键词搜索：使用 OpenSearch，基于频率、接近度和匹配度
- **27:15** - 向量搜索：比较向量距离，返回 Top-K 相似结果
- **28:00** - 多模态嵌入：语义理解，但嵌入特定于模型
- **28:45** - 搜索意图分析示例：
  - "Brandon 和另外两个人在舞台上" → 视觉权重 0.9
  - "Robert 谈论搜索技术" → 转录权重高，人物识别权重高
- **30:00** - 动态权重调整：使用 LLM 分析搜索词，自动调整不同数据源的权重
- **30:45** - 架构：API Gateway → Lambda → LLM（分析意图）→ 生成嵌入 → 查询多个数据存储

### **知识图谱与关系理解 (31:00 - 35:30)**
- **31:00** - 问题：搜索返回大量结果，但缺乏上下文关系
- **31:30** - 解决方案：使用知识图谱（AWS Neptune Analytics）
- **32:00** - 图数据库原理：节点（人物、资产）+ 关系（边）
- **32:30** - 关系类型：时间关系、身份关系、领域关系
- **33:00** - 图谱应用：
  - 人物出现在资产中
  - 资产从其他资产创建
  - 故事发生在特定地点
- **33:45** - 实际价值：识别衍生内容关系
  - 例如：从 4 个原始资产创建 1 个新资产，再为 YouTube/TikTok 创建更多版本
  - 搜索时可识别这 7 个资产实际上是同一内容
- **34:30** - 虚拟实体概念：不仅搜索资产，更搜索人物、主题等实体
- **35:00** - 数据填充：分析数据同时填充 OpenSearch 和知识图谱，使用相同方式查询

### **AI 代理与内容创建 (35:30 - 结束)**
- **35:30** - 最终目标：从搜索到内容创建
- **36:00** - AI 代理角色：编排复杂的内容工作流
- **36:30** - 代理使用相同的搜索接口，确保一致的用户体验
- **37:00** - 工作流示例：
  - 代理接收创建请求
  - 搜索相关内容片段
  - 利用知识图谱理解关系
  - 组装生成新内容
- **37:45** - 强调：所有内容基于真实档案，非合成生成
- **38:15** - 平台优势总结：
  - 模块化设计，易于替换组件
  - 版本化管理，支持 A/B 测试
  - 多数据库架构，灵活应对不同查询
  - 可观测性：监控质量、性能和成本

### **总结与关键要点 (结束前)**
- **核心架构原则**：松耦合、可版本化、可替换
- **技术栈**：AWS S3、Lambda、Bedrock、OpenSearch、Neptune Analytics、Rekognition、Transcribe
- **成功关键**：平台设计优先于模型选择，为变化做好准备
- **Bloomberg 的独特方法**：在引入服务前就考虑如何移除它（可抛弃式 AI 策略）
