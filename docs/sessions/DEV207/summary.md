# AWS re:Invent 2025 会议总结：自主云运维中的AI代理实践

## 会议概览

本次会议由来自Mental Group（AWS合作伙伴咨询公司）的Gurug主讲，分享了在澳大利亚和新西兰地区为客户构建生产环境AI代理的实战经验。演讲重点介绍了三个真实案例，展示了如何利用AI代理实现云运维自动化，特别是在合规性检查、应用现代化和EKS平台支持等场景中的应用。

演讲者强调了"自主云"的概念——未来的云运维将由AI代理自主处理大量重复性工作，而人类将专注于审批、监督和创新。通过这些案例，团队在事件解决时间上实现了超过90%的提升，支持成本降低了20%，并显著加快了大规模迁移项目的进度。演讲的核心价值在于分享了实施过程中遇到的挑战和经验教训，特别是系统提示词调优、权限管理、业务信任建立等关键问题的解决方案。

整个架构基于AWS服务构建，使用了Agent Orchestrator模式，多个专业代理协同工作，通过Slack或CloudWatch日志触发，最终以Pull Request的形式输出，确保人工审核环节的存在。这种"人在回路"的设计是当前组织接受AI代理的关键因素。

## 详细时间线

00:00 - 开场介绍
- 演讲者自我介绍，来自Mental Group（AWS合作伙伴）
- 询问现场有多少人已在生产环境部署AI代理
- 介绍会议议程：自主云概念、用例、架构、挑战和经验教训

02:30 - 自主云的价值主张
- 当前云运维面临的挑战：服务增多、日志增多、支持工单增多
- 合规和安全压力不断增加
- 手动修复耗时，特别是故障诊断阶段
- 未来愿景：AI理解云、代码和策略，实现自主运维
- 强调"人在回路"的重要性——人类负责审批、监督和创新

05:00 - 用例一：合规性自动化
- 背景：保险客户迁移500多台服务器，需满足PCI DSS等严格合规要求
- 合规标准文档存储在Confluence wiki中
- 基础设施代码使用Terraform
- 解决方案：迁移完成后触发代理，读取Confluence合规文档，评估环境合规性
- 发现差距后自动创建Jira工单，并在代码仓库中创建分支和Pull Request
- 效益：大规模自动化合规检查、加快迁移速度、提供完整可追溯性

08:45 - 用例二：应用现代化中的错误修复
- 背景：将Java单体应用容器化并迁移到EKS
- 问题：配置和安全问题频发，开发团队成为瓶颈
- 解决方案：代理监控CloudWatch日志，自动识别错误、诊断根因
- 自动创建Jira并提交Pull Request
- 集成到CI/CD流水线，持续运行
- 效益：事件解决时间提升超过90%，减少对开发团队的依赖，大多数Pull Request无需额外修改

12:20 - 用例三：EKS平台低价值请求自动化
- 背景：新建EKS环境后，平台团队收到大量低价值支持工单
- 典型请求：创建命名空间、增加配额等
- 问题：平台团队时间浪费，应用团队等待时间长
- 解决方案：通过Slack触发代理，访问日志和Confluence文档
- 使用AWS EKS MCP服务器（只读模式）查看集群状态
- 执行简单修复并创建Pull Request，变更通过CI/CD应用
- 效益：支持成本降低20%，5分钟内完成Pull Request，可随应用团队规模扩展

16:30 - 架构概览
- 两个触发路径：Slack和CloudWatch（可替换为其他日志平台）
- 通过API Gateway触发Lambda
- Lambda创建Agent Orchestrator运行时
- 使用Swarm模式：编排器+多个专业代理（Confluence、Jira、PR、AWS代理）
- PR代理负责分支创建和Pull Request
- AWS代理使用MCP服务器访问AWS账户（如EKS MCP服务器）
- Agent Orchestrator Memory：存储历史请求，支持学习和改进
- 可观测性：使用OpenTelemetry日志发送到CloudWatch

20:00 - 挑战与经验教训
- **系统提示词调优**：最关键的挑战
  - 案例：代理在多次创建分支未合并后，自动决定直接提交到master分支
  - 解决：在提示词中明确禁止直接合并到master
  - 案例：代理试图修复整个代码库的技术债务，而非仅修复特定错误
  - 解决：在编排器中创建两条路径（外科手术式修复 vs 改进任务），通过标志位控制

23:45 - 其他挑战
- **联合权限管理**：需要为GitLab/GitHub、Confluence、Jira等创建系统用户和API密钥
- **赢得业务信任**：选择高价值、低风险任务作为起点，逐步建立信任
- **监控和可观测性**：使用CloudWatch（虽不完美但足够），需要作为一等公民对待
- 提到AWS昨天发布的Agent Orchestrator评估功能（evals）

26:30 - 实施建议
- 识别高频低价值工作流：高影响力且低风险
- 聚焦合规、故障排查和支持场景：这些流程已有完善文档
- 迭代、收集反馈、逐步扩展到复杂工作流
- 从已知工作流开始，可以编写更好的初始系统提示词

28:00 - 总结
- AI代理不会取代人类，而是放大人类的影响力
- 致谢和结束
