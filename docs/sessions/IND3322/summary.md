# AWS re:Invent 2025 会议总结：构建面向AI的现代数据平台

## 会议概述

本次会议由AWS专业服务团队的两位资深架构师Razan Wang和Sam Gordon主讲，他们基于过去两年与大型金融服务机构（FSI）客户的实际合作经验，深入探讨了如何构建面向AI时代的现代数据平台。

会议的核心观点是：**没有坚实的数据基础，就没有生产级AI智能体系统的未来**。当前所有企业都在讨论构建AI智能体，但现实是这些智能体需要记忆、上下文、实时访问高质量数据以及复杂的推理能力，而所有这些都依赖于强大的数据基础。Gartner预测，到2027年，40%的AI智能体项目将会失败，另有研究表明52%的首席数据官认为当前的数据基础还未为AI消费做好准备。

演讲者提出了构建数据基础的四大核心要素：打破数据孤岛、建立数据信任、建立跨组织治理、创建灵活的数据消费模式。会议通过"客户360洞察"这一实际用例，系统性地展示了如何利用AWS SageMaker Unified Studio等服务来解决这些挑战，实现70-80%的问题解答时间缩短、80%的数据可发现性、90%的手动治理流程减少，以及5-10倍的AI/ML实验增长。

## 详细时间线与关键要点

### 开场与问题定义 (00:00 - 05:30)

00:00 - 会议开始，Razan Wang自我介绍，来自悉尼的AWS专业服务团队，专注于数据领域，过去两年与大型金融服务客户密切合作

01:15 - Sam Gordon介绍，同样来自专业服务团队的ML顾问

01:45 - 提出核心问题：如果你了解所运营的一切数据状态，并能在这些数据之上进行推理，你能解锁什么机会、解决什么问题？

02:30 - 使用冰山比喻说明数据与AI的关系：AI智能体是表面，但底层需要强大的数据基础支撑

03:15 - 强调关键观点：**没有坚实的数据基础，生产级AI智能体系统就没有未来**

### 四大数据基础要素 (05:30 - 08:00)

05:30 - 介绍构建数据基础的四大核心要素：
- 打破数据孤岛
- 建立数据信任
- 建立跨组织治理
- 创建灵活的数据消费模式

06:45 - 引用研究数据：Gartner预测40%的AI智能体项目将在2027年前失败；52%的CDO认为数据基础尚未为AI做好准备

07:30 - 说明这不仅是技术问题，更是组织能力建设问题

### 障碍一：数据孤岛 (08:00 - 20:00)

08:00 - 介绍"恶性循环"：数据孤岛导致信任问题，信任问题导致治理困难，治理困难限制数据消费，有限的消费又强化了孤岛

09:00 - 展示"飞轮"概念：这是整个演讲的议程框架，将逐一解决四大障碍

09:45 - 提出现代数据平台的预期成果：
- 70-80%更快的问题解答时间
- 80%的数据可发现性
- 90%的手动治理流程减少
- 5-10倍的AI/ML实验增长

10:30 - 引入"客户360洞察"用例：展示跨业务单元统一客户视图的挑战

11:15 - 描述碎片化领域场景：零售银行使用本地数据仓库和Oracle，而其他部门使用不同的云技术栈

12:30 - 指出问题：需要9-12个月才能完成跨域集成，这还不包括构建ML模型的时间

13:45 - 介绍解决方案：**SageMaker Unified Studio**作为中心化服务，特别是其目录功能

14:30 - 展示架构转变：从混乱的点对点连接转变为以统一目录为中心的清晰架构

15:15 - 详细介绍SageMaker Catalog的能力：域结构创建、数据资产发布、数据发现、自动化访问工作流

16:00 - 识别数据孤岛的三大核心挑战：
- 没有明确的数据真实来源
- 碎片化的数据发现
- 到处都是重复数据

17:30 - 提出解决方案飞轮：运营模型与RACI矩阵、多账户策略、数据所有权、统一目录、跨账户访问

18:15 - 展示运营模型：定义中央治理和域级别的角色与职责

19:00 - 介绍多账户策略：将组织结构映射到技术架构，每个域拥有独立账户以实现安全隔离、成本分配和技术自由

### 数据所有权与数据合约 (20:00 - 25:00)

20:00 - 引入**数据合约**概念：数据生产者和消费者之间的正式协议

20:45 - 数据合约包含的内容：所有权、模式结构、验证规则、治理要求

21:30 - 展示数据合约演进：从简单的所有者标记开始，逐步添加更多元数据

22:15 - 强调迭代改进：第一次不需要完美，可以持续添加信息

23:00 - 介绍统一目录解决数据发现问题：SageMaker Unified Studio提供业务数据目录能力，不仅是技术元数据

24:00 - 跨账户访问：SageMaker Unified Studio通过Lake Formation自动管理跨账户访问

### 障碍二：数据信任 (25:00 - 38:00)

25:00 - 引入第二个障碍：数据信任问题

25:30 - 强调：AI智能体的有效性取决于数据质量

26:00 - 识别四大核心挑战：数据质量问题、血缘追溯、PII暴露、访问治理

26:45 - 引用统计：40%的分析项目因数据质量问题而失败

27:30 - 描述发现问题的时间成本：需要数周才能发现数据质量问题

28:15 - 提出解决方案飞轮：数据质量、数据血缘、PII合规、访问治理

29:00 - 数据质量解决方案：
- 在数据合约中定义质量规则
- 使用AWS Glue Data Quality进行检查
- 支持ML驱动的质量检查和异常检测

30:30 - 展示在SageMaker Unified Studio中查看数据质量分数

31:15 - 数据血缘解决方案：
- 从Glue作业启用血缘捕获（需要Glue 5.0版本）
- 支持Apache Airflow通过Open Lineage插件
- 支持dbt通过Open Lineage

33:00 - 展示配置示例：只需启用参数并提供域ID

34:30 - 在SageMaker Unified Studio中查看完整的血缘图谱

35:00 - PII检测解决方案：
- 在数据合约中标记PII列
- 使用Glue Visual ETL进行ML驱动的敏感数据检测

36:30 - 访问治理解决方案：
- Bronze层（原始数据）：仅限数据工程师、运维和合规团队
- Silver层（清洗数据）：数据工程师、分析师和科学家
- Gold层（业务对齐）：细粒度访问控制，包括行列过滤

### 障碍三：跨组织治理 (38:00 - 45:00)

38:00 - Sam Gordon接手演讲，介绍第三个障碍：跨组织治理

38:30 - 总结前两个障碍的成果：已定义业务单元、数据已上云、质量已改善

39:00 - 指出新问题：作为消费者，我不知道有什么数据可用，或者不理解数据的含义

39:45 - 描述典型场景：
- 第一天就发现没有数据源
- 需要请求访问，等待数周或数月
- 收到的是CSV文件，没有描述信息
- 大量来回沟通，非常耗时

40:30 - 强调可发现性和元数据丰富的重要性

41:00 - 介绍解决方案：通过SageMaker Unified Studio实现数据发现和元数据管理

42:00 - 展示自动化访问工作流：消费者可以直接请求访问，所有者批准后自动授权

43:15 - 强调Lake Formation在后台处理所有权限管理

44:00 - 讨论业务术语表和元数据丰富：帮助用户理解数据含义

### 障碍四：数据消费模式 (45:00 - 结束)

45:00 - 介绍最后一个障碍：灵活的数据消费模式

45:30 - 说明不同角色有不同的消费需求：
- 数据分析师需要BI工具
- 数据科学家需要笔记本环境
- AI智能体需要API访问

46:15 - 展示多种消费方法：
- Amazon QuickSight用于BI可视化
- SageMaker Studio用于ML开发
- Athena用于SQL查询
- API用于智能体访问

47:00 - 强调统一目录支持所有这些消费模式

47:45 - 总结完整的飞轮效应：解决四个障碍后，形成良性循环

48:30 - 回顾关键成果指标和客户成功案例

49:00 - 结束语：强调这些都是基于真实客户经验的实践方案
