# AWS re:Invent 2025 会议总结：业务驱动的AI创新与数据产品化

## 会议概述

本次会议由Starburst首席数据官Adrian Estala和MUFG企业数据负责人Raja Palaniswamy共同主讲，重点探讨了如何通过数据产品化和语义层构建来赋能业务团队驱动AI创新。会议强调了让业务团队在生成式AI和对话式AI解决方案中发挥主导作用的重要性，而非传统的IT主导模式。

演讲者分享了MUFG银行在数据战略转型方面的实践经验，展示了如何从传统的物理数据整合模式转向逻辑数据产品模式，通过联邦查询、业务导向的语义层和数据产品直接应用这三个核心层次，实现业务团队的自助服务能力。会议特别强调了"数据需要AI，AI需要数据"的循环关系，以及如何通过适当的治理、指导和护栏来降低AI实验的时间成本和经济成本。

## 详细时间线与关键要点

### 0:00-3:00 开场介绍与背景设定
- Adrian Estala介绍自己作为Starburst CDO的背景，强调从生态系统角度而非单一产品角度讨论策略
- Raja Palaniswamy介绍自己在MUFG负责企业数据和数据战略的职责
- Adrian通过现场调研了解听众构成，包括首席产品官和业务自助服务推广团队

### 3:00-7:00 业务赋能AI的三层框架
- **第一层：数据联邦访问** - 允许团队访问数据所在位置，无需迁移，支持云端和本地环境
- **第二层：业务导向语义层** - 将数据组织成业务团队可理解的逻辑数据产品，"让业务团队走进房间就能识别他们的数据"
- **第三层：数据产品应用** - 在BI或AI解决方案中直接采用这些数据产品

### 7:00-10:00 MUFG的数据战略转型经验
- Raja分享"数据需要AI，AI需要数据"的核心理念
- 从传统物理数据整合转向逻辑数据产品创建
- 建立三种角色：业务数据生产者、数据精英和业务用户
- 核心数据产品与功能数据产品的区别：核心产品来自多个数据源的整合，功能产品是针对特定业务需求的衍生产品

### 10:00-13:00 业务团队赋能与AI应用
- **转型触发因素**：缩短上市时间，提高收入生成者的能力，实现业务自助服务
- **语义层实践**：在MUFG建立客户、渠道、银行等核心数据产品，各功能团队可组合创建衍生产品
- **业务元数据的重要性**：在列级、表级、角色级提供真实的业务描述，提高LLM分析的准确性和性能
- **三个G原则**：指导(Guidance)、护栏(Guardrails)、治理(Governance)

### 13:00-16:00 AI与数据的双向应用
- **数据用于AI**：将结构化和非结构化数据输入LLM模型进行学习和洞察生成
- **AI用于数据**：使用AI进行数据标记、血缘关系创建、数据分析等数据管理任务
- **对话式数据访问**：业务用户可以通过聊天方式查询数据，如"给我所有中端市场银行客户"
- **集成认证机制**：确保数据访问的安全性和权限控制

### 16:00-16:30 总结与问答
- 强调业务团队在数据能力提升中的关键合作伙伴作用
- 通过Starburst提高数据质量指标，为AI建模提供更丰富的高质量数据
- 会议在Starburst展台550继续深入讨论的邀请中结束
