# AWS re:Invent 2025 会议总结：Global Payments 主机迁移至 AWS 的统一运营实践

## 会议概述

本次分享由 AWS 技术客户经理总监 Steven Clark 和 Global Payments 数字体验技术副总裁 Ganesh Semuscondan 共同呈现，重点介绍了 Global Payments 如何将其关键任务型支付处理系统从大型机迁移到 AWS，以及如何通过 AWS 新推出的统一运营计划（Unified Operations Plan）成功运营这些系统。

Global Payments 是美国、英国、加拿大、爱尔兰和中国的头号发卡处理商，每年处理超过 750 亿笔交易，其中 350 亿笔来自发卡业务。这些系统覆盖从信用卡申请、实时审批、卡片打印、PIN 嵌入到交易授权的完整支付生命周期。在感恩节周末，他们单日处理了 15 亿笔交易，系统必须在毫秒级完成响应，任何延迟都会直接影响消费者体验和商业运营。

会议宣布 AWS 统一运营计划正式全面上线（GA），这是一个将传统支持模式转变为集成运营模式的创新服务。通过配备专属 AWS 专家团队，该服务帮助客户在问题发生前预防、快速响应突发事件，并提供深度定制化的运维支持，确保关键业务系统在 AWS 上稳定高效运行。

## 详细时间线与关键要点

### 开场与背景介绍
[00:00 - 05:30]
- Steven Clark 以现金使用习惯引入话题，指出现代社会对无现金支付系统的依赖
- 强调每小时全球处理超过 1 亿笔借记卡和信用卡交易
- 介绍 Global Payments 作为全球领先的支付处理商的关键地位

### AWS 支持服务演进历程
[05:30 - 12:00]
- **企业支持（Enterprise Support）**：提供 15 分钟响应时间和主动指导
- **2022 年推出事件检测与响应（Incident Detection and Response）**：直接接入客户告警平台，5 分钟内响应
- **倒计时高级服务（Countdown Premium）**：为关键事件提供规划和实时支持
- **客户需求三大目标**：避免问题发生、快速解决问题、提供有上下文的响应

### 统一运营计划正式发布
[12:00 - 15:30]
- **今日宣布**：AWS 统一运营计划（Unified Operations Plan）正式 GA
- **核心理念**：从支持计划转变为客户与 AWS 运营团队之间的集成运营模式
- **服务内容**：
  - 专属 AWS 专家团队成为客户团队的延伸
  - 预防性问题识别和架构建议
  - 故障模式分析和可观测性设计
  - 定制化运行手册（Runbook）开发与测试
  - 计划事件支持和突发事件快速响应

### Global Payments 公司介绍
[15:30 - 20:00]
- Ganesh 介绍 Global Payments 作为美国、英国、加拿大、爱尔兰和中国第一发卡处理商的地位
- 年处理 750 亿笔交易，其中 350 亿笔来自发卡业务
- 业务覆盖 100 多个国家
- 完整支付生命周期：账户开立、实时审批、卡片制作、PIN 嵌入、交易授权、在线银行、B2C/B2B 应用
- 感恩节周末单日处理 15 亿笔交易

### 云平台愿景与目标
[20:00 - 25:00]
- **2020 年启动云转型**，所有客户当时均在本地生产环境
- **平台目标**：
  - 快速上市能力
  - 客户信心保障
  - 统一控制平面
  - 多种云集成模式
  - 选择 AWS 作为首选合作伙伴

### 主机授权平台现代化
[25:00 - 32:00]
- **旗舰平台特点**：高吞吐量、低延迟
- 每月处理 36 亿笔交易，支持 100 多个客户
- **授权流程**：交易从支付网络（Visa/Mastercard）到发卡行，必须在毫秒级完成
- **处理环节**：30 多个检查点，包括欺诈检测、风险评分、地理位置验证等
- **性能要求**：消费者在收银台没有耐心等待，任何延迟都会影响体验

### 现代化平台架构目标
[32:00 - 38:00]
- **目标性能**：5000+ TPS（当前为 2500-3000 TPS）
- 整体延迟控制在 200 毫秒以内
- 多区域高可用架构
- 微服务化：欺诈、风险、数字钱包等独立服务
- 支持消费者和商业平台
- 避免回到单体架构

### 多区域架构设计
[38:00 - 42:00]
- **AWS 服务选择**：
  - DynamoDB：管理交易数据
  - Amazon RDS：客户配置管理
  - MSK（Managed Streaming for Kafka）：事件处理
- 合成故障转移区域设计
- DynamoDB 全球表副本
- 强调良好的故障模式控制和可观测性

### 生产就绪挑战
[42:00 - 45:00]
- 数百个客户在本地稳定运行多年
- 客户了解现有系统的基线性能
- 必须提供信心保证，确保云平台达到或超越现有标准
- 需要客户和组织共同推进现代化进程

### 统一运营四大支柱
[45:00 - 52:00]

1. 主动指导（Proactive Guidance）
- 架构审查：从设计到 NFR（非功能性需求）全流程
- 故障模式分析：硬故障和灰度故障场景
- AWS 设计专家工程师深度参与组件布局审查

2. 告警与可观测性
- 识别并调整噪音告警
- 确保真实告警不被忽略
- 防止重复问题，建立预防机制
- 通过 RCA（根因分析）提升未来可见性

3. 测试检测与响应
- 联合桌面演练
- 针对每个微服务的系统对比

4. 计划事件策略
- 零停机补丁更新
- 边缘条件洞察
- 容器平台持续补丁管理

### 弹性运营成果
[52:00 - 56:00]
- **关键工作负载审查**：确保安全性、弹性、性能和可观测性
- **中断日响应**：
  - 与 AWS 团队的参与时间少于 2 分钟
  - 在客户致电前主动掌握系统状态
- **成果**：
  - 避免财务影响
  - 维护信任和声誉
  - 确认架构和运营的弹性

### 真实案例：EKS 集群升级
[56:00 - 63:00]
- **场景**：100 多个 EKS 集群需要从 1.30 升级到 1.31
- **挑战**：生产前最后时刻遇到 CoreDNS 问题
- **解决过程**：
  - AWS 专家团队了解完整架构模式
  - 快速排除路由、DNS 等各层问题
  - 精准定位特定 CoreDNS 组件问题
  - 因为从第一天就参与架构设计，具备完整上下文
- **结果**：
  - 成功升级所有 100+ EKS 集群
  - 零生产影响和停机
  - 按时履行客户承诺
  - 保持业务连续性

### 持续改进机制
[63:00 - 68:00]
- **告警优化**：35+ 次告警中仅 7 次升级为事件
- **自动修复**：实施自动愈合机制（如 EKS 容器连接自动重建）
- **运行手册更新**：引入新组件时持续更新 GP 和 AWS 运行手册
- **工作负载审查**：随客户基数和交易量增长持续评估可扩展性
- **成果**：卓越运营、团队效率提升、客户体验改善、风险管理强化

### 对比案例：无统一运营的早期经历
[68:00 - 75:00]
- **场景**：早期关键客户云迁移，某天突然 API 无响应
- **问题**：
  - 无告警、无可观测性工具
  - 团队逐个排查："我这边一切正常"
  - 类似调试老式电视天线的盲目尝试
  - **耗时 9 小时才定位根因**
- **教训**：没有适当支持、架构审查和工具，故障排查效率极低

### 统一运营后的同类问题处理
[75:00 - 结束]
- 建立 AWS 协作后，同类 MSK Connect 告警问题得到快速解决
- 强调提前规划和架构审查的重要性
- 复杂系统需要完善的可观测性和响应机制
