# AWS re:Invent 2025 会议总结：从技术信号到业务洞察的可观测性转型

## 会议概述

本次会议由 Coralogix 的开发者关系团队成员主讲，探讨了可观测性领域中一个经常被忽视的重要话题：如何将技术遥测数据转化为具有业务价值的洞察。演讲者通过实际案例展示了如何从最简单的非结构化日志开始，通过数据解析、enrichment（数据增强）和转换，最终生成能够向 C 级高管传达的业务影响报告。

会议的核心论点是：可观测性之所以被认为"昂贵"，并非因为价格过高，而是因为其价值不够清晰。许多组织将技术团队和业务团队割裂开来，导致技术事件无法有效地转化为业务决策依据。演讲者提出了一个四级可观测性成熟度模型，从基础的遥测覆盖到最终的业务洞察生成，强调了数据组织、跨团队协作和工具整合的重要性。

演讲者还介绍了 Coralogix 平台的独特优势，包括不到 30 秒的中位响应时间、完全支持 OpenTelemetry、以及能够直接查询存储在客户 S3 中的数据的 DataPrime 查询引擎。通过多个客户案例（TradeWeb、SoftBet、Kalora 等），展示了如何通过成熟的可观测性实践实现业务价值，包括加速审计流程、提高团队发布速度、降低基础设施成本等。

### 核心问题：信息孤岛 (14:30 - 20:00)

14:30 - 16:30 - 技术与业务的错误分离
- 常见误解：公司分为"技术侧"和"业务侧"
- 这种分离在技术成为业务核心驱动力的今天已不再适用
- 技术现在是横向赋能者，最佳技术能力直接提升业务竞争力
- 但许多工具仍然按照这种旧模式设计（技术界面 vs 业务界面）
- 信息孤岛导致难以进行跨领域关联分析

16:30 - 18:00 - 传统信息流动模式的问题
- 技术事件沿着 CTO → 工程总监 → 工程经理的路径上报
- 只有"足够重要"的事件才会传播出去，但"足够重要"的定义往往不明确
- 营销洞察只到达 CMO，销售洞察只到达首席销售官/首席收入官
- 工程师可能对销售洞察感兴趣（如客户反馈产品不支持某些销售流程）
- 营销指标（页面跳出率、停留时间）实际上是技术测量，应该跨公司共享

18:00 - 20:00 - 避免依赖临时协作
- 大多数公司依赖临时的、基于个人关系的协作
- 观众调查显示：在公司工作 10 年以上的人会认识大多数人，依赖个人关系而非流程
- 这种模式基于少数人的善意和友谊，不可持续
- 需要通过整合工具来建立自然的信息流动机制

### 技术事件的业务影响 (20:00 - 23:00)

20:00 - 21:30 - 技术事件影响所有人
- API 减速会影响公司的每个部分
- 页面加载变慢，营销落地页受影响
- 整个技术栈可能产生连锁反应
- 技术时刻需要以有意义的方式呈现

21:30 - 23:00 - 当前问题
- CPU 峰值等技术事件通常只沿技术路径上报
- 不会自然地横向传播到其他部门
- 需要改变这种模式

### 实践案例：从日志到业务洞察 (23:00 - 32:00)

23:00 - 25:00 - 起点：非结构化日志
- 展示一个普通的非结构化日志示例
- 介绍 Coralogix 的日志摘要功能（按空格键触发）
- 自动识别元数据，如用于追踪关联的 trace ID
- 日志本身信息有限："RPC not found error"对非技术人员无意义

25:00 - 27:00 - 添加业务上下文
- 该错误发生在结账流程中
- 业务含义：每次发生此错误，就有客户无法完成购买
- 日志中的 ID 字段可用于过滤和分析
- 提取关键值以便索引、查询、过滤和聚合

27:00 - 29:30 - 数据增强（Enrichment）
- Enrichment 是一个被低估的技术
- 通过 ID 字段关联元数据，将其添加到日志中
- 创建包含所有必要上下文的单一文档
- 对深度分析至关重要

29:30 - 32:00 - 转化为可用数据
- "可用"意味着可以向 C 级高管传达
- 目标：让高管理解"这是重要的事情，原因如下"
- 实现信息流动的非临时化

### DataPrime 和数据集 (32:00 - 38:00)

32:00 - 34:00 - 创建查找表
- Coralogix 的"数据集"概念类似数据库表
- 存储相似数据的集合，不必完全相同结构
- 不必是时间序列数据
- 示例：产品详情查找表，包含产品 ID 和相关信息

34:00 - 36:30 - DataPrime 查询语言
- Coralogix 的分析语言
- 可在单个查询中查询日志、指标和追踪
- 支持 SQL 类似功能：连接、过滤、聚合、计数等
- 示例：将产品详情数据集与原始日志连接
- 动态提取产品 ID 字段并进行连接
- 从查找表中获取所有相关信息

36:30 - 38:00 - 业务价值计算
- 增强后的日志包含产品价格信息
- 结合结账失败的上下文，可以计算损失金额
- 对比效果：
  - "我们有一个 RPC not found 错误" - 难以引起重视
  - "过去 15 分钟我们因这个 bug 损失了 $1,500" - 极具说服力
- 数据的准备和整理对多个目标至关重要

### 数据分层和存储优化 (38:00 - 40:30)

38:00 - 40:30 - 打破传统模式
- 传统模式：所有数据进入高性能索引存储（如 OpenSearch + SSD）
- 2 周后移至低成本压缩存储
- 假设所有数据同等重要 - 这是错误的
- 不同数据有不同的价值时间窗口
- 某些日志只在一段时间后才有用
- 某些追踪只在发生后 10 秒内有用
- 必须区别对待数据
- 重要细节：示例中的查询直接从客户的 S3 存储中执行

### 可观测性成熟度模型 (40:30 - 52:00)

40:30 - 42:00 - 模型介绍
- Coralogix 为所有客户提供的北极星愿景
- 四个级别的成熟度模型
- 代表可观测性的发展方向

42:00 - 44:30 - 级别 1：遥测覆盖
- 获取全栈遥测：日志、指标、追踪、性能分析、事件等
- 为现有系统提供覆盖
- 为什么是级别 1？数据存在大量缺口时很难进步
- 当前问题：许多供应商的客户为了成本优化而不发送日志
- 这使得因果分析变得非常困难
- 避免数据缺口至关重要

44:30 - 47:00 - 级别 2：工程智能
- 大多数组织达到的级别
- 级别 1 和 2 是通用问题
- 围绕数据库、基础设施、服务器、进程等整合遥测
- 定义 SLO、跨团队一致的测量标准
- 当前 SRE 的黄金标准
- 其他可观测性成熟度模型通常止步于此

47:00 - 50:00 - 级别 3：生产智能
- 解决"价值不足"的感觉
- 将工程知识置于生产上下文中
- 不仅仅是"API 违反了 500ms SLO"
- 而是"对客户的影响是什么？"
- APM 在此变得非常有用，可以随时间测量
- DataPrime 允许以新的自定义方式处理数据
- 需要灵活、丰富的查询语言
- 必须能够访问大量数据
- 许多客户因成本或扩展问题不生成追踪
- 追踪提供服务间通信的可见性
- 帮助识别瓶颈和性能问题
- 使根因分析更容易
- 日志对因果分析也很重要（仅有指标难以理解"为什么"）

50:00 - 52:00 - 级别 3 的实践应用
- 将 SLO 与 SLA 放在同一仪表板上
- SLO 设置为略低于 SLA
- SLO 成为违反 SLA 的领先指标
- 这是应用此逻辑的优秀示例

### 级别 4：业务洞察 (52:00 - 56:00)

52:00 - 54:00 - 与前三级的区别
- 级别 1-3 关于响应事件
- 级别 4 是主动深入数据寻找洞察
- 示例中的错误只是背景噪音，没有重大中断或峰值
- 主动调查发现了业务影响
- DataPrime 的强大之处：增强和装饰数据以产生业务影响洞察

54:00 - 56:00 - 业务洞察的本质
- 将分散的技术信号转化为任何人都能使用的信息
- 之前提到的两个原因之一

### AI 代理和数据组织 (56:00 - 59:30)

56:00 - 57:30 - AI 代理的兴起
- 询问观众是否在遥测数据上使用 AI 代理
- 少数观众举手
- AI 代理与遥测交互时的关键挑战：
  - 遥测数据量大
  - 代理的上下文窗口有限
  - 一次只能查看有限数据

57:30 - 59:30 - 组织良好的数据的重要性
- 良好组织的数据让代理能够决定查看哪部分
- Coralogix 的 Ollie（可观测性代理）：
  - 知道如何查询日志、指标和追踪
  - 执行完整的根因分析
  - 当数据组织成数据集时，知道在哪里查找
  - 知道查看特定字段
- 组织数据使 AI 工具更容易进行分析
- 常见误解：组织数据只是为了人类
- 实际上：组织数据是为了任何需要通信的实体
- 遥测数据组织常被视为不必要
- 但组织后会带来巨大收益

### 成熟度模型总结 (59:30 - 65:00)

59:30 - 61:00 - 两个阶段
- 级别 1-2：关于数据覆盖和工程洞察
- 级别 3-4：将工程洞察置于业务上下文中

61:00 - 63:00 - 平台要求
- 需要具备必要功能的平台
- 行业正在围绕以下功能整合：
  - APM（应用性能监控）
  - RUM（真实用户监控）
  - 安全监控
  - 基础设施监控
  - 日志分析
- 没有这些功能的平台很难取得进展

63:00 - 65:00 - 成熟数据 + 成熟团队
- 前半部分主要关于数据
- 后半部分关于实践和流程
- 个人经历：在博彩公司工作时发现两个赛马场在亏损
- 发现后没有采取行动，只是继续编码
- 许多人有类似经历：发现问题但认为"不是我的事"
- 6 周后产品经理才发现同样的问题
- 如果有人告诉他应该向谁报告，对公司会有重大影响
- 这就是 DevRel 团队以特定方式与客户互动的原因
- 许多团队只需要一点推动就能前进
- 公司内部的动力至关重要

### 教育和参与的重要性 (65:00 - 68:30)

65:00 - 67:00 - 建立动力的方法

- 有趣的互动活动
- 加强与客户的关系
- 让团队有效使用平台

67:00 - 68:30 - 可观测性的小众性
- 可观测性是一个小众领域
- 爱好者容易忘记这一点
- 大多数人只关心服务器正常运行时间
- 只想保持系统运行
- 教育是可观测性的基本组成部分

- DevRel 团队的核心衡量标准：
  - 与客户的互动次数
  - 互动的创造性和及时性

### 客户成功案例 (68:30 - 76:00)

68:30 - 71:00 - TradeWeb
- 每天发送 130 TB 数据
- 业务问题：审计流程耗时很长
- 使用传统的重新水化模型（数据保留几周后压缩存储）
- Coralogix 解决方案：
  - DataPrime 查询引擎允许无限次查询
  - 数据存储在客户账户的云对象存储（S3）中
  - 无额外成本
- 结果：审计查询从两周缩短到几秒钟
- 这是一个业务问题，不是典型的可观测性用例

71:00 - 73:30 - SoftBet
- 幻灯片显示 65 TB，实际发送约 140 TB
- 全栈日志、指标和追踪
- 最初关注 MTTA（平均确认时间）和 MTTR（平均恢复时间）
- 但更关注采用方面
- 大型分布式公司内部出现基层动力
- 90% 的仪表板是 Coralogix 仪表板
- 包括业务人员在内的 90% 公司员工使用 Coralogix 仪表板
- 意义：整个公司共享相同的洞察
- 因为有共享平台
- 附加好处：MTTA 和 MTTR 降低 30%
- 帮助团队更快发布软件
- 在游戏行业，这是残酷竞争的关键差异化因素

73:30 - 76:00 - Kalora
- 有趣的案例
- Coralogix 审查了他们的 OpenTelemetry 配置
- 发现配置效率低下
- 不影响功能，但可以优化
- 花了几天时间重写 OTel 配置
- 结果：节省了 25% 的遥测基础设施成本
- 发送 14 TB 数据（即将增加）
- 14 TB 的数据处理架构成本相当可观
- 这是因为关注级别 1
- 必须做好基础工作，因为这是通往级别 4 的基础
- 如果只考虑级别 1-2，可能会接受"有点粗糙但能用"
- 但知道北极星愿景后，明白一开始就需要非常干净的基础
