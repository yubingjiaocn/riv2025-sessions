# AWS re:Invent 2025 分会场总结：Red Panda 与 AI 代理的数据流集成

## 会议概述

本次分会场由 Red Panda 公司主讲，重点介绍了流数据处理平台如何与 AI 代理（Agentic AI）技术相结合。Red Panda 是一个基于 Kafka 协议的流数据平台，服务于纽约证券交易所等大型金融机构和零售企业，帮助组织在云端或本地环境中高效、可靠地移动和处理海量数据。

演讲者深入探讨了 AI 代理部署面临的核心挑战，包括信任建立、数据访问授权、审计追踪和多代理编排等问题。Red Panda 提出的解决方案是构建一个"代理数据平面"（Agentic Data Plane, ADP），将流数据处理的成熟技术应用于 AI 代理场景。该平台通过统一的数据层管理所有代理通信，提供不可变日志记录、重放能力和细粒度的访问控制，从而解决"影子 AI"问题。演讲还强调了从已知流程开始、在安全 VPC 环境中部署、以及选择低延迟应用场景（如制造业、广告技术、游戏）作为切入点的实践建议。

## 详细时间线

### 开场与 Red Panda 介绍
- **00:00-01:30** - 开场致辞，感谢参会者从展览区抽身参加会议
- **01:30-03:00** - 介绍 Red Panda 作为流数据平台的定位，区别于视频流媒体，专注于企业数据流动
- **03:00-04:30** - 说明会议议程：AI 与流数据的交叉点、架构深入探讨、实用指导

### Red Panda 平台能力
- **04:30-06:00** - Red Panda 基于 Kafka 协议运行，服务纽约证券交易所等金融机构处理高频交易数据
- **06:00-07:30** - 支持零售企业的会话数据联合和分发，可部署在自托管、任何云或混合环境
- **07:30-09:00** - 展示单一二进制部署形式，云无关特性，支持从高安全性（PII/PHI 医疗数据）到普通零售数据的广泛场景
- **09:00-11:30** - 介绍生态系统扩展能力：从 MySQL/PostgreSQL 等数据库拉取数据，CDC 用例，提供持久化不可变日志，同步到 Snowflake、Databricks、Iceberg 等分析平台
- **11:30-12:30** - 说明部署形态：自托管裸机或云环境中的 Kubernetes

### 代理数据平面（ADP）架构
- **12:30-14:30** - 发布代理数据平面（Agentic Data Plane）产品，定义 AI 代理三要素：意图提示、LLM 模型访问、工具集
- **14:30-16:30** - 架构说明：在 AWS VPC 或自托管数据中心运行，提供运行时服务、网关访问多种 LLM（Bedrock、OpenAI、Anthropic/Claude）
- **16:30-18:00** - 强调模型无关性，允许灵活切换 LLM 以优化代理性能
- **18:00-19:30** - 处理跨 VPC 的数据互联，支持 VPC 对等连接，管理敏感信息的出入站流量
- **19:30-21:00** - 所有服务输出到可观测性日志，提供重放能力，这是 Red Panda 7 年来的核心优势

### 流数据与 AI 代理的协同
- **21:00-23:00** - 将流数据处理经验应用于代理：代理间大量消息传递（如 MCP 协议调用）都可写入持久化日志
- **23:00-25:00** - 讨论 AI 代理面临的挑战：治理、安全、访问控制，归根结底是信任问题
- **25:00-26:30** - 引用海明威名言："建立信任的最好方式是先给予一些信任"，提出实践方法

### 分层架构与影子 AI 治理
- **26:30-28:30** - ADP 采用分层方法：行动层（安全代理、编码代理、制造/IoT 代理）和推理层（LLM 访问）
- **28:30-30:00** - 所有代理通过统一数据平面，实现全面追踪和日志记录
- **30:00-31:30** - 解决"影子 AI"问题（类似过去的"影子 IT"），通过单一平面进行授权认证和凭证管理
- **31:30-33:00** - 强调流数据已无处不在，覆盖网络安全、医疗、零售、分析等各个垂直领域

### 流数据处理原则应用于 AI
- **33:00-35:00** - 利用现有数据流基础设施，已有的认证和服务账户体系
- **35:00-37:00** - 流数据原语适配代理需求：发布-订阅模式、异步操作、分布式消息传递
- **37:00-39:00** - 架构解耦：工具、LLM、推理层分离，需要追踪、持久化存储和灾难恢复能力
- **39:00-40:30** - 实时能力：帮助客户从批处理迁移到实时应用，代理需要实时上下文以影响决策

### 问题空间与解决方案
- **40:30-42:30** - 识别关键问题：上下文构建与维护、授权治理、代理行为审计、重放验证、多代理编排
- **42:30-44:00** - 这些都是流数据已解决的问题，现在增强应用于代理场景
- **44:00-45:30** - 治理和重放：收集追踪数据，不可变存储，内置审计路径
- **45:30-47:00** - 多代理编排：类似现有的跨 15 个服务收集日志、过滤的能力

### 上下文管理架构示例
- **47:00-49:00** - 架构图展示：代理部署在 Kubernetes pods 中，在 VPC 内可扩展
- **49:00-50:30** - 代理需要访问外部数据库，形成复杂的互联网络
- **50:30-52:00** - 通过单一数据平面强制执行策略，便于 DevOps 管理和追踪
- **52:00-53:00** - 特别适用于多代理访问相同 MCP 工具或数据集的场景

### 实践采用指南
- **53:00-54:30** - 总结三大交叉点：数据（访问、认证、授权）、AI（推理模型访问）、治理（统一空间管理）
- **54:30-56:30** - 实践建议第一步：与基础设施和安全团队沟通数据访问权限，这是最耗时的环节
- **56:30-58:00** - 确定哪些数据可用（非零级、一级敏感数据），是通过 MCP 查询还是 CDC 连接器
- **58:00-59:30** - 部署建议：从安全 VPC 和私有网络开始，作为安全护栏
- **59:30-01:01:00** - 在本地 VPC 内锁定和记录代理行为，直到充分理解其非确定性行为
- **01:01:00-01:03:00** - 选择已知流程而非新用例：从有明确 30 步文档的流程开始，更易追踪和衡量
- **01:03:00-01:04:30** - 目标是让一个人通过代理增强变成三个人的生产力

### 制造业用例详解
- **01:04:30-01:06:00** - 聚焦低延迟用例：广告技术、制造业、游戏，秒级响应至关重要
- **01:06:00-01:08:00** - 制造业场景：工厂设备通过 MQTT 协议聚合数据到工厂管理系统和运营数据库
- **01:08:00-01:10:00** - 边缘部署：Red Panda 可在边缘运行，代理辅助技术人员，监控数据流并访问设备手册
- **01:10:00-01:12:00** - 边缘用例：代理根据实时信息提供决策标准或根本原因分析（RCA），指导现场维修
- **01:12:00-01:14:30** - 运营数据库侧用例：50 台同型号设备，安装相同供应商备件后，一台出现故障
- **01:14:30-01:16:30** - 代理推理能力：基于上下文（如备件 PDF 文档）推断第二台设备可能存在相同问题，建议预防性停机
- **01:16:30-01:17:30** - 这种 24/7 运行、人在回路中的代理可发现传统预防性维护或 PID 控制器无法发现的问题
