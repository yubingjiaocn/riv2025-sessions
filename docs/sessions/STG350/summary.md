# AWS re:Invent 2025 安全团队规模化运营会议总结

## 会议概述

本次会议由 AWS 首席信息安全官 Amy 主讲,重点探讨了在 AI 时代如何扩展和优化安全团队的运营能力。Amy 强调,随着生成式 AI 技术的快速发展,传统的确定性安全防护方法已不再足够,安全团队必须适应非确定性系统带来的挑战。会议分享了 AWS 内部如何通过将安全专业知识嵌入产品开发全生命周期、利用 AI 技术提升响应速度(将日志分析时间从 4 小时缩短至 11 分钟)以及构建大规模主动防御系统来应对这些挑战。

会议还介绍了 AWS 安全代理(AWS Security Agent)这一重要产品发布,该工具能够将安全意图转化为自动化机制,在设计和代码审查阶段自动检测安全问题。礼来公司(Eli Lilly)的安全负责人 Andrea Abel 分享了他们如何通过威胁建模方法来管理供应链安全风险的实践案例。Amy 总结了安全团队规模化的三大核心原则:将安全专业知识嵌入业务工作流、保持对风险的敏锐适应能力、以及与业务部门紧密合作解决实际问题。

## 详细时间线与关键要点

00:00 - 开场介绍
- Amy(AWS CISO)介绍会议主题:在 AI 时代扩展安全团队规模
- 强调技术变革(特别是 AI)对安全工作的重大影响

02:30 - AI 对威胁态势的影响
- 生成式 AI 降低了攻击者制作针对性攻击的成本
- 钓鱼攻击变得更具说服力和针对性
- 攻击者开始针对 AI 工具链进行提示注入攻击
- AI 代理的自主性增加带来新的安全挑战

05:15 - AI 为防御者带来的机遇
- AI 帮助安全团队快速处理海量信息,发现"针堆中的针"
- AWS 内部将深度日志分析时间从约 4 小时缩短至 11 分钟
- 凭证泄露事件的自动化响应:自动检索 CloudTrail 数据、发现受影响实体、检测横向移动

08:45 - 安全团队规模化的三大核心原则
1. 将安全专业知识嵌入业务开发和运营工作流
2. 适应不断变化的风险环境
3. 与业务部门紧密合作,解决具体业务问题

10:20 - 原则一:嵌入安全专业知识
- 反对"左移"(shift left)概念,因为产品开发不是线性过程
- 强调在整个产品开发生命周期中嵌入安全能力

12:00 - AWS 内部的三种安全构建方式
1. 创建安全原语(primitives)简化复杂的安全工作
2. 在整个生命周期中嵌入安全指导和工具
3. 优化安全团队自身的构建体验

14:30 - 安全原语案例:S2N TLS
- 2015 年发布,将 OpenSSL 的 50 万行代码精简至 6000 行
- 2022 年发布 S2N Quic 实现 QUIC 传输协议
- 同年添加后量子密钥交换支持

17:45 - API 自动化测试
- 使用 AI 代理自动分析 API 变更并创建全面的测试用例
- 覆盖正常和边缘情况,确保测试一致性

20:10 - 合规性自动化
- 利用生成式 AI 自动化评估准备工作
- 减少对开发人员的重复询问
- 利用已知的产品构建信息进行初步评估

23:00 - 主动防御系统规模数据
- Blackfoot:每天处理 312 万亿次流量转换
- MadPot:每天监控 5.5 亿次恶意活动
- Mithra:每天处理 20 万个恶意域名
- Sonaris:每天阻止近 50 亿次扫描

26:30 - 主动防御系统的客户价值
- 为 AWS Shield、Route 53 Resolver DNS Firewall 提供保护
- 为 WAF 和 Network Firewall 提供托管规则集
- 向 GuardDuty 和 Inspector 提供威胁情报

29:15 - 拒绝方筛查(Denied Party Screening)案例
- 每天处理超过 20 亿次交易
- 使用 Bedrock Agent Core 实现自动化决策
- 准确率 96%,精确率 96%,召回率 100%
- 自动化处理 60% 以上的交易量

32:40 - Neha Runta 登场介绍 AWS Security Agent
- 区分"意图"(intentions)与"机制"(mechanisms)
- 意图需要持续努力和审计,可能失败
- 机制是被编码和内置的,无需人工记忆和执行

35:20 - Block Public Access (BPA) 示例
- BPA 使用策略分析引擎确保资源永不公开
- 多个服务集成 BPA 能力(S3、DynamoDB、EFS)
- 新服务如何知道需要集成 BPA 是一个挑战

37:50 - AWS Security Agent 功能介绍
- 上下文感知,了解组织的安全要求
- 检查设计文档和代码中的安全问题
- 执行按需上下文渗透测试
- 将安全意图转化为机制

40:30 - 设计文档审查演示
- 不到 1 分钟分析所有安全要求
- 识别 11 个不适用、5 个合规、2 个需要更多信息、1 个不合规的要求
- 在编写代码前的设计阶段发现 BPA 缺失问题

43:15 - 代码审查集成
- 内部安全原语 Daffodil(形式化验证的 IAM 库)
- Security Agent 分析代码提交,标记未集成 Daffodil 的情况
- 集成 GitHub,可在 Pull Request 中使用
- 安全要求只需定义一次,可应用于设计和代码审查

46:00 - 可定制性
- 客户可以编码自己的内部安全要求
- 适应特定公司、行业、合规和市场需求
- 指南变更时可更新要求,消除滞后时间

48:30 - 原则二:适应不断变化的风险
- 外部威胁环境持续演变
- 内部开发工作流也在变化

50:00 - APT 29 水坑攻击案例
- AWS 网络威胁情报团队(ACTI)协助破坏 APT 29(Midnight Blizzard)的水坑攻击
- 攻击者入侵合法网站,注入 JavaScript 重定向约 10% 访客
- 使用随机化、Base64 编码、Cookie 等规避技术
- AWS 快速隔离受影响实例并与合作伙伴共享信息

53:20 - 开源包供应链攻击
- Amazon Inspector 团队贡献了 3 起开源包供应链攻击的研究
- Shihuludworm 利用开发者工作流窃取凭证并自我传播
- 部署新的检测规则,结合 AI 识别可疑包模式
- 识别超过 15 万个与特定攻击活动相关的包
- 与 OpenSSF 协调响应和披露

56:45 - 开发工作流的变化
- 生成式 AI 增加了开发者与运行代码之间的距离
- 这为安全团队创造了新机会
- 未来需要更强大的测试安全网
- 安全团队可以在不依赖开发者的情况下自动修复问题

59:30 - 适应性的关键:正确的度量
- 关注风险结果而非过程
- 从双向门学习和实验开始
- 仔细思考度量指标和潜在的不良激励

1:02:00 - Agent 设计经验
- 特定用途的 Agent 集合优于通用 Agent
- 在性能和安全性方面都更优
- 更容易实现最小权限模型

1:05:15 - 度量建议
- 不要只度量容易度量的内容(如发现数量)
- 关注修复时间、P50 时间、最长尾部、开发者操作数量
- 全面度量才能实现更好的自动化和规模化

1:08:00 - 原则三:与业务部门合作
- 解决实际业务问题,而非泛泛的安全工作
- AWS 的核心问题是大规模运营

1:09:30 - Target for Investigation (TFI) 系统
- 识别、分类和升级可能需要安全团队参与的工单
- 支持每年约 150 万个案例,覆盖 285+ 个服务
- 从 2022 年的关键词和正则表达式演进到机器学习,再到生成式 AI

1:12:00 - CVE 管理规模化
- CVE 发布量预计同比增长 22%(从 4 万增至近 5 万)
- 单个 CVE 可能影响数千万资产
- 新发布的漏洞在 5 分钟内完成首次评估
- 使用生成式 AI 将深度评估时间从 27 小时(P90)缩短至 10 分钟以内
- 深度分析覆盖率提升 520%

1:15:30 - 基于 AI 的持续评估引擎
- 持续监控所有 CVE
- 考虑网络可达性、实例详情、身份、数据分类等因素
- 以细粒度方式描述风险
- 减少误报,跨数百万资产正确描述风险

1:17:45 - API 安全评估方法
- 从最大机会开始,粗粒度处理,然后逐步细化
- 迭代循环持续改进
- 从 SDK 理解开始,到 API 使用分析,再到自动化系统上下文分析

1:20:00 - 给客户的建议
- 深入理解业务目标
- 识别业务的特定问题和约束
- 从粗粒度解决大机会开始
- 端到端提问,注意变化

1:22:30 - Agent 安全需求
- 安全构建、配置、执行和扩展 Agent
- 确保 Agent 之间的隔离
- 记住过去的交互并学习
- 细粒度访问控制
- 执行复杂工作流并连接工具
- 全面的遥测数据

1:25:00 - Agent Core 介绍
- 模块化服务,可混合匹配
- 提供将 Agent 安全投入生产所需的一切

1:27:15 - Andrea Abel(礼来公司)登场
- 礼来致力于更快地向患者提供改变生命的药物
- 面临复杂的供应链和 AI 带来的威胁与机遇

1:29:00 - 网络杀伤链原则
- 强调可见性的重要性:"无法防御看不见的东西"
- 必须在网络事件影响业务前阻止或检测

1:30:45 - 供应链安全挑战
- 供应链是可见性最薄弱的环节
- 批评通过问卷调查确定安全态势的做法
- 使用 AI 和机器人更快回答问卷而非提升安全是错误方向

1:32:30 - 威胁建模方法
- 礼来对连接到关键系统的第三方或供应商软件进行威胁建模
- 识别明文密码、共享服务、横向移动风险等问题
- 提供缓解选项和优先级

1:35:00 - 三方合作模式
- 业务部门、安全团队和供应商(在 AWS 上)形成"三条腿凳子"
- 与 AWS 客户经理(Matt Sater 和 Dave Reed)合作
- 共同制定缓解计划,从最关键问题开始

1:37:30 - 供应链安全的未来
- AI 和 Agent 提供了改变供应链安全的机会
- 采用威胁驱动方法
- 目标是在不产生意外后果的情况下实现技术价值

1:39:45 - 正向飞轮效应
- 从业务成果出发并减少开发者摩擦会创造正向循环
- 安全是业务向客户交付的一部分
- 业务框架有助于跨组织协作

1:41:30 - 主动防御团队的起源
- 从业务需求出发(S3 团队减少意外数据发现)
- 跨学科合作:数据科学、分布式系统、威胁情报、软件开发

1:43:00 - 三大原则的实践建议
1. 嵌入专业知识:成为构建者,不要害怕创建所需工具
2. 适应变化:度量正确的指标(安全质量、修复速度、错误率)
3. 业务合作:从业务成果出发,理解摩擦点,明智地设置摩擦

1:45:30 - AI 的作用总结
- AI 带来挑战但也带来机遇
- 开启新机会(如自动部署修复)
- 是帮助安全团队持续改进的又一工具
