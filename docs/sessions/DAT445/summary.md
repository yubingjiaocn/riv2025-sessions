# AWS re:Invent 2025 - AWS Zero ETL 集成深度解析

## 会议概述

本次会议是 AWS re:Invent 2025 第四天的技术分享,主讲人 Dave Gardner(数据库专家解决方案架构师)深入介绍了 AWS Zero ETL 集成技术。会议主要面向数据架构师、数据科学家和 ETL 程序员,旨在帮助他们简化从操作型数据库到分析系统的数据管道构建过程。

Zero ETL 是 AWS 推出的托管服务,旨在消除传统 ETL 流程中的复杂性和脆弱性。通过 Zero ETL,用户可以将事务数据从 DynamoDB、Aurora、RDS Oracle 等源系统自动复制到 Redshift、S3 数据湖和 OpenSearch 等分析系统,无需编写和维护复杂的数据管道代码。会议通过实际案例展示了如何利用 Zero ETL 结合生成式 AI 进行情感分析和客户反馈处理,强调了将事务数据快速引入分析系统对业务洞察的重要性。

Zero ETL 的核心优势包括:简化设置流程、自动处理 schema 变更、支持跨账户和跨区域复制、提供秒级数据新鲜度,以及减少运维负担。这使得数据团队能够将更多精力投入到数据分析和 AI 应用开发上,而不是维护数据管道基础设施。

### **业务用例与架构模式 (05:30 - 12:00)**
- **05:30** - 为什么需要分析事务数据:客户关系管理、欺诈检测、游戏排行榜、库存优化、情感分析、产品洞察
- **06:45** - 实际案例:Aurora MySQL 中的客户评论数据,使用生成式 AI 进行情感分析
- **07:30** - 完整工作流展示:Zero ETL 将数据从 Aurora 复制到 Redshift → 连接产品和评论表 → 调用 LLM 进行情感分析 → 根据情感生成个性化响应(正面推荐新产品,负面提供折扣)
- **09:15** - CQRS 架构(命令查询职责分离):事务系统需要高可用性和性能,分析系统需要复杂查询能力
- **10:30** - 传统架构的复杂性:使用 Lambda、DMS、ETL 工具从 DynamoDB/DocumentDB/RDS 提取数据,加载到 S3、Redshift、OpenSearch

### **Zero ETL 核心概念 (12:00 - 15:30)**
- **12:00** - Zero ETL 的价值主张:简化数据管道设置和管理,支持 PB 级事务数据分析
- **13:15** - 传统 ETL 的痛点:中间层复杂、脆弱、需要大量维护工作
- **14:00** - Zero ETL 作为托管服务,将"无差异化的繁重工作"交给 AWS 处理
- **14:45** - 使数据团队能够专注于高价值分析和 AI 应用开发

### **DynamoDB Zero ETL 详解 (15:30 - 30:00)**
- **15:30** - DynamoDB 数据复杂性:NoSQL 键值对、嵌套 JSON、单表设计(项目集合)
- **17:00** - 目标系统差异:Redshift(列式存储、分布键、SUPER 数据类型)、S3 Lakehouse(Iceberg 表)、OpenSearch(文档和索引)
- **18:30** - 前置要求:启用时间点恢复(PITR)、配置表资源策略允许 Redshift 和 Glue 访问
- **19:45** - 在 AWS Glue 控制台创建集成:选择 DynamoDB 作为源,Redshift 或 Lakehouse 作为目标
- **21:00** - 跨账户支持:开箱即用,可将事务账户数据复制到分析账户
- **22:15** - "为我修复"功能:自动创建所需的 IAM 策略和资源策略
- **23:30** - 输出设置:选择展开顶层字段、展开所有字段或不展开;配置分区键(使用 DynamoDB 原生键或自定义键)
- **25:00** - 数据映射示例:Redshift 使用分布键和 SUPER 数据类型存储嵌套数据;Lakehouse 创建 Iceberg 表
- **27:00** - 安全配置:KMS 加密密钥、一次性快照或持续复制
- **28:30** - 数据新鲜度:每 15 分钟同步一次,使用 DynamoDB S3 导出和 PITR

### **DynamoDB 到 OpenSearch (30:00 - 36:00)**
- **30:00** - 使用场景:实时搜索(如租车系统中搜索"拉斯维加斯 12 月第一周的红色特斯拉")
- **31:00** - 技术实现:使用 DynamoDB Streams,启用新旧镜像
- **32:15** - 数据新鲜度:秒级更新,适合实时搜索需求
- **33:00** - 在 OpenSearch 中创建管道蓝图,需要编写 JSON 映射配置
- **34:30** - 映射配置包括:源表定义、S3 中间存储桶、目标 OpenSearch 集群和索引、字段映射函数
- **35:15** - 容量规划:每个 OCU(OpenSearch 计算单元)处理 1MB/秒,相当于 10 个 DynamoDB WCU;支持自动扩展
- **35:45** - 支持死信队列处理错误

### **Aurora Zero ETL 深度解析 (36:00 - 45:00)**
- **36:00** - Aurora Zero ETL 的独特优势:在存储层实现,对计算实例无影响
- **37:15** - 支持版本:MySQL 3.5.2+ 和 PostgreSQL 16.14+
- **38:00** - 技术架构:直接并行导出从 Aurora 存储到 Redshift 存储;使用 MySQL 增强 binlog 或 PostgreSQL 逻辑复制进行 CDC
- **39:30** - 数据新鲜度:秒级同步
- **40:15** - 数据映射:关系型数据库到列式存储,支持 SUPER 数据类型
- **41:30** - 过滤功能:支持包含/排除特定数据库、表或行(基于客户反馈在两年前发布后添加)
- **42:45** - 过滤示例:只复制"mysteries"类别的书籍,或排除特定作者
- **43:30** - 重要提示:"为我修复"功能可能需要重启数据库(启用逻辑复制),建议在维护窗口执行
- **44:15** - Redshift 端的"为我修复":自动配置大小写敏感性

### **其他 RDS 引擎与新功能 (45:00 - 52:00)**
- **45:00** - RDS Oracle Zero ETL:使用 redo 归档日志而非 binlog,从 EBS 复制到 Redshift/Lakehouse
- **46:30** - Oracle Database@AWS:在 AWS 数据中心运行的 OCI Exadata,支持 Zero ETL 到 Redshift 和数据湖
- **47:45** - Oracle@AWS 配置要求:启用 redo 日志、配置网络、Secrets Manager、KMS、IAM 策略、SYSDBA 用户权限
- **49:00** - DocumentDB 到 OpenSearch:上季度发布,类似 DynamoDB 流程
- **50:00** - DocumentDB 限制:每个管道只支持一个集合;暂不支持跨区域/跨账户;不支持弹性集群
- **51:00** - 重大发布:上周宣布 AWS Glue Zero ETL 支持自管理数据库(MySQL、SQL Server、Oracle、PostgreSQL),包括本地和 EC2 上的数据库

### **监控、管理与优势 (52:00 - 57:00)**
- **52:00** - 总结:8 种 Zero ETL 连接选项,覆盖 AWS 托管服务、EC2 数据库、本地数据库和第三方云
- **53:15** - Schema 变更自动处理:开发团队添加列时,Zero ETL 自动在目标系统创建对应列,无需手动干预
- **54:30** - 传统 ETL 痛点:Schema 变更导致管道中断或数据丢失,需要深夜紧急修复
- **55:00** - Zero ETL 优势:DDL/DML 变更自动传播,大批量更新自动同步
- **56:00** - CloudWatch 监控:延迟、行数、性能指标;Redshift 提供额外的监控表
