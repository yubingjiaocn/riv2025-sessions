# AWS re:Invent 2025 会议总结：使用 Agentic AI 重新构想大规模迁移规划

## 会议概述

本次会议主题为"使用 Agentic AI 重新构想大规模迁移规划"，由 AWS 迁移转型团队的 Tushar Saxena 和 Jonathan Shapiro Ward 主讲。会议深入探讨了如何利用生成式 AI 和 Agentic AI 技术来彻底改变传统的云迁移规划流程。

传统的迁移规划是一个高度手动化的过程，需要大型团队协作、使用 Excel 电子表格进行规划，并且耗时数月甚至数年。随着生成式 AI 和 Agentic AI 助手的出现，这一过程正在发生根本性变革。AI 代理可以替代人工助手，使迁移规划变得更加无缝和高效。演讲者强调，在开始迁移之前，企业必须明确迁移的原因（如降低成本、缩短上市时间、提高创新能力等），定义可衡量的成功指标，并获得领导层的支持。会议重点介绍了 AWS Transform 这一新的 Agentic AI 服务，它能够端到端地协助从发现、评估到实际迁移的全过程，大幅缩短迁移周期。

## 详细时间线与关键要点

### 开场与背景介绍
[00:00 - 02:30]
- 会议主题确认：使用 Agentic AI 重新构想大规模迁移规划
- 演讲者介绍：Tushar Saxena（AWS 迁移转型服务团队负责人）和 Jonathan Shapiro Ward（AWS 迁移转型发现评估与规划负责人）
- 传统迁移规划的痛点：手动流程、耗时长、信息分散、依赖大型团队和 Excel 电子表格

### 迁移的"为什么"（Why）
[02:30 - 05:45]
- 强调在开始迁移前必须明确迁移动机
- 常见迁移原因：
  - 降低运营成本
  - 时间驱动因素（如数据中心退出、许可证到期）
  - 提升创新能力，摆脱技术债务
- 定义成功指标的重要性：
  - 成本节约
  - 缩短上市时间
  - 增加创新型员工比例
- 获得领导层和业务部门支持的必要性

### 成功迁移的要素
[05:45 - 07:30]
- 领导层支持是先决条件
- 需要明确的约束条件（如时间限制）
- 引用帕金森定律：如果不设定时间限制，工作会无限扩展
- 将非结构化的人类概念（目标、指标、约束）转化为 AI 代理可理解的输入

### 迁移规划的基本原则
[07:30 - 11:00]
- Jonathan 接手讲解迁移规划的四个高层步骤：
  1. **发现（Discovery）**：收集本地环境的基础数据（技术和非技术数据）
  2. **评估（Assessment）**：进行 TCO（总拥有成本）评估和其他专项评估
  3. **波次规划（Wave Planning）**：将应用程序分组并排序为可独立迁移的批次
  4. **敏捷规划（Agile Planning）**：将迁移视为敏捷项目，设置仪式、管理路线图和待办事项

### 迁移规划的循环性质
[11:00 - 13:30]
- 强调迁移规划不是一次性过程，而是循环迭代的
- 初始阶段可能从简单的 RV Tools 文件或 CMDB 导出开始
- 随着迁移进展，需要更详细的数据来进行复杂应用的规划
- 持续更新发现数据 → 评估 → 波次规划 → 敏捷执行的循环

### 评估类型详解
[13:30 - 16:45]
- **TCO 评估**：最关键的评估类型，成本管理不善是迁移失败的主要原因之一
- **专项评估**：
  - 合规性审查
  - 安全评估
  - 通常由 AWS 合作伙伴或专业服务团队执行
- **人员与流程评估**：
  - 组织战略分析
  - 变更管理流程
  - 财务和运营模式转型

### 成本优化与右侧调整
[16:45 - 19:30]
- 本地工作负载通常严重过度配置（为峰值需求配置）
- 零售行业案例：为黑色星期五或圣诞销售高峰配置基础设施
- 直接迁移会导致成本过高
- **右侧调整（Right-sizing）**最佳实践：
  - 匹配实际使用需求曲线
  - 利用云的弹性（如自动扩展组）
  - 手动分析困难，AI 可自动化此过程

### 依赖关系分析
[19:30 - 23:00]
- 展示复杂的 TCP 跟踪可视化图
- 理解通信模式和依赖关系至关重要
- 避免破坏关键应用的通信链路
- 手动分析依赖关系：
  - 使用电子表格和图形分析
  - 耗时且容易出错
  - 遗漏依赖可能导致回滚
- 生成式 AI 和 LLM 可以解决这一难题

### 应用程序组合与波次规划
[23:00 - 27:30]
- **识别应用程序**：定义业务价值单元
- **依赖关系分组**：
  - 共享数据库的应用必须一起迁移
  - 数据库依赖关系是关键
  - HTTP/Web API 依赖通常更能容忍延迟
- **术语说明**：
  - 应用程序（Applications）：业务价值单元
  - 移动组/依赖组（Move Groups/Dependency Groups）：有依赖关系的应用集合
  - 波次（Waves）：移动组的优先级排序列表

### 波次规划可视化
[27:30 - 30:00]
- 从服务器/基础设施 → 应用程序 → 移动组 → 波次的层次结构
- 初期波次可能包含 1 个移动组（10 台服务器）
- 随着信心增强，后期波次可扩展到 10 个移动组（150 台服务器）
- 以甘特图形式展示执行计划

### 波次规划策略
[30:00 - 33:00]
- **初始波次**：
  - 选择小型但有意义的应用（非纯开发测试环境）
  - 较少依赖关系
  - 更高的 RTO/RPO 容忍度（可容忍更多停机时间）
  - 展示价值以获得持续支持
- **中期波次**：更复杂的应用
- **后期波次**：核心关键任务应用，低 RTO/RPO 要求

### 敏捷方法论的重要性
[33:00 - 35:30]
- 反复强调：将迁移视为敏捷项目，而非瀑布式项目
- 许多客户错误地采用线性项目计划
- 必须能够根据新信息演进计划
- 循环迭代过程至关重要

### 传统规划方法的挑战
[35:30 - 37:00]
- 传统方法依赖电子表格、甘特图、Jira 看板
- 手动跟踪工作量巨大
- 见过 50 个标签页的迁移规划电子表格
- 版本控制混乱（多人编辑、邮件传递）
- 生成式 AI 时代可以做得更好

### AWS Transform 服务介绍
[37:00 - 40:00]
- 现场调查：很少人熟悉 AWS Transform
- AWS Transform 是 Agentic AI 服务，用于加速迁移和现代化
- **功能范围**：
  - 迁移（本次会议重点）
  - Windows/.NET/SQL Server 全栈现代化
  - 大型机现代化（COBOL 到 Java）
  - 自定义转换（编程语言和框架升级）- 昨日新发布

### AWS Transform Assessments 演示
[40:00 - 48:00]
- **传统评估方法**：Migration Evaluator、OLA 评估（人工主导，需约一周）
- **AWS Transform Assessments**：
  - 使用 Agentic AI 自动生成 TCO 评估
  - 自动生成代码并执行评估
  - 输出格式：PowerPoint、PDF、Excel
  - 最快的迁移评估方式
  - 支持 x86 计算、附加存储、对象存储（S3）、SAN 存储（FSx）

### 实时演示操作
[48:00 - 55:00]
- 上传 RV Tools 文件
- 选择目标区域（俄勒冈）
- 约 10 分钟生成评估
- **评估结果展示**：
  - 计算、存储、年化成本
  - 预留实例成本
  - 详细假设说明
- **AI 聊天功能**：
  - 询问"为什么推荐专用主机？"
  - AI 解释 Windows 许可证优化和 BYOL（自带许可证）数学计算
  - 基于评估数据的知识库回答
- **导出格式**：
  - PowerPoint：便于向利益相关者展示
  - Excel：用于进一步分析和可视化
  - PDF：文档归档
- **无缝过渡**：评估完成后直接转入迁移执行阶段

### AWS Transform for Migrations 新功能
[55:00 - 会议结束]
- 周一（re:Invent 第一天）发布大量新功能
- 完全重构的迁移体验
- 从评估数据自动填充到迁移作业
- 支持依赖关系分析和详细迁移规划
- 将传统需要数月甚至数年的手动流程缩短至数周
