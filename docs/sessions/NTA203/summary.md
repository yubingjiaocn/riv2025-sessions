# AWS re:Invent 2025 - Agentic AI 架构与实践分享会总结

## 会议概述

本次 AWS re:Invent 2025 分享会深入探讨了 Agentic AI（智能体 AI）在企业中的应用与实践。会议由 Eric Espth（美国企业客户解决方案架构师团队负责人）主持，联合 Akquil Melakunta（高级解决方案架构师）、Jay Vel（高级 SA 领导者）和 Kalpana Mahesh（Agentic AI 专家领导者）共同分享。

会议首先明确了 Agentic AI 与传统生成式 AI 的区别：生成式 AI 主要用于内容生成、摘要和翻译等任务，而 Agentic AI 的核心目标是完成特定任务，特别是解决需要处理大量数据、涉及主观判断的复杂业务问题。演讲者强调这两种技术是互补而非替代关系。

会议通过医疗、零售电商和金融服务三个垂直行业的实际案例，展示了 Agentic AI 如何从试点项目走向生产环境。特别值得注意的是，Gartner 预测到 2027 年将有超过 40% 的 Agentic AI 项目因成本上升、业务价值不明确或风险控制不足而失败，因此会议重点强调了战略方法和价值模型的重要性。AWS 的愿景是成为构建和部署世界上最值得信赖和最有用的智能体的最佳平台，帮助组织通过 Agentic AI 重新构想其业务。

### Agentic AI 与生成式 AI 的区别
[05:30 - 10:45]
- 核心问题：是否应该用 Agentic AI 替代生成式 AI？答案是否定的
- 生成式 AI 适用场景：信息摘要、内容生成、可视化、翻译等
- Agentic AI 的主要目的：完成特定任务，解决类似人类处理的复杂问题，特别是涉及主观决策和大量数据处理的场景

### 医疗行业应用案例
[10:45 - 18:20]
- **生成式 AI 在医疗领域的应用**：
  - 患者体验：聊天机器人提供自助护理分诊、主动沟通、患者友好报告生成
  - 护理管理：记录摘要、风险识别、患者群体细分
  - 医学研究：临床试验患者匹配、研究出版物摘要、医学文献问答

- **Agentic AI 在医疗领域的转型**：
  - 智能患者体验：自主协调多个系统（EHR、实验室结果、影像库、调度系统）
  - 医院员工生产力：多智能体系统自主管理临床工作流程，实时分配资源，预测人员需求
  - 供应链优化：持续监控库存、预测需求模式、自主触发采购流程
  - 关键区别：编排层使智能体能够跨多个领域协同工作，实时做出自主决策并执行操作

### 跨行业机遇与挑战
[18:20 - 22:15]
- 各行业都存在大量未开发的潜力，Agentic AI 技术能够解决行业特定挑战
- **Gartner 预测警告**：到 2027 年，超过 40% 的 Agentic AI 项目将因以下原因失败：
  - 成本不断上升
  - 业务价值不明确
  - 风险控制不足
- AWS 愿景：成为构建和部署世界上最值得信赖和最有用智能体的最佳平台

### 智能体架构基础（零售电商视角）
[22:15 - 30:40]
- **智能体的核心功能要素**：
  - LLM 推理能力
  - 明确的目标设定（例如：客服智能体在 2 分钟内解决查询，客户满意度达 4.5）
  - 工具访问能力（MCP 服务器扩展功能）
  - 记忆系统（短期和长期记忆）
  - 企业数据访问
  - 端到端可观测性和护栏机制

- **单智能体 vs 多智能体**：
  - 零售购买场景示例：客户查询产品信息
  - 单智能体可以完成，但多智能体并行处理更高效
  - 多智能体分工：客户认证、库存查询、促销信息、最近门店定位
  - 结果：提供综合性响应，包含用户认证、个性化推荐、促销信息和最近门店位置

### 从反应式到自主式智能体
[30:40 - 35:50]
- **反应式智能体**：等待用户输入（如传统聊天机器人）
- **自主/主动式智能体**：自主行动，无需用户输入
- **动态定价案例**：
  - 传统上需要团队协作完成定价策略
  - 使用多智能体协作解决复杂定价问题

### 智能体间通信协议
[35:50 - 38:30]
- **Agent-to-Agent 协议**：
  - 开放标准协议，为智能体通信创建抽象层
  - 解决智能体间互操作性问题
  - 支持传输无关通信：HTTP with JSON RPC、gRPC（高性能场景）、REST（Web 应用）

### 多智能体协作模式
[38:30 - 48:20]
- **模式一：智能体作为工具（Agents as Tools）**：
  - 定价平台市场情报团队示例
  - 编排器智能体位于顶层，调用专业领域智能体
  - 竞争对手价格抓取智能体作为可调用函数

- **模式二：动态路由（Dynamic Routing）**：
  - 动态客户定价场景
  - 根据客户历史购买记录进行细分
  - 智能体节点决定通信路径（如购买超过 1000 单位的客户路由到批量折扣智能体）

- **模式三：对等协作（Peer Collaboration）**：
  - 价格审批变更场景
  - 智能体作为对等方共享共同知识库和记忆
  - 基于共享决策而非层级结构

- **模式四：工作流模式（Workflow Pattern）**：
  - 将定价变更推送到平台
  - 确定性流程：基础成本 → 市场定位 → 规则应用 → 验证
  - 一个智能体的输出作为下一个智能体的输入
  - 适用于退货订单管理等确定性工作流

- **复合架构**：实际应用中通常采用多种模式组合

### 生产环境路径（金融服务视角）
[48:20 - 52:30]
- 现场调查：约 20% 参会者来自金融行业
- **Gartner 预测**：到 2027 年，40% 的金融机构将使用 AI 智能体自动化工作流程
- **何时开始**：立即开始
- **从哪里开始**：从业务问题出发，与业务负责人沟通，反向推导解决方案

### 业务价值模型框架
[52:30 - 58:45]
- **成本要素**：
  - 模型 API 访问（20-25%）
  - 模型优化（微调）
  - 基础设施
  - 数据策略（关键差异：智能体可使用企业自有数据）
  - 人才/开发团队培训
  - 运营支持和道德 AI（GDPR、HIPAA 等合规要求）

- **收益要素**：
  - **直接收益**：按访问次数收费、订阅模式
  - **间接收益**：销售收入增长
  - **无形收益**：
    - 增强客户体验
    - 提升员工生产力（如呼叫中心智能体辅助）
    - 客户满意度评分提升
    - 新客户转化率提升

- **ROI 计算示例**（投资咨询智能体）：
  - 假设：每天 10,000 次查询，50% 由机器人处理
  - 人工成本：$30/小时；机器人成本：$0.10/次查询
  - ROI：年节省 1080 万美元 + 新交易增长 + 成本降低 + 客户体验改善 + 员工生产力提升

### 生产环境挑战
[58:45 - 65:30]
- **通用挑战**：
  - **性能**：金融行业需要毫秒级响应，处理数百万笔交易
  - **可扩展性**：交易时段（8-5 点）需要扩展，非交易时段缩减
  - **安全性**：每个智能体有自己的安全要求，需要建立智能体间和底层数据的安全机制
  - **上下文管理**：智能体需要访问完整客户信息以提供准确响应
  - **治理**：跨所有智能体建立统一的治理模型（GDPR、HIPAA 合规）

- **智能体孤岛（Agent Silos）问题**：
  - 现场调查：仅 5-10% 听说过智能体孤岛
  - **场景说明**：金融交易流程
    - 市场分析智能体：分析社交媒体情绪
    - 交易执行智能体：按指定价格和数量执行交易
    - 风险管理智能体：根据风险评分管理投资组合
  - **问题根源**：每个智能体使用自己的服务、在自己的位置写入数据
  - **孤岛挑战**：
    - 集成困难：智能体间无法有效通信
    - 安全性：难以建立统一安全策略
    - 数据不一致：不同位置、不同时间框架的数据
    - 低效扩展：各自独立扩展，无法共享资源

### 生产级架构解决方案
[65:30 - 75:20]
- **应用层**：
  - Web 和移动应用
  - IDE（如 Kira）加速开发
  - 聊天界面（越来越流行的智能体访问方式）

- **认证层**：
  - Amazon Cognito 或其他认证服务

- **智能体和 MCP 服务层**：
  - MCP 服务：协议，用于访问外部工具/数据源
  - LLM（大语言模型）：通过 Amazon Bedrock API 访问

- **Agent Core（智能体核心）**：
  - 大规模部署智能体的关键服务
  - 提供可观测性框架：跨层和层内监控
  - 会话级安全：为每个用户提供专用安全会话，防止数据泄露

- **语义层**：
  - 数据处理和转换
  - AWS Glue：ETL 作业
  - Amazon Neptune：创建数据图和连接关系

- **数据层**：
  - 不一定需要数据湖
  - 关键是了解数据位置、内容和转换需求
  - 维护全面的数据源清单

- **可观测性**：
  - 跨所有层监控
  - 智能体间通信监控

- **安全性**：
  - AWS 的首要优先级
  - 会话级安全隔离
