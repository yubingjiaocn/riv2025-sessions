# AWS技术会议总结：利用Agentic AI实现电信行业自主网络

## 会议概述

本次会议由AWS电信垂直领域首席技术专家Ishwar Perulkar主持，重点介绍了如何利用Agentic AI技术帮助电信运营商实现网络自主化运营。会议邀请了英国电信(British Telecom)移动网络总监Razar Ranama分享了BT的愿景和面临的挑战，以及AWS首席解决方案架构师AJ Ravindra Nathan详细讲解了技术实现方案。

电信运营商在网络运营方面投入巨大的人力和资金成本，运营成本可高达收入的五分之一。传统的网络运营主要依赖人工操作，故障诊断耗时长，效率低下。通过引入AWS的Agentic AI技术栈，特别是Amazon Bedrock和Agent Core等服务，可以帮助运营商构建智能化、自动化的网络运营系统，显著降低运营成本并提升网络性能。

会议展示了BT与AWS合作开发的数据驱动运营(DDOPS)框架，该框架以数据为核心，通过清洗和整合来自网络各层的海量数据，利用机器学习模型和AI代理实现异常检测、根因分析、服务影响分析等关键用例，最终目标是实现L4级别的自主网络运营。

## 详细时间线与关键要点

### 开场介绍 (00:00 - 02:30)
- **00:00** - Ishwar Perulkar开场，介绍会议主题：利用Agentic AI实现电信行业自主网络
- **00:30** - 说明会议结构：AWS挑战介绍、BT愿景分享、技术方案讲解、预期收益和后续步骤

### 电信行业面临的三大挑战 (02:30 - 05:00)
- **02:45** - 挑战一：高昂的网络运营成本 - 可达收入的1/5，大量操作依赖人工，故障诊断耗时
- **03:30** - 挑战二：5G承诺未完全实现 - 5G设计为动态可编程网络，但行业尚未充分利用其潜力
- **04:15** - 挑战三：数据未充分利用 - 电信运营商拥有海量数据但利用率低，需转化为数据产品

### 网络生命周期与自主化成熟度 (05:00 - 08:00)
- **05:15** - 介绍网络生命周期四个阶段：规划工程、部署配置、服务激活、运营优化
- **06:00** - 本次工作重点关注运营和服务激活阶段
- **06:45** - 介绍TM Forum定义的自主网络成熟度模型(L0-L5)，当前工作目标为L4级别
- **07:30** - 强调该框架创建于2019年AI浪潮之前，现在通过实际工作赋予其更具体的含义

### AWS AI技术栈介绍 (08:00 - 11:00)
- **08:15** - 基础设施层：包括Nvidia GPU和AWS自研芯片Trainium/Inferentia
- **08:45** - AI开发层：包括Bedrock托管服务、Agent Core原语套件、多种第三方模型和Amazon Nova模型
- **09:30** - 应用层：开箱即用的代理和工具，AWS Marketplace潜力
- **10:15** - 强调Agent Core是构建企业级可扩展Agentic系统的关键组件

### BT公司背景与网络规模 (11:00 - 14:30)
- **11:15** - Razar Ranama介绍BT历史：成立于1846年，一直是电信创新领导者
- **12:00** - BT网络包括固定网络(铜缆、光纤)和移动网络两大层
- **12:45** - 网络规模数据：3000万用户、每日250万活跃用户、关键国家基础设施(紧急服务)
- **13:30** - 网络包括20,000个宏基站、大量小基站、分布式容器化核心网

### BT的三大战略支柱 (14:30 - 17:00)
- **14:45** - Build(建设)：大规模部署光纤网络，2029-2030年实现99%的5G SA覆盖
- **15:30** - Connect(连接)：提供Wi-Fi 7和5G SA融合网络，首家在英国推出Wi-Fi 7的运营商
- **16:15** - Accelerate(加速)：降低成本、加快服务开发，通过AI运营实现目标

### 未来网络愿景 (17:00 - 19:30)
- **17:15** - AI驱动：网络不仅使用AI运营，也是未来AI应用的最佳承载网络
- **18:00** - 意图驱动：基于用户需求和应用类型自动提供相应服务(如游戏切片、安全增强)
- **18:45** - 自主化：自愈网络，5G SA全IP化、API驱动，利用NEF/NSSF等技术实现切片和应用管理

### BT面临的运营挑战 (19:30 - 23:00)
- **19:45** - 网络复杂度高：20,000宏基站、数千小基站、分布式容器化核心网
- **20:30** - 数据爆炸：每个基站监控4,000个KPI，容器化环境产生海量日志
- **21:15** - 运营压力：每天12,000个事件、数百次变更，小变更可能引发连锁故障
- **22:00** - 历史包袱：人员技能从网络工程向软件工程转型困难
- **22:45** - 数据孤岛：数据分散、不清洁、缺乏统一管理，工具繁多形成"工具博物馆"

### DDOPS数据驱动运营框架 (23:00 - 26:00)
- **23:15** - 提出DDOPS(Data-Driven Operations)概念，以数据为核心
- **23:45** - 第一步：修复数据、清洗数据、统一数据摄取
- **24:30** - 五大支柱：
  - 快速定位故障原因
  - 评估影响范围和受影响客户
  - 从数据中学习节点行为模式
  - 自动化一切可自动化的操作
  - 持续改进，确保问题不再重复发生
- **25:30** - 通过数据可视化和分析帮助人类做出更快决策

### 选择AWS的三大原因 (26:00 - 28:30)
- **26:15** - 文化契合：AWS的客户至上文化，"空椅子"理念帮助BT聚焦最终用户体验
- **27:00** - 技术加速：利用AWS现有技术(Bedrock、Agent Core、Nitro)加快开发进程
- **27:45** - 领域理解：AWS团队快速理解电信术语和业务场景(如RAN KPI)，沟通高效

### BT的成功标准 (28:30 - 30:30)
- **28:45** - 数据为基础：不急于实现单一用例，而是构建完整的数据工程路线图
- **29:15** - 安全设计：考虑网络安全威胁，将DDOPS能力扩展到网络优化和网络安全领域
- **29:45** - 全面自动化：不放过任何可自动化的环节，构建端到端用户体验视图
- **30:15** - 可扩展性：避免工具碎片化，构建统一的生态系统

### 解决方案架构 - 北极星愿景 (30:30 - 33:30)
- **30:45** - AJ Ravindra Nathan介绍三层架构愿景
- **31:15** - 网络层：AI原生网络，网络数据源提供经ML模型预处理的数据
- **32:00** - 数据产品层：AI驱动的数据产品生命周期管理，数据摄取、清洗、产品化
- **32:45** - 应用层：数据驱动的AI网络应用，包括GenAI和优化的ML模型
- **33:15** - 核心理念：将数据转化为洞察，将意图转化为行动

### 数据产品生命周期管理 (33:30 - 37:00)
- **33:45** - 数据源：性能计数器、告警、拓扑、配置、事件、变更、知识库
- **34:30** - Agentic数据工程：使用AI代理加速数据工程和特征工程流程
- **35:15** - Agentic语义层：统一定义KPI、告警、关联规则，运行时动态使用
- **36:00** - 开放数据格式：使用Iceberg on S3存储数据
- **36:30** - 图数据库：使用图数据库表示网络连接关系，便于遍历和分析
- **36:50** - 向量存储：将非结构化运营文档和知识转化为可被GenAI使用的向量

### 核心数据产品 (37:00 - 38:30)
- **37:15** - RAN和核心网KPI及告警数据产品
- **37:40** - 客户体验指标数据产品
- **38:00** - 跨域网络和服务拓扑数据产品
- **38:20** - 向量化运营文档数据产品
- **38:30** - 网络数字孪生：高阶数据产品,整合网络模型、拓扑、性能和告警,提供历史和实时视图

### AI应用层与代理 (38:30 - 40:00)
- **38:45** - 超优化模型：异常检测模型、变更影响预测模型
- **39:15** - 因果图：记录代理执行的操作和分析的根因,用于未来学习
- **39:40** - 核心代理：
  - 根因分析代理
  - 服务影响分析代理
  - 故障排查和诊断代理
  - RAN优化代理
- **39:50** - 支持意图驱动编排,将高层意图转化为网络配置变更

### 三大核心用例 (40:00 - 41:00)
- **40:10** - 核心网和RAN异常检测：实时检测网络异常
- **40:30** - 核心网和RAN根因分析：快速定位故障根本原因
- **40:50** - 服务影响分析：评估故障对服务和用户的影响范围

### 技术架构详解 (41:00 - 45:00)
- **41:15** - 数据摄取层：
  - Amazon MSK处理流式数据
  - Amazon EMR处理批量数据
  - AWS Glue Data Catalog管理元数据和业务目录
  - Lambda和EventBridge实现事件驱动架构
  - Amazon S3存储Iceberg格式数据
- **42:30** - 数据存储层：
  - Redshift和ClickHouse存储时序数据(冷/热/超热数据)
  - Amazon Neptune和Neptune Analytics存储拓扑数据
  - Amazon Aurora存储告警和事件数据
- **43:45** - 图分析算法：广度优先搜索、深度优先搜索、社区检测、中心性分析
- **44:30** - 强调使用正确的工具完成正确的工作,针对不同数据类型选择最优存储和处理方案

### 会议总结要点
- **44:45** - 整体架构实现从原始数据到洞察的完整转化
- **45:00** - 支持意图驱动的网络编排,实现高层运营意图到网络配置的自动转换
- **45:15** - 通过Agentic AI实现L4级别的自主网络运营目标
