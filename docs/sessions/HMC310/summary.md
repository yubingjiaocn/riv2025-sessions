# AWS re:Invent 2025 会议总结：数字主权与混合边缘服务

## 会议概述

本次会议深入探讨了AWS如何通过混合云和边缘服务帮助客户实现数字主权目标。演讲者Ben Lavasani（AWS混合云专家）和Mallerie共同介绍了在当今互联互通的数字世界中，组织如何在保持对数据完全控制的同时，充分利用云计算的变革力量。会议指出，82%的公司正在考虑主权解决方案，这反映了一个基本事实：组织既需要云的强大功能，又需要保持对数据的完全控制权。

数字主权不仅仅关乎数据存储位置，更涉及数据访问权限、数据处理方式以及在保持云创新速度的同时维护运营自主权。从欧洲GDPR标准到各行业特定法规，合规要求日益严格。AWS通过其混合和边缘服务组合，为客户提供了实用的解决方案，使其能够在保持对数据位置、访问和处理的绝对控制的同时，充分发挥AWS的全部能力。

会议还特别介绍了沙特阿拉伯支付解决方案提供商Jadia的案例，展示了如何在严格的金融监管环境下，利用AWS Outposts和区域服务构建符合主权要求的支付系统。整个演讲强调了AWS对数字主权承诺的四大支柱：数据位置控制、可验证的数据访问控制、全方位加密以及云的弹性。

### 数字主权框架与挑战（5:30-10:00）

5:30-7:00 - 数字主权两大支柱：
- 数据主权：数据驻留（地理位置控制）、合规性、操作员访问限制、基于角色的访问控制、审计跟踪和加密
- 运营主权：系统弹性、灾难恢复、关键系统冗余、独立性和透明度

7:00-8:30 - 客户面临的三大挑战：
1. 合规挑战：不断演变的监管环境、跨境数据传输限制（如GDPR）、行业特定法规（电信、医疗、金融服务）
2. 安全挑战：平衡强大的安全和数据访问控制与可用性、复杂的加密标准、审计跟踪和监控

8:30-10:00 - 运营挑战：跨IT环境（本地、云、边缘）保持一致管理、确保业务连续性同时保持主权、不同市场的数据本地化要求差异

### AWS混合边缘服务介绍（10:00-18:00）

10:00-11:30 - AWS服务连续体概念：从区域到边缘提供一致的基础设施、服务和API，确保无论在何处运行都有一致的开发者体验

11:30-13:00 - AWS Local Zones介绍：
- AWS部署和运营的基础设施，位于大型都市区（如拉斯维加斯）
- 提供按需弹性服务，按使用付费
- 与区域服务通过AWS骨干网集成
- 全球扩展计划：美国、欧洲、中东、非洲、亚太地区

13:00-15:00 - Dedicated Local Zones（DLZ）：
- 为有更严格主权要求的组织设计
- 专为单个客户或客户社区构建
- 可部署在客户指定的数据中心
- 支持多租户功能，可在不同实体间实现隔离
- 可由位于特定主权领土的AWS人员运营

15:00-18:00 - AWS Outposts介绍：
- 将AWS云扩展到本地数据中心或托管环境
- 提供42U数据中心机架
- 支持核心服务：EC2、EKS、EBS、S3、RDS
- 可跨多个机架扩展，AWS处理机架间网络
- 提供2U和1U服务器选项用于小型部署

### Nitro系统深度解析（18:00-24:00）

18:00-19:30 - Nitro系统概述：
- 自2018年起在所有现代EC2实例中可用
- 是所有混合边缘服务的基础
- 重新定义云基础设施交付方式

19:30-21:00 - Nitro架构三层：
1. Nitro硬件层：将虚拟化功能卸载到专用硬件，使用AWS定制芯片
2. Nitro虚拟机管理程序：轻量级虚拟化层，提供接近裸机的性能
3. 实例操作系统和应用层

21:00-23:00 - Nitro安全四大支柱：
1. 全面加密：所有组件间通信都加密
2. 安全启动：启动时加密验证所有安全密钥
3. 运行时补丁：无需中断工作负载即可持续修补安全层
4. 无远程访问：AWS或外部人员无法访问客户环境

23:00-24:00 - 第三方验证：全球网络安全咨询公司NCC发布独立报告，确认Nitro设计上不存在操作员访问

### Nitro安全密钥与硬件细节（24:00-27:00）

24:00-25:30 - Nitro安全密钥（NSK）机制：
- 是Outpost上所有静态数据的信任根
- 拔出安全密钥后，Outpost被加密粉碎
- 维护时可将密钥交给客户，确保数据安全

25:30-27:00 - 符合NIST媒体清理标准800-88，通过加密方式实现传统物理粉碎的效果，运营上更加便捷

### 客户案例：Jadia（27:00-31:00）

27:00-28:30 - Jadia公司背景：
- 沙特阿拉伯支付解决方案提供商
- 占据沙特POS业务75%市场份额
- 扩展至埃及和阿联酋

28:30-30:00 - 面临的挑战：
- 沙特阿拉伯货币管理局（SAMA/央行）对数据出境有严格限制
- 需要在AWS区域宣布前快速创新

30:00-31:00 - 解决方案：
- 结合阿联酋区域和沙特、埃及的Outposts构建通用技术栈
- 分阶段迁移：先将后端系统迁移到区域，然后扩展并转向云原生架构

### AWS数字主权承诺（31:00-36:00）

31:00-32:00 - Mallerie接手演讲，回顾Jadia创新案例

32:00-33:30 - AWS数字主权承诺四大支柱：
1. 数据位置控制：可选择在特定区域、可用区、Local Zones或Outposts运行
2. 可验证的数据访问控制：Nitro系统无管理员访问，消除人为错误和篡改
3. 全方位加密：支持静态、传输和内存加密，所有AWS服务默认支持加密

33:30-36:00 - 第四支柱：云的弹性
- 所有AWS区域都有多个完全隔离的可用区
- 客户可跨多个可用区和区域分区应用
- Outposts可归属到同一父区域的不同可用区或不同区域

### 数据边界与Control Tower（36:00-43:00）

36:00-37:30 - 数据边界定义：
- AWS环境中的权限护栏集
- 确保只有受信任的身份从预期网络访问受信任资源
- 作为始终在线的边界保护数据

37:30-39:00 - 数据边界实施：
- 通过VPC端点策略、资源控制策略（RCP）和服务控制策略（SCP）组合实现
- 与AWS Well-Architected框架设计原则协同工作

39:00-41:00 - AWS Control Tower介绍：
- 完全托管的服务，用于大规模治理和建立多账户环境
- 建立安全着陆区，基于AWS最佳实践设置身份、治理和日志记录
- 提供超过1000个AWS编写的控制措施

41:00-43:00 - 控制目录三种类型：
1. 预防性控制：使用IAM SCP阻止不合规操作（如阻止关闭CloudTrail日志）
2. 检测性控制：使用AWS Config规则识别已发生的不合规操作
3. 主动性控制：使用CloudFormation Hooks在部署时验证合规性

### 实施示例与自定义策略（43:00-50:00）

43:00-45:00 - Outpost机架上的Control Tower应用：
- 着陆区跨父区域和本地数据中心部署
- 预防性控制自动阻止违规操作
- 策略以纯文本编写，易于理解

45:00-47:30 - 自定义SCP示例：
- 场景：Outpost位于国家1，归属于不同国家的AWS父区域
- 目标：防止从Outpost复制数据回父区域
- 策略拒绝各服务的区域复制（镜像、卷、快照等）

47:30-50:00 - S3命名空间分离设计：
- S3 on Outpost与区域S3有独立的IAM命名空间
- 创建清晰的服务边界
- 便于编写模块化、最小权限策略
- EC2也采用类似设计模式（EC2 Outpost ARN）

### Outposts上的服务详解（50:00-58:00）

50:00-51:30 - Outposts服务概览：
- 14个本地运行的服务（计算、存储、网络）
- 通过AWS服务链接连接到父区域的所有AWS服务

51:30-53:30 - 计算服务：
- EC2实例：通用型、计算密集型、内存密集型
- Amazon EBS：持久化存储，支持GP2（所有类型）和GP3（第二代机架）
- 可从区域复制AMI到Outpost，确保部署一致性

53:30-55:00 - EBS功能：
- 支持启动卷和数据卷
- 可动态增加卷大小，无性能影响
- 本地快照功能
- 默认加密所有卷和快照

55:00-58:00 - S3 on Outpost核心安全特性：
1. 独立的IAM命名空间
2. 默认启用加密（SSE-S3或SSE-C）
3. 仅通过VPC端点访问
4. 默认启用且无法更改的公共访问阻止
5. CloudTrail审计（对象和存储桶级别）
6. CloudFormation集成

### S3 on Outpost架构与路由（58:00-结束）

58:00-60:00 - 数据分离架构：
- 父区域存储：非对象数据（存储桶元数据、策略、CloudWatch指标、CloudTrail日志）
- 本地存储：对象数据和对象标签信息
- 默认情况下对象永不离开Outpost，除非明确移动

60:00-结束 - 请求路由机制：
- 创建存储桶等管理操作路由到父区域
- 对象数据操作保持在本地
- 通过不同的端点和命名空间实现清晰分离
