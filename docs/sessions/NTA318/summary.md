# AWS re:invent 2025 - SAP Rise 与非 SAP 工作负载迁移安全基础架构会议总结

## 会议概述

本次会议是 AWS re:invent 2025 第一天的 NTA-318 技术分享，由 AWS 高级解决方案架构师团队（Shafley、Hajjo Kra 和 Noah Parker）主讲。会议重点探讨了如何为 SAP Rise 和非 SAP 工作负载的云迁移构建安全可靠的 AWS 云基础架构。

演讲者首先介绍了企业面临的常见挑战，包括基础设施管理复杂性、多个支持合同以及数据中心整合需求。SAP Rise with SAP on AWS 作为一个综合性业务转型解决方案，将 SAP S/4HANA 私有云、AWS 基础设施和托管服务整合为单一解决方案，帮助客户将资源从传统基础设施管理转移到业务创新上。会议详细讲解了云迁移策略（包括七种 R 策略）、网络连接选项（Direct Connect、Site-to-Site VPN、Transit Gateway 和互联网连接）、灾难恢复模式以及全面的安全架构。

通过 Mondly International 和 CSL 等真实客户案例，演讲者展示了如何通过 AWS 控制塔、Landing Zone Accelerator 等工具建立多账户架构，实现工作负载隔离、自动化合规控制和统一安全管理。会议强调了 AWS 的共享责任模型，以及如何利用 Nitro 系统、加密服务、网络防火墙、Amazon GuardDuty 和 AWS Security Hub 等原生安全服务构建企业级安全态势。

## 详细时间线

### 开场与背景介绍
- **00:00:00 - 00:02:30** - 会议开场，介绍演讲团队成员，调查现场观众关于 SAP 环境现代化和非 SAP 工作负载迁移的情况
- **00:02:30 - 00:03:45** - 会议目标说明：学习如何设置 AWS 云基础、迁移策略、弹性与灾难恢复、网络连接选项和安全环境配置

### SAP Rise 解决方案介绍
- **00:03:45 - 00:05:20** - 本地 SAP 环境面临的挑战：多个商业合同、多个支持合同、各自的 SLA 以及基础设施管理的繁重工作
- **00:05:20 - 00:07:15** - SAP Rise with SAP on AWS 定义：将 SAP S/4HANA 私有云、AWS 基础设施和托管服务整合为单一解决方案，包含基础设施管理、软件支持和技术托管服务

### 业务驱动因素与客户挑战
- **00:07:15 - 00:09:30** - 客户面临的业务挑战：业务转型、客户服务增强、供应链优化、成为数据驱动型组织
- **00:09:30 - 00:11:00** - 强制性事件：成本削减目标、数据中心退出或整合、并购导致的系统整合、硬件/软件生命周期终止、生成式 AI 创新需求

### 云迁移策略
- **00:11:00 - 00:13:45** - 云迁移策略的三个考虑领域：从本地 SAP 环境迁移到 Rise（SAP 托管环境）、非 SAP 工作负载迁移到 AWS（客户托管环境）、连接性考虑
- **00:13:45 - 00:16:20** - 七种 R 迁移策略详解：Rehost（重新托管）、Relocate（重新定位）、Replatform（重新平台化）、Refactor（重构）、Repurchase（回购）、Retain（保留）、Retire（淘汰）
- **00:16:20 - 00:17:30** - 实际案例：SAP ECC 转换为 Rise、CRM/EWM 组件替换为 S/4、BW 非 HANA 回购为 Data Sphere、Vertex PLM 重新托管或重构、自定义应用保留或淘汰

### 客户案例：Mondly International
- **00:17:30 - 00:19:45** - Mondly International（美国食品饮料公司）案例：处理超过 80 个分散的 SAP 生产系统，选择 Rise with SAP on AWS 实现一致的运营支持和最佳安全性、弹性、可靠性

### AWS 云基础架构最佳实践
- **00:19:45 - 00:22:00** - 常见错误做法：客户通常从三个大型 AWS 账户（开发、预发布、生产）开始，导致工作负载隔离困难、IAM 策略复杂、IP 地址耗尽、账单分离困难
- **00:22:00 - 00:24:30** - 最佳实践：为每个团队创建独立的 AWS 账户，实现安全和账单层面的绝对隔离

### AWS 控制塔与 Landing Zone
- **00:24:30 - 00:27:15** - AWS 账户作为基础容器，AWS Control Tower 提供初始多账户结构、账户工厂、组织单元配置、预防性和检测性护栏
- **00:27:15 - 00:29:30** - AWS Control Tower 的优势：Landing Zone 最佳实践、集中式身份和访问日志、自动化合规账户配置、建立预防性/检测性/主动性控制
- **00:29:30 - 00:31:00** - Landing Zone Accelerator：集成额外 AWS 安全服务、DevOps 管道、配置管理，AWS 专业服务团队和合作伙伴提供支持

### 网络连接架构（Hajjo 主讲）
- **00:31:00 - 00:33:45** - 四种主要连接选项概述：AWS Direct Connect、AWS Site-to-Site VPN、AWS Transit Gateway、互联网连接
- **00:33:45 - 00:36:30** - AWS Direct Connect 详解：专用连接（1/10/100/400 Gbps）或合作伙伴托管连接（50 Mbps 至 25 Gbps），全球 130 多个位置，支持高可用性冗余连接
- **00:36:30 - 00:38:15** - AWS Site-to-Site VPN：成本效益高、快速部署、IPSec 加密、每个 VPN 隧道最高 5 Gbps 带宽，可与 Direct Connect 结合实现冗余
- **00:38:15 - 00:40:45** - AWS Transit Gateway：支持 ECMP 聚合多个 VPN 隧道、支持最多 5000 条路由、支持 IPv4 和 IPv6、从 VPN Gateway 迁移简单
- **00:40:45 - 00:42:00** - 互联网连接：使用 AWS WAF 检查入站连接、通过 Application Load Balancer 路由流量到 SAP Web Dispatcher、TLS 1.2 或更高版本加密

### Direct Connect 架构深入解析
- **00:42:00 - 00:46:30** - 详细架构说明：本地数据中心（站点 A 和 B）连接到两个独立的 Direct Connect 位置（冗余设计）、单个 AWS 区域包含客户托管账户和 SAP 托管账户、Transit Gateway 作为中心连接枢纽、连接第三方 SaaS 应用和 SAP BTP

### Site-to-Site VPN 架构
- **00:46:30 - 00:48:15** - VPN 架构：本地位置通过客户网关建立两个 IPSec 隧道（256 位 AES 加密）到 AWS Transit Gateway，连接客户托管账户和 SAP Rise VPC

### 连接选项决策树
- **00:48:15 - 00:52:30** - 决策树指导：评估是否有现有 AWS 基础设施、是否需要专用连接、带宽需求（超过 1.25 Gbps 考虑 ECMP）、是否需要加密（VPN over Direct Connect）、是否需要多 VPC/多账户架构、是否需要跨区域连接
- **00:52:30 - 00:53:30** - AWS 专业网络和 SAP 解决方案架构师团队提供决策支持

### 运营模型与责任划分
- **00:53:30 - 00:56:45** - SAP Rise on AWS 运营模型：客户和合作伙伴负责应用层（应用和变更管理、功能支持、SAP Basis 应用支持）、SAP 和客户共同负责连接、SAP 负责应用环境和托管（操作系统和数据库维护、SAP Basis 基础设施支持、备份管理）、AWS 负责云弹性（物理基础设施、硬件维护、数据中心）
- **00:56:45 - 00:58:30** - 与 AWS 共享安全责任模型对齐：AWS 负责云的安全（物理基础设施、计算、存储、网络、区域、可用区）、SAP 和客户负责云中的安全（身份访问、配置管理、数据、功能支持）

### 弹性与灾难恢复能力
- **00:58:30 - 01:01:15** - AWS 弹性能力：高度安全的私有网络互连 SAP Rise、BTP、AWS 和第三方服务、多个 AWS 区域全球覆盖、长距离和短距离 DR 选项、Rise 和 BTP 之间的低延迟高性能网络
- **01:01:15 - 01:03:45** - 备份能力：默认加密的 Amazon EBS 卷和备份、AWS Backup Agent for SAP HANA 自动备份、生命周期管理优化存储、AWS Backup 服务提供集中数据保护、勒索软件恢复（逻辑隔离保险库）、AWS KMS 集成加密、AWS Backup Audit Manager 合规监控

### 灾难恢复模式
- **01:03:45 - 01:06:30** - 短距离 DR：同一 AWS 区域内跨可用区同步复制、零 RPO（零数据丢失）
- **01:06:30 - 01:08:15** - 长距离 DR：跨 AWS 区域异步复制、应用服务器可跨可用区部署实现区域弹性、RPO 为分钟级
- **01:08:15 - 01:11:00** - AWS Elastic Disaster Recovery：支持本地到 AWS、AWS 区域内、跨可用区、其他平台到 AWS、持续块级复制、最小化停机和数据丢失、暂存区存储、分钟级恢复
- **01:11:00 - 01:13:00** - 多账户 DR 架构：支持跨账户复制和恢复、账户级隔离确保数据保护

### 安全能力（Noah 主讲）
- **01:13:00 - 01:15:30** - AWS 安全理念：安全是 DNA 的一部分、核心基础设施满足军事、全球银行等高敏感组织的安全要求、企业级安全适用于所有规模客户
- **01:15:30 - 01:18:45** - Nitro 系统安全优势：将虚拟化功能分解为独立的专用处理器单元（Nitro 卡）、降低成本、提供性能优势、增强虚拟化资源的安全性和隔离、无操作员访问设计、加密验证每次启动序列、无中断更新

### Rise 账户安全特性
- **01:18:45 - 01:21:30** - SAP 和非 SAP 账户分离、数据安全保护、默认静态加密（文件、块、对象存储）、Amazon S3 支持的不可变备份、传输中加密使用 mutual TLS、SAP 提供防火墙即服务、短距离和长距离 DR 选项

### 非 SAP 账户安全服务
- **01:21:30 - 01:24:45** - AWS Key Management Service (KMS)：大规模创建、控制和管理加密密钥、保护所有应用和工作负载数据、维护合规性、完全控制密钥访问
- **01:24:45 - 01:26:00** - AWS CloudFront：为用户提供更近的安全内容交付、用户与边缘位置及源之间的加密
- **01:26:00 - 01:27:30** - AWS Network Firewall：大规模深度数据包检查、过滤和监控、专注于威胁管理而非基础设施管理
- **01:27:30 - 01:28:30** - AWS Web Application Firewall (WAF)：实时缓解恶意机器人和攻击者

### 流量检查架构
- **01:28:30 - 01:32:15** - 南北流量和东西流量检查的重要性、AWS Gateway Load Balancer 使用 AWS PrivateLink 支持的端点、集中检查服务、组织范围安全态势、透明拦截流量（bump in the wire）、检查 VPC 使用 Gateway Load Balancer 端点保护 VPC 间通信
- **01:32:15 - 01:33:30** - 第三方提供商支持：开源解决方案、第三方网络防火墙和设备、本地硬件、AWS 原生安全服务

### Rise 账户安全服务集成
- **01:33:30 - 01:35:45** - AWS Network Firewall 作为第一道防线、Amazon Macie 使用机器学习和模式匹配发现和保护敏感 SAP 数据（PII、财务数据）、Amazon GuardDuty 持续监控恶意或未授权活动、所有 GuardDuty 发现集中在 S3 存储桶
- **01:35:45 - 01:36:45** - 第三方网络防火墙合作伙伴与 AWS Gateway Load Balancer 原生集成

### 统一安全管理平台
- **01:36:45 - 01:40:30** - 传统挑战：多个安全工具、各自的界面和工作流、手动分析安全信号之间的连接
- **01:40:30 - 01:43:00** - 统一平台方法：消除工具蔓延、降低运营复杂性、自动关联威胁检测、漏洞和安全发现、提供业务影响和风险级别的丰富上下文、自动化响应工作流
- **01:43:00 - 01:46:15** - AWS Security Hub：统一威胁和漏洞、控制和配置、敏感数据在单一视图中、优先处理真正重要的风险、监控安全趋势、统一资源可见性、快速大规模响应
- **01:46:15 - 01:48:00** - 开放网络安全模式框架 (OCSF)：与第三方合作伙伴和工具无缝集成、减少数据规范化需求、连接其他工具或 AWS 原生服务增强分类和响应

### 集中日志与监控架构
- **01:48:00 - 01:52:30** - SAP 和非 SAP 工作负载日志集中到安全账户、Amazon GuardDuty 持续监控和检查、监控 AWS CloudTrail 检测可疑 API 活动、使用 VPC Flow Logs 和 Route 53 DNS 日志分析网络流量、与 AWS Step Functions 集成自动处理防火墙日志、阻止可疑主机流量
- **01:52:30 - 01:54:00** - Rise with SAP on AWS 的优势：早期检测入侵尝试、通过合规验证满足要求、使用 AWS Lambda 和 AWS Security Hub 自动化事件通知、响应和修复

### 客户案例：CSL Project Unifi
- **01:54:00 - 01:58:00** - CSL 三阶段方法：第一阶段评估现有 SAP 基础设施（识别三个孤立的 SAP 实例）、将现有应用 Lift and Shift 到 Rise with SAP 实现敏捷性、灵活性和降低 TCO；第二阶段使用 SAP Signavio（专门在 AWS 上运行）规划变更路线、通过 SAP BTP 服务现代化解锁新洞察和数据链接、使用生成式 AI 加速开发者创新；第三阶段 AWS 帮助 CSL 通过合作伙伴关系重新构想业务、构建包含自动化的路线图
