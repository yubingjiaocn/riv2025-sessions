# AWS re:Invent 2025 会议总结：使用 Amazon SageMaker Unified Studio 实现 AI 创新

## 会议概述

本次会议由 AWS 技术产品营销经理 Reggu 主持,联邦银行(Commonwealth Bank of Australia, CBA)数据平台总经理 Terry Sutherland 和 AWS 首席分析解决方案架构师 Pravin Kumar 共同分享了联邦银行如何通过 Amazon SageMaker Unified Studio 构建现代化数据平台的转型之旅。

会议核心聚焦于数据基础建设的重要性。根据哈佛商业评论的研究,虽然 89% 的首席信息官计划构建 AI 和生成式 AI 应用,但 52% 的人认为他们还没有准备好,主要原因是对自身数据缺乏足够的信任和了解。因此,在追求 AI 创新之前,企业必须回归基础,建立健全的数据基础设施。

联邦银行作为澳大利亚最大的银行,服务着 1750 万客户(占澳大利亚总人口的三分之一),处理全国 50% 的交易。该银行成功完成了一项雄心勃勃的数据转型项目:在 9 个月内将 61,000 条本地数据管道(相当于 10 PB 数据)迁移到 AWS 云平台,并建立了基于联邦模式的数据网格生态系统,使 40 个业务部门能够独立运作,同时保持严格的治理标准。这一成功实践使联邦银行被评为全球 AI 成熟度前四的银行。

## 详细时间线与关键要点

### 00:00 - 05:30 | 开场与数据基础的重要性
- **演讲者介绍**:Reggu(AWS 技术产品营销经理,在 AWS 工作近 10 年)介绍会议议程
- **核心挑战**:引用哈佛商业评论研究 - 89% 的 CIO 有 AI 计划,但 52% 认为数据准备不足
- **关键观点**:在追求 AI 创新之前,必须建立健全的数据基础,这包括提高数据素养和建立数据文化

### 05:30 - 12:00 | 提高数据素养的七大行动
- **构建数据清单**:建立所有数据资产的目录(表、S3 存储桶、仪表板、模型等)
- **数据注释**:为技术数据添加业务上下文和描述
- **PII 检测**:识别个人身份信息,避免在模型训练中使用
- **数据分类**:按敏感度级别对数据进行分类
- **数据质量评估**:评估数据的好坏程度
- **数据血缘追踪**:跟踪数据来源和转换过程
- **发布新数据资产**:建立持续更新数据清单的实践

### 12:00 - 15:30 | 建立数据文化
- **社交平台类比**:将数据文化比作社交媒体平台
- **三方角色**:数据生产者、数据消费者、中央团队(提供平台、护栏和最佳实践)
- **可持续性**:数据文化需要融入组织 DNA,能够经受高管变动的考验

### 15:30 - 20:00 | Amazon SageMaker 平台介绍
- **三层架构**:
  - 底层:湖仓架构(基于 Apache Iceberg)
  - 中层:数据和 AI 治理能力
  - 顶层:统一数据体验(SQL、数据处理、生成式 AI 应用开发)
- **灵活起点**:客户可以从任何层开始使用,不必全部采用
- **未来功能**:2026 年将推出流处理、BI 和搜索分析功能

### 20:00 - 28:00 | 联邦银行背景与战略
- **银行规模**:澳大利亚最大银行,服务 1750 万客户,处理全国 50% 交易
- **全球排名**:按市值计算全球第 13 大银行,AI 成熟度全球前四
- **三大战略支柱**:
  1. 人员:将数据工程师和数据科学家分散嵌入各业务部门
  2. 保障措施:在数据和 AI 生命周期的每个阶段内置治理和控制
  3. 技术:与 AWS 合作建立数据网格生态系统

### 28:00 - 35:00 | 数据网格与联邦模式
- **生产者-消费者模型**:40 个业务部门作为数据生产者和/或消费者独立运作
- **统一数据市场**:通过单一界面发现、请求和使用整个 AWS 生态系统的数据
- **数据引力原则**:数据存储位置决定了工作位置,因此需要迁移本地数据到云端

### 35:00 - 42:00 | 大规模迁移项目
- **迁移规模**:61,000 条数据管道,10 PB 数据
- **时间周期**:9 个月完成
- **测试强度**:每条管道至少测试 3 次,共计 229,000 次测试
- **100% 准确性**:每个表、每行、每列、每个数字都必须与本地平台完全一致
- **变更管理**:通过 200 场定制培训会议,培训超过 1000 名工程师

### 42:00 - 48:00 | 数据市场建设
- **技术基础**:构建联邦数据网格平台,支持规模化、治理和分散所有权
- **抽象层**:统一访问数据,提供计算和 UI 选择的灵活性,同时维护严格治理标准
- **AWS DataZone**:实现跨组织的发现、访问和共享
- **SageMaker Unified Studio**:提供单一界面体验,从分析师到高管都能查看和使用所需数据

### 48:00 - 52:00 | 转型成果与收益
- **本地开发环境**:工程师可直接从本地 VS Code 连接,不再局限于远程平台
- **单一界面**:统一的仪表板用于数据发现、分析和监控
- **Spark UI 集成**:轻松访问,实时跟踪查询性能
- **专用计算资源**:工作流可靠运行,故障排查更简单
- **SageMaker 集成**:数据实验室输出直接在平台内可用,消除手动步骤

### 52:00 - 56:00 | 经验教训
- 无法通过中心化运营模式和单体平台实现数据和 AI 规模化
- 需要培训人员理解新的联邦运营模式,并为工程师和数据科学家提供新工具认证时间
- MVP、钢线(steel threads)和迭代是发现路径和实现客户价值的最快方式
- 转型始终是学习曲线,需要适应未知

### 56:00 - 1:05:00 | 技术架构深入解析(Pravin 主讲)
- **迁移前架构**:两个大型 Hadoop 集群,一个运行管道,另一个服务交互式工作负载
- **迁移后架构**:
  - 数据迁移到 Amazon S3
  - 计算由 Amazon EMR 提供支持
  - 存储与计算分离,支持多种云原生分析和 ML 引擎
- **联邦设置**:40 个业务部门各有专用 AWS 账户,使用 S3 存储 Apache Iceberg 格式数据

### 1:05:00 - 1:12:00 | 抽象层与关键需求
- **两大关键需求**:
  1. 企业级业务数据目录:生产者发布资产,消费者搜索发现和请求访问
  2. 集成构建体验:为数据工程师、分析师、科学家提供统一界面
- **SageMaker Unified Studio 优势**:
  - 内置治理与 SageMaker 目录
  - 域单元复制组织层级
  - 项目隔离工作负载并映射到数百个 AWS 账户
  - 发布-订阅模型集成第三方引擎

### 1:12:00 - 1:18:00 | 单体与联邦平台整合
- **挑战**:如何将已迁移的单体平台(数万张表)与新的联邦数据市场整合
- **两步解决方案**:
  1. 为每张迁移表识别业务部门所有者,提供必要治理
  2. 构建轻量级 ETL 作业,将元数据从 RDS Hive metastore 复制到 Glue 数据目录
- **结果**:数据消费者可在中心位置浏览、发现、请求访问,并通过查询就地架构跨数百个 AWS 账户构建应用

### 1:18:00 - 1:35:00 | 实时演示
- **演示场景**:为金融服务机构构建欺诈检测模型
- **多账户设置**:
  - 共享服务账户(中央团队,SageMaker 域所在地)
  - 数据生产者账户(零售银行团队)
  - 数据消费者账户(金融犯罪团队)
- **用户角色 1 - Sam(数据工程师)**:
  - 创建客户档案表,合并两个现有表
  - 丰富元数据(添加 README、数据质量、血缘)
  - 发布到目录供他人发现
  - 批准访问请求
- **用户角色 2 - Javier(数据科学家)**:
  - 在目录中发现数据集
  - 请求访问 Sam 创建的表
  - 进行交互式分析
  - 使用 SageMaker AI API 构建 ML 模型

### 1:35:00 - 结束 | 平台功能展示
- **目录浏览**:语义搜索功能,域单元层级映射到业务部门
- **项目概念**:将项目映射到用例,分组团队可访问的计算和数据
- **项目角色 ARN**:映射团队在项目中的计算和数据访问权限
- **数据标签**:显示团队可访问的所有数据集(Glue 表、Redshift、S3 存储桶)
- **计算标签**:显示可用计算资源(Redshift、Glue Spark、HyperPod、MLflow 等)
- **成员管理**:显示项目团队所有成员
