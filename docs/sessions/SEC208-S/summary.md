# AWS re:Invent 2025 安全设计会议总结

## 会议概述

本次技术会议由Native公司创始人Gail和AWS身份团队的Sajania共同主讲，重点探讨了"安全设计"(Security by Design)的重要性及其在AWS环境中的实现方法。Gail曾在AWS工作并领导Security Hub产品，基于丰富的云安全经验，两位演讲者深入分析了从被动检测控制转向主动预防控制的必要性。

会议强调了云提供商虽然提供了强大的安全能力，但客户在充分利用这些能力时面临巨大挑战。演讲者通过具体的EBS卷加密场景，生动展示了传统检测方法的局限性，并提出了基于AWS原生控制机制的解决方案。会议最后介绍了Native公司如何简化跨云环境的安全设计实施过程。

## 详细时间线与关键要点

### 0:00-2:30 开场介绍与背景
- Gail介绍自己作为Native公司创始人的身份，以及与AWS Security Hub的工作经历
- 强调云提供商具备强大安全能力，但客户难以充分利用这些能力
- 介绍会议议程：安全设计重要性、AWS能力、实施步骤及Native解决方案

### 2:30-5:00 检测控制的局限性场景
- Sajania通过EBS卷加密实例说明"修复跑步机"问题
- 安全团队发现1000个未加密卷后的繁琐修复过程
- 检测方法的反应性特点：等待问题发生后再修复
- 组织在检测和修复期间面临的安全风险

### 5:00-7:30 安全设计的预防性方法
- 从检测方法转向预防方法的核心理念
- 通过策略预防新的未加密EBS卷创建
- 安全设计的基础：从反应性转向预防性
- 将加密控制扩展到EFS、S3、KMS等其他AWS资源

### 7:30-10:00 安全设计的应用领域
- 配置管理之外的应用场景
- 云足迹管理：控制AWS资源的区域访问，防止影子IT
- 外部访问管理：合作伙伴和第三方集成的安全控制
- 数据边界框架的引入
- 特权操作控制：数据库删除、安全服务防篡改等

### 10:00-12:30 AWS原生控制机制
- 服务控制策略(SCPs)：为所有主体设置最大权限边界
- 资源控制策略(RCPs)：从资源角度控制访问
- 声明性策略：定义和强制AWS资源配置
- VPC端点策略、基于资源的策略等补充机制
- EBS和EFS加密策略的具体实现示例

### 12:30-14:30 数据边界与复杂性挑战
- 数据边界的三个维度：身份、资源、网络
- 可信身份、可信资源、可信网络的边界定义
- 多种策略类型之间的平衡与协调复杂性
- 跨云环境时复杂性的进一步增加

### 14:30-15:24 Native解决方案与总结
- Native公司基于AWS原生能力的解决方案理念
- 完整的控制实施流程：发现、规划、模拟、实施、运营化
- 跨云提供商的统一治理能力
- 自适应覆盖和持续更新支持
- 联系方式和进一步了解渠道介绍
