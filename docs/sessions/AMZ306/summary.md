# AWS re:Invent 2025 - AMC 306 会议总结

## 会议概述

本次会议主题为"利用AI提升流媒体质量：Prime Video的创新方法"（Elevate Streaming Quality with AI: Prime Video's Innovative Approach）。会议由AWS首席解决方案架构师Tulip Gupta主持，Prime Video的首席工程师Brian Breck和高级数据工程经理Mona分享了两个核心用例。

Prime Video作为全球流媒体平台，服务超过2亿Prime会员，覆盖240多个国家和地区，支持30多种语言，拥有数百万标题内容。随着规模的不断扩大（如2025年NFL比赛峰值观众达到1800万），Prime Video面临着如何高效扩展和保持高质量服务的挑战。为应对这些挑战，团队采用了生成式AI（GenAI）和AI代理（AI Agents）技术，特别是AWS的Strands Agents框架，成功实现了艺术作品质量审核和流媒体质量监控的自动化。

会议展示了AI代理如何从简单的生成式AI助手演进为能够独立推理、规划和执行任务的智能系统。通过Amazon Bedrock、Strands Agents等AWS服务，Prime Video团队不仅大幅提升了运营效率，还显著改善了用户体验。Brian的团队将人工审核工作量减少了88%，将审核时间从数天缩短到不到一小时；而Mona的团队则构建了能够自主检测、定位和缓解流媒体质量问题的智能系统。

### Brian的用例：艺术作品质量审核 (18:00 - 42:00)

18:00 - Brian自我介绍：Prime Video合作伙伴体验团队首席工程师

18:30 - 团队职责：与内容创作者合作，摄取内容并准备流媒体资产（包括艺术作品、元数据、预告片）

19:00 - 现场互动：询问有多少人使用多个设备观看流媒体

19:45 - 挑战说明：需要支持多种设备形态、30多种语言、200多个地区

20:30 - 艺术作品用途：代表电影、电视节目、频道、轮播，出现在流媒体网站和营销材料中

21:15 - 安全区域（Safe Zone）概念介绍：需要在艺术作品周边保留空间以放置Logo或操作按钮

22:00 - 示例对比：《Lioness》艺术作品的合格与不合格案例

23:00 - 其他缺陷类型：成人内容、像素化、本地化问题、色盲可访问性等，共30多种缺陷类型

24:00 - 传统挑战：
- 人工审核流程耗时数天
- 需要为30多种缺陷训练ML模型，耗时长
- 数据质量问题：不同审核员标准不一致
- 评估周期长：需要数天运行数据集

26:30 - 评估流程说明：使用数千张图像，人工标注真实结果（Ground Truth），对比自动化结果

27:45 - 示例：《Free Guy》艺术作品中模型误判Ryan Reynolds头发位置的情况

29:00 - 发现LLM和多模态模型可以检测某些缺陷

29:45 - 初步尝试：使用Q CLI（现已整合到Curo）生成算法，测试不同基础模型

30:30 - 初步结果：精度不够高，但显示出潜力，需要快速迭代

31:00 - 解决方案：构建评估框架用于缺陷检测

32:00 - 评估框架输入：带有真实标注的数据集、初始配置

32:30 - 评估框架输出：结果、改进反馈、艺术作品视图（移动端vs网页端）、基准统计数据

33:30 - 后端架构介绍：使用Strands编排的多代理系统

34:00 - 架构流程：
- 用户通过CloudFront和负载均衡器提交请求
- API将数据存储到配置表（包括初始提示、数据集链接、模型配置）
- 编排器将任务分配给评估代理
- 评估代理执行缺陷检测
- 结果写入S3存储桶

36:00 - 结果计算器生成统计数据并确定下一步行动

36:45 - 提示改进代理（Prompt Improver Agent）：分析结果并建议改进（修改提示、更换模型、调整配置）

37:30 - Strands的三大优势：
1. 简化与LLM的交互
2. 轻松创建代理间关系
3. 提供开箱即用的工具

38:15 - 内置工具：图像读取器、文件读写、代理作为工具

39:00 - 自定义工具：裁剪图像工具（用于安全区域检测）、透明度检查工具（用于可读性和可访问性）

40:00 - Judge（评判代理）机制：提供定性结果和额外上下文，帮助理解失败原因

41:00 - Judge工作流程：配置→Judge配置器代理→Judge代理→提供额外上下文用于提示改进

41:45 - 成本考虑：仅在必要时使用Judge，因为成本较高

42:00 - Strands的整体优势：简化流程、基于文本的接口、集中访问图像、支持回归测试

### Brian用例成果与经验 (42:00 - 48:00)

42:00 - 扩展应用：将框架应用于文本审核（如剧情简介）

43:00 - 数据质量问题发现：人工审核员使用不一致的标准导致精度瓶颈

44:00 - 解决方案：创建标准操作程序（SOP），确保人工和自动化系统使用一致标准

44:45 - 改进后实现自动调优机制：系统可以自主改进，无需人工干预

45:30 - 生产环境架构：
- 配置存储在AWS App Config
- 合作伙伴通过门户上传艺术作品
- 通过API Gateway处理
- 并行运行多个缺陷检测模块
- 从App Config读取配置并调用Bedrock
- 几乎实时生成结果（不到一分钟）

46:45 - 合作伙伴可以覆盖结果，不同意的内容进入人工审核队列

47:15 - 关键成果：
- **精度从37%提升到78%**（安全区域检测）
- **假阳性和假阴性减少70%**
- **审核时间从数天缩短到不到一小时**
- **人工审核工作量减少88%**

48:00 - 经验教训总结

### Brian的经验教训 (48:00 - 51:00)

48:00 - 教训1：不要一次做太多事情，将问题分解为单个缺陷类型，按发生频率和人工工作量排序逐个解决

49:00 - 教训2：在整个生命周期中使用生成式AI（从概念验证到系统设计、开发、评估、监控），约85%的工作使用了GenAI，节省数月工程时间

49:45 - 教训3：LLM能够有效改进自己的提示（使用Claude审查Claude的提示）

50:15 - 教训4：建立稳健的评估循环至关重要，能够快速迭代而无需花费大量时间调查

50:45 - 教训5：人工评估困难且容易出错，值得投入时间生成高质量数据集

51:00 - 未来计划：已完成安全区域，正在推进Logo和文本放置、成人内容、文本可读性、本地化和可访问性，年底前全部投入生产
