# AWS re:Invent 2025 Apache Iceberg 会议总结

## 会议概述

本次会议由Snowflake的媒体娱乐行业首席架构师William Rashon和华纳音乐集团数据工程总监Michael Jones共同主讲，重点探讨了Apache Iceberg在复杂企业环境中的实际应用。会议深入分析了华纳音乐集团如何利用Iceberg作为"连接组织"，在多引擎、多平台的分布式环境中实现数据互操作性。

华纳音乐集团面临着典型的大型企业挑战：拥有75年音乐历史的复杂组织结构，包含众多独立运营的厂牌和子厂牌，技术栈分散在Snowflake、Databricks等多个平台上。通过Iceberg，他们成功构建了统一的数据层，实现了跨平台的无缝数据访问，支持每月数十亿次查询，处理数百TB数据。

## 详细时间线与关键要点

### 00:00-05:00 会议开场与Iceberg基础介绍
- William Rashon介绍会议议程和背景
- Apache Iceberg定义：数据湖表格式，为各种查询引擎提供抽象层
- 核心特性：分区演进、隐藏分区、资产合规、时间旅行、灵活性
- 媒体娱乐行业Iceberg使用量激增：上季度查询量增长300%，表创建增长400%

### 05:00-10:00 湖仓架构与技术生态
- 湖仓架构三层结构：存储层、目录层、计算层
- 存储层作为基础，实现跨系统兼容性
- 目录层提供数据集的一致视图和事务严格性
- 计算层执行查询和数据处理工作
- GitHub贡献者增长趋势显示Iceberg发展速度超越Delta Lake

### 10:00-15:00 华纳音乐集团背景与挑战
- Michael Jones介绍华纳音乐集团：75年音乐历史，复杂的分布式组织
- 技术挑战：多引擎多平台环境，缺乏标准化
- 现有架构：Snowflake支持传统湖仓和BI分析，Databricks构建新数据网格
- 核心问题：如何在不同引擎间实现大量数据的无缝访问

### 15:00-20:00 具体用例场景分析
- WMG Pulse：面向艺术家和词曲作者的分析洞察应用
- BI和营销分析：基于Snowflake的数十亿月查询量
- 机器学习团队：偏好MLflow和Databricks，需要Snowflake数据
- 第三方集成：需要与Spotify等流媒体平台交换数百TB数据

### 20:00-25:00 WMG Opus案例研究
- Opus：华纳音乐最常用的内部工具，700用户使用
- 技术架构：React前端 + Legacy Java Spring后端 + Snowflake直接查询
- 挑战：需要集成Databricks数据网格中的新流媒体产品
- 解决方案选择：从API迁移、Delta集成到Uniform Iceberg的评估

### 25:00-30:00 Iceberg实施策略
- 采用Uniform Iceberg方案：将Delta表导出为Iceberg格式
- 关键洞察：将Snowflake带到数据所在位置，而非移动数据
- 平台引擎无关的数据层设计理念
- 初步成功：每月10亿次查询通过Snowflake表访问Iceberg数据

### 30:00-35:00 双向数据流与联邦目录
- 数据产品从Snowflake读取bronze和silver层数据
- 从联邦查询转向联邦目录，降低计算成本
- 实现Delta表作为Iceberg发布，Snowflake直接读取
- 性能影响：仅10%的查询性能下降，被应用并行化吸收

### 35:00-40:00 扩展应用与统一互操作
- Snowflake原生Iceberg管理表的使用
- 第三方数据共享简化：无需复杂的计算处理
- 统一的跨数据平台互操作性：3.2PB Snowflake + 4PB Databricks
- Iceberg作为系统间API的概念

### 40:00-44:00 经验教训与成果总结
- 技术挑战：无故障安全、转换计算需求、分区文件大小优化
- 运营经验：并行运行监控、自动刷新问题、元数据管理复杂性
- 业务成果：EC2使用量减少70%，单一一致数据层，缩短洞察时间
- 未来愿景：支持团队自主性的平台引擎无关开放生态系统
