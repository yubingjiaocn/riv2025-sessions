# AWS re:Invent 2025 NatWest 数据转型之旅会议总结

## 会议概述

本次会议由NatWest银行的技术负责人分享了该银行正在进行的大规模数据转型项目。演讲者强调了在AI时代，特别是代理式AI(Agentic AI)快速发展的背景下，从事技术工作是一个难得的机遇。NatWest作为英国最受尊敬的银行品牌之一，拥有2000万客户和6万名员工（其中一半是技术和工程人员），正在经历从传统银行架构向现代化、云原生、以客户为中心的数字银行的转型。

这次转型的核心驱动力是建立世界级的数据基础架构，使银行能够真正了解客户，提供超个性化的服务体验。NatWest面临的挑战包括：拥有300年历史的银行、50年历史的大型机系统、2亿行COBOL代码、近30个分散的客户数据库，以及因收购而形成的碎片化技术格局。通过与Accenture和AWS的战略合作，NatWest正在构建基于AWS云服务的现代数据网格(Data Mesh)架构，目标是在2027年底前完成主要转型工作。

演讲者特别强调，这不仅仅是技术转型，更是文化转型——从"我认为"的主观判断文化转变为"我知道，因为数据告诉我"的数据驱动文化。通过采用AWS SageMaker Unified Studio、AWS Glue、Lake Formation和Bedrock等服务，NatWest正在建立支持代理式AI的基础平台，实现从传统数据仓库到现代湖仓架构的演进。

## 详细时间线与关键要点

[00:00 - 02:30] 开场与背景介绍
- 演讲者欢迎参会者，提到当天上午精彩的主题演讲
- 强调现在是从事技术行业的最佳时机，特别是在AI快速发展的时代
- 介绍将讨论NatWest的数据转型之旅

[02:30 - 05:45] 个人背景与加入NatWest的原因
- 演讲者分享了加入NatWest的三个原因：具有挑战性的问题、优秀的合作伙伴、有意义的使命
- 提到之前在医疗行业工作3.5年，在BP工作10年的经验
- 强调金融服务行业正在经历巨大变革，客户希望被当作人对待，特别是在最需要帮助或最快乐的时刻

[05:45 - 08:30] NatWest银行概况
- NatWest是英国最受尊敬的品牌之一
- 拥有2000万客户和6万多名员工
- 一半员工是技术和工程人员，另一半是客户支持人员
- 每天处理英国超过一半的支付交易
- 目标是将传统银行平台现代化，适应代理式AI和数字化未来

[08:30 - 11:15] 转型的起点与挑战
- NatWest是一家拥有300年历史的银行
- 技术基础设施包括：50年历史的大型机、2亿行COBOL代码
- 由于收购形成的碎片化格局：零售银行(NatWest)、私人银行(Coutts)、商业银行
- 关键问题：无法识别同一客户在不同银行品牌中使用的产品

[11:15 - 14:00] 转型愿景：成为超个性化银行
- 核心理念：了解客户是实现超个性化的起点
- 建立世界级的数据基础架构(data spine)
- 目标：简单化、标准化、自动化、云原生
- 强调简化不仅是关闭技术系统，更是简化客户旅程和体验

[14:00 - 17:30] 具体转型目标
- 将近30个客户数据库减少到3个或更少
- 从需要数周甚至数月的发布周期，提升到每天数千次发布（对标新兴银行）
- 简化监管合规流程
- 商业银行客户入职时间从17周大幅缩短

[17:30 - 20:00] 合作伙伴的重要性
- 认识到内部缺乏足够的工程和产品能力
- 转型时间紧迫，需要外部合作伙伴加速
- 选择Accenture和AWS作为战略合作伙伴
- 强调合作伙伴选择的关键：使命一致、互补能力、创造1+1=3的乘数效应

[20:00 - 23:45] 数据基础架构目标
- 定义了银行的目标参考架构
- 数据现代化是NatWest转型的核心
- 四个关键应用场景：
  1. 商业银行：缩短客户入职时间（从17周）
  2. 零售银行：快速推出超个性化产品
  3. 财务：简化复杂的财务报告流程（目前有3万个Informatica批处理作业）
  4. 金融犯罪防控：通过AI实现世界级的欺诈检测和预防

[23:45 - 27:30] 技术架构演进
- 起点：传统本地部署的Teradata数据仓库、Snowflake架构
- 目标：现代化数据网格(Data Mesh)架构
- 数据网格的四个特征：
  1. 分布式数据所有权
  2. 数据即产品
  3. 自助式基础设施
  4. 分布式自动化治理

[27:30 - 31:00] AWS技术栈应用
- 用AWS Lakehouse替代Teradata
- 逐步减少Snowflake的使用
- 采用AWS Glue和Lake Formation进行治理
- 通过数据市场(Data Marketplace)实现网格架构
- 关键创新：通过语义层和自然语言实现数据产品访问
- 为代理式AI建立基础平台

[31:00 - 34:15] 文化转型
- 从"我认为"文化转变为"我知道"或"我不知道但会查数据"的文化
- 案例：从高管猜测客户数量到基于数据的准确回答
- 通过语义层、自然语言和代理实现自动化数据消费
- 与Accenture和AWS的合作重点是规模化和速度
- 目标：2027年底前完成主要转型

[34:15 - 37:00] 转型的复杂性
- 5500个应用程序
- 应用程序的可发现性很低
- 传统的管理信息通过电子表格访问
- 转型不仅是技术变革，更是工作方式的变革
- 利用标志性项目推动实践现代化

[37:00 - 40:30] 架构细节
- 展示了经典的网格架构与现代云原生湖仓架构共存
- 一系列过渡架构支持Teradata退役和Snowflake缩减
- 三个核心组件：
  1. 数据市场（网格引擎）
  2. 治理层
  3. 云原生数据湖（用于管理信息和报告）

[40:30 - 43:45] 与AWS产品团队的合作
- AWS SageMaker Unified Studio是新产品，过去12个月持续演进
- 与AWS密切合作开发产品功能和工业强度
- 与Bedrock和Agent Core产品团队合作构建代理式架构
- 强调AWS产品团队的开放性和响应能力

[43:45 - 47:30] 成功的关键因素
- 使命和目标的清晰性
- 获得CEO级别的高管支持
- 跨三个合作伙伴组织的目标一致性
- 多年期资金承诺，避免年度预算削减
- 保持课程修正的灵活性（例如适应代理式AI的兴起）

[47:30 - 50:15] 工业化交付方法
- 坚持工业化的交付方法和模式
- 建立标准化的交付模式
- 采用工厂思维模式
- 将方法编码化和自动化以实现最快速度

[50:15 - 54:00] 经验教训
- **技术方面**：AWS SageMaker Unified Studio作为新产品需要持续开发；与产品团队的协作至关重要
- **机遇方面**：联邦模式赋予数据消费者自主权；将数据所有权和责任下放到数据创建和管理的地方
- **挑战方面**：最大挑战不是技术，而是运营模式的清晰性；需要提前建立交付和消费的运营模式；确保人员学习技术、术语和工作方式；建立基础后可以通过代理式AI实现规模化自动化
