# AWS re:Invent 2025 会议总结

## 会议概述

本次AWS re:Invent 2025分会场聚焦于零售和消费品行业(CPG)如何利用云计算和人工智能技术推动业务创新和竞争优势。会议由AWS北美零售和CPG业务发展团队负责人Aperna Galaso主持,分享了两个客户案例:Mondelēz International(亿滋国际)和Grainger。

会议核心主题围绕IT部门如何从传统的被动服务提供者转变为战略合作伙伴。在当前技术快速变化、消费者期望不断提高、利润空间收窄的环境下,CPG公司面临前所未有的挑战。传统的IT角色已经不再适用,IT领导者需要与业务领导者并肩作战,共同制定战略决策。两家公司通过不同路径实现了相同目标:Mondelēz通过构建自动化平台成为敏捷创新伙伴,Grainger则通过FinOps和AI代理技术提升决策能力。

这两个案例展示了如何通过技术创新将IT从成本中心转变为价值创造中心,使IT成为企业战略决策桌上不可或缺的声音。

## 详细时间线与关键要点

### 开场介绍 (00:00 - 05:30)

00:00 - 01:30 - Aperna Galaso开场致辞
- 介绍会议主题:零售和CPG行业如何利用云和AI推动竞争优势
- 将分享Mondelēz和Grainger两个客户案例

01:30 - 03:45 - 生成式AI发展现状
- GenAI已从概念验证阶段发展到生产应用,企业开始关注业务价值
- 但技术演进速度太快(从GenAI到智能体再到多智能体编排),企业感觉难以跟上
- 引用《爱丽丝梦游仙境》中红皇后赛跑的比喻:企业拼命奔跑却停留在原地

03:45 - 05:30 - CPG行业面临的挑战
- 市场变化空前,消费者期望史无前例地高
- 利润空间收窄,服务成本上升
- 技术变革速度持续加快而非减缓
- C-suite高管了解消费者但不熟悉算法和技术演进

### Mondelēz案例:构建云自动化平台 (05:30 - 28:00)

05:30 - 07:00 - 战略转型的必要性
- IT角色从被动服务提供者转变为战略合作伙伴
- 未来CPG公司不仅销售产品,更要构建数字优先的商业模式
- IT和业务领导者必须共同出现在决策桌前

07:00 - 09:30 - Mondelēz公司介绍
- Chris介绍Mondelēz(亿滋国际)
- 年收入360亿美元
- 拥有Oreo、Cadbury等知名品牌
- 2012年从Kraft Foods分拆成立,但品牌历史悠久

09:30 - 12:00 - 云迁移历程
- 2018年开始云迁移之旅,采用托管服务提供商模式进行提升和迁移
- 2020年底最大的SAP系统迁移失败,至今仍在数据中心
- 2021年成立云工程团队,从零开始构建
- 2021-2024年团队从1人增长到30多名全球工程师
- 2024年与AWS签署战略合作伙伴关系
- 2025年宣布所有SAP RISE环境将在AWS上运行

12:00 - 14:30 - 平台设计指导原则
- 为开发者提供自助服务平台
- 默认安全
- 一切自动化
- 一切基础设施即代码(IaC)
- 微分段架构
- 从第一天就考虑扩展到1000个账户的场景

14:30 - 18:00 - 平台架构核心理念
- 将云的力量以安全、集中的方式交到有能力的开发者手中
- 采用"黄金路径"和"赋能约束"的平台工程理念
- 常青方法:确保护栏和控制措施到位,使变更快速且不会过时

18:00 - 22:00 - 自动化配置流程
- 单个请求自动配置:GitHub代码仓库、分支、CI/CD管道、AWS账户
- 使用Account Factory for Terraform自动创建账户
- 自动配置标准角色:DevOps Owner、DevOps Contributor、Read-only
- 20-30分钟内完成所有配置
- 访问请求自动路由到业务负责人,无需中央团队审批

22:00 - 24:30 - GitOps强制执行
- 所有角色仅具有AWS只读访问权限
- 开发者通过Pull Request提交更改
- Spacelift自动触发Terraform plan并提供即时反馈
- 使用OPA(Open Policy Agent)策略强制执行安全标准
- 可以在单一位置更改策略,影响数千个CI/CD堆栈和数百个AWS账户

24:30 - 26:00 - 平台优势示例
- 发现负载均衡器未强制TLS 1.3时,通过策略快速修复所有环境
- Terraform许可变更后,快速迁移到OpenTofu
- 一键启用Amazon Q Developer进行PR审查
- 强制协作:不符合标准的请求被拒绝或促进模块改进

26:00 - 28:00 - 成本管理和未来规划
- 使用AWS CUDOS仪表板实现成本透明
- 微分段架构使每个账户的成本清晰可见,无需依赖标签
- 未来计划:推进产品和平台组织转型、完成遗留云迁移、关闭数据中心、重构和无服务器化、深化AWS战略合作

### Grainger案例:FinOps与AI代理 (28:00 - 50:00)

28:00 - 30:00 - Grainger公司介绍
- JC介绍Grainger
- 北美领先的MRO(维护、维修和运营)公司
- 2024年收入172亿美元
- 服务450万客户,提供3000多万种产品
- 26,000名员工,自1927年起"让世界保持运转"

30:00 - 32:30 - 战略挑战
- 关键洞察被困在复杂系统中
- 利益相关者忙碌且缺乏专业知识解读数据
- 没人访问仪表板
- 通过邮件、Slack、会议等手动沟通无法跟上企业增长速度
- 需要从集中式成本优化转向企业级FinOps民主化

32:30 - 35:00 - 初始解决方案
- 目标受众:产品和技术领域领导者(控制大量云使用和优先级)
- 解决方案:执行摘要,发送到收件箱
- 无需新工具,无需行为改变
- 由FinOps专家手动生成
- 领导者可在5分钟内了解云使用、增长和优化机会

35:00 - 37:00 - 不可能的数学问题
- 每份摘要需要1个完整工程日
- 初始6个领域领导者,最终目标30-50个接收者
- 每年需要4,800工程小时仅服务一个角色
- 无法扩展,但领导者确实参与并采取行动

37:00 - 39:30 - 评估三种方法
- 手动:质量高但无法扩展
- 程序化:可扩展但缺乏细微差别、定制指标和趋势分析
- 生成式AI:在正确约束、模板化提示和组织数据支持下,可达到专家质量并实现企业规模

39:30 - 42:00 - 生产化挑战
- 传统IaC方法(Terraform、Boto3)跟不上GenAI创新速度
- Terraform笨重,AWS provider缺少所需配置
- 解决方案:采用LangGraph Agents在源代码中定义代理
- 需要确定性输出而非可变响应(不是聊天机器人)
- 开发无头MCP客户端,无需人工交互

42:00 - 44:30 - 企业约束和技术选择
- 必须满足安全、采购、法律要求
- 选择Amazon Bedrock Agent Core(当时仍在预览阶段)
- 提供企业级要求、配置便捷性、身份验证、可观察性和监控
- 缺少文档,需要深入Agent Core源代码寻找答案

44:30 - 47:00 - Project Nightingale架构
- 使用Amazon Bedrock Agent Core和Claude模型
- 两个关键代理:FinOps分析师和执行摘要撰写者
- 多代理架构实现企业规模
- 自定义MCP服务器和工具
- 数据源:组织数据、Cost Optimization Hub推荐、云成本管理工具账单数据

47:00 - 50:00 - 一致性输出和结果
- 需要一致、确定性的输出,不能有幻觉或错误
- 通过模板化提示和约束实现
- 成功实现分析师质量的FinOps云成本智能
- 企业规模部署
- 演示和未来展望(会议结束)

会议总结要点:
- IT必须从服务提供者转变为战略合作伙伴
- 自动化和基础设施即代码是敏捷性的基础
- 微分段和GitOps强制执行提供安全性和可扩展性
- 生成式AI可以将专家知识民主化到企业规模
- 成功的关键是满足用户需求而非强迫行为改变
