# AWS re:Invent 2025 会议总结：生成式 AI 代理与 MCP 在软件开发中的应用

## 一、会议概述

本次 AWS re:Invent 2025 分会场由 Derek、Pavatra 和 Gita 三位讲师共同主讲，主题聚焦于生成式 AI 代理和模型上下文协议（MCP）如何帮助客户在软件开发中获得更好的结果和更高的开发速度。会议分为三个主要部分：首先介绍 AI 代理的组成原理和基础概念，然后深入讲解如何使用 Kiro 工具构建和部署自定义代理，最后通过 FINRA 的实际案例展示企业级应用场景。

会议强调了规范驱动开发（Spec-driven Development）的重要性，以及如何通过自定义代理来优化开发团队的工作流程。讲师们特别指出，随着 AI 代理技术的快速发展，开发团队正在创建针对不同任务的专用代理，并在团队内部共享这些工具，从而持续改进开发效率。整个会议内容既包含理论框架，也提供了实践指导和真实案例，为与会者提供了全面的 AI 代理应用视角。

## 二、详细时间线与关键要点

### **开场介绍 (00:00 - 02:30)**
- **00:00** - Derek 开场欢迎，介绍三位讲师及会议主题：生成式 AI 代理在 MCP 中的应用
- **00:30** - 说明讲师团队与全球客户合作，帮助他们采用代理式软件开发工具
- **01:00** - 介绍会议议程：代理组成原理、使用 Kiro 构建自定义代理、FINRA 案例分享

### **AI 代理组成基础 (02:30 - 08:00)**
- **02:30** - Derek 开始讲解代理组成原理，引用 Simon Willis 的定义：LLM 代理在循环中运行工具以实现目标
- **03:00** - 扩展定义：强调代理在循环中构建上下文的重要性
- **03:30** - 展示代理循环的可视化流程：输入 → 上下文窗口 → LLM → 工具选择 → 工具执行 → 结果反馈
- **04:30** - 讲解 LLM 的两个核心能力：推理能力和知识储备（通过预训练获得的潜在知识）
- **05:30** - 强调 LLM 是无状态的，需要通过上下文窗口提供工作记忆
- **06:00** - 详细说明上下文窗口的组成：系统提示、上下文文件、聊天历史、源代码、工具调用输出等
- **07:00** - 介绍 MCP（模型上下文协议）作为工具集成的标准，类比为模型的"USB 接口"
- **07:30** - 说明 MCP 服务器可以提供技能、资源和变更操作

### **Kiro 工具介绍 (08:00 - 15:00)**
- **08:00** - Pavatra 接棒，介绍 Kiro 作为最新的代理式 AI 开发工具
- **08:30** - 现场调查：询问有多少人尝试过 Kiro
- **09:00** - 介绍 Kiro 基于规范驱动开发（Spec-driven Development）的设计原则
- **09:30** - 用类比说明：如果只给开发者几个要点就让他们编码，会导致混乱和不符合预期的结果
- **10:30** - 对比传统 Vibe Coding：代理可能会自行做决策，导致开发者需要审查大量生成的代码
- **11:30** - 强调企业级应用需要标准化流程：需求 → 技术设计 → 项目计划 → 任务分解 → 实施
- **12:30** - Kiro 的工作流程：将提示转换为清晰的规范（specs）→ 生成系统设计 → 创建任务分解
- **13:30** - 强调这种方法为复杂分布式系统提供了治理层

### **Kiro Powers 和自定义代理 (15:00 - 25:00)**
- **15:00** - 介绍昨天刚发布的 Kiro Powers 功能
- **15:30** - Kiro Powers 为代理提供专门的上下文，减少 MCP 上下文过载的问题
- **16:00** - 演示如何在 kiro.dev 网站上一键安装 Powers，或创建自己的 Power
- **16:30** - 深入讲解 Kiro CLI 中的自定义代理功能
- **17:00** - 定义自定义代理：可为特定用例、领域和工作流配置的 AI 助手
- **18:00** - 类比：自定义代理像"数字黏土"，可以塑造成任何需要的形态
- **19:00** - 详细说明自定义代理的组成部分：
  - 模型选择（如 Haiku、Opus、Claude Sonnet）
  - 模型提示（定义代理的行为和个性）
  - 预批准工具（MCP 服务器等）
  - 工具配置（读写权限、访问范围）
- **21:00** - 介绍资源部分：上下文文件和钩子（Hooks）
- **22:00** - 说明代理钩子：在执行特定点触发的步骤（如启动前记录日志）
- **23:30** - 强调护栏（Guardrails）的重要性：确保代理符合安全和合规要求

### **自定义代理演示 (25:00 - 35:00)**
- **25:00** - 演示创建自定义代理的两种方式：向导模式和配置文件
- **25:30** - 向导模式演示：通过 /agent 命令启动，逐步提示输入代理名称、描述、作用域
- **26:30** - 展示 AWS Rust 代理的完整配置文件示例
- **27:00** - 配置文件包含：MCP 工具访问权限、工具别名、允许的工具列表
- **28:30** - 展示工具设置：允许写入的路径、可使用的 AWS 服务（S3、Lambda）
- **29:30** - 展示资源配置：readme.md 文件作为上下文、文档文件夹
- **30:30** - 展示代理钩子示例：启动时运行 git status 命令
- **31:00** - 说明使用 Claude Sonnet 4 作为模型

### **代理作用域：本地 vs 全局 (35:00 - 42:00)**
- **35:00** - 解释本地代理和全局代理的区别
- **35:30** - 本地代理：仅在特定项目目录及其子目录中可用
- **36:00** - 全局代理：跨多个项目可用
- **36:30** - 全局代理的使用场景：
  - 跨项目的通用工作流（如基础设施即代码助手）
  - 通用开发工具（如代码审查器、文档编写器）
  - 工具特定工作流（如 Docker、Git 工作流）
- **38:30** - 本地代理的使用场景：
  - 项目特定配置（如微服务开发代理）
  - 个人生产力工具（如每日站会报告生成器）
  - 独特的开发环境（前端/后端测试代理）
  - 测试和实验（如演示数据生成器）

### **自定义代理的优势 (42:00 - 45:00)**
- **42:00** - 工作流优化：为特定任务创建定制代理
- **42:30** - 减少中断：预配置上下文和工具，减少重复询问
- **43:00** - 上下文感知：预加载所需上下文，减少重复解释
- **43:30** - 更好的团队协作：共享配置文件，实现统一标准
- **44:00** - 安全控制：预定义工具访问权限，增强安全性

### **Amazon Bedrock Agent Core (45:00 - 48:00)**
- **45:00** - 介绍在 AWS 云上运行代理的解决方案：Amazon Bedrock Agent Core
- **45:30** - 定义为综合性代理平台，提供模块化服务以大规模运行代理
- **46:00** - 宣布最近两个月推出的两项新服务：策略和评估
- **46:30** - 展示 Agent Core 的模块化服务：长期记忆、短期记忆、身份、可观测性、网关、运行时等
- **47:30** - 说明可以根据需求选择所需的服务模块

### **行业用例概览 (48:00 - 49:00)**
- **48:00** - 展示关键行业垂直领域的代理用例
- **48:30** - 强调代理空间正在快速增长，出现越来越多的智能代理应用

### **FINRA 案例介绍 (49:00 - 55:00)**
- **49:00** - Gita Ramachandran 上台，介绍 FINRA 的自定义代理之旅
- **49:30** - 介绍演讲内容：FINRA 简介、AI 在 SDLC 中的应用、AI 代理实验、演示、经验教训
- **50:00** - FINRA 简介：金融业监管局（Financial Industry Regulatory Authority）
- **50:30** - FINRA 的使命：保护投资者、促进市场诚信
- **51:00** - 监管范围：监督 3,200 家证券公司的 630,000 名经纪交易商
- **51:30** - 数据规模：每天处理高达 1.45 万亿个数据事件

- **52:30** - 监控活动：欺诈、内幕交易、市场操纵等
- **53:30** - FINRA 的 AI 之旅目标：负责任地探索、有计划地扩展、在软件构建和交付中找到真正价值
