# AWS re:Invent 2025 会议总结：构建者思维模式

## 会议概述

本次分享由AWS Serverless Hero、Seammens公司杰出工程师Ben Kho主讲，深入探讨了作为构建者应该具备的思维模式。演讲的核心观点是：构建者不应该将自己与工具或输出物绑定，而应该专注于结果（outcomes）而非产出（outputs）。Ben强调，软件开发不是工匠艺术，而是解决问题的手段。真正有价值的是软件功能本身，而实现代码只是必要的负债。

演讲通过持续部署、无责备事后分析等实践类比，说明了问责制、谦逊和内省对于构建者思维的重要性。Ben指出，构建者的本质是问题解决者，编写代码只是工具箱中的主要工具。如果能在不编写代码的情况下解决问题，那才是更好的解决方案。在AI时代，这种思维模式变得更加重要——责任仍然在人类身上，我们需要理解什么是"好"，才能有效地使用AI工具。

## 详细时间线与要点

### 开场与思维模式定义 (00:00 - 05:30)

00:00 - 会议开始，Ben Kho自我介绍，说明本次演讲关注构建者思维模式

01:15 - 定义思维模式：思维模式是你如何看待世界的方式，包括显性和隐性的思考方式

02:00 - 提出三个揭示思维模式的问题：
- 我是谁？（身份问题）
- 我做什么？（功能问题）
- 我如何应对变化？（响应问题）

03:30 - 解释为什么讨论思维模式很重要，引入持续部署的类比

### 持续部署与思维模式的类比 (05:30 - 10:00)

05:30 - 持续部署的价值：强制你在测试、回滚等方面做到优秀

06:45 - 提出"周五部署"的观点：如果你不敢在周五部署，说明你其他时间的部署也不可靠

07:30 - 三个关键概念：问责制（accountability）、谦逊（humility）、内省（introspection）

08:15 - "You build it, you run it"模式的重要性：开发团队对运维负责

09:00 - 无责备事后分析（blameless postmortems）：优先理解"为什么"而非追究责任

### 谦逊与自信的关系 (10:00 - 11:30)

10:00 - 澄清：谦逊和自信并不互斥

10:30 - 区分自我（ego）和自信：自我是固守不变，自信是承认可以改进

11:00 - 谦逊意味着承认自己可能出错，并愿意发现和改进

### 典型的构建者思维模式问题 (11:30 - 15:00)

11:30 - 问题一：与工具认同（identifying with tools）

12:00 - 举例：VI vs Emacs的争论，SQL vs NoSQL开发者的身份认同

13:00 - 问题二：与输出物认同（identifying with output）

13:30 - "Not Invented Here"综合症：将自己创造的东西视为自己的一部分

14:00 - 问题三：与更大目标隔离（isolated from greater purpose）

14:30 - 只关注狭窄的交付范围，而不理解在组织中的位置和对客户的价值

### 更好的思维模式：我们不是工匠 (15:00 - 22:00)

15:00 - 核心观点：我们不是工匠（artisans）

15:45 - 软件开发已经变得容易，但做好软件仍然困难

16:30 - 我们的爱好恰好可以赚钱，但工作目的与爱好目的不同

17:00 - 机械手表的类比：昂贵和困难本身可以成为价值，但软件不是这样

18:00 - 雕塑的类比：雕塑不需要被使用就有价值，但软件必须被使用才有价值

19:00 - 关键原则：专注于结果（outcomes）而非产出（outputs）

20:00 - 软件功能是资产，实现代码是负债

20:45 - 实现代码最多只能完全符合预期，现实中总会减损价值

21:30 - 构建者的工作是构建功能，而不是编写代码

### 结果导向与组织定位 (22:00 - 25:30)

22:00 - 通过关注结果，可以更容易追溯到客户价值

22:45 - 在多层组织中，结果可以串联起整个价值链

23:30 - 客户需求的理解：人们不买钻头，他们买的是孔

24:15 - 需要深入理解客户真正想要的结果，而不仅仅是他们提出的解决方案

25:00 - 与客户合作找到更好的方式来产生他们想要的结果

### 应对变化与价值链 (25:30 - 28:00)

25:30 - 当客户需求变化时，整个价值链会发生变化

26:15 - 理解自己在价值链中的位置，有助于应对组织变化

27:00 - 可以主动询问："我不理解，请帮助我理解"

27:45 - 理解价值链有助于确定需要创建什么样的输出来实现结果

### 身份重新定义：构建者是问题解决者 (28:00 - 30:00)

28:00 - 回到"我是谁"的问题

28:30 - 核心定义：构建者是问题解决者（problem solvers）

29:00 - 构建是工具箱中的主要工具

29:30 - 如果能在不构建任何东西的情况下解决问题，那是更好的解决方案

29:45 - 避免实现带来的负债，直接获得功能资产

### 应用思维模式：技术选择 (30:00 - 37:00)

30:00 - 将思维模式应用于技术选择

30:30 - 不要与工具认同，将工具视为达到目的的手段

31:00 - 理解自己是可适应的（adaptable）

31:45 - 避免脆弱性：不要说"我的开发环境必须是特定方式"

32:30 - 关键指标：始终关注总拥有成本（Total Cost of Ownership, TCO）

33:15 - TCO包括：个人、团队、组织、公司的成本

34:00 - AWS账单只是系统成本的一部分

34:30 - 需要平衡运维成本和云服务成本

35:00 - 代码运行的次数远多于构建次数，应优化持续运维成本

36:00 - 技术债务（tech debt）的重新理解

36:30 - 技术债务不是完全坏事，就像财务债务一样是杠杆

### 技术债务的战略价值 (37:00 - 39:30)

37:00 - 零债务意味着行动不够快

37:30 - 可以战略性地承担技术债务以更快进入市场

38:00 - 需要有计划：既要支付利息（维持运行），也要偿还本金（长期修复）

38:45 - 有时应该"咬紧牙关"用正确的方式做

39:15 - 可以就技术债务进行明智的对话和选择

### 生产力感觉的陷阱 (39:30 - 45:00)

39:30 - 重要观点：生产力的感觉是实际生产力的糟糕代理指标

40:00 - 编码时的生产力感觉关注的是输出，而非结果

40:45 - 在心流状态中感觉不到代码是否真正有用

41:30 - 要小心假设"感觉低效"就是真的低效

42:00 - 案例研究：2017年的Step Functions

42:30 - 当时Step Functions很难使用（JSON、JSON Path、文档不足）

43:00 - 使用它感觉不高效、不有趣、没有心流状态

43:45 - 但一旦完成，系统就能自动运行，几乎不需要维护

44:15 - 相比之下，其他实现方式需要持续维护和更新

44:45 - 虽然不有趣，但TCO更低，能以更少的输出获得想要的结果

### 技术转型：抛弃代码 (45:00 - 49:00)

45:00 - 从技术选择（快照）到技术转型（应对变化）

45:30 - 当需求变化时，可能需要抛弃已有的东西

46:00 - 困难之处：你投入了大量时间、心血和汗水

46:30 - 抛弃它感觉像抛弃自己的一部分

47:00 - 关键观点：你不是你的输出，那个东西不是你

47:30 - 如果它不再有用,你可以去做其他有用的事情

48:00 - "抛弃代码应该是一件快乐的事"

48:30 - 当功能不再需要，或有第三方商品化解决方案，或AWS推出新服务时

49:00 - 可以将负债从资产负债表上移除，去创造新东西

### 组织转型 (49:00 - 51:30)

49:00 - 组织转型包括重组、做不同的事、用不同的方式做事

49:45 - 理解自己在价值链中的位置变得有用

50:15 - 当前提供的结果 vs 有能力提供的结果

50:45 - 如果不是你的工具，改变工具就不会威胁到你

51:15 - 可能在价值链中上移或进入不同的价值链

51:30 - 需要理解为组织服务的新方式

### AI的影响 (51:30 - 58:00)

51:30 - 必须讨论AI的影响

52:00 - 关键原则：将责任转移给AI不等于转移问责制

52:30 - 如果AI出错，问责制仍然在你身上

53:00 - AI是一个工具，需要对结果负责

53:30 - 对AI生成的代码应用相同的思维：为什么需要样板代码？

54:00 - 如果AI生成样板代码，它仍可能出错

54:30 - 减少样板代码意味着减少出错的机会

55:00 - 让AI做更少的工作，创建更小的实现

55:30 - 让AI专注于结果而非输出

56:00 - 人类也会犯错，创造东西本身就会出错

56:30 - 未来可能问责制会转移到AI，但现在还很远

57:00 - AI对初级和高级工程师的不同影响

57:30 - 历史上新技术让初级工程师能与高级工程师竞争

### AI时代的技能发展挑战 (58:00 - 62:00)

58:00 - AI目前不是这样工作的

58:30 - 需要知道"好"是什么样子才能发现问题并修复

59:00 - 初级工程师还没有这种能力

59:30 - AI为他们做事，阻碍了学习过程

60:00 - AI在高级工程师手中更有效

60:30 - 最终高级工程师会退休，需要培养新人

61:00 - 初级工程师需要找到方法学习必要的知识

61:30 - 同时平衡"AI在做，也许我不需要理解"的诱惑

### 软件质量与AI (62:00 - 66:00)

62:00 - 重要观点：高质量软件本身不是目的

62:30 - 软件质量水平是业务需求的一部分

63:00 - 有些地方需要极高质量，有些地方可以很多bug

63:30 - AI改变了成本-质量曲线

64:00 - Vibe coding（氛围编码）可以以极低成本获得能工作的东西

64:30 - AI降低高质量软件的成本，但降幅远小于低质量软件

65:00 - 这改变了我们对创造什么的思考

65:30 - "以十分之一的成本创造一半好的东西"可能是正确选择

66:00 - 回到"我们不是工匠"的观点

### 质量与市场策略 (66:00 - 68:30)

66:00 - 软件不是必须高质量的工艺品

66:30 - 业务可能要求更快进入市场，即使质量较低

67:00 - 可以随时间改进

67:30 - 这个选项应该在桌面上

68:00 - 有些业务永远不接受低质量

68:30 - 在竞争环境中会如何演变尚不清楚

### 总结 (68:30 - 70:00)

68:30 - 承认演讲节奏有点快

69:00 - 核心总结：结果优于产出的思维模式使我们能够服务于被雇佣的目的

69:30 - 这与周末的爱好不同，爱好中我们完全控制做什么

70:00 - 在工作中，我们需要满足业务要求，而不仅仅是追求个人兴趣

会议核心要点总结：
- 构建者 = 问题解决者，构建只是工具
- 专注结果（outcomes）而非产出（outputs）
- 功能是资产，实现是负债
- 不要与工具或输出物认同
- 抛弃代码应该是快乐的事
- 总拥有成本（TCO）是关键指标
- 技术债务可以是战略杠杆
- 生产力感觉≠实际生产力
- AI时代责任仍在人类
- 软件质量是业务需求，不是目的本身
