# AWS re:Invent 2025 会议总结：使用 AI 代理加速传统应用现代化

## 会议概述

本次会议重点探讨了如何利用生成式 AI 和 AI 代理技术来加速传统应用的现代化进程。演讲者 Phil Devel(AWS 首席产品经理)和 Matt Lewis(IBM Consulting 的 AWS 首席架构师)共同展示了 AI 驱动的应用现代化方法。

会议指出,目前约 65% 的企业应用被视为传统应用,这些应用依赖过时技术,难以维护和扩展,且占用了大部分 IT 预算。传统的现代化方法通常需要漫长的手动项目,但随着生成式 AI 的出现,AI 代理能够进行推理、使用工具并执行多步骤工作流,从而显著提高现代化任务的速度和质量。

演讲者通过七个实际演示展示了 AWS Transform、Amazon Q Developer 和 Amazon Q IDE 等工具如何帮助企业实现从逆向工程到正向工程的全流程现代化,包括代码分析、业务逻辑提取、技术文档生成、运行时升级、跨平台移植等关键环节。这些 AI 代理能够自动化处理大量重复性任务,同时保持人工审核环节,确保现代化过程的质量和可控性。

## 详细时间线

### 开场与背景介绍
[00:00 - 02:30] 会议开场,介绍传统应用现状
- 分析研究显示 65% 的企业应用属于传统应用
- 传统应用依赖过时技术,难以维护和扩展,消耗大部分 IT 预算
- CIO 需要现代化这些应用以提高敏捷性和效率

[02:30 - 03:45] 介绍演讲者和会议议程
- Phil Devel:AWS 首席产品经理,专注于使用 AWS Transform 构建 AI 代理
- Matt Lewis:IBM Consulting 的 AWS 首席架构师、AWS Hero 和大使
- 议程包括:现代化方法、AI 代理模式、七个实际演示

### 应用现代化方法论
[03:45 - 06:30] Matt 介绍现代化方法的高层视角
- 企业 IT 环境通常由多样化的系统、技术、语言和框架组成
- 现代化的四大驱动因素:成本节约(从 CapEx 转向 OpEx)、敏捷性(采用 DevOps)、创新(利用新功能)、弹性(利用云特性如自动扩展、多可用区)

[06:30 - 08:15] 两种互补的现代化方法
- **逆向工程**:分解现有系统以理解其工作原理,包括分析系统、更新技术文档、提取业务规则和逻辑、分解为模块化组件、规划迁移
- **正向工程**:从需求出发构建新系统,包括应用转换、构建部署管道、创建 API、测试阶段、规划切换

### 现代化模式详解
[08:15 - 11:30] Phil 介绍应用栈现代化模式
- **Rehost(重新托管)**:将应用从操作系统层面进行提升和迁移到现代化计算平台
- **Replatform(重新平台化)**:更改运行时和数据库类型,利用托管服务提高运营效率
- **Refactor(重构)**:更改完整的应用栈包括语言和框架,消除过时技术
- **进一步重构**:改变业务功能以适应不断演变的业务需求

[11:30 - 14:00] 应用分解模式
- **技术分解**:从大型主机应用到分布式单体应用,再到模块化单体和微服务
- 将应用分层为表示层、业务逻辑层和数据层
- **业务分解**:使应用更好地与业务需求对齐,实现更细粒度的团队协作
- 微服务的优势:技术选择灵活、更好的可扩展性、更快的上市时间
- 微服务的挑战:增加复杂性、延迟、分布式事务、最终一致性

[14:00 - 15:30] Strangler Fig 模式
- 通过引入网关和集成层,逐步从传统单体应用中提取功能到现代化环境
- 随时间推移将查询重新路由到新环境,实现平滑过渡
- 如果现代化不合适,可以回退到旧应用,降低风险

### AI 代理在现代化中的应用
[15:30 - 18:00] AI 代理的关键特性
- 具有用户界面理解目标
- 能够推理和分解目标为可执行任务
- 可以调用其他代理
- 能够观察资产(如识别应用语言和框架)
- 访问工具执行特定操作(如转换)
- 访问知识库获取最佳实践
- 支持反馈循环进行迭代改进

[18:00 - 20:00] AI 代理工作流程
- 用户与编排层(编排代理)交互
- 代理理解目标并分解为可执行任务
- 调用专门的代理(分析代理、转换代理、测试代理)
- 每个代理观察输入、执行现代化活动、提供输出
- 大规模处理:将应用工件放入存储库,代理访问并逐步现代化

[20:00 - 22:00] AWS 现代化工具介绍
- **AWS Transform**:用于大规模现代化的 AI 代理服务,支持主机现代化、.NET 现代化、VMware 现代化,新增自定义转换功能
- **Amazon Q Developer**:IDE 内的编码助手,支持软件开发生命周期的各种任务
- **Amazon Q IDE**:最新的代理式 IDE,优化了代理协作,专注于规范驱动开发

### 演示 1:代码分析与规范生成
[22:00 - 27:30] Matt 演示从代码生成设计规范
- 问题:直接从一种编程语言翻译到另一种会导致语义漂移
- 传统应用包含大量针对平台限制的代码,迁移时不应带入技术债务
- 最佳实践:基于设计规范进行现代化

[27:30 - 32:00] 使用 Step Functions 的自动化流程
- 架构:S3 存储桶 → EventBridge → Step Functions 状态机
- 四个任务:读取文件内容 → 调用 Bedrock LLM 生成规范 → 写入 Markdown 文件到 S3 → 发送 SNS 通知
- 使用预签名 URL 让业务分析师无需 AWS 凭证即可下载规范
- 人工参与循环进行审核

[32:00 - 35:00] Prompt 模板和 Bedrock Converse API
- 使用 Prompt 模板定义指令和指导
- 使用 Bedrock Converse API 统一调用不同模型
- 可以调整推理参数(如 temperature 控制创造性,最大输出令牌数)
- 演示:将 REXX(主机脚本语言)代码转换为规范文档

### 演示 2:业务逻辑提取与技术文档生成
[35:00 - 38:00] Phil 介绍代码理解和业务逻辑提取
- 传统应用缺乏文档,难以找到了解应用的人员
- 使用 AWS Transform 理解代码并提取业务逻辑

[38:00 - 42:00] AWS Transform 的代理工作流
- 用户提供目标,Transform 识别所需代理
- 分析代理:提取原始分析数据
- 业务逻辑提取代理:提取业务规则和逻辑
- 技术文档生成代理:生成技术文档
- 聊天代理:通过自然语言查询访问信息

[42:00 - 48:00] 演示 AWS Transform 界面
- 创建主机作业,指定 S3 中的应用位置
- 分析完成后显示应用分解信息
- 业务逻辑提取:应用级、功能组级、组件级、程序级的多层次信息
- 显示业务摘要、工作流程、决策点、分支、输入输出数据源
- 完整的业务规则列表确保现代化的全面性

[48:00 - 52:00] 技术文档生成和聊天功能
- 选择要文档化的应用部分
- 生成 PDF 包含技术信息:集成、输入输出、错误处理
- 聊天界面:通过自然语言查询快速找到特定信息(如"哪个数据集被使用")
- 提供 JSON 格式输出,便于在其他工作流中重用(创建规范、测试用例等)

### 演示 3:运行时和依赖项升级
[52:00 - 55:00] Matt 介绍自行车许可证应用案例
- 五年前为 AWS 网络研讨会编写的应用,在 GitHub 上逐渐过时
- 使用 Java 11、Spring Boot、AWS SDK v1,均已严重过时
- 目标:升级到支持的版本

[55:00 - 58:00] 使用 Q Developer 和 AWS Transform Custom
- Q Developer 聊天:识别过时的库和框架(Java、Spring Boot 为关键,AWS SDK 为高优先级)
- AWS Transform Custom:新发布的功能,支持自定义转换

[58:00 - 63:00] Transform CLI 演示
- 克隆存储库到本地,安装 Transform CLI
- 运行 atx 命令,输入自然语言提示:"升级我的 Java 应用"
- CLI 查找预构建的转换定义(8 个预构建转换)
- 识别 Java 版本升级转换
- 用户补充:同时升级 AWS SDK v1 到 v2 和 Spring Boot

[63:00 - 67:00] 规划和执行代理
- 规划代理:创建 plan.json 文件,包含步骤、验证命令
- 用户审核并批准计划
- 执行代理:逐步执行,检测编译错误并自动修复
- 示例:检测到 Spring Boot 新接口方法的破坏性更改,自动添加新方法

[67:00 - 70:00] 验证和 Git 集成
- 转换成功完成,应用运行在长期支持版本上
- 多代理系统:每个步骤由专门的子代理执行
- Git 集成:在初始化的 Git 存储库中工作,每步创建新提交
- 推送到 GitHub,创建新分支,可以查看所有提交并创建 Pull Request

[70:00 - 72:00] 集成测试
- 启动服务器,验证运行在最新版本的 Spring Boot 和 Java
- 创建虚拟自行车许可证,添加积分
- 在 DynamoDB 中查看事件历史,确认功能正常

### 演示 4:.NET 跨平台移植

[78:00 - 82:00] 构建验证
- 转换过程中执行构建验证更改的功能性
- 如果构建失败,尝试不同机制或提供构建错误输出

[82:00 - 85:00] Visual Studio 演示

- 提交作业:将项目从 .NET 3 移植到 .NET 8
- 分析所有包,识别需要完成的工作
- 提供详细计划供审核
