# AWS re:Invent 2025 会议总结：Jefferies 基于 Amazon Bedrock 的 AI 交易助手

## 会议概览

本次会议由 AWS 资本市场首席行业专家 Alex Mari 和 Jefferies 电子交易高级副总裁 Sanjay 共同主讲，重点介绍了 Jefferies 如何利用 AWS Amazon Bedrock 构建 AI 交易助手，以解决前台交易员面临的数据访问和分析难题。

Jefferies 是一家拥有 60 年历史的全方位投资银行，秉承"客户至上"的理念。该公司与 AWS 的合作始于 2022 年，目前 AWS 已深度融入 Jefferies 的各项创新业务中，包括固定收益算法交易平台、股票期权定价计算以及企业级 AI 平台 Gen Jeff AI。交易助手项目是双方合作的重要成果之一，通过生成式 AI 技术为交易员提供实时洞察，显著减少了分析师和开发人员的工作负担。

该解决方案采用对话式分析界面，允许交易员使用自然语言查询海量交易数据。系统利用 Amazon Bedrock 的 Titan 嵌入模型生成 SQL 查询，并通过可视化图表呈现结果。在 Beta 测试阶段，该系统已为约 50 名用户节省了 80% 的常规分析任务时间，并计划向全球交易团队推广。

00:05:30 - 交易助手项目的起源
- 解决股票交易员面临的核心挑战：难以实时获取客户行为、交易模式和市场趋势的洞察
- 数据量巨大且分散：每天数百万笔交易存储在多个数据存储和可视化工具中
- 交易员缺乏时间和编码能力来生成和维护洞察系统
- 目标：降低获取洞察的障碍，赋能交易员通过生成式 AI 进行实时对话式分析

00:07:15 - 解决方案核心能力（Alex 讲解）
- 易用性和加速洞察时间是设计关键
- 使用 Titan 嵌入模型作为底层 LLM，生成 SQL 查询
- 支持多种输出形式：文本、表格、图表
- 对话式分析界面，允许交易员深入探索数据
- 系统维护对话上下文，提供相关洞察和建议
- 使用 Amazon Bedrock Agents（Strands agents），最小化代码量
- 灵活选择不同 LLM 模型
- 高级安全防护：行级数据权限控制、防止意外访问敏感数据、完整审计日志以满足合规要求

00:09:00 - 演示环节（Sanjay）
- 用户界面集成在现有 BI 平台 Global Flow Monitor (GFM) 中
- GFM 包含实时交易数据，交易员每天依赖该平台
- 登录时验证凭证，确保用户仅看到授权数据
- 演示查询："给我今天美国交易的行业细分"
- 后台流程：LLM 生成 SQL 查询，在 GridGain 内存数据网格上执行，瞬间检索数据
- Python 库将原始数据转换为可视化图表和表格
- 对话式分析与丰富可视化的结合受到交易员社区好评

00:11:00 - 架构详解（Alex）
- 步骤 1：通过 AWS Direct Connect 连接交易助手与 Jefferies 本地 BI 平台 GFM
- 步骤 2-4：使用 AWS EKS 构建服务，进行用户认证、创建会话和查询 LLM agents
- 步骤 3：Bot 服务建立用户会话并调用查询 agent
- 步骤 4：查询 agent 是 Strands agent，与多个 MCP 工具交互，决定使用哪个数据源
- 从 LangChain 迁移到 Strands agents，性能显著提升
- Strands 使用 Titan 嵌入模型的高级推理能力规划和执行 agent 步骤
- 步骤 6：使用 Amazon Bedrock Knowledge Base 作为向量存储
- 查询执行器服务查询底层数据，LLM 选择可视化方式，使用 Markdown UI 库渲染

00:14:30 - 项目影响与成果（Sanjay）
- Beta 测试阶段向约 50 名销售和交易运营用户推出
- 常规分析任务时间减少 80%，直接转化为收入生成能力提升
- 采用率高，用户满意度高
- 计划向全球交易用户群推广
- 减少了为多个原型和交易台定制仪表板的技术负担
- 自助服务能力降低了对技术资源的依赖
- 数据访问民主化：业务用户可用自然语言查询数百万条股票交易记录
- 架构具有未来适应性：自学习、持续改进、易于集成、可扩展

00:16:45 - 未来规划
- 全球推广战略的三大支柱：
  1. 多产品扩展：从股票扩展到其他产品类型和交易台
  2. 全球部署：将效率提升带到国际交易运营
  3. 增强治理：加强可观测性和审计能力以满足监管合规要求
- 探索高级代码生成：从基于 UI 的 Java 工具过渡到 NLP 驱动的代码生成
- 将解决方案转化为通用 AI API，供全公司使用

00:18:00 - 关键经验教训
- 不要依赖 LLM 生成可视化，以避免幻觉风险（使用 LLM 选择输出类型，但由 Python Markdown 库生成实际可视化）
- 使用更快的数据存储（如内存数据库）以最大化结果输出速度
- 使用 Python 构建 LLM 交互以获得灵活性，其他部分使用 Java 以便移植现有代码
