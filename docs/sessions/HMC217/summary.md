# AWS re:Invent 2025 会议总结：Rivian 使用 AWS Outposts 的制造业实践

## 会议概述

本次 AWS re:Invent 2025 分会场由 AWS 高级解决方案架构师 Matt Reed 和 Rivian 高级 AWS 解决方案架构师 Vinnie Makachek 共同主讲，重点介绍了 AWS Outposts 在制造业环境中的应用。会议面向拥有制造设施、管理 IT/OT 基础设施或希望通过技术提升流程效率和降低成本的企业。

Matt Reed 首先介绍了制造业面临的挑战，包括低延迟需求、边缘计算、实时分析和 AI/ML 推理等。他详细讲解了 AWS Outposts 作为全托管服务的优势，特别是第二代 Outposts 的新特性，如解耦网络架构、增强的网络弹性和更灵活的连接选项。随后，Vinnie Makachek 分享了 Rivian 在实施 Outposts 过程中的真实经历，包括初期失败的教训、自建平台的挑战，以及最终成功采用 Outposts Gen 2 的完整历程。

Rivian 的案例展示了从传统虚拟机架构迁移到现代容器化平台的复杂性，强调了网络弹性架构设计、应用恢复策略和标准化的重要性。通过三年多的探索和迭代，Rivian 最终找到了适合其制造业务的混合云解决方案，实现了平台整合、工作负载可移植性和认知简化。

## 详细时间线

### 开场与背景介绍 (00:00 - 05:30)

00:00 - 01:30 - Matt Reed 开场，介绍会议主题和目标受众
- 面向制造设施所有者、IT/OT 基础设施管理者
- 关注效率提升和成本节约

01:30 - 03:45 - 行业趋势和挑战
- Research and Markets 研究显示未来 5 年相关领域支出将增长 4 倍
- 制造业三大趋势：低延迟需求、边缘计算、弹性改进
- 主要挑战：单位毫秒级延迟要求、连接可靠性、传统基础设施协调复杂

03:45 - 05:30 - AWS Outposts 产品介绍
- 全托管服务，将 AWS 基础设施和服务带到本地边缘位置
- 提供一致的混合云体验
- 形态因子：42U 机架（数据中心）和 1U/2U 服务器（小型设施）

### AWS Outposts 核心功能 (05:30 - 09:15)

05:30 - 07:00 - Outposts 主要优势
- 使用相同的 CLI、SDK 构建和部署云原生应用
- 自定义管理平面、治理策略和安全控制
- AWS 提供全托管服务，包括补丁和更新

07:00 - 09:15 - 第二代 Outposts 新特性
- 网络架构完全重新设计，计算与网络解耦
- 不再需要一对一配置，单一网络栈处理所有计算
- 提高成本效率和部署灵活性
- 支持按需扩容（infill requests）
- 增强网络弹性，支持设备故障转移
- 新增域路由组（domain routing groups）和 BYOIP 支持

09:15 - 09:45 - 全球可用性和行业领先地位
- 多个地区提供第一代和第二代 Outposts
- Gartner 2024 魔力象限：AWS 在混合基础设施管理、边缘计算、保障工作负载和 AI/ML 四个领域排名第一

09:45 - 10:30 - Vinnie Makachek 自我介绍和 Rivian 概况
- Vinnie 担任 Rivian 高级 AWS 解决方案架构师

### Rivian 面临的挑战 (12:00 - 15:30)

12:00 - 13:30 - 传统制造业 IT 挑战
- 大量传统虚拟机托管栈
- 无法利用内部 Kubernetes 平台团队的专业知识
- 技能孤岛：不同团队需要学习不同技能
- 无法互相支持轮值或故障处理

13:30 - 15:30 - 运维挑战和工作负载类型
- 传统虚拟机环境维护可用性困难
- 补丁管理需要停机
- 工作负载包括：制造执行系统（MES）、工业自动化、AI/ML、计算机视觉

### 第一次尝试：Outposts Gen 1 (15:30 - 18:45)

15:30 - 16:30 - 初步实施方案
- 将虚拟机快照迁移到 Outposts
- 部署少量 RDS 实例
- 最初认为项目成功

16:30 - 18:45 - 遇到的问题
- 网络弹性不足
- "Outpost"名称具有误导性，让人以为可以完全独立运行
- 实际上需要与区域保持网络连接才能提供服务和编排
- 缺乏应用恢复策略
- 未能充分了解新平台的故障模式
- 缺乏标准化，仍在使用虚拟机而非容器
- 最终导致内部信任丧失

### 第二次尝试：自建平台 MCP (18:45 - 23:00)

18:45 - 20:00 - 制造容器平台（MCP）构建
- 团队决定自行构建解决方案
- 组合多个供应商工具
- 分别采购存储、网络设备、虚拟化/Kubernetes 平台
- 各系统需要集成但并非原生设计为统一整体

20:00 - 23:00 - 自建平台的挑战
- "工单地狱"：每个组件由不同供应商和团队负责
- 故障排查困难，难以确定根本原因
- 容量规划复杂，仍需采购和部署硬件
- 从需求到交付需要 6 个月到 1 年时间
- 缺乏标准化：与 EKS 存在差异
- 缺少 IAM roles for service accounts 等功能
- 无法直接将云工作负载迁移到本地
- 需要在本地进行开发，消耗有限的空间、电力和冷却资源

### 第三次尝试：Outposts Gen 2 成功实施 (23:00 - 28:30)

23:00 - 24:30 - 重新评估和决策
- 团队重新审视前两次失败的原因
- 意识到未能为弹性进行架构设计
- 决定全面采用 Outposts Gen 2

24:30 - 26:30 - 新架构设计
- 使用 Direct Connect 替代公网连接
- 建立 Rivian 骨干网连接到主要数据中心提供商
- 实现冗余、可靠和安全的区域连接
- 采用 EKS 实现功能对等
- 利用 Kubernetes 平台团队的深厚专业知识

26:30 - 28:30 - 实施时间线回顾
- 2020 年中：启动 Outpost Gen 1 概念验证
- 2021 年末：三个月内发生三次网络故障（包括圣诞树安装导致光纤损坏）
- 2022-2024：尝试自建平台
- 2024 年末：进行平台对比测试，达成共识采用 Outposts

### 成果与经验教训 (28:30 - 31:00)

28:30 - 29:30 - 实施成果
- 平台整合
- 工作负载可移植性
- 低延迟
- 认知简化：全公司使用统一平台（AWS）

29:30 - 31:00 - 关键经验教训
1. 为弹性而架构：必须从一开始就考虑弹性设计
2. 容量管理仍然重要：需要提前规划，建立报告仪表板监控容量消耗
3. 使用 RAM 共享 VPC：避免创建大量 VPC，在单一数据中心使用共享 VPC 可大幅降低运维开销
4. 信任难得易失：需要赢得内部客户的信任，这是最重要的

31:00 - 会议结束，开放问答
