# AWS re:Invent 2025 会议总结：数字资产做市商的GenAI新闻分析平台

## 会议概览

本次技术分享由AWS解决方案架构师David Adab和数字资产市场微观结构独立研究员West共同呈现，重点介绍了数字资产做市商如何利用AWS Bedrock服务构建GenAI驱动的新闻分析平台。会议深入探讨了做市商在加密货币市场中面临的独特挑战，特别是如何在极短时间窗口内（通常30分钟内）对市场动态新闻做出反应。

演讲者详细阐述了从传统字典匹配方法到统计机器学习，再到大语言模型的技术演进历程。他们展示了如何通过推理优化将输出token速度从80提升至180 tokens/秒，实现端到端延迟控制在10秒以内。该解决方案采用代理架构（Agentic Architecture），结合精细调优的嵌入模型和通用推理模型，成功处理加密货币市场中的高频新闻流，帮助交易员在Trump推文、Elon Musk社交媒体动态等非结构化信息冲击下快速调整交易策略，有效管理做市风险。

## 详细时间线与关键要点

00:00 - 开场介绍
- David Adab介绍自己是专注于金融服务和数字资产交易系统的解决方案架构师
- 会议主题：使用AWS Bedrock服务构建GenAI新闻分析平台

00:30 - 做市商基础概念（West讲解）
- 免责声明：观点仅代表个人，不代表任何组织立场
- 交易生态系统介绍：买家、卖家、交易所和做市商的角色
- 做市商职责：持续报价买卖订单，确保市场流动性

02:00 - 订单簿机制说明
- 订单簿结构：左侧为买单（bid），右侧为卖单（ask）
- Y轴表示该价格水平的累积规模
- 做市商在两侧同时挂多个买卖订单

03:00 - 做市商面临的风险
- 突发新闻可能导致市场单向波动
- 示例：所有卖单被执行但买单未成交，做市商需要以更高价格回购库存
- 这会导致"低卖高买"的亏损情况

04:00 - 理想做市场景
- 市场在可控波动范围内小幅波动
- 买卖订单在短时间内都能成交
- 虽然单笔利润小，但换手率高可累积收益

05:00 - 价差与成交频率的权衡
- 不能无限扩大价差来提高单笔利润
- 价差过大可能导致一天内无法成交
- 做市商倾向于高频小利润模式

06:00 - 波动性的重要性
- 波动性是定价公式的关键组成部分
- 如果预知市场将高度波动，可提前扩大价差
- 影响因素示例：FOMC利率决议

07:00 - 新闻分析的时效性要求
- 需要在几秒钟内完成所有分析
- 必须快速判断新闻对市场的影响方向
- 立即在市场中采取行动

08:00 - Trump推文的市场影响
- 引用IG Group的Chris：Trump推文的即时影响通常持续约30分钟
- Trump关税推文和Elon Musk加密货币推文的案例
- 示例：Elon Musk发推"我喜欢某种狗"，狗狗币价格立即上涨

09:30 - 传统新闻vs社交媒体新闻
- FOMC会议结果：可预测时间、数值化、易于比较
- 社交媒体推文：不可预见、非数值化、需要智能解读
- 人工判断耗时过长，算法交易环境不可接受

10:30 - LLM的必要性
- 需要LLM判断新闻是否会影响市场
- 需要判断影响是正面还是负面
- 过渡到David讲解技术方案

11:00 - 历史技术方法回顾（David讲解）
- 第一代：字典匹配方法，使用看跌/看涨/崩盘等情感词典
- 局限性：词汇量有限，无法处理上下文
- 示例：大规模空头清算事件"会被误判为负面，实际应为正面

12:00 - 第二代：统计机器学习
- 使用标注数据的监督学习
- 模型：朴素贝叶斯、FinBERT（金融语料库训练）
- 优势：更好的泛化能力，理解新闻上下文
- 劣势：需要大量标注数据，成本高、上市时间长

13:00 - 第三代：大语言模型
- 基于Transformer的多模态推理
- 使用Claude、DeepSeek等模型
- 优势：上下文感知，最少或无需微调
- 可以推理特定领域事件（如协议漏洞的跨链影响）

14:00 - 推理性能优化的重要性
- 动态市场中的新闻分析需要极短时间窗口
- 必须进行推理优化

14:30 - 推理优化演进历程
- 2025年2月：SageMaker JumpStart + P5N实例 → 80 tokens/秒
- 2025年4月：切换到vLLM，启用深度多token预测、混合精度、线性注意力、分布式专家并行 → 140 tokens/秒
- 2025年8月：切换到SGLang，使用推测解码 → 180 tokens/秒

16:00 - 性能提升的实际意义
- 每分钟数千个事件，每毫秒延迟都会累积
- 从80到180 tokens/秒意味着在相同时间窗口内处理能力翻倍
- 实现端到端延迟低于10秒，避免不良选择

17:00 - Day 1架构介绍
- 底层：新闻流API持续摄取新闻到S3存储桶
- EventBridge触发Lambda函数
- Lambda基于资产、紧急程度和情感进行分类
- 调用DeepSeek模型标注元数据
- 存储到Aurora PostgreSQL和OpenSearch

18:30 - QCI终端集成
- 交易员和业务分析师可与新闻源交互
- 可以询问LLM最新的Trump公告或过去10小时Elon Musk的X动态

19:30 - 加密货币市场的隐藏挑战
- 同一新闻被报道50次
- 示例：以太坊升级延迟出现在Twitter、Reddit、Discord、Telegram等多个平台
- 如果每个重复都通过昂贵的LLM处理，会浪费资金并增加延迟

20:30 - 去重管道设计
- 步骤1：Lambda调用嵌入模型BGE-M3计算嵌入
- 步骤2：相似度检查，>0.75视为重复，存入OpenSearch重复集合，停止处理
- 步骤3：<0.5视为可能唯一，与历史新闻比对
- 步骤4：真正唯一的新闻进行近实时预测分析

22:00 - 分析输出
- 价差扩大建议
- 资产影响评估
- 价格波动概率
- 生成预测报告发送到交易台Slack频道
- 由人工决定是否采取行动

23:00 - 嵌入模型微调的必要性
- 通用嵌入模型对加密货币特定重复检测效果不佳
- 展示散点图：开箱即用的BGE-M3在数千个查询-文档对上的表现
- 绿点：重复新闻（期望相似度>0.75）
- 红点：非重复新闻（期望相似度<0.5）
- 问题：0.5-0.75之间存在大量重叠的"混乱中间地带"

24:30 - 微调后的效果
- 使用数千条标注的加密货币新闻微调BGE-M3
- 绿点和红点明显分离
- 混乱中间地带消失，0.3-0.6之间有清晰分界
- 仅560M参数的小型嵌入模型即可实现

25:30 - 代理架构的优势
- LLM无需微调，仅需提示工程
- 这是代理架构的强大之处

26:00 - 实时演示
- 右侧：交易台接收新闻频道
- 左侧：新闻流实时处理
- 示例1：SEC文件分析，推理模型判断不包含市场动态信息，非重大监管文件，不发送到交易台
- 持续摄取和分析新闻，大部分被分类为常规新闻

27:30 - 重要新闻触发
- Trump发布关于中国变得非常敌对的新闻
- 管道识别为高影响力新闻
- 推送到Telegram供交易员决定是否扩大价差保护投资组合

28:30 - 关键要点总结
1. 使用代理架构而非微调：使用通用推理模型处理分层任务决策
2. 专业化嵌入层：更具成本效益，更快上市时间
3. 通过架构消除偏差：教系统如何推理新事件
4. 人在回路中不可妥协：利用交易员反馈循环持续改进系统
5. 结果：7×24实时覆盖，配合人工监督
