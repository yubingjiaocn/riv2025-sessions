# AWS re:Invent 2025 SEC 222 会议总结

## 会议概述

本次会议主题为"从代码到云端：使用 AWS 构建应用安全(AppSec)程序"(SEC 222)，由 AWS 高级安全工程师 Daniel Begimer 和首席安全工程师 Patrick Gaul 主讲。这是一场 200 级别的技术分享会，时长 60 分钟，于周四下午 4 点举行。

两位讲师分享了在 AWS 构建和运行应用安全程序的实践经验和框架。Daniel 在 AWS 工作超过 5 年，拥有 13 年安全领域经验，涵盖事件响应、应用安全和 AI 安全。Patrick 在 AWS 工作约 3.5 年，此前曾在初创公司担任安全副总裁，并有丰富的应用安全工程团队管理经验。

会议核心内容围绕应用安全的四个阶段展开：规划与准备、执行和规模化。讲师们强调了"左移"(Shift Left)安全理念，即在软件开发生命周期(SDLC)的早期阶段集成安全实践，以及分布式安全责任的重要性。会议提供了实用的工具、框架和 AWS 服务演示，包括威胁建模、Amazon CodeWhisperer(Kira)、Amazon Inspector 等，帮助参会者在各自组织中实施或改进应用安全程序。

## 详细时间线与关键要点

### 开场介绍 (0:00 - 3:30)
- **0:00** - 会议开始，Daniel Begimer 自我介绍，AWS 高级安全工程师，5 年以上 AWS 工作经验，13 年安全领域经验
- **1:15** - Patrick Gaul 介绍，AWS 首席安全工程师，3.5 年 AWS 经验，曾任初创公司安全副总裁
- **2:00** - 会议定位：200 级别分享会，60 分钟深入探讨应用安全程序构建

### 应用安全定义与原则 (3:30 - 12:00)
- **3:30** - 会议议程概览：应用安全概述、原则、程序路线图(四个阶段)
- **4:45** - 应用安全定义：在软件开发生命周期所有阶段评估软件安全属性的人员、流程和技术集合
- **5:30** - AWS 应用安全的两个核心维度：分布式责任(每个人都对安全负责)和左移(在 SDLC 早期集成安全实践)
- **6:45** - 威胁建模作为核心机制的介绍
- **7:15** - 应用安全的重要性：更快交付功能、降低成本、减少风险
- **8:30** - AWS 应用安全的五大核心原则：
  1. 设定清晰期望(定义风险容忍度和安全标准)
  2. 强大的培训机制(赋能开发者)
  3. 自动化(一致应用安全要求)
  4. 指标驱动(持续测量和改进)
  5. 组织层面支持(从高层到个人贡献者)
- **10:30** - Patrick 分享个人经验教训：选择工具时必须考虑开发者是否真正会使用
- **11:15** - "简单按钮"理念：在开发者所在的地方满足他们，而不是增加额外流程

### 应用安全程序路线图 (12:00 - 18:00)
- **12:00** - 行为经济学洞察：人类天生倾向于不采取行动，因此必须让安全变得简单
- **13:00** - 四阶段实用路线图介绍：规划、准备、执行、规模化
- **13:45** - 规划阶段：利益相关者分析、识别风险、建立目标和指标
- **14:30** - 准备阶段：代码扫描、沟通期望、培训开发者进行威胁建模
- **15:30** - 执行阶段：威胁建模成为标准流程、在 IDE 中发现和修复问题、在 CI/CD 管道中部署安全测试
- **16:45** - 规模化阶段：识别系统性问题、构建安全设计模式、提供"黄金路径"、收集反馈

### 规划阶段详解 (18:00 - 22:30)
- **18:00** - 识别和接触利益相关者的重要性
- **18:45** - 关键建议：获得领导层支持、寻找对安全友好的组织快速取得成功
- **19:30** - 重新定义思维方式：在开发者所在的地方满足他们
- **20:15** - 理解高层风险：了解业务背景、运营环境、优先级
- **21:00** - 像攻击者一样思考，识别潜在攻击途径
- **21:45** - 制定项目章程、获得利益相关者认同、分布式责任

### 准备阶段与沟通策略 (22:30 - 28:00)
- **22:30** - 准备阶段目标：获得可见性、部署测试、识别系统性问题
- **23:30** - EAST 模型介绍(沟通框架)：
  - E (Easy) - 保持简单
  - A (Attractive) - 使其有吸引力、游戏化
  - S (Social) - 社会化、利用社会证明
  - T (Timely) - 及时性、识别并消除行动障碍
- **24:45** - 沟通案例对比：传统方式 vs 优化方式
- **25:30** - 传统沟通示例：冗长、要求行动、增加负担
- **26:30** - 优化沟通示例：使用默认启用(需要退出而非加入)、展示 90% 开发者认为简单、利用社会证明

### 执行阶段 - 威胁建模 (28:00 - 34:00)
- **28:00** - Daniel 接手讲解执行阶段
- **28:30** - 威胁建模的重要性：对所有新功能和产品进行威胁建模
- **29:15** - 目标：在 SDLC 早期识别风险，在白板上而非生产环境中进行更改
- **30:00** - 四问题框架介绍：
  1. 你在做什么？(新服务、应用、功能)
  2. 可能出什么问题？(最坏情况、CIA 三要素)
  3. 我们要做什么？(缓解措施、工具、服务)
  4. 我们做得好吗？(审查、迭代改进)
- **32:00** - 威胁建模是迭代过程，需要从其他项目和安全问题中学习
- **32:45** - AWS 的实践：威胁建模是发布新产品/功能/服务的必要环节
- **33:15** - 威胁建模实施步骤：准备图表和文档、召集跨职能团队、记录决策
- **33:45** - 推荐工具：Threat Composer(AWS GitHub 开源工具)

### 执行阶段 - 代码扫描 (34:00 - 45:00)
- **34:00** - 代码扫描和软件组成分析(SCA)的重要性
- **34:30** - Amazon CodeWhisperer (Kira) 介绍："简单按钮"的典范
- **35:15** - Kira 功能：提高生产力、运行代码扫描、执行安全测试、在 IDE 中直接提供安全分析
- **36:00** - 现场演示开始：Fry Factory(法式炸薯条在线商店)示例应用
- **36:45** - 演示 SQL 注入漏洞：通过订单页面获取敏感信息
- **37:30** - 查看源代码：Flask 应用、硬编码凭证、安全漏洞
- **38:15** - Kira Steering Documents 介绍：定义可重用的提示、标准、安全要求和模板
- **39:00** - 展示 Steering Documents 示例：文档标准、威胁建模(STRIDE)、安全标准
- **40:00** - 安全标准文档重点：密钥管理、注入防护
- **40:45** - Kira Agents 介绍：定义具有不同角色的代理
- **41:30** - 威胁建模代理示例：使用 STRIDE 进行威胁建模、摄入 Steering Documents
- **42:15** - 安全扫描代理：集成自动化安全助手(ASH)工具
- **43:00** - 演示 Kira CLI 运行安全扫描：简单提示"运行安全扫描"
- **44:00** - 扫描结果：基于发现、工具输出和安全标准生成上下文化安全报告
- **44:45** - 进一步使用 Kira 生成具体可操作的修复建议

### 执行阶段 - Amazon Inspector (45:00 - 50:00)
- **45:00** - Amazon Inspector 介绍：另一个"简单按钮"示例
- **45:30** - Inspector 功能：漏洞管理工具、扫描容器、计算资源、代码仓库
- **46:00** - 集成优势：与 GitHub/GitLab 集成、集中可见性、无需开发者额外操作
- **46:45** - Inspector 演示：Amazon Inspector 控制台
- **47:15** - 创建扫描配置：几次点击即可集成 GitHub 仓库
- **47:45** - 配置参数：扫描频率(每周)、每次提交时扫描
- **48:15** - 扫描类型选择：基础设施即代码扫描、静态代码分析、软件组成分析
- **48:45** - 仓库过滤选项、按需扫描功能
- **49:30** - 演示按需扫描执行
