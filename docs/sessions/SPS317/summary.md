# AWS re:Invent 2025 - AWS 统一运营(Unified Operations)会议总结

## 会议概述

本次会议由AWS现场支持组织总监Jason Bower主持,联合航空公司(United Airlines)企业架构实践总经理Nenos Gabriel和AWS技术专家共同分享了AWS统一运营(AWS Unified Operations)的创新方法。会议重点探讨了如何通过将AWS专家团队与AI驱动工具相结合,从根本上改变传统的被动式运维支持模式,转向主动式卓越运营。

根据Splunk的研究数据,全球2000强企业每年因运营中断损失超过4000亿美元,相当于平均公司利润的9%。企业平均每年面临29次计划外运营事件,每次事件平均耗时超过5小时,造成1350万美元的损失。AWS统一运营正是为解决这些痛点而设计,承诺在5分钟内响应关键事件,通过详细的运营就绪审查、全天候警报监控和持续改进来预防问题发生。

联合航空作为首批采用者,在其复杂的技术环境中(包括2100个应用程序、4300多个微服务和API、每日50亿次数字渠道访问)成功实施了统一运营。通过12周的试点项目,涵盖预订管理系统、机场运营、机队维护和重量平衡四个关键应用领域,联合航空实现了关键案例响应时间改善超过70%,平均故障恢复时间(MTTR)降低7%,客户净推荐值(NPS)提升5分,并实现了1000多次自动化跨区域故障转移。

## 详细时间线与关键要点

00:00 - 开场与背景介绍
- Jason Bower介绍自己作为AWS现场支持组织总监的背景,拥有15年帮助全球大型企业在AWS上构建、部署和运营工作负载的经验
- 互动环节:询问观众是否曾在半夜因运营中断被叫醒,是否遇到超过2小时才解决的故障,以及是否在事件中难以召集合适的人员

02:30 - 行业挑战数据
- 引用Werner Vogels的名言:"一切都会时刻失败"
- 披露关键统计数据:全球2000强企业每年因中断损失4000亿美元(占平均公司利润的9%)
- 企业平均每年面临29次计划外事件,每次事件耗时超过5小时,成本1350万美元

05:00 - 传统运营流程的问题
- 描绘典型场景:运行跨多个可用区甚至多个区域的关键任务应用程序
- 传统流程的痛点:花费时间将问题路由到正确团队、建立上下文、升级到主题专家,同时客户受影响、收入流失
- 99.9%的可用性仍意味着每月43分钟的停机时间

08:00 - 传统运营流程分析
- 详细说明传统流程步骤:检测问题→启动事件桥接→召集SRE和开发团队→向AWS开设支持案例→AWS询问基本问题→故障排除→最终找到根本原因
- 强调需要在软件开发生命周期的更早阶段(左移)进行运营设计

11:00 - AWS统一运营介绍
- 宣布AWS统一运营:结合AWS专家和AI驱动工具的新方法
- 核心承诺:预防问题、在5分钟内响应关键事件、持续改进运营
- 三大支柱:主动指导(通过运营就绪审查)、快速事件响应(全天候警报)、持续改进(工作负载优化和主动评估)

15:00 - 主动支持方法
- 强调在事件发生前、期间和之后与团队合作
- 执行针对各种故障模式的工作负载审查
- 开发应用程序特定的运行手册
- 进行测试和演练模拟,确保团队知道如何应对

18:00 - 与AI工具集成
- 介绍新的AWS DevOps代理和支持助手
- 利用关于工作负载、警报和运行手册的知识为AI提供更多上下文
- 帮助工程师进行故障排除,加速主题专家解决问题

20:00 - 联合航空案例介绍
- Nenos Gabriel登台,介绍联合航空背景
- 关键数据:即将成为百年企业、每年运送1.75亿乘客、360个目的地、每日5000个航班
- 技术复杂性:2100个应用程序、4300+微服务和API、每日50亿次数字渠道访问
- 异构环境:从大型机到中间件层,再到云端AI创新

25:00 - 联合航空的DTOP框架
- 介绍数字技术运营平台(DTOP)
- 目标:简化和提高效率,在SDLC中左移
- 分析SDLC中的27项能力,识别手动步骤并实现自动化
- 关键能力包括:安全代码开发、代码辅助、防止新漏洞引入生产环境

30:00 - 工程标准化的重要性
- 强调可重复解决方案的重要性
- 数据始终无边界,解决方案可在整个组织中重复使用
- 确保与架构框架保持一致的一致性

33:00 - 统一运营如何帮助联合航空
- 效率驱动因素:客户希望准时到达
- 关键洞察:如果保持准时,客户推荐联合航空的可能性高3倍
- 统一运营在5分钟或更短时间内加入呼叫
- AWS工程师了解应用程序和AWS堆栈,减少故障排除时间,加快解决速度

37:00 - 自动化故障转移成果
- 实现1000多次自动化跨区域故障转移
- 重点从故障排除转向恢复服务,确保客户到达目的地
- MTTR降低7%
- NPS分数提高5分

40:00 - AWS技术专家深入讲解
- 介绍统一运营如何满足联合航空的复杂需求
- 强调这不仅是新的支持层级,而是AWS与客户共同处理关键任务运营的方式

42:00 - 试点项目范围
- 选择四个关键应用领域:预订管理系统、机场运营、机队维护、重量平衡
- 选择标准:不同的架构模式和常用AWS服务
- 预订管理系统:每小时数千笔交易,15分钟停机可能影响数百名乘客

45:00 - 实施时间表
- 第1-2周:奠定基础,集成AWS支持案例系统与联合航空的ServiceNow事件管理系统和监控工具
- 第3-4周:深入四个应用程序的架构,执行架构审查和运营就绪审查,识别改进机会
- 第5-6周:演练阶段,进行真实事件模拟,测试团队协作和沟通
- 第7周起:主动支持开始,持续改进和实施最佳实践

50:00 - 人员整合
- 联合航空方面:运营团队、事件管理人员、应用团队、平台团队、架构团队
- AWS方面:技术客户经理、解决方案架构师(企业支持已有)
- 统一运营新增角色:事件管理工程师(24/7待命,5分钟内响应)和服务领域专家(专注于一个或多个AWS领域)

55:00 - 工具集成的三种方式
- 直接关键事件创建:从ServiceNow直接创建AWS支持案例,无需登录控制台
- 快速警报注入:监控工具警报自动触发AWS支持案例创建,无需人工干预
- AWS事件通知:AWS服务故障通知直接发送到联合航空的监控工具

60:00 - 流程对齐
- 主动方面:服务领域工程师执行架构和运营审查,提供改进建议并协助实施
- 被动方面:事件管理工程师5分钟内响应,携带运行手册、上下文和业务影响信息

63:00 - 具体改进示例
- 机队维护应用:启用ECS容器应用的可用区重新平衡,提高容错能力
- 数据库:启用时间点恢复,防止数据损坏
- 依赖项故障转移:确保核心应用和依赖项同时故障转移,避免双峰行为

66:00 - 试点成果
- 关键案例响应时间:平均参与时间改善超过70%
- 改进了桥接呼叫程序、升级指标和沟通框架
- 为AWS支持代理提供定制建议,包含应用程序上下文和架构学习

70:00 - 未来路线图
- 在未来几个月将统一运营扩展到更多关键应用程序
- 将试点期间的学习成果应用到其他应用程序
- 支持联合航空的左移计划:应用程序在上线前完成运营审查和架构审查
- 建立反馈循环:从事件中发现的改进机会反馈到DTOP框架

73:00 - 总结与建议
- 强调运营转型不仅是更快解决问题,更是预防问题和持续改进
- 推荐参加其他相关会议:AWS支持AI、运营、可观测性、卓越运营和弹性主题
