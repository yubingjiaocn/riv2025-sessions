# AWS re:Invent 2025 会议总结

## 会议概述

本次技术分享由来自Reply系统集成公司的Wrao主讲，重点探讨了企业在采用生成式AI时面临的重要架构设计选择。演讲核心围绕如何在实际业务系统中选择合适的AI工作流集成方案，包括函数调用(Function Calling)、单一自主代理(Single Agent)和多代理系统(Multi-Agent System)三种不同范式。

演讲者通过旅行规划的实际场景贯穿全场，从简单的天气查询到复杂的多领域协作任务，逐步展示了不同架构的适用场景。他强调，选择AI架构不应追求最先进的技术，而应根据任务复杂度、可预测性需求、成本延迟考量、业务逻辑控制和自主性要求等关键因素做出决策。演讲最后提醒开发者始终从简单方案开始，重视可观测性建设，并充分利用AWS提供的Amazon Bedrock、Lambda、Step Functions等端到端工具来构建AI解决方案。

## 详细时间线

00:00 - 开场介绍
- 演讲者Wrao自我介绍，来自Reply系统集成公司，专注于生成式AI和AI领域
- 提出核心议题：企业在采用生成式AI时如何选择正确的集成方法

01:30 - 函数调用(Function Calling)基础
- 介绍最简单的方法：函数调用功能
- 解释现代LLM(如Amazon Bedrock上的模型)如何通过函数调用与外部系统交互
- 强调函数调用提供结构化和确定性输出，替代不可预测的文本生成

03:00 - 函数调用实际应用场景
- 银行助手示例：查询账户余额时触发后端函数而非猜测答案
- 函数调用是概率推理(LLM特性)与确定性软件API之间的桥梁

04:00 - 旅行场景案例：拉斯维加斯行李准备
- 标准LLM给出通用建议列表
- 配备函数调用的模型可调用天气API获取实时预报，提供更准确的个性化建议

05:30 - 函数调用的局限性
- 对已知和简单交互非常高效
- 但在企业级工作流中，涉及相互依赖的函数和动态需求时变得复杂
- 需要在LLM和应用程序之间频繁往返通信

06:30 - ReAct模式与代理(Agent)概念
- 介绍ReAct模式(推理与行动)
- LLM不仅能预测文本或直接调用函数，还能进行推理
- 代理可以动态选择工具、规划多步骤、分析中间结果并相应调整行动

08:00 - 代理架构定义
- 代理是超越函数调用的自主AI流程
- 接收目标、推理解决方案、分解任务为多个步骤
- 自主决定何时调用函数、使用工具或获取外部数据

09:00 - 扩展旅行场景：多方面自主处理
- 代理配备多个工具：天气API、航班搜索、酒店预订、日历集成
- 关键区别：不再手动编排步骤，代理自主推理并决定何时调用每个工具
- 示例：规划拉斯维加斯五日游，代理自动查询天气、航班、酒店并推荐行李物品

11:00 - 多代理系统(Multi-Agent System)介绍
- 某些业务问题过于复杂，需要分解为专业化角色
- 多个代理协作，每个专注于特定领域，拥有专属工具和指令

12:00 - 多代理旅行场景
- 天气代理：专注预报
- 航班代理：搜索航班
- 酒店代理：处理住宿
- 活动代理：建议游玩项目
- 协调代理：在顶层编排所有代理
- 类似人类组织中不同专家通过管理者协作

14:00 - 两种基础多代理架构模式
- **层级模式(Hierarchical/Supervisor)**：中央编排器分解任务并分配给专业工作代理，收集整合结果
- **网络模式(Network/Swarm)**：多个代理作为对等节点在网格中操作，共享黑板，直接通信协调

16:00 - 多代理系统的挑战
- 编排复杂性：谁与谁对话、何时对话、以什么顺序
- 延迟和成本：更多代理意味着更多LLM调用和更长响应时间
- 调试困难：需要详细的可观测性追踪代理决策链
- 安全性：每个代理访问敏感数据，权限必须严格控制

18:00 - 真实故障场景示例
- 两个代理无限递归调用
- 由于提示不一致或工具API未通知更改导致
- 强调使用OpenTelemetry等工具进行可观测性监控的重要性

19:30 - 架构选择的关键驱动因素

20:00 - 因素1：任务复杂度
- 简单问答流程：函数调用足够
- 多步骤任务：考虑代理
- 涉及并行领域(如法律合规和财务)：多代理系统更合适

21:30 - 因素2：可预测性vs适应性
- 函数调用：完全可预测，确定性控制
- 代理：更强适应性，基于用户意图和上下文动态决策
- 多代理：允许涌现行为，但可预测性较低

23:00 - 因素3：成本和延迟
- 函数调用：便宜快速，最少token使用
- 代理：更昂贵，多步骤推理
- 多代理：成本和延迟倍增

24:00 - 因素4：业务逻辑控制
- 函数调用：开发者控制一切
- 代理：逻辑部分嵌入提示和推理步骤
- 多代理：将问题解决委托给代理

25:00 - 因素5：自主性
- 函数调用：被动响应
- 代理：主动分解目标、决策步骤、提出后续问题
- 多代理：协作自主，分配角色和任务，最少人工干预

26:30 - 决策矩阵总结
- 低复杂度+低自主性：函数调用
- 中等复杂度+适应性推理：单一代理
- 高复杂度+多领域协作：多代理编排

27:30 - 实际用例映射
- **函数调用**：FAQ机器人、货币转换、订单状态查询
- **单一代理**：旅行规划助手、个人研究助手、内容摘要
- **多代理**：复杂合同生成、企业级故障排查、项目管理、法律尽职调查

29:00 - 总结与最佳实践
- 始终从简单开始，不要过度复杂化
- 规划可观测性：系统会出错，需要成本控制和治理
- 利用AWS工具：Amazon Bedrock、Lambda、Step Functions、Agent Core等
