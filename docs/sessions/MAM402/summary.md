# AWS re:Invent 2025 会议总结：打破遗留系统壁垒，使用生成式AI加速现代化

## 会议概述

本次会议由AWS的两位资深解决方案架构师Anila和Sri Kumaran Naya主讲，重点探讨了如何利用生成式AI技术加速企业遗留应用的现代化转型。会议以一个实际的电商应用"Unicorn Store"为案例，展示了如何将传统的.NET单体应用逐步转型为基于微服务的云原生架构。

演讲者强调，数字化转型是许多企业的关键优先事项，但遗留IT应用往往成为业务增长的障碍。传统的现代化改造不仅复杂，而且需要大量的成本和人力投入。本次会议展示了如何结合架构方法论、最佳实践以及生成式AI的能力，来有效地转型这些遗留应用。

会议采用了七阶段的现代化方法论，包括应用分析、运行时和框架转换、构建验证、分解分析、架构设计、重构和现代化。演讲者详细演示了如何使用AWS的两个核心AI工具——Amazon Q Developer（Kiro）和AWS Transformation——来自动化和加速整个现代化流程。这种方法不仅能够提升应用的性能和安全性，还能显著降低Windows许可成本，提高业务敏捷性。

### 分解分析理论 (38:00 - 45:00)
- **38:00** - 介绍分解分析阶段：现代化过程中最困难的部分
- **38:45** - 讨论常见陷阱一：分布式单体（通过水平分层导致）
- **40:00** - 分布式单体的问题：将功能相关的模块组合在一起，但仍继承单体的设计问题
- **41:00** - 讨论常见陷阱二：纳米服务（通过垂直分层导致）
- **41:45** - 纳米服务的问题：按UI页面拆分功能，导致数百个微小服务，增加运营复杂度和通信开销
- **43:00** - 强调正确粒度的重要性：需要结合业务领域分析和技术分析
- **44:00** - 引入架构方法论：领域驱动设计（Domain-Driven Design, DDD）

### 领域驱动设计（DDD）介绍 (45:00 - 52:00)
- **45:00** - 介绍Eric Evans的领域驱动设计理念
- **45:45** - DDD的核心思想：技术不应是主要关注点，业务功能才是
- **46:30** - DDD帮助业务所有者和应用所有者达成共同理解
- **47:15** - DDD的关键概念：领域（Domain）
- **47:45** - 示例：Unicorn Store的领域是零售（Retail）
- **48:15** - 子域（Subdomain）：订单管理、履行、配送
- **49:00** - 三种子域类型：核心子域（业务差异化）、支持子域（支持核心功能）、通用子域（如身份认证）
- **50:00** - 核心子域的上下文：订单子域包含商品、用户、支付等不同上下文
- **51:00** - 这些上下文最终定义边界，边界转化为服务
- **51:45** - 引出问题：如何开始理解这些子域和上下文？

### 事件风暴（Event Storming）方法论 (52:00 - 60:00)
- **52:00** - 引用Alberto Brandolini的观点：软件反映的是开发者的理解或误解
- **52:45** - 介绍事件风暴：一种协作工作坊方法论
- **53:30** - 事件风暴的目的：让业务所有者和应用所有者共同理解领域
- **54:15** - 通过识别领域事件及其生产者和消费者来理解领域
- **55:00** - 事件风暴的七个关键元素（用不同颜色的便签表示）
- **55:30** - 元素一：领域事件（Domain Event）- 用过去式表示已发生的事情，如"订单已创建"
- **56:15** - 元素二：命令（Command）- 触发事件的操作，如"下单"、"取消订单"
- **57:00** - 元素三：参与者（Actor）- 发出命令的人或系统
- **57:30** - 元素四：聚合（Aggregate）- 理解命令并执行业务逻辑的实体
- **58:00** - 元素五：业务策略（Business Policy）- 业务验证规则，如库存验证、支付验证
- **58:45** - 元素六：外部系统（External System）- 如支付网关、库存系统
- **59:30** - 元素七：读模型（Read Model）- 用于查询和显示数据
