# AWS re:Invent 2025 - Amazon EKS Capabilities 会议总结

## 会议概述

本次会议由 Amazon EKS 服务团队的首席产品经理 Jesse Butler 和产品经理 Shiram Ranganadan 主讲,重点介绍了 Amazon EKS 的全新功能层 —— EKS Capabilities。这是 EKS 服务推出七年以来首次引入的新功能层,标志着 EKS 从单纯的集群生命周期管理向帮助客户构建和扩展 Kubernetes 平台的方向演进。

EKS Capabilities 的核心理念是通过提供完全托管的平台服务,帮助客户实现 GitOps 最佳实践,降低运维负担。该功能基于开源标准,目前推出了三个核心能力:Argo CD(用于 GitOps 自动化)、ACK(AWS Controllers for Kubernetes,用于基础设施配置)和 Crossplane(用于创建平台抽象)。这些能力完全由 AWS 在服务账户中运行和管理,不占用客户集群的计算资源,同时保持了 Kubernetes 原生体验和可移植性。

会议强调了随着企业 Kubernetes 使用规模的扩大,平台工程团队面临的挑战日益增加。EKS Capabilities 通过提供标准化、声明式配置、持续协调和自动修复等特性,帮助客户从第一天起就建立可扩展的基础,避免后期重构的痛苦。该功能已在所有 EKS 商业区域正式发布(GA)。

## 详细时间线与关键要点

### 开场与背景介绍 (00:00 - 05:30)

- **00:00** - Jesse Butler 和 Shiram Ranganadan 介绍自己,均来自 Amazon EKS 服务团队
- **01:15** - 介绍会议主题:Amazon EKS Capabilities,这是团队数月来一直在开发的新功能
- **02:30** - 引用 Alan Kay 的名言:"简单的事情应该简单,复杂的事情应该可能",强调抽象的重要性
- **03:45** - 引用 Dijkstra 的观点:抽象的目的不是模糊,而是创建一个可以绝对精确的新语义层

### Kubernetes 作为抽象层 (05:30 - 10:00)

- **05:30** - 阐述 Kubernetes 本身就是一个抽象层,使复杂的事情成为可能
- **06:15** - 强调 Kubernetes 民主化了大规模分布式计算
- **07:00** - 引用数据:80% 的企业在生产环境中使用 Kubernetes,另有 13% 正在试点或调研
- **08:20** - 讨论简单性、规模一致性和可扩展性是现代平台的核心需求
- **09:30** - 指出 CNCF 拥有超过 200 个项目,数百个兼容工具,生态系统庞大

### EKS 的演进历程 (10:00 - 15:30)

- **10:00** - 回顾 EKS 在 2017 年 re:Invent 上发布,最初目标是让 Kubernetes 更易于 AWS 客户使用
- **11:00** - 展示数据平面和节点管理的演进:从自管理节点 → 托管节点组 → Auto Mode(去年发布)
- **12:30** - 强调 EKS 的使命是持续改进集群生命周期管理
- **13:45** - 指出当客户扩展到多个集群、多个账户和多个区域时,真正的挑战是管理和扩展 Kubernetes 本身
- **14:30** - 讨论大多数客户使用 Terraform、CloudFormation 或 Kubernetes IaC 进行集群管理

### 规模化挑战与平台工程 (15:30 - 22:00)

- **15:30** - 分析外部命令式 IaC(主要是 Terraform)的局限性
- **17:00** - 指出在多集群场景下会遇到摩擦点:管道过多、故障排查困难、所有权不清
- **18:15** - 强调客户在多集群阶段需要投资专门的平台工程资源
- **19:30** - 列举云原生平台的核心特征:声明式配置、持续协调、可编程的可发现性和可观测性、主动漂移检测和自动修复、基于标准
- **21:00** - 指出 Kubernetes 本身就是这些特征的参考实现

### 平台基础组件与 GitOps (22:00 - 28:00)

- **22:00** - 介绍成功的大规模 EKS 客户共同使用的三个基础平台组件:工作负载、云资源基础设施、集群基础设施
- **23:30** - 定义 GitOps 作为参考实现,强调其四个核心特征
- **24:45** - GitOps 的四个特征:期望状态声明式表达、期望状态不可变且版本化、期望状态从源自动应用、期望状态持续协调
- **26:00** - 强调持续协调是 GitOps 最关键的特征,使 Kubernetes 成为理想选择
- **27:15** - 指出虽然不能为每个人构建理想系统,但可以构建理想的平台

### GitOps 工具的挑战 (28:00 - 32:00)

- **28:00** - 讨论开源 GitOps 工具(如 Argo CD、ACK、Crossplane)的使用
- **29:15** - 指出随着规模扩大,自管理的分散式 GitOps 环境会带来新的管理负担
- **30:30** - 展示 EKS 集群内功能的演进:Helm Charts → EKS Add-ons → EKS Capabilities
- **31:45** - 宣布 EKS Capabilities 是 EKS 首次推出的新功能层

### EKS Capabilities 核心概念 (32:00 - 38:00)

- **32:00** - 介绍 EKS Capabilities 的愿景:开放标准、Kubernetes 原生体验、减少摩擦、支持新创新
- **33:30** - 强调所有能力都完全托管,永远不会在客户集群中运行
- **34:45** - 介绍三个初始能力:Argo CD(GitOps 引擎)、ACK(基础设施配置)、Crossplane(平台抽象)
- **36:00** - 对比自管理平台架构:客户需要在集群中安装和管理控制器
- **37:15** - 展示 EKS Capabilities 架构:所有控制器在 AWS 服务账户中运行,不占用客户集群资源

### Shiram 接手:EKS Capabilities 详解 (38:00 - 42:00)

- **38:00** - Shiram 开始讲解,重申 EKS Capabilities 是可扩展的平台功能集
- **38:45** - 强调 AWS 完全管理所有能力,无需客户担心安装、配置、补丁、升级
- **39:30** - 说明控制器在 AWS 服务账户中运行,客户在集群中找不到它们
- **40:15** - 介绍创建能力的通用模式:选择能力类型、命名、选择集群、传递能力角色
- **41:30** - 说明可通过 CLI、EKS kubectl、IaC 工具或控制台创建能力

### IAM 角色配置 (42:00 - 45:00)

- **42:00** - 介绍新的服务主体:capabilities.eks.amazonaws.com
- **42:45** - 强调信任策略必须信任此服务主体,而非 OIDC 提供者
- **43:30** - 说明不同能力需要不同的权限:Argo CD 需要访问 CodeCommit/ECR/Code Connections,ACK 需要相应 AWS 资源权限,Crossplane 本身不需要特定权限

### Argo CD 能力详解 (45:00 - 55:00)

- **45:00** - 介绍 Argo CD 作为基于 GitOps 的持续部署工具
- **46:15** - 说明 Argo CD 支持漂移检测和自动协调
- **47:30** - 展示中心化架构:一个 Argo CD 实例可连接多个集群(跨账户、跨区域)
- **48:45** - 强调 EKS Capabilities 完全管理网络连接,无需客户配置 VPC 对等或 Transit Gateway
- **50:00** - 演示通过 CLI 创建 Argo CD 能力的示例
- **51:15** - 介绍与 AWS Identity Center 的集成,实现 Argo UI 和 CLI 的单点登录
- **52:30** - 展示通过控制台创建能力的界面,大部分信息已预填
- **53:45** - 说明可直接通过控制台创建 IAM 角色,适合快速入门

### Argo CD 核心资源 (55:00 - 58:00)

- **55:00** - 展示能力创建后的控制台界面,显示基本详情和健康状态
- **55:45** - 说明客户仍可访问 Argo UI 和 CLI,EKS 控制台提供超链接
- **56:30** - 介绍 Argo CD Application 资源:定义源代码位置和目标集群
- **57:15** - 说明需要通过创建 Kubernetes Secret 并添加特殊标签来注册集群
