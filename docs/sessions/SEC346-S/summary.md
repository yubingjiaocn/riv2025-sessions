# AWS re:Invent 2025 Vega 分会场总结

## 会议概述

本次会议由 Vega 公司的联合创始人兼 CTO Ellie 和威胁检测与研究主管 Aron 主讲。Vega 是一家成立于 2024 年 1 月的初创公司,已从 Cyber Starts、Excel、Wet Point 和 CRV 等投资方筹集了 6500 万美元,致力于彻底改变安全运营和安全分析的方式。

演讲重点介绍了如何利用 AWS S3 存储桶重新构想 SIEM(安全信息和事件管理)架构,通过联邦化架构方法,帮助企业在不增加成本的情况下提高安全可见性。传统的单体式 SIEM 架构要求将所有日志集中化并按字节收费,导致成本高昂。Vega 提出的解决方案是直接在数据存储位置(如 S3 存储桶)进行索引和查询,无需将数据传输到集中式 SIEM,从而实现 60-80% 的成本削减。

演讲者分享了两个实际案例:一家大型电商平台通过采用 Vega 的方案完全淘汰了传统 SIEM,实现了 70% 的成本降低;一家财富 500 强制药公司拥有 500 多个 AWS 账户,通过在 S3 上直接进行安全操作,避免了数据传输成本。Vega 的核心技术 SAM(安全分析网格)支持跨多个存储库进行关联查询,提供统一的安全视图,并兼容现有的 SIEM 和数据湖解决方案。

## 详细时间线

0:00 - 0:45 | 公司介绍
- Vega 公司成立于 2024 年 1 月,筹集了 6500 万美元
- 演讲者介绍:Ellie(CTO,拥有 15 年以上网络安全经验)和 Aron(威胁检测与研究主管,10 年事件响应和威胁狩猎经验)

0:45 - 2:30 | 当前 SIEM 面临的三大挑战
- 安全遥测数据量持续增长,尤其是云迁移和多云环境带来的 CloudTrail、VPC Flow Logs 等日志
- 成本限制导致企业难以充分利用所有安全数据
- 多数据存储库分散(有些企业拥有多达 9 个不同的 SIEM)导致管理复杂

2:30 - 4:15 | 挑战传统 SIEM 架构假设
- 质疑为何必须集中化所有安全遥测数据
- 质疑为何存储和检测工程必须耦合在同一个单体系统中
- 提出为何不能利用多个数据存储库(如 S3 存储桶)实现传统 SIEM 的功能

4:15 - 6:45 | 传统方法的成本分析
- 传统方式需要六个成本环节:AWS 服务费、AWS 存储费、数据传输费(egress)、SIEM 摄取费、SIEM 存储费、安全模块费
- 对于拥有 200-750 个 AWS 账户的企业,成本极其高昂

6:45 - 8:30 | Vega 联邦化架构方案
- 直接在 S3 存储桶中对日志进行索引,无需数据传输
- 使用同一云区域内的索引计算资源
- Vega 控制台可直接查询存储和索引,执行检测、威胁狩猎和事件响应
- 成本降低 60-80%

8:30 - 10:45 | 案例一:大型电商平台
- 原本将所有日志发送到 SIEM,同时备份到 S3
- 采用 Vega 后,直接在 S3 上索引所有数据源(防火墙、EDR、Office 365 等)
- 完全淘汰了传统 SIEM,实现 70% 的总成本降低
- 获得更长的数据保留期和更多数据源接入能力

10:45 - 12:30 | 案例二:财富 500 强制药公司
- 拥有 500 多个 AWS 账户,每天数十 TB 的数据量
- 无法承担将所有 AWS 日志发送到 SIEM 的成本
- 采用 Vega 索引所有 AWS 日志(CloudTrail、VPC Flow Logs、GuardDuty、CloudWatch、RDS 等)
- 实现完整可见性和检测能力,无数据传输成本

12:30 - 14:00 | 解决数据碎片化问题
- Vega 可连接现有 SIEM 和数据湖
- 支持跨存储关联查询,避免在多个控制台工作
- 三大核心能力:跨存储关联、混合数据湖、统一安全视图

14:00 - 17:30 | SAM(安全分析网格)架构详解
- 可连接到任何现有投资:EDR 平台、现有 SIEM(Splunk、Microsoft Sentinel、Google SecOps、Elastic)、下一代 SIEM/XDR(SentinelOne、CrowdStrike)
- 支持数据湖(Databricks、Snowflake)
- 核心优势:在对象存储中直接索引数据,无需数据传输
- 支持多云环境,在本地区域内索引数据

17:30 - 19:15 | 产品特性与 AI 能力
- 使用 KQL(Kusto 查询语言)作为统一查询接口,易于上手
- 集成 AI 和 LLM 能力简化查询
- 提供单一语义数据层,为 AI 原生安全运营奠定基础
- AI 检测和响应能力取决于可访问的数据质量
