# AWS re:Invent 2025 技术会议总结

## 会议概述

本次会议由AWS高级经理Chong Mimmoon和韩国顶尖科技公司Toss Securities的Nam Cho共同主讲，重点介绍了Toss Securities如何利用AI技术和AWS服务改造其超级应用的运维操作。Toss是一款涵盖银行、投资、购物、游戏和出行等几乎所有日常生活场景的韩国超级应用。Toss Securities在推出仅3年后就成为韩国第一大海外股票交易平台，但快速增长也带来了巨大的运维挑战。

会议首先介绍了Amazon Bedrock和Amazon Nova的核心能力。Amazon Bedrock是AWS的全托管GenAI应用开发平台，提供模型选择、成本优化、数据定制、企业级安全和智能代理等五大核心功能。Amazon Nova是AWS自研的基础模型家族，包括文本理解模型（Micro、Lite、Pro、Premier）、创意内容生成模型（Canvas、Reel）、语音模型（Sonic）和自动化任务模型（Act）。Toss Securities主要使用Nova Micro和Lite进行测试自动化。

Nam Cho详细分享了Toss Securities面临的实际挑战：每天发布数十个新版本，涵盖银行、支付、安全、商务等多个领域，测试速度跟不上发布速度；跨国家、设备和操作系统版本的测试极其复杂；远程办公使得设备访问困难；夏令时调整时需要工程师通宵手动测试交易系统。为解决这些问题，团队构建了一个基于AI的自动化测试平台，实现了远程设备控制、网络包检测、自动化工作流创建和调度等功能，并通过AI代理实现自然语言交互式测试场景创建。

## 详细时间线

00:00 - 会议开场
- Chong Mimmoon介绍自己是AWS高级经理
- 介绍合作伙伴Toss Securities
- 说明会议议程：先介绍AWS服务，再由Toss分享实际案例

01:30 - Amazon Bedrock介绍
- 介绍Bedrock的五大核心能力：模型选择、成本与性能优化、简易定制化、数据安全、智能代理
- 强调数据隐私和企业级安全
- 说明代理可以处理超越简单聊天机器人的复杂任务

03:00 - Amazon Nova模型家族介绍
- Nova Micro：文本输入输出
- Nova Lite/Pro/Premier：多模态输入（文本、图像、视频）生成文本输出
- Nova Canvas：图像生成
- Nova Reel：视频生成
- Nova Sonic：语音转语音
- Nova Act：在浏览器中自动执行任务
- 强调Toss Securities使用Nova Micro和Lite进行测试自动化

04:30 - Toss超级应用介绍
- Nam Cho介绍Toss涵盖银行、投资、购物、游戏、出行等日常生活各个方面
- Toss Securities在3年内成为韩国第一大海外股票交易平台
- 快速增长带来了严重的运维痛点

05:30 - 面临的核心挑战
- 每天发布数十个新版本，测试速度跟不上
- 单元测试和集成测试都非常耗时
- 跨国家、设备、操作系统版本的测试极其复杂
- 远程办公导致设备访问困难，需要寻找特定设备型号
- 夏令时调整时需要工程师在凌晨7点手动测试交易系统开盘

08:00 - 挑战总结为三大类
- 难以在快速部署的同时保持质量
- 跨多种设备和环境的测试极其复杂
- 缺乏实时可见性来监控、调度和验证日志

08:30 - 自动化测试平台演示
- 展示Web界面远程控制设备，无延迟（播放YouTube视频演示）
- 可以检查测试期间生成的所有网络数据包
- 可以验证是否产生了预期的日志
- 可以创建自动化工作流并按需调度执行

10:00 - 工作流创建演示
- 演示自动启动YouTube并切换标签页的工作流
- 展示可点击元素的识别
- 演示通过点击切换标签页

11:30 - AI代理自然语言交互演示
- 演示用户用自然语言要求"启动naver.com"（韩国版Google）
- 通过聊天创建的测试会保存为可重用的工作流
- 演示调用MCP工具
- 展示实时构建和调整工作流的能力

13:00 - 工作流实时调整演示
- 演示要求每5秒发送一条"hello world"的Thread DM
- 然后要求改为3秒
- 工作流立即更新并按要求执行

14:00 - 系统架构介绍
- 使用完全隔离的AWS环境
- 通过Direct Connect实现办公室和家庭的安全访问
- 服务部署在EC2上
- 代码和工作流脚本通过AWS的Git服务器管理
- 为符合韩国金融监管要求，使用Amazon Bedrock的隔离模型实例

15:30 - Provider组件架构
- Provider可以是员工PC或数据中心服务器
- 手机通过USB连接到Provider
- 自动在设备上安装Mobile Controller
- Provider通过gRPC与Controller通信控制设备
- Provider通过HTTP WebSocket与自动化平台服务器通信
- 用户可以远程控制多个设备并运行现有测试脚本

17:00 - 工作流系统设计理念
- 现有工作流解决方案过于复杂、难以本地化、性能不足
- 设计目标：足够快、足够灵活、足够简单
- 不依赖复杂的拓扑排序算法，仅使用连接节点信息（前一个、后一个节点和触发器ID）
- 体现"简单即最好"的开发理念

18:30 - 系统灵活性设计
- 支持运行时添加新节点并立即应用到工作流
- 选择TypeScript是因为其类型系统便于定义和验证输入参数
- 可以为输入参数设置默认值

19:30 - AI在项目中的应用
- 用户说"告诉我亚马逊的价格"时，代理首先执行RAG检索相似测试用例
- 如果没有找到相似示例，代理回退到直接推理
- 使用Nova Lite进行多模态推理，确定屏幕上需要点击的元素
- 首先分析设备的dump文本和结构化数据
- 如果找不到元素，回退到空间图像分析，通过图像理解直接识别元素

21:00 - AI代理的四个核心步骤
- 规划（Plan）
- 分析（Analyze）
- 行动（Action）
- 审查（Review）
- 工作流代理遵循相同原则：分析现有工作流、识别缺失组件或所需更改、直接应用

22:00 - 构建代理的类比
- 构建代理就像创建人形机器人
- 需要"眼睛"来理解上下文
- 需要"手臂"来执行必要的操作
- 这些"手臂"可以通过MCP等方法有效实现
- 本质上是将现有的部分组合在一起

23:00 - 平台带来的影响
- 不再受物理位置限制，可以随时随地远程访问任何设备
- 可以在任何需要的时间和地点自动运行和验证测试，确保一致的质量
- 团队可以专注于有意义的工作，而不是重复性测试

24:00 - 未来规划
- 计划与Figma等设计系统集成，为代理提供更丰富的产品上下文
- 构建Atlas地图，自动捕获和分析应用中检测到的每个新屏幕和转换
- Atlas项目像应用的神经地图，将所有转换链接在一起，使代理更加智能
- 最终目标：构建能够理解自然语言测试场景并自主测试所有可能交互的代理

25:30 - 会议结束
- 强调这些技术最终让工作和生活更轻松
- 有了编码代理的帮助，可以独立构建全栈项目
- 感谢听众的时间和关注
